# Phase 06 — UI Design Contract

> Visual and interaction contract for the Foundations + Preview Mode phase. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

This is an Obsidian community plugin running inside the Obsidian desktop app. The plugin owns chrome (custom `ItemView`, settings tab, sticky header bar) and inherits all chrome-adjacent surfaces from Obsidian (tab strip, scroll container, focus rings, theme palette). Body content rendered into the preview tab is owned by `MarkdownRenderer.render` — the spec does NOT prescribe styles for problem-statement text, examples, code blocks, or LC HTML fragments; those follow the user's active Obsidian theme.

**Scope covered by this contract:**
1. New `ProblemPreviewView` (`ItemView`, `viewType: 'leetcode-preview'`) — center tab, sticky header bar, body area
2. Sticky header bar — title heading, difficulty pill, topic chips, primary action button
3. New "Preview" settings section — `Preview › Click behavior` toggle in `SettingsTab.ts`
4. Modified `ProblemBrowserView` row click affordance — single-click → preview, shift-click → open
5. Right-click context menu on `ProblemBrowserView` rows — "Preview problem" entry (PREVIEW-01)
6. `Notice` toasts — preview load failures, fetch-on-cache-miss timeout
7. Tab title format + tab icon for the preview leaf

**Explicitly out of scope** (deferred to later phases):
- AI Debug button, AI Review section, AI streaming modal — Phase 07–09
- Preview "I have N notes in this cluster" coverage hint — depends on Phase 11
- Topic chip click-through to filter `ProblemBrowserView` — backlog post-v1.1
- Onboarding modal of any kind — explicitly rejected in CONTEXT.md
- Right-click → "Start Problem (skip preview)" — shift-click already provides this

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Obsidian-native) |
| Preset | not applicable — Obsidian plugin, no React/Next/Vite stack |
| Component library | Obsidian API primitives (`ItemView`, `PluginSettingTab`, `Setting`, `Notice`, `Menu`, `MarkdownRenderer`, `setIcon`) |
| Styling approach | Class-scoped plain CSS in `styles.css` rooted at `.leetcode-preview` (preview view) and `.leetcode-settings` (existing settings tab). Reuses shared `lc-diff--{easy\|medium\|hard}` selectors from `styles.css` lines 184–186 |
| Token system | Obsidian CSS variables (`--background-primary`, `--background-secondary`, `--text-normal`, `--text-muted`, `--interactive-accent`, `--color-green`, `--color-yellow`, `--color-red`, `--background-modifier-border`). **Never hardcoded hex.** |
| Icon library | Lucide (Obsidian built-in) — `setIcon(el, name)`. Phase 06 uses `eye` (preview tab) with `book-open` as fallback if `eye` is unavailable in the runtime Lucide set; `play` (Start Problem); `external-link` (Open Problem); `code-2` (browser, unchanged) |
| Font | Inherit Obsidian's `--font-interface` and `--font-monospace`. No custom web fonts. |
| DOM constructor | `createEl()` / `createDiv()` / `createSpan()` only. `innerHTML` forbidden (`eslint-plugin-obsidianmd@^0.3.0` enforced). |
| MarkdownRenderer signature | `MarkdownRenderer.render(this.app, markdown, body, '', this)` where `this` = the `ProblemPreviewView` instance (extends `ItemView` which extends `Component`). Satisfies `no-plugin-as-component` rule. |

**Rationale (theme-portability hard rule):** Obsidian ships dozens of user themes; plugin colors MUST come from theme variables so the plugin reads native in light mode, dark mode, and every community theme. Hardcoded hex breaks community-plugin review.

## Spacing Scale

Obsidian-aligned 4-point scale. Reuses tokens already used by `ProblemBrowserView` and existing `styles.css`. Phase 06 does NOT introduce new spacing tokens.

| Token | Value | Usage in Phase 06 |
|-------|-------|-------------------|
| xs | 4px | Topic chip horizontal padding, gap inside difficulty pill, gap between chips inside chip-cluster |
| sm | 8px | Gap between sticky-header header row and chip row, action button horizontal padding, body block top padding |
| md | 12px | Sticky header bottom padding, gap between difficulty pill and topic chip cluster |
| md2 | 16px | Outer padding of `.leetcode-preview` container (matches `.leetcode-browser` line 5 of `styles.css`), sticky-header outer padding |
| lg | 24px | Vertical gap between sticky header and rendered body's first heading (when MarkdownRenderer output is present) |

Exceptions:
- **Sticky header total height: 56–72px** depending on whether topic chips wrap to a second row. Width is the leaf's width minus 32px outer padding. Header is `position: sticky; top: 0` and stays pinned while the body scrolls.
- **Action button min-height: 30px** — matches the existing neutral-button token from `.lc-fm__picker` / `.leetcode-code-action-run` (`styles.css` lines 661, 956). No new button height in Phase 06.
- **Difficulty pill: 22px height** (matches Phase 1 pill convention referenced in v1.0 `01-UI-SPEC.md`); horizontal padding 8px; reuses `lc-diff--{easy|medium|hard}` color tokens.
- **Topic chip: 22px height**, padding 0 8px, font 12px, weight 500 — matches existing `.lc-fm__picker-pill` (`styles.css` lines 349–356) for visual consistency with the filter modal's chip language.

## Typography

Obsidian-native sizes via CSS variables. Phase 06 declares ONLY plugin chrome typography (sticky header + settings rows). Body content inside the preview is owned by `MarkdownRenderer` — its typography follows Obsidian's reading-mode defaults and is NOT specified here.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Heading | 18px (`var(--h2-size)` fallback to 18px) | 600 (semibold) | 1.3 | `{id}. {title}` heading in sticky header |
| Body | 14px (`var(--font-ui-small)`) | 400 (regular) | 1.5 | Settings row descriptions, dropdown labels, action-button label |
| Label | 12px (`var(--font-ui-smaller)`) | 500 (medium) | 1.4 | Difficulty pill label, topic chip label, tab title text |

**Two declared weights:** 400 (regular) for body + 600 (semibold) for the title heading. 500 (medium) is the existing chip-label weight (precedented in `lc-fm__picker-pill` and `lc-row__diff`). If the executor prefers strict 2-weight, fall back to 400 for chip labels — but the existing 500 in `styles.css` is the locked precedent.

**Monospace:** none in Phase 06 chrome. (Body MarkdownRenderer output may contain monospace; that's owned by Obsidian.)

**Line-height rationale:** 1.3 for the title heading mirrors the v1.0 settings heading (`01-UI-SPEC.md`); 1.5 for body matches `01-UI-SPEC.md` row body line height.

## Color

**Hard rule (carried from v1.0):** every color value in Phase 06 CSS references an Obsidian CSS variable. **No raw hex. No `rgba()` with literal numbers.** Non-negotiable.

| Role | Token | Usage in Phase 06 |
|------|-------|-------------------|
| Dominant (60%) | `var(--background-primary)` | `ProblemPreviewView` body background, sticky-header background (so it visually merges with the body and reads as a tab, not a banner) |
| Secondary (30%) | `var(--background-secondary)` | Topic chip background, action-button hover state background, settings-section row background (Obsidian default) |
| Accent (10%) | `var(--interactive-accent)` | **Start Problem** primary action button when no note exists. ONLY this surface in Phase 06. |
| Accent text | `var(--text-on-accent)` | Text inside the `Start Problem` accent-filled button |
| Neutral button | `var(--background-primary)` border `var(--background-modifier-border)` | **Open Problem** action button (no accent — neutral, matches existing `.leetcode-code-action-run` pattern in `styles.css` lines 954–965) |
| Destructive | `var(--text-error)` | Not used in Phase 06 — preview has no destructive actions |
| Text normal | `var(--text-normal)` | `{id}. {title}` heading, action-button neutral label, settings labels |
| Text muted | `var(--text-muted)` | Topic chip label color, error-state secondary text, settings-row description text |
| Border | `var(--background-modifier-border)` | 1px bottom border on sticky header (separates from body); border on `Open Problem` neutral button |
| Difficulty — Easy | `var(--color-green)` | Difficulty pill text + 12% background via `color-mix(in srgb, var(--color-green) 15%, transparent)` (matches v1.0 D-09 chip pattern `styles.css:184` + `01-UI-SPEC.md` Difficulty pill spec) |
| Difficulty — Medium | `var(--color-yellow)` | Same pattern (`styles.css:185`) |
| Difficulty — Hard | `var(--color-red)` | Same pattern (`styles.css:186`) |

**Accent (`var(--interactive-accent)`) is reserved EXCLUSIVELY for in Phase 06:**
1. The `Start Problem` action button when no note exists for the previewed slug.
2. (Inherited from v1.0 — `Log in via embedded window` settings button; active filter chip background.)

**NOT accent in Phase 06:**
- `Open Problem` button (neutral — note already exists, "open" is a routine navigation, not a primary action)
- Topic chips (muted; they are informational, not interactive in v1.1 base ship)
- Sticky header background (matches `--background-primary`; reads as continuation of the tab, not a banner)
- Tab icon, tab title — these inherit Obsidian's tab-strip styling

**60/30/10 mapping:**
- 60% `--background-primary`: preview tab body, sticky header
- 30% `--background-secondary`: topic chips, action-button hover, settings rows
- 10% `--interactive-accent`: Start Problem button only

**Difficulty pill `lc-diff--*` is REUSED, not redefined.** The existing CSS rules at `styles.css:184–186` (`.leetcode-browser .lc-diff--easy/medium/hard`) define color only, not background. Phase 06 ports the pill into the preview header by adding a parallel selector cluster: `.leetcode-preview .lc-diff--easy/medium/hard` with the same `color: var(--color-{x})` rule plus a `color-mix` background. Executor may consolidate via a non-`.leetcode-browser`-scoped variant if desirable, but MUST NOT change v1.0's existing browser-scoped rules.

## Copywriting Contract

All user-facing strings are **sentence case** (existing v1.0 convention; enforced by review). No trailing periods on button labels, headings, chip labels. Notices end with a period because they are full sentences.

### Preview tab — strings

| Element | Copy |
|---------|------|
| Tab title (preview leaf) | `Preview: {id}. {title}` (e.g. `Preview: 1. Two Sum`). Display title set via `getDisplayText()` |
| Tab tooltip | `Preview: {id}. {title}` (same as title) |
| Sticky-header heading text | `{id}. {title}` (no "Preview:" prefix here — tab title carries that signal) |
| Difficulty pill label | `Easy` / `Medium` / `Hard` — verbatim, no abbreviation (NOT `Med.`; the browser's `Med.` abbreviation is local to `ProblemBrowserView` for narrow sidebar; preview is a center tab with room) |
| Topic chip label | `{topic-slug → Title Case}` — same display transform v1.0 uses for browser topic display. Plain text, not a wikilink. |
| Action button (no note exists) | `Start Problem` |
| Action button (note exists) | `Open Problem` |
| Loading state heading (cache miss) | `Loading problem…` |
| Loading state body | `Fetching {id}. {title} from LeetCode.` |
| Fetch-failure heading | `Couldn't load problem` |
| Fetch-failure body | `Check your internet connection and try again.` |
| Fetch-failure action | `[Retry]` button |

### Settings tab — new "Preview" section strings

| Element | Copy |
|---------|------|
| Section heading | `Preview` |
| Setting name | `Click behavior` |
| Setting description | `What happens when you click a problem in the LeetCode browser. Shift-click always opens the note directly.` |
| Dropdown option (default) | `Preview first` (value: `preview`) |
| Dropdown option | `Open note directly` (value: `open`) |

### Right-click context menu (PREVIEW-01)

| Element | Copy |
|---------|------|
| Context menu item | `Preview problem` |
| Context menu icon | `eye` (Lucide) — same icon as the tab |

### `Notice` messages (Phase 06)

| Trigger | Copy | Duration |
|---------|------|----------|
| Preview cache miss + fetch failed | `Couldn't fetch {id}. Check your connection.` | 4000 ms (Obsidian default) |
| User clicks `Start Problem` and note creation fails | `Couldn't create note. {error reason}.` | 6000 ms — actionable, surfaced from existing NoteWriter error path; reuse v1.0 error copy |
| User clicks `Open Problem` and the existing note is gone (raced) | `Couldn't find the note for {id}. Click Start Problem to recreate.` | 6000 ms |

Phase 06 inherits all v1.0 Notice copy verbatim — no v1.0 strings rewritten.

### Primary CTA per surface

| Surface | Primary CTA | Verb + Noun |
|---------|-------------|-------------|
| Preview tab (no existing note) | `Start Problem` | Start (problem) |
| Preview tab (existing note) | `Open Problem` | Open (problem) |
| Settings — Preview section | none | (toggles only — no CTA) |
| Browser row context menu | `Preview problem` | Preview (problem) |

### Destructive actions

**Phase 06 has NO destructive actions.** Preview is read-only; Start/Open both delegate to existing v1.0 NoteWriter pipelines which themselves are non-destructive (they idempotently re-open existing notes). No confirmation modals are introduced in Phase 06.

## Interaction Contracts

### `ProblemBrowserView` row interactions (modified)

| Interaction | Contract |
|-------------|----------|
| **Left-click row** (default) | Calls `plugin.routeProblemClick(slug, status, intent)` with `intent='preview'`. Preview tab opens or reuses existing preview leaf. Row visual state unchanged. |
| **Shift + left-click row** | Calls `routeProblemClick(slug, status, intent='open')`. Always bypasses preview, regardless of `Preview › Click behavior` setting. (CONTEXT.md decision A.) |
| **Settings: `Preview › Click behavior = open`** | Plain left-click resolves to `intent='open'`; shift-click is a no-op modifier in this mode (still opens). Preview is reachable only via right-click or palette. |
| **Right-click row** (PREVIEW-01) | Obsidian `Menu` opens at the cursor with one entry: `Preview problem` (icon: `eye`). Selecting it dispatches `routeProblemClick(slug, status, 'preview')`. No other entries in v1.1 base ship. |
| **Random-pick (shuffle button)** | UNCHANGED from v1.0 — still calls `plugin.openProblem(slug, status)` directly with no preview indirection. (CONTEXT.md `<code_context>`.) |
| **Hover row** | UNCHANGED — `var(--background-modifier-hover)` background; cursor `pointer` (existing `.lc-row:hover` rule). |
| **Keyboard `Enter` on focused row** | Resolves to the configured click behavior (`preview` or `open`) — no shift-key modifier on keyboard. |

### `ProblemPreviewView` (new)

| Interaction | Contract |
|-------------|----------|
| **Open via row click (cache hit)** | Synchronous: `workspace.getLeavesOfType('leetcode-preview')` — if a leaf exists, swap its content via `setViewState({type: 'leetcode-preview', state: {slug}})`; otherwise `workspace.getLeaf('tab')` + `setViewState`. Reveal the leaf. ONE preview tab at any time. |
| **Open via row click (cache miss)** | Open or reuse the preview tab; immediately render the loading state (heading + body copy from § Copywriting). Trigger a `getProblemDetail(slug)` fetch in parallel. On resolve → render; on reject after `requestUrl` failure → render the fetch-failure state with `[Retry]`. |
| **Repeat preview (different slug)** | Reuse existing preview leaf; do NOT open a second tab. Animation: instant DOM swap; no fade. Tab title updates. |
| **Click `Start Problem` button** | Disable button + change label to `Starting…` (12px label transition; same fixed width to avoid jitter); call `plugin.openProblem(slug, status)` (existing v1.0 path); on resolve → wait `setWindowTimeout(100ms)` (popout-aware via `src/shared/timers.ts`); call `this.leaf.detach()`. The note tab is now focused (NoteWriter reveals it). On reject → re-enable button, restore label, surface Notice with v1.0 error copy. |
| **Click `Open Problem` button** | Identical to Start path mechanically: disable + label `Opening…`; `plugin.openProblem(slug, status)`; 100 ms delay; `leaf.detach()`. NoteWriter's existing-file branch (`NoteWriter.ts:227`) handles re-open without overwrite. |
| **Click sticky-header title** (no UI element — text only) | Static, non-interactive. (Topic chips are also non-interactive in v1.1 base; CONTEXT.md decision C.) |
| **Scroll body** | Sticky header pins to top of viewport. `position: sticky; top: 0; z-index: 10` on `.leetcode-preview__header`. Body scroll uses Obsidian's view native scroll container. |
| **Tab close (X)** | Standard Obsidian leaf close — `onClose()` lifecycle runs; cancel any in-flight fetch via the throttled fetcher's existing abort path; clear the rendered DOM. No prompt. |
| **Plugin unload** | `onunload` detaches all `leetcode-preview` leaves (best-effort `getLeavesOfType('leetcode-preview').forEach(l => l.detach())`). |
| **Tab title context menu** (Obsidian native — pin tab, etc.) | Inherited from Obsidian; preview leaf supports pin / move / close like any other leaf. |

### Settings — new "Preview" section

| Interaction | Contract |
|-------------|----------|
| **Section position** | Inserted in `SettingsTab.display()` between the existing `Notes` heading block (lines 149–173) and `Knowledge graph` heading block (line 182). New `new Setting(containerEl).setName('Preview').setHeading()` then a single dropdown row. |
| **Click-behavior dropdown** | `Setting.addDropdown()`. Two options: `Preview first` (value `preview`, default) and `Open note directly` (value `open`). On change → `await plugin.settings.setPreviewClickBehavior(v)` (planner confirms exact field name on `PluginData`). Persists immediately, no Save button. No reload required — `ProblemBrowserView` reads the setting on each row click via `plugin.settings.getPreviewClickBehavior()`. |
| **Default value** | `preview` for both fresh installs and v1.1 upgraders (CONTEXT.md decision A — single default, no upgrader branch). |

### `routeProblemClick(slug, status, intent)` decision flow (consumed by ProblemBrowserView)

```
intent === 'open'  → existing plugin.openProblem(slug, status) verbatim
intent === 'preview':
  let cfg = plugin.settings.getPreviewClickBehavior()  // 'preview' | 'open'
  if cfg === 'open' && intent !== 'preview-via-rightclick'  → openProblem(slug, status)
  else                                                       → openOrReusePreview(slug)
```

**Right-click → preview ALWAYS opens preview**, regardless of the click-behavior setting (right-click intent is explicit, not the default click affordance). The router treats right-click as a separate `intent='preview'` call carrying a `force: true` flag (planner determines exact API; researcher recommends an extra param `force` rather than a fourth intent value).

## Layout — `ProblemPreviewView` (center tab)

```
┌─────────────────────────────────────────────────────────────┐  ← Obsidian tab strip (native)
│ [eye] Preview: 1. Two Sum     ×                             │
├─────────────────────────────────────────────────────────────┤  ← Obsidian view header (native)
│                                                             │
│ ┌──────────────────────────────────── sticky header ──────┐ │  ← .leetcode-preview__header.is-sticky
│ │ 1. Two Sum                                              │ │  ← H2 18px/600, --text-normal
│ │                                                         │ │
│ │ [Easy]  [Array]  [Hash Table]              [Start Problem] │  ← row 2: pill + chips + action button
│ └─────────────────────────────────────────────────────────┘ │     pill+chips left-aligned, button right
│ ─────────────────────────────────────────────────────────── │  ← 1px --background-modifier-border
│                                                             │
│  ## Description (rendered by MarkdownRenderer.render)       │  ← body region, scrollable
│  Given an array of integers `nums` and an integer …        │     vertical scroll under the sticky header
│                                                             │
│  ## Example 1:                                              │
│      Input: nums = [2,7,11,15], target = 9                  │
│      Output: [0,1]                                          │
│      Explanation: …                                         │
│                                                             │
│  …                                                          │
└─────────────────────────────────────────────────────────────┘
```

**Sticky-header row order (top to bottom):**
1. Title row — `<h2 class="lc-preview__title">{id}. {title}</h2>`. Single line; ellipsis if it overflows the leaf width minus the action button's reserved width.
2. Chip row — left cluster (difficulty pill + topic chips, gap 8px between pill and first chip, 4px between chips, wraps to a second line if topics overflow) + right cluster (action button, fixed width 140px reserved).

**Sticky-header dimensions:**
- Outer padding: 16px (matches `.leetcode-browser` padding `styles.css:5`)
- Internal vertical gap: 8px between title row and chip row
- Sticky `top: 0`, `z-index: 10` (same as `.lc-progress` `styles.css:204`)
- 1px bottom border in `var(--background-modifier-border)` (matches `.lc-progress` border-bottom `styles.css:208`)

**Body region:**
- 16px outer padding (continues from sticky header's 16px outer padding so visual columns align)
- 24px top padding above the first MarkdownRenderer-rendered heading to give breathing room from the sticky header's bottom border
- Vertical scroll uses `ItemView`'s default content scroll container (no custom overflow rule; planner confirms inherited behavior)

**Empty / loading / error layouts** (full-tab, no sticky header until problem data is loaded):
```
┌──────────────────────────────────────────────────────┐
│                                                      │
│              Loading problem…                        │  ← H3 16px/600, --text-normal
│              Fetching 1. Two Sum from LeetCode.      │  ← 14px --text-muted
│                                                      │
└──────────────────────────────────────────────────────┘
```
On fetch failure:
```
┌──────────────────────────────────────────────────────┐
│                                                      │
│              Couldn't load problem                   │  ← H3
│              Check your internet connection and try again. │
│                                                      │
│              [Retry]                                 │  ← neutral button
│                                                      │
└──────────────────────────────────────────────────────┘
```
Both states reuse the existing `.lc-empty` rule pattern from `ProblemBrowserView` (`renderEmptyState`, `styles.css:303`). Phase 06 does NOT introduce a new empty-state class.

## Accessibility

| Requirement | Contract |
|-------------|----------|
| Keyboard navigation | Sticky header: title is plain text (not focusable). Action button is keyboard-focusable (`<button>`). Tab order: action button first, then body content (which has its own internal tab order owned by Obsidian's reading view). |
| Focus rings | **Do not override Obsidian's default focus styling.** Any `outline: none` is a bug. The action button inherits theme focus ring. |
| ARIA on action button | `aria-label="{Start\|Open} problem {id}: {title}"`. Button text alone is sufficient; the aria-label adds context for screen readers when multiple preview tabs are open. |
| ARIA on context menu item | Obsidian `Menu` provides accessibility automatically. |
| ARIA on sticky header | None needed — header content is plain text + chips + button; semantic `<h2>` for the title; `role` defaults are correct. |
| Color contrast | Inherited from Obsidian theme variables — all themes meet WCAG AA per Obsidian guidelines. Difficulty pill backgrounds via `color-mix` at 15% maintain contrast against `--background-primary` per existing v1.0 chip pattern (validated in light + dark themes during v1.0 dogfood). |
| Screen reader on tab | Tab title `Preview: {id}. {title}` is read by Obsidian's tab strip. `getDisplayText()` returns the same string. |
| Motion | No animations beyond Obsidian's built-in transitions. The `Start Problem → Starting…` label transition is instant text swap (no fade); button width is held constant by `min-width` to prevent jitter. |
| Reduced motion | `@media (prefers-reduced-motion: no-preference)` is respected in any future polish; Phase 06 has no decorative motion to gate. |
| Sticky header z-index | `z-index: 10` matches v1.0 `.lc-progress` precedent. Does not collide with Obsidian's chrome (which uses higher z-indices). |
| Focus management on tab reuse | When `setViewState({type, state: {slug}})` swaps the leaf's content, focus stays on whatever element was focused before (typically the row in the browser). Phase 06 does NOT auto-focus the action button — that would steal focus from the user's keyboard navigation. |

## State machine — `ProblemPreviewView`

```
                         ┌──────────────┐
                         │ uninitialized │
                         └──────┬───────┘
                                │ onOpen() / setViewState({slug})
                                ▼
                        ┌──────────────────┐
                        │ resolving cache  │ (synchronous getProblemDetail)
                        └──┬────────┬──────┘
                  cache hit│        │ cache miss
                           ▼        ▼
                  ┌──────────┐   ┌──────────┐
                  │ rendered │   │ loading  │
                  └──┬───────┘   └──┬───────┘
        click action │              │ fetch resolve
                     │              ▼
                     │           ┌──────────┐
                     │           │ rendered │
                     │           └────┬─────┘
                     │                │ click action
                     ▼                ▼
              ┌────────────────────────────┐
              │ pending (button disabled)  │
              └──┬───────────────────┬─────┘
       openProblem│                  │ openProblem
              resolve            reject
                  ▼                  ▼
        ┌────────────────┐    ┌──────────┐
        │ leaf detached  │    │ rendered │
        │ (note focused) │    │ (Notice) │
        └────────────────┘    └──────────┘

(cache miss + fetch reject path:)
              ┌──────────┐
              │ loading  │
              └────┬─────┘
                   │ requestUrl reject
                   ▼
              ┌─────────────┐
              │ fetch-error │ → click [Retry] → loading
              └─────────────┘
```

**Tab-reuse swap state:** when the user clicks a different row while a preview is rendered, the view goes through `setViewState({state: {slug}})` which calls `setState(state)` internally; the view re-enters `resolving cache` for the new slug. No intermediate "swapping" state needed — DOM swap is synchronous on cache hit.

## Edge cases (locked decisions)

| Case | Behavior |
|------|----------|
| User clicks a row, preview opens, user immediately clicks another row | Existing preview leaf reused; DOM swapped to new slug. No second tab. |
| User has the preview tab open and clicks `Start Problem` | Note created; `leaf.detach()` after 100 ms; note tab is now focused. Preview is gone. |
| User has the preview tab open and clicks `Open Problem` | Existing note revealed; `leaf.detach()` after 100 ms. |
| User clicks `Start Problem` while preview cache is loading | Button is disabled during the loading state — `Start Problem` only renders after content is ready. (Loading state has no action button.) |
| User has the preview pinned (Obsidian "pin tab" feature) | Pin is respected — `leaf.detach()` still runs; pin doesn't block detach. (Standard Obsidian behavior; no Phase 06 override.) |
| User triggers preview while no internet | Cache hit → render normally (offline behavior is preserved per CLAUDE.md offline constraint). Cache miss → loading → fetch reject → error state with `[Retry]`. |
| User has `Preview › Click behavior = open` and right-clicks → `Preview problem` | Right-click intent overrides setting; preview opens. (Right-click is explicit.) |
| User shift-clicks while `Preview › Click behavior = open` | Open (same as without shift in this mode; no double-modifier semantics). |
| Preview leaf is the only open leaf and user clicks `Start Problem` | Note opens in a fresh leaf via `workspace.getLeaf('tab')` (NoteWriter's existing behavior); preview detaches. Obsidian opens an empty-state pane if the workspace has no other leaves — standard Obsidian, not a Phase 06 concern. |
| Two Obsidian windows open (popout) — preview triggered from sidebar in main window | Preview opens in the same window where the click happened. `workspace.getLeaf('tab')` inherits the active window context. (Plugin uses `setWindowTimeout` from `src/shared/timers.ts` to remain popout-safe per v1.0 WR-02.) |
| Plugin reload while preview tab is open | Obsidian persists the leaf with `viewType: 'leetcode-preview'` and `state: {slug}`. On plugin reload, `registerView` re-registers the constructor, and Obsidian re-creates the view via `setViewState`. The view re-enters `resolving cache` from scratch. **No serialized state beyond `slug` is persisted.** |
| Slug in persisted leaf state is invalid (e.g. cache cleared) | Loading state → fetch attempt → on reject, error state with `[Retry]`. |
