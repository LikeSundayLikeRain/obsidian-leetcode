<objective>
Close 17-UAT.md Issue 3 (Test 12 fm reactivity asymmetry): Java → Python3 fm change correctly reconfigures the child editor's syntax highlighter, but Python3 → Java in the reverse direction does NOT — chevron updates, fence opener stays put per D-14, but the child's syntax coloring stays Python3.

Root cause (validated against `src/main.ts:2613-2641` `readActiveFenceSlug`): Gate 3 in `handleFmChangeForLanguageReactivity` (lines 2557-2568) calls `this.readActiveFenceSlug(file)` which parses the slug from the parent's FENCE OPENER TAG. Per D-14, the fm-reactivity listener does NOT rewrite the fence opener — so the opener tag NEVER changes via this path. After the first Java → Python3 fm swap, the child editor reconfigures to Python3 but the parent fence opener still says `\`\`\`java`. On the second swap (Python3 → Java in fm), Gate 3 reads `currentSlug = readActiveFenceSlug(file) = 'java'` (from the unchanged opener tag), compares to `fmLangRaw = 'java'`, finds them equal, and returns early — no Compartment.reconfigure dispatch fires.

The bug is structural: D-14 explicitly says the opener tag is decoupled from fm in this listener path, so reading "current applied child language" from the opener tag is unsound. Gate 3 must read from the actual child language state, not the parent doc.

Fix: introduce a per-child `WeakMap<EditorView, string>` on the plugin instance (`this.childLanguageTracker`) that records the slug currently applied to the child's `languageCompartment`. Update it from BOTH dispatch sites: the Phase 16 chevron switch path (`dispatchChildLanguageReconfigure` at `src/main.ts:~2484-2490`) AND the fm-reactivity dispatch (`handleFmChangeForLanguageReactivity` at `src/main.ts:~2574-2585`). Replace Gate 3's `readActiveFenceSlug` call with a tracker read.

Purpose: Closes 17-UAT.md Issue 3 (Test 12) — round-trip fm swaps are now symmetric. Does NOT regress D-14 (opener tag still not rewritten by listener), Pitfall 3 dedupe (same-slug fm writes still no-op via tracker comparison), or the chevron switch path (which already updates the parent opener — this plan only changes WHERE Gate 3 reads from).

Output: Plugin-instance `childLanguageTracker: WeakMap<EditorView, string>`, tracker writes at both Compartment.reconfigure dispatch sites, Gate 3 reads from the tracker, regression test that exercises round-trip fm swaps and fails on current main.

CM6 Compartment lifecycle (must understand for tracker placement):
  - Compartment.reconfigure(extension) returns a StateEffect.
  - Dispatching it on an EditorView updates the compartment's "current value" inside that view's state.
  - There is NO public API to READ the current Compartment value (it's an internal map). The plugin must track applied slugs externally — hence the WeakMap.

  // No cleanup needed — WeakMap GCs entries when the EditorView is destroyed (registry destroy + browser GC).
