<objective>
Close 17-UAT.md Issue 4 (Tests 13 + 14): the themed HighlightStyle from Plan 17-05 paints tokens (so the binding works — tokens are colored, not fallback gray), but the same colors appear regardless of Obsidian's active theme — token palette does NOT track dark↔light theme switches, and is identical across two vaults running different community themes. Only the BACKGROUND flips on theme switch; foreground tokens remain identical.

Root cause analysis (from 17-UAT.md hypothesis + source inspection):
1. `src/main/childEditorTheme.ts:42-56` defines a HighlightStyle binding Lezer tags to `var(--code-keyword)`, `var(--code-string)`, etc.
2. The child editor's DOM is mounted via `Decoration.widget({ block: true })` (`src/main/nestedEditorExtension.ts:222-228`) — a fresh DOM subtree under a `<div class="lc-nested-editor">` container (`src/main/nestedEditorExtension.ts:101-103`).
3. Obsidian defines `--code-*` variables on selectors that target the markdown source view's code-block scope. The plugin's widget DOM is INSIDE `.cm-editor` (Obsidian's outer editor) but the `--code-*` cascade reaches it ONLY for variables defined at `body` or `:root` level — and in 2026's Obsidian build set, several themes scope `--code-*` to `.markdown-source-view .HyperMD-codeblock`, which the widget DOM does NOT match.
4. Where `var(--code-keyword)` resolves to the SAME value regardless of theme, it's because either (a) the variable resolves to a higher-scoped fallback that doesn't differ between themes, or (b) the variable is never defined for the widget's CSS scope and the browser resolves the unresolved `var(...)` to a HighlightStyle-internal default that's hardcoded.
5. `styles.css:1899-1923` already has `.cm-editor .lc-nested-editor` scoping for the child editor's container, but it does NOT define any `--code-*` variables — only `background`, `font-family`, etc. that don't drive the highlight palette.

Fix: Add theme-scoped CSS rules in `styles.css` that define the `--code-*` variable palette under `.theme-light .lc-nested-editor` and `.theme-dark .lc-nested-editor`. This forces the variables to resolve to a known, theme-dependent value inside the widget DOM regardless of how the active community theme scopes its own `--code-*`. The defaults are sensible (similar to GitHub's light/dark code palette) — community themes that DO scope `--code-*` to a selector covering `.lc-nested-editor` will still win via normal CSS specificity (more specific selectors override our `.theme-X .lc-nested-editor` baseline).

Purpose: Closes 17-UAT.md Issue 4 (Tests 13 + 14) — palette tracks Obsidian's base theme + community themes that override `--code-*`. The fix is cosmetic-only and entirely contained in CSS + a verifying source-level test.

Output: Two new theme-scoped CSS blocks in styles.css; a source-level test that asserts both blocks exist and the keyword color differs between them.
</objective>

Obsidian theme convention (HIGH confidence — used by every Obsidian theme + the default theme):
  - body classes: .theme-light, .theme-dark (mutually exclusive; switched by Obsidian on theme change)
  - rules under these classes cascade to all descendants
  - community themes that define --code-* typically scope under .theme-light / .theme-dark or :root
  - the fix uses these classes so it tracks Obsidian's mode without JS

These palettes are starting points — the plan does not lock specific hex codes. The KEY invariant is: light-mode and dark-mode --code-keyword resolve to DIFFERENT values (test asserts inequality, not specific values). Community themes that define --code-* on .theme-light / .theme-dark / :root will win via CSS cascade because they're authored at the same or higher specificity.

CRITICAL specificity note:
  - The plugin's rules go to `.theme-light .lc-nested-editor` (specificity 0,2,0).
  - Obsidian's own `:root` definitions (specificity 0,1,0) lose to ours — but Obsidian rarely puts --code-* on :root for child-editor consumption; this is fine.
  - Community themes that put --code-* on `body.theme-dark` (specificity 0,2,0 — same as ours) will tie; the LATER rule wins. styles.css loads AFTER community themes in Obsidian's load order — so the plugin DEFAULT may override a community theme's choice. This is a tradeoff: the plan accepts slightly more aggressive defaults to ensure the bug is fixed in stock Obsidian, and community-theme users can be served separately if a specific community theme reports a regression.
  - Mitigation: prefer scoping the plugin defaults at LOWER specificity than community-theme overrides. Use `:where(.theme-light) .lc-nested-editor` (specificity is `:where` = 0,1,0 because `:where` has zero specificity contribution, then `.lc-nested-editor` = 0,1,0 → total 0,1,0). This loses to any community theme rule with higher specificity. RECOMMENDED — the executor should use `:where()` to lower specificity to "default fallback" levels.

styles.css load order (verified by inspection of plugin manifest):
  - The plugin's styles.css is injected by Obsidian after the active theme.css.
  - Our rules at `.theme-X .lc-nested-editor` would WIN over any earlier-loaded community theme rule with the same selector. Using `:where()` reduces specificity so community themes win.
