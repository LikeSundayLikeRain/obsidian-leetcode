<objective>
Close two compounding regressions in fence auto-recovery surfaced by user manual testing on 2026-05-23 (round 2 — post 17-07..17-11 gap closure):

**Bug 1 (runtime trigger gap):** `repairFenceStructure` is wired into `createChildSyncExtension` as a CHILD-side `EditorView.updateListener` (childEditorSync.ts:87-127) that only fires on CHILD doc dispatches. When the user damages the fence in the PARENT (e.g., types over the closer line in Source Mode), the child-side listener never observes the change — there is no parent-side trigger. The user reports "only fires on reload" because on next mount, the child→parent sync's first dispatch trips the listener which then triggers repair against the broken parent state. This means runtime parent-side fence damage goes unrepaired until the user reloads or types in the child.

**Bug 2 (missing-closer recovery shape regression):** User reproduction shows that after deleting the fence closer line and reloading, the recovered state is wrong: instead of inserting a single bare ` ```` ` closer line immediately above `## Notes` to cap the original body, the repair pathway produces a duplicate fence pattern — the original body remains intact above (with closer), then a duplicate ` ```java + body + ` (or no closer) ` block appears below, leaving the doc with TWO fences where there should be ONE. This contradicts the round-1 fix (17-02) which the regression tests assert is correct, so the round-1 logic is sound in isolation but interacts incorrectly with either the post-repair full-replace dispatch (childEditorSync.ts:113-122), the lastBodyLine detection at line 471-477 (when trailing blank lines precede `## Notes`), or a re-entry path where repair fires multiple times in one parent transaction cycle.

Purpose: Plan 17-02 closed the marker-disambiguation root cause for offline repair invocation. Round-2 testing reveals the wiring + idempotency gap that prevents 17-02's fix from reaching the user during normal editing flow. Without this plan, runtime fence damage stays unrecovered until reload, and the recovery path itself can produce a worse-than-broken doc shape on the user's reproduction. Both bugs must be addressed together because the runtime trigger (Bug 1) is what surfaces Bug 2 to the user — testing them in isolation does not match the field reproduction.

Output: A round-2 debug doc enumerating Bug 1 + Bug 2 hypotheses with confirmed root cause(s); at least 3 new regression tests in `tests/main/childEditorSync.repair.test.ts` covering (1) parent-side runtime trigger, (2) the user's exact duplicate-fence reproduction, (3) re-entry idempotency; a fix scoped to what the debug doc confirms (parent-side trigger wiring + missing-closer correctness or re-entry guard, depending on Bug 2's confirmed cause); a UAT entry for manual verification.
</objective>
