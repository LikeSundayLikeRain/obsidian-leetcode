<objective>
Close the Source Mode phantom-duplicate-render bug surfaced by 17-UAT.md Tests 2 (PASTE-02) and 8 (SRCLIV-01). In Source Mode while the child editor has focus, structural changes (paste, Enter+type a new line) cause a phantom duplicate render of the parent fence body below the child editor; existing-line edits do NOT cause duplication; duplication disappears when the child editor loses focus.

Root cause (per 17-UAT.md Test 2 hypothesis, validated by source inspection of `src/main/nestedEditorExtension.ts:284-299`): the parent's `buildNestedEditorExtension` StateField has a fast-path at update() that detects `userEvent.startsWith('leetcode.')` and returns `old.map(tr.changes)` instead of rebuilding decorations via `buildNestedDecorations(tr.state, plugin, registry)`. When the child→parent mirror dispatches a line-adding change with `userEvent: 'leetcode.child-sync'` (childEditorSync.ts:108-114, addToHistory.of(false)), the fast-path runs — but `RangeSet.map(tr.changes)` only shifts existing decoration positions; it does NOT extend the line-hide coverage to the new line(s). Result: the new line's `## Code` parent doc text becomes visible beneath the child editor (the widget is anchored at openerLine.to, the original line-hides cover only the original line range, and the newly inserted line falls between them with NO hide decoration). Once the user changes focus, the StateField receives a non-`'leetcode.*'` transaction (or a `reconfigured` event), the rebuild path runs, line-hide is rebuilt over the full fence, and the phantom disappears.

Fix: narrow the `'leetcode.*'` fast-path so it covers only transactions that DO NOT change line count inside the fence — line-count-changing leetcode-userEvent transactions take the rebuild path. Equivalently: rebuild on docChanged when the changes touch any line inside `[fence.openerLine, fence.closerLine]`. Existing-line edits keep the fast-path (no regression for the 99% case).

Purpose: Closes 17-UAT.md Issue 1 (Tests 2 + 8) — the only Source-Mode-specific render failure in the UAT report.

Output: Patched `update()` in `buildNestedEditorExtension`'s StateField, regression test that fails on current main and passes after the fix.
</objective>

Transaction.changes API (CM6 standard):
  tr.changes.iterChanges((fromA, toA, fromB, toB, inserted) => { ... })
  tr.changes.iterChangedRanges((fromA, toA, fromB, toB) => { ... })  // simpler — only ranges
  tr.startState.doc.lineAt(offset).number — line count of pre-state
  tr.state.doc.lineAt(offset).number       — line count of post-state
  Use these to detect "did the change cross a line boundary".

Decision rule for the fix (CHOSEN):
  Trigger rebuild via buildNestedDecorations when ALL of these hold:
    1. tr.docChanged (changes exist)
    2. tr.annotation(Transaction.userEvent)?.startsWith('leetcode.') is true (current fast-path scope)
    3. tr.changes shifts any line boundary inside [tr.startState fence's openerLine, closerLine] OR adds/removes lines anywhere (simplest: any line-count delta).
  Otherwise keep old.map(tr.changes) for the fast-path so single-character existing-line edits stay cheap.
  Implementation: count lines pre vs post via tr.startState.doc.lines vs tr.state.doc.lines — equality means existing-line-only edit.
