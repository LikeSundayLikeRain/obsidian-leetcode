## Multi-case Run semantics (issue 2.a + 2.b)

### Q1: How should Run handle multiple test cases?

| Option | Description | Selected |
|--------|-------------|----------|
| A: Batch all tabs as ONE LC call (LC.com behavior) | One `interpret_solution` call, server returns arrays, renderer splits per-case tabs. Matches LC.com. | ✓ |
| B: Loop client-side (one LC call per tab) | N REST calls, aggregate results. Simpler split, more network, no partial progress. | |
| C: Status quo (active tab only) | Send only active tab; show all tabs as result tabs with PASS/FAIL only on the run case. | |

**User's choice:** A — Batch all tabs as ONE LC call (D-01)

### Q2: How should we determine input arity (lines-per-case)?

| Option | Description | Selected |
|--------|-------------|----------|
| A: Derive from `sampleTestCase` (LC's official source) | Newlines + 1 = arity. Single source of truth = LC. | ✓ |
| B: Track arity per-tab in EphemeralTabStore | Each tab stores its own line count. No LC ground truth. | |
| C: Punt arity — require 1 tab = 1 case | Simplest impl, hostile to copy-paste users. | |

**User's choice:** A — Derive from sampleTestCase (D-02)

### Q3: When user clicks '+' to add a custom case, what should happen?

| Option | Description | Selected |
|--------|-------------|----------|
| A: Append below sample cases (additive) | Sample cases stay as Cases 1..N; new tabs become N+1, N+2... | ✓ |
| B: Tag custom cases visually + persist across sessions | Breaks D-08/CF-06 from Phase 5 (in-memory only). | |
| C: Keep additive, mark custom-case results visually different | No persistence change but adds custom chips. | |

**User's choice:** A — Additive, status quo on persistence (D-03)

### Q4: How should per-tab PASS/FAIL chips be computed?

| Option | Description | Selected |
|--------|-------------|----------|
| A: Per-case strict string compare (LC.com) | `code_answer[i] === expected_code_answer[i]` per-case. | ✓ |
| B: Use only top-level correct_answer | All tabs same color. Less informative. | |
| C: Tolerance hook for set-output edge cases | Future-proof but LC.com doesn't do this. | |

**User's choice:** A — Per-index strict string compare (D-04)
