## Arity Plumbing â€” Decision Record

**`sampleTestCase` is NOT plumbed through the renderer.** Two arity dimensions exist:

| Dimension | Source | Used for |
| --- | --- | --- |
| **Case count** | `Math.max(code_answer.length, expected_code_answer.length, 1)` | Number of case tabs to emit (D-05) and the case-pass mask iteration |
| **Lines per case** | `parseMetaData(metaData)?.params.length` (when metaData parses) â†’ else `Math.ceil(totalLines / arity)` (when metaData is absent) | `splitInput` chunking for the Input section (D-08) |

**Why response-driven case count, not metaData-driven:** if a user submits a single custom-input case to a multi-param problem (e.g., one tab with `[1,2,3]\n4` for two-sum), `metaData.params.length === 2` but the response carries `code_answer.length === 1`. The user expects ONE tab labeled `Case 1`, not two phantom tabs that have no output. The response array is authoritative on case count.

**Why metaData drives lines-per-case:** LC's wire format is one value per line in `metaData.params` order. For a 2-case Two Sum run with `dataInput = '[2,7,11,15]\n9\n[3,2,4]\n6'`, the joined input is 4 lines; `splitInput(joined, 2)` recovers `['[2,7,11,15]\n9', '[3,2,4]\n6']` â€” one chunk per case, each chunk carrying `params.length` lines. Each chunk's lines map 1-to-1 with `metaData.params` for the labeled-row rendering.

**Fallback chain when metaData is absent (production today):**
1. Compute `linesPerCase = Math.ceil(totalLines / caseCount)` â€” best-effort split.
2. `splitInput(joined, linesPerCase)` produces per-case chunks.
3. `renderInputSection` falls through to the raw-dump branch (single `<pre>`) since `md` is null.
4. The user sees `[2,7,11,15]\n9` verbatim per tab â€” informative, just not labeled.

The current renderer treats the empty string as `passMask[i] = null` (no chip) since `expected[1] === ''` triggers the null branch. Need a live-smoke fixture (Plan 05) to confirm LC's actual wire shape for partial-failure runs and adjust the mask logic if expected stays populated.

2. **DetailCacheEntry plumbing for metaData / sampleTestCase:** see the Â§ProblemDetail Cache Shape â€” Surfaced Gap section above. Production users will hit the raw-dump fallback until this lands. Tests pass via the synthetic fixture; live usage degrades gracefully.

5. **Aggregate-pass reading on `code_answer = []` + `expected_code_answer = []`:** this is the timeout/error path, but it's also defensible to assert "no cases yielded any output" should treat aggregate as fail. Currently `passMask.every((m) => m !== false) === true` (vacuously true), which paints the title `Accepted`. Plan 05 should validate.
