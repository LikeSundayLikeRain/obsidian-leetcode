## User Constraints (from CONTEXT.md)

### Locked Decisions

**Multi-case Run wire format**
- **D-01:** Run = ONE batched `interpret_solution` call. RunModal joins all tabs' inputs into one `dataInput` string; the response's `code_answer[]` and `expected_code_answer[]` arrays are split per-case for rendering. Matches LC.com.
- **D-02:** Per-case arity is derived from `sampleTestCase` (LC GraphQL field on questionData). Count newlines + 1 = lines per case. This becomes the splitter for both the request join and the response split. Single-source-of-truth = LC.
- **D-03:** Adding a custom case is purely additive — sample cases stay as Cases 1..N, custom tabs append as N+1, N+2... `Reset to sample cases` wipes custom tabs and re-seeds from `exampleTestcases`. Status quo on persistence: EphemeralTabStore stays in-memory only (D-08 from Phase 5 / CF-06 unchanged).
- **D-04:** Per-tab PASS/FAIL chip = `code_answer[i].trim() === expected_code_answer[i].trim()` (per-index strict string compare). LC's aggregate `correct_answer` is ignored (it's just AND of per-case anyway).

**Run verdict modal layout**
- **D-05:** Layout matches LC.com screenshot exactly — title row with verdict + runtime inline, tab strip always visible (even with N=1, per LC behavior), three stacked sections (Input → Output → Expected). Footer: Close on the right (`mod-cta`).
- **D-06:** All colors come from Obsidian semantic CSS vars only — `var(--text-error)`, `var(--text-success)`, `var(--text-muted)`, `var(--text-normal)`, `var(--background-secondary)`. NO hardcoded LC palette. Adapts to every theme automatically.
- **D-07:** State-driven coloring (refines D-06):
  - **Title:** `var(--text-error)` on WA/TLE/MLE/RE/CE; `var(--text-success)` on AC; muted gray on Run-pending.
  - **Output value text:** `var(--text-error)` only when this case is failing; `var(--text-normal)` when passing.
  - **Expected value text:** `var(--text-success)` only when the corresponding case is failing; `var(--text-normal)` when the case passes.
  - **Section labels (`Input`, `Output`, `Expected`):** always `var(--text-muted)`.
  - **Section surfaces:** always `var(--background-secondary)`.
- **D-08:** Input section parses `metaData.params[]` and renders per-param `name = value` (e.g., `nums =\n[2,2,3,2]`). Param name in muted small text, value in monospace on its own line. Falls back to raw input dump if `metaData` is missing or malformed.

**Button polish**
- **D-09:** All non-CTA buttons (Cancel, Close, Reset, +, ×, tab buttons) port the `.lc-fm__picker` look already established by Phase 5.3 chevron polish — `min-height: 30px`, `1px solid var(--background-modifier-border)`, `border-radius: 6px`, `background: var(--background-primary)`, hover → `border-color: var(--text-muted)`. Primary buttons (Run, Submit, Close in modal footer) keep `mod-cta`.
- **D-10:** RunModal's tabs visually match the VerdictModal's tab strip — same shape, same `.is-active` highlight, same `×` delete look, same `+` add-tab pill size.
- **D-11:** Footer hierarchy stays primary-right / secondary-left, using today's `space-between` flex layout. No structural change.
- **D-12:** Inline action row gets the same polish.
  - **D-12a:** Inline Run/Submit buttons port the new neutral-button style.
  - **D-12b:** Chevron glyph changes from Unicode `▼` triangle to Lucide `chevron-down` via `setIcon(span, 'chevron-down')`. Chevron button shape becomes a select-input pill — matches the look of the problem-browser's filter dropdown (`.lc-filter-modal .lc-fm__picker` style).

**Verdict modal info density**
- **D-13:** Run verdict header chrome is `{verdict}  Runtime: N ms` on a single row — verdict in bold (state-colored), runtime in muted text. NO problem title in chrome.
- **D-14:** Submit verdict modal stays as-is. `renderSubmitVerdict` (and its existing runtime/memory/percentile row) is unchanged. Only the Run path gets the LC-style overhaul.
- **D-15:** Run-mode compile_error / runtime_error render as a single error state — title in `var(--text-error)`, full error text in a mono error block, no per-case tabs. No "Copy error" button (defer if requested later).
- **D-16:** Asymmetric Copy-failing-testcase behavior:
  - **Run verdict:** NO copy button. Drop from `renderRunResult`.
  - **Submit verdict:** KEEP existing Copy button. On click, append the failing case as a new tab in RunModal (additive per D-03), then close the verdict modal. Keep `onCopyFailingInput` shape; confirm append target is RunModal's `EphemeralTabStore`.

### Anti-Patterns to Avoid

- **`innerHTML` for code_answer / expected_code_answer values** — flagged by `eslint-plugin-obsidianmd`; XSS risk; will fail community plugin review. Use `setText(pre, value)` (already the pattern in `makeSection`).
- **Global `document` in renderer** — renderer uses `parent.ownerDocument` to stay popout-window safe. Same posture as Phase 5.3 chevron widget.
- **Forking the renderer into `runVerdictRenderer.ts`** — would force `VerdictModal.ts` to dispatch on payload shape (Run vs. Submit) at TWO sites. Keep one renderer; expand the Run branch.
- **Hardcoding LC colors (red/green hex)** — D-06 explicitly forbids; theme adaptation breaks.
- **Querying `code_answer.length` to derive arity** — UNRELIABLE. LC may return fewer outputs than inputs on early-exit (compile_error during sample 2 of 3). Use `metaData.params.length` (D-08) as authoritative arity; fall back to `sampleTestCase` newlines+1 (D-02) only if metaData absent.

## Common Pitfalls

### Pitfall 1: `code_answer.length !== expected_code_answer.length` on partial failures

**What goes wrong:** When LC's interpreter throws mid-sample (e.g., runtime error on sample 2 of 3), `code_answer` may have fewer entries than `expected_code_answer`. Per-index compare at index >= code_answer.length silently treats the case as "missing output" rather than "errored mid-execution".

**Why it happens:** LC fills `code_answer` lazily; `expected_code_answer` is pre-computed.

**How to avoid:** Use authoritative arity from `metaData.params` count or `sampleTestCase` newline arithmetic — NOT `code_answer.length`. For each case slot i in 0..arity-1: if i >= code_answer.length, render that tab as "no output" (likely the failing case).

**Warning signs:** Per-case PASS/FAIL chip shows `passed` for indexes that LC never executed.

`[VERIFIED via fixture inspection: tests/solve/fixtures/run-sample.json shows code_answer.length === expected_code_answer.length === 3 on a 3-case happy path]`

### Pitfall 2: `code_answer` shape is `string | string[]` — not always array

**What goes wrong:** Renderer assumes `Array.isArray(res.code_answer)` and never handles the single-string case.

**Why it happens:** The TypeScript type at `types.ts:78-79` is `string | string[]`. The runtime renderer at `verdictModalRenderer.ts:359-366` (`isRunResponse`) requires array AND non-empty for the Run branch — single-string responses fall through to submit branch.

**How to avoid:** Use the existing `toLines(v: string | string[] | undefined): string[]` helper at `verdictModalRenderer.ts:376-380` which normalizes both shapes to `string[]`. Always go through this helper for D-04 per-index compare.

**Warning signs:** TypeError on `.map` / `.length` over a string primitive.

### Pitfall 3: Test selectors that lock to literal `▼` glyph

   - What we know: Submit-mode CE has `compile_error` + `full_compile_error`. Run-mode shape is documented identically (per `RunCheckResponse` typing — `types.ts:84-85`).
   - What's unclear: Whether LC returns Run-mode CE on user code with syntax errors, or routes it through a different status_code.
   - Recommendation: Sniff for both `compile_error` AND status_code 20 in `renderRunResult`. Matches the typing.

3. **Per-case error: does runtime_error on case 2 of 3 still populate `code_answer[0]`?**
   - What we know: A2 assumes happy-path equal lengths. Edge case is unverified.
   - What's unclear: Whether sample 1 output is preserved when sample 2 errors.
   - Recommendation: Capture during live smoke. Renderer iterates `Math.max(outputs.length, expected.length, arity)` and shows "(no output)" for missing slots — robust to either outcome.
