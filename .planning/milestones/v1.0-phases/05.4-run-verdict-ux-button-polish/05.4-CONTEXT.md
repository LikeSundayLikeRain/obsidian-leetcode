# Phase 5.4: Run Verdict UX + Button Polish - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5.4 polishes the visible UX layer between clicking Run/Submit and seeing results. Three surfaces:

1. **RunModal** (where users choose test cases) ‚Äî tab strip restyled, buttons restyled, `+`/`√ó` polished.
2. **Run-mode VerdictModal** (the LeetCode-style per-case result page) ‚Äî header redesigned, per-case tabs added, Input/Output/Expected sections introduced with theme-aware coloring, error states cleaned up.
3. **Inline `## Code` action row** (Phase 5.1 + 5.3 chevron + Run + Submit) ‚Äî buttons restyled to match the new modal style; chevron glyph + shape redesigned to look like a select-input.

**In scope:**
- Multi-case Run via batched `interpret_solution` call (D-01, D-02): one REST call per Run click, results split per-case using `sampleTestCase` arity.
- Per-case PASS/FAIL chips computed via per-index string compare against `expected_code_answer` (D-04).
- Verdict modal layout matches LC.com screenshots (D-05): header inline, tab strip always visible, stacked Input/Output/Expected sections.
- Theme-aware colors using only Obsidian semantic vars (D-06, D-07): title color shifts state-by-state, Output value color shifts only on WA, Expected value color shifts only on WA.
- Input section uses `metaData.params[]` to render `name = value` per param (D-08).
- Button styling: non-CTA buttons port the `.lc-fm__picker` look from Phase 5.3 chevron polish (D-09); RunModal tabs match VerdictModal tabs (D-10); footer keeps primary-right / secondary-left (D-11).
- Inline action row Run/Submit + chevron get the same polish (D-12); chevron uses Lucide `chevron-down` glyph instead of `‚ñº` triangle and renders as a select-input pill.
- Run-mode header chrome: `{verdict}  Runtime: N ms` inline, no problem title (D-13).
- Run-mode error states (compile / runtime) render single error block with no per-case tabs (D-15).
- Submit-mode "Copy failing testcase" button preserved + extended to append the failing case as a new RunModal tab (D-16).

**Explicitly out of scope:**
- **Submit verdict modal redesign** ‚Äî `renderSubmitVerdict` stays untouched (D-14). Only the Run path gets the LC-style overhaul.
- Custom AI-generated test cases ‚Äî separate phase if desired.
- Verdict history / share-result links ‚Äî separate phase.
- Run percentile chips ‚Äî LC's `interpret_solution` doesn't return percentiles (D-13 scope note).
- Persisting custom cases across note re-opens ‚Äî D-08 from Phase 5 (CF-06) is preserved (D-03 keeps EphemeralTabStore in-memory only).
- Section locking inside lc-slug notes ‚Äî that's Phase 5.5.
- Order-insensitive answer comparison ‚Äî strict per-index string compare only, matching LC.com (D-04 rejected option C).

</domain>

<decisions>
## Implementation Decisions

### Multi-case Run wire format
- **D-01:** Run = ONE batched `interpret_solution` call. RunModal joins all tabs' inputs into one `dataInput` string; the response's `code_answer[]` and `expected_code_answer[]` arrays are split per-case for rendering. Matches LC.com.
- **D-02:** Per-case arity is derived from `sampleTestCase` (LC GraphQL field on questionData). Count newlines + 1 = lines per case. This becomes the splitter for both the request join and the response split. Single-source-of-truth = LC.
- **D-03:** Adding a custom case is purely additive ‚Äî sample cases stay as Cases 1..N, custom tabs append as N+1, N+2... `Reset to sample cases` wipes custom tabs and re-seeds from `exampleTestcases`. Status quo on persistence: EphemeralTabStore stays in-memory only (D-08 from Phase 5 / CF-06 unchanged).
- **D-04:** Per-tab PASS/FAIL chip = `code_answer[i].trim() === expected_code_answer[i].trim()` (per-index strict string compare). LC's aggregate `correct_answer` is ignored (it's just AND of per-case anyway).

### Run verdict modal layout
- **D-05:** Layout matches LC.com screenshot exactly ‚Äî title row with verdict + runtime inline, tab strip always visible (even with N=1, per LC behavior), three stacked sections (Input ‚Üí Output ‚Üí Expected). Footer: Close on the right (`mod-cta`).
- **D-06:** All colors come from Obsidian semantic CSS vars only ‚Äî `var(--text-error)`, `var(--text-success)`, `var(--text-muted)`, `var(--text-normal)`, `var(--background-secondary)`. NO hardcoded LC palette. Adapts to every theme automatically.
- **D-07:** State-driven coloring (refines D-06):
  - **Title:** `var(--text-error)` on WA/TLE/MLE/RE/CE; `var(--text-success)` on AC; muted gray on Run-pending.
  - **Output value text:** `var(--text-error)` only when this case is failing; `var(--text-normal)` when passing.
  - **Expected value text:** `var(--text-success)` only when the corresponding case is failing (so the user can spot the right answer); `var(--text-normal)` when the case passes.
  - **Section labels (`Input`, `Output`, `Expected`):** always `var(--text-muted)`.
  - **Section surfaces:** always `var(--background-secondary)`.
- **D-08:** Input section parses `metaData.params[]` and renders per-param `name = value` (e.g., `nums =\n[2,2,3,2]`). Param name in muted small text, value in monospace on its own line. Falls back to raw input dump if `metaData` is missing or malformed.

### Button polish
- **D-09:** All non-CTA buttons (Cancel, Close, Reset, +, √ó, tab buttons) port the `.lc-fm__picker` look already established by Phase 5.3 chevron polish ‚Äî `min-height: 30px`, `1px solid var(--background-modifier-border)`, `border-radius: 6px`, `background: var(--background-primary)`, hover ‚Üí `border-color: var(--text-muted)`. Primary buttons (Run, Submit, Close in modal footer) keep `mod-cta` (Obsidian's theme-aware accent).
- **D-10:** RunModal's tabs visually match the VerdictModal's tab strip ‚Äî same shape, same `.is-active` highlight, same `√ó` delete look, same `+` add-tab pill size. Verdict and Run modal tabs feel like the same surface.
- **D-11:** Footer hierarchy stays primary-right / secondary-left, using today's `space-between` flex layout. No structural change.
- **D-12:** Inline action row (Phase 5.1 Run/Submit + Phase 5.3 chevron) gets the same polish. Two refinements:
  - **D-12a:** Inline Run/Submit buttons port the new neutral-button style (replace today's plain Obsidian buttons).
  - **D-12b:** Chevron glyph changes from Unicode `‚ñº` triangle to Lucide `chevron-down` (via `setIcon(span, 'chevron-down')`). Chevron button shape becomes a select-input pill ‚Äî matches the look of the problem-browser's filter dropdown (`.lc-filter-modal .lc-fm__picker` style). Specific tokens: `min-height: 30px`, `1px solid var(--background-modifier-border)`, `border-radius: 6px`, `background: var(--background-primary)`, hover `border-color: var(--text-muted)`. The chevron is no longer button-shaped ‚Äî it reads as "click me to pick".

### Verdict modal info density
- **D-13:** Run verdict header chrome is `{verdict}  Runtime: N ms` on a single row ‚Äî verdict in bold (state-colored), runtime in muted text. NO problem title in chrome (the modal opened from THE active note; user knows what they're looking at).
- **D-14:** **Scope refinement** ‚Äî Submit verdict modal stays as-is. `renderSubmitVerdict` (and its existing runtime/memory/percentile row) is unchanged. Only the Run path gets the LC-style overhaul. This shrinks the implementation surface.
- **D-15:** Run-mode compile_error / runtime_error render as a single error state ‚Äî title in `var(--text-error)`, full error text in a mono error block (`var(--background-secondary)` surface, scrollable on overflow), no per-case tabs. No "Copy error" button (defer if requested later).
- **D-16:** Asymmetric Copy-failing-testcase behavior:
  - **Run verdict:** NO copy button. The user authored the input themselves in RunModal ‚Äî copying it back to themselves is redundant. Drop the button from `renderRunResult`.
  - **Submit verdict:** KEEP the existing Copy button. On click, append the failing case (`last_testcase` from LC's submit response) as a new tab in RunModal (additive per D-03), then close the verdict modal. Today's renderer already wires `onCopyFailingInput` for Submit mode ‚Äî keep that shape, just confirm the append target is RunModal's `EphemeralTabStore`.

### Claude's Discretion
- Specific CSS class names for the new verdict-modal sections (e.g., `.leetcode-verdict-case-tab`, `.leetcode-verdict-input`, `.leetcode-verdict-output-value`) ‚Äî planner picks based on house style established in Phase 5.3 (`leetcode-` prefix is the convention).
- Whether the new verdict layout reuses today's `verdictModalRenderer.ts` or a new `runVerdictRenderer.ts` ‚Äî planner decides based on testability and shared-helper economics.
- The exact splitter algorithm for arity-aware case-block joining/splitting ‚Äî planner reads sampleTestCase examples and decides between line-counting and block-counting.
- Whether to expose `data-status="pass"|"fail"` attributes on case tabs / sections (rejected from D-06 path C ‚Äî not needed since theme vars handle every theme ‚Äî but planner may still add them for testability).
- Lucide icon name fallback if `chevron-down` doesn't render in older Obsidian builds (e.g., `caret-down`) ‚Äî planner verifies via Phase 5.3's `setIcon` precedent.

</decisions>

## Existing Code Insights

### Reusable Assets
- `.lc-fm__picker` CSS class (Phase 5.3 chevron polish) ‚Äî port to all non-CTA buttons + the chevron container per D-09 / D-12b.
- `setIcon` from Obsidian (already imported in `VerdictModal.ts` and `languageChevronWidget.ts`) ‚Äî use for `chevron-down` glyph (D-12b) and any Lucide-backed icons in the new verdict layout.
- `EphemeralTabStore.addTab(slug, input)` (or the equivalent `setTabs` shape) ‚Äî D-16 Submit-side copy-failing routes through this.

- `appendEl` / `setText` / `clear` / `addClass` helpers in `verdictModalRenderer.ts` and `VerdictModal.ts` ‚Äî already happy-dom-safe; reuse for any new DOM nodes.

### Established Patterns
- **Renderer + Modal split** ‚Äî `verdictModalRenderer.ts` is environment-agnostic (testable under happy-dom); `VerdictModal.ts` wraps it with Obsidian chrome (`setIcon`, focus management, Notice). Plan 5.4 preserves this split for testability.
- **State-driven CSS class additions** ‚Äî `addClass(el, 'leetcode-verdict-pending')` etc. are removed via `clearPendingStateClass()` on transition. Plan 5.4 follows the same pattern for AC/WA/error state classes.
- **`data-lc-role="close"` attribute on Close buttons** ‚Äî `VerdictModal.focusCloseButton()` uses this selector to wire Close click + default focus. New Run-verdict layout MUST keep this attribute on its Close button.
- **CF-07 (no innerHTML, no script-parsing sinks)** ‚Äî every DOM node built via `createElement` + `textContent`. Plan 5.4 preserves.

### Integration Points
- `runFromActive()` in `src/main.ts` ‚Äî the production entry point. Builds `SubmissionContext`, calls `interpretSolution`, hands off to verdict-modal. D-01 changes the dataInput shape (multi-case batched); the orchestrator probably doesn't need to change since it's just a string.
- `pollingOrchestrator.ts` ‚Äî handles backoff polling for Run + Submit. No change for Plan 5.4 ‚Äî the per-case rendering is purely a renderer concern after polling completes.
- `tests/solve/verdictModalRenderer.test.ts` ‚Äî fixture-based tests under happy-dom. Plan 5.4 expands these for the new Run-mode layout (per-case tabs assertion, color class assertion, error state assertion).
- `tests/solve/RunModal.test.ts` ‚Äî drives the existing tab-strip selectors (`.leetcode-run-tab`, `.is-active`, `.leetcode-run-tab-add`, `.leetcode-run-tab-delete`, `.leetcode-run-textarea`, `.leetcode-run-submit`). Plan 5.4 D-01/D-09/D-10 may add new selectors (e.g., `.leetcode-run-tab.is-custom` if the planner adds visual distinction) ‚Äî keep existing selectors stable for backward compat.

<specifics>
## Specific Ideas

- User-provided LC screenshots (Wrong Answer + Accepted) ARE the design spec. Planner should keep them visible during impl.
- User explicitly wants the chevron to look like the problem-browser filter dropdown (`.lc-fm__picker` styled select-input with `chevron-down` glyph).
- D-16 Submit-side copy-failing: the user worded it as "submit only show 1 failure at a time, so we can just append the test case to run" ‚Äî the append-to-RunModal-as-new-tab flow is the definitive UX.
- Q2 from the cycle-2 UAT (copy-onto-no-Code-heading auto-creates the heading) is accepted behavior ‚Äî Plan 5.4 should NOT regress that auto-create path; it's used by the D-16 Submit-copy flow when the failing case has no `## Code` heading yet (rare but possible).
- The new chevron pill should NOT look like a button ‚Äî it should read as a select-input. The user's phrase: "make the button looks more like a text box".

</specifics>
