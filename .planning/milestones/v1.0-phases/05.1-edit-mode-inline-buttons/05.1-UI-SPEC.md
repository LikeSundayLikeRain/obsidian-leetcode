## Spacing Scale

Declared values (all multiples of 4, verified against shipped styles.css rules):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Widget `margin-top: 4px` from closing fence line (shipped `.leetcode-code-actions` rule) |
| sm | 8px | Gap between Run and Submit buttons (`gap: 8px`); button horizontal padding (`padding: 0 8px`); `margin-bottom: 8px` below row |
| md | 16px | not used in this surface |
| lg | 24px | Button row height (`height: 24px`) â€” fits Obsidian's compact-button visual language |
| xl | 32px | not used |

**Exceptions:**
- Button height `24px` â€” compact-button height matching Obsidian's pill/chip visual language; verified shipped in Reading Mode; NOT a 44px touch-target because desktop-only plugin (`isDesktopOnly: true`) + pointer-primary input.
- Conditional tweak budget: if live-smoke reveals visual misalignment, add `.cm-editor .leetcode-code-actions { margin-top: 8px }` (still 8-scale-compliant). This is a contingent addition flagged in RESEARCH Open Question #3, not a default.

**Widget spacing relative to the closing fence:** The widget is anchored at `state.doc.line(closerLine).to` with `side: 1` (inline, non-block). This renders at the end of the closing fence line, flowing in-line. The existing `margin-top: 4px` rule creates 4px of vertical breathing room between the fence's baseline and the button row's top edge in Reading Mode; in Edit Mode (Live Preview + Source Mode) the same rule applies via CSS cascade â€” live-smoke verifies actual rendered spacing matches. If it doesn't visually match, the spec permits a `.cm-editor`-scoped override of `margin-top: 8px` as the only sanctioned deviation.

## Color

All colors via Obsidian CSS variables â€” no raw hex, no rgba literals (conforms to project convention locked in `styles.css` header comment at line 2).

| Role | Value | Usage in this phase |
|------|-------|---------------------|
| Dominant (60%) | `--background-primary` (Obsidian theme var) | Note editor background surrounding the widget â€” not painted by this phase, inherited from Obsidian |
| Secondary (30%) | `--background-secondary` (Obsidian theme var) | Button default background (`.leetcode-code-action-run`, `.leetcode-code-action-submit`) |
| Accent (10%) | `--background-modifier-hover` + `--text-normal` (Obsidian theme vars) | Button hover state only |
| Destructive | not used | No destructive action in this surface |
| Border | `--background-modifier-border` (Obsidian theme var) | Button `1px` border (subtle affordance) |
| Text default | `--text-muted` (Obsidian theme var) | Button label in default state |
| Text hover | `--text-normal` (Obsidian theme var) | Button label on hover |

**Accent reserved for:**
- Run button hover background + label color uplift (`--background-modifier-hover` + `--text-normal`)
- Submit button hover background + label color uplift (same)
- **NOT** used as a primary-call-to-action color. Buttons are deliberately **neutral** (no `mod-cta`, no `--interactive-accent` background) â€” matches Reading-Mode D-13 shipped design.

**Why neutral, not primary:** Locked by Phase 5 D-13 (and mirrored here): the button row is a *subtle* affordance that lives in the document flow. A primary (accented) button would dominate visual hierarchy and compete with the user's code. Neutral buttons sit quietly next to the fence and lift on hover â€” the commanded action is the hover-to-normal color transition, not a always-on visual shout.

**Theme verification method:**
- **Light theme:** Obsidian Settings â†’ Appearance â†’ Theme: "Default" light mode. Verify `--background-secondary` renders the button as a pale-gray pill against white editor background; hover lifts to a slightly darker gray; text transitions from muted (~#6b7280-equivalent) to normal (~#1f2937-equivalent).
- **Dark theme:** Settings â†’ Appearance â†’ Theme: "Default" dark mode. Verify the same pill renders as a darker-than-background-primary panel against the dark editor; hover lifts to `--background-modifier-hover`; text transitions from muted to normal (lighter gray).
- **Sampling cadence:** After Wave 1 implementation passes unit tests, during D-10 live-smoke, toggle Settings â†’ Appearance â†’ Theme between light and dark with an `lc-slug` note open in Live Preview. Verify buttons legible and hover responsive in both. Repeat in Source Mode. Repeat in Reading Mode (regression check on 05-05's shipped surface).

## Copywriting Contract

| Element | Copy | Source |
|---------|------|--------|
| Primary CTA 1 | `Run` | Locked from Phase 5 Plan 05 (`src/main/codeBlockButtonRow.ts` line 17) â€” shipped in Reading Mode |
| Primary CTA 2 | `Submit` | Locked from Phase 5 Plan 05 (`src/main/codeBlockButtonRow.ts` line 27) â€” shipped in Reading Mode |
| Empty state heading | not applicable | Widget is either rendered (on `lc-slug` notes with a `## Code` fence) or not rendered â€” no inner empty state |
| Empty state body | not applicable | Same â€” no inner empty state |
| Error state | no inline error rendering | Click errors surface via Notice through existing `runFromActive` / `submitFromActive` handlers â€” see "Error Surfacing" below |
| Destructive confirmation | not applicable | Neither Run nor Submit is destructive. Submit writes to LC, not to the vault. No confirmation gate on click |
| Loading state | no loading state | No spinner, no disabled state during pending Run/Submit â€” see "Loading / Pending State" below |
| Focus / keyboard hint | not applicable | Buttons rely on native `:focus-visible` browser behavior â€” see "Keyboard & Focus Behavior" |

**Error Surfacing (inherited from Phase 4 D-06 via CONTEXT canonical_refs):**

- Click dispatch = command-palette-flow semantics â†’ **Notice-only** error surface. No inline toast, no inline error text, no red button state.
- `plugin.runFromActive()` / `plugin.submitFromActive()` already own the full error-routing pipeline (CF-19 locked â€” no new Notice copy in this phase).
- Error conditions handled by the downstream handlers: session expired, 429 rate-limit, network timeout, no active editor, empty code block, LC server error. All surface as Obsidian `Notice` per the locked copy from Phase 5 Plan 03 (`src/ui/SessionExpiredNotice.ts` + main.ts error-mapping).
- The widget itself renders nothing conditional on click outcome. Buttons remain in default visual state through pending â†’ complete â†’ error.

**Loading / Pending State (deliberate non-decision, inherited):**

- **No spinner.** **No disabled state while awaiting response.** **No visual change on click.**
- Rationale (locked to match Reading Mode, which also shows no inline spinner):
  1. `plugin.runFromActive` / `plugin.submitFromActive` already route through `guardSingleFlight` â€” rapid re-clicks are safely coalesced by the handler layer.
  2. Feedback channel is Notices + VerdictModal â€” these carry the complete pending/complete/error story. An inline spinner would duplicate information and risk drift with the handler's single-flight guard.
  3. Matches Reading-Mode parity (Phase 5 Plan 05 shipped with no spinner). Dual modes behave identically.
- **Contract:** If a future phase wants inline pending feedback, it MUST add it to BOTH modes (Reading and Edit) atomically â€” no asymmetric state between modes.

**Keyboard & Focus Behavior:**

- Buttons are native `<button>` elements emitted by `buildCodeBlockButtonRow` via `doc.createElement('button')`. This makes them keyboard-focusable and activatable by default (Space / Enter).
- CM6 widget with `side: 1` (inline, non-block) **does** include the widget in the editor's tab order unless `ignoreEvent` returns `true` for focus events. Phase 5.1's `CodeActionsWidget.ignoreEvent()` returns `false` per RESEARCH Pitfall 3 â€” click events propagate. Tab order inclusion is the result.
- **Decision:** Accept the default tab order. Users Tab-navigating through the editor will land on Run, then Submit, after the `## Code` fence's contents. This is the expected accessible behavior for an inline action surface.
- **Focus indicator:** Browser-default `:focus-visible` ring. The shipped `.leetcode-code-actions` rules do NOT set `outline: none` â€” native focus ring is preserved. Verify during D-10 live-smoke that the ring is visible against both light and dark themes (it should be; Obsidian's theme vars don't stomp native `:focus-visible`).
- **No custom key bindings.** No new hotkeys. Users who want a keyboard shortcut bind one via Obsidian Hotkeys settings on the existing `leetcode:run` / `leetcode:submit` commands (CF-05 + FND-03).

**Visual States (complete list):**

| State | Visual Treatment | Implementation |
|-------|------------------|----------------|
| Default | `background: var(--background-secondary)`, `color: var(--text-muted)`, `border: 1px solid var(--background-modifier-border)` | Shipped in `styles.css` lines 867-877 |
| Hover | `background: var(--background-modifier-hover)`, `color: var(--text-normal)`, transition `100ms ease` | Shipped in `styles.css` lines 878-882 |
| Focus-visible | Native browser focus ring (no override) | Not explicitly styled â€” inherited from UA default |
| Active (pressed) | Not explicitly styled â€” browser default `:active` (momentary flash) | Inherited from UA default |
| Disabled | **Not used.** Buttons never disable. Single-flight guard lives in the handler, not the DOM | By omission |
| Pending / Loading | **Not used.** No visual change during pending response | By omission |
| Error | **Not used.** Errors surface via Notice, not in the button surface | By omission |

## Interaction Contract

### Mount Points

| Context | Mount Path | Source-of-Truth File |
|---------|-----------|---------------------|
| Reading Mode | `MarkdownPostProcessor` via `registerCodeBlockActionProcessor(plugin)` | `src/main/codeActionsPostProcessor.ts` (shipped in Phase 5 Plan 05 â€” **UNTOUCHED in 5.1**) |
| Edit Mode (Live Preview) | CM6 editor extension via `plugin.registerEditorExtension(buildCodeActionsEditorExtension(plugin))` | `src/main/codeActionsEditorExtension.ts` (NEW in Phase 5.1) |
| Edit Mode (Source Mode) | Same CM6 editor extension as Live Preview â€” no branching | `src/main/codeActionsEditorExtension.ts` (NEW in Phase 5.1) |

Both mount points emit the **same DOM** (from `buildCodeBlockButtonRow`) and apply the **same CSS** (`.leetcode-code-actions`). Visual parity is guaranteed by DOM/CSS reuse, not by independent re-specification.

### Gate: when the buttons appear

Buttons render **if and only if** all of the following are true:

1. The active note has non-empty `lc-slug` frontmatter (verified via `app.metadataCache.getFileCache(file)?.frontmatter`).
2. The note body contains an `## Code` H2 heading followed (before the next H2) by a fenced code block (triple-backtick opener + closer).
3. The editor view is rendering (CM6 extension only attaches in Edit Mode; MarkdownPostProcessor only runs in Reading Mode).

Notes without `lc-slug`, notes with `lc-slug` but no `## Code` fence, and notes with fences ONLY outside `## Code` (e.g., example fences under `## Problem`) all show **zero** buttons.

### Click Dispatch

| Button | Handler | Error Path |
|--------|---------|-----------|
| `Run` | `plugin.runFromActive()` (direct method call) | Notice via existing error-mapping in handler |
| `Submit` | `plugin.submitFromActive()` (direct method call) | Notice via existing error-mapping in handler |

**Forbidden dispatch:** `executeCommandById('leetcode:run' / ':submit')`. Pattern locked out by CONTEXT D-05 + Phase 5 Plan 05 live-smoke finding (Pitfall 6 from RESEARCH).

### Idempotency

- One widget per `## Code` fence per editor view. Never duplicate.
- `CodeActionsWidget.eq(other)` returns `true` for widgets with the same plugin reference â†’ CM6 reuses the rendered DOM across transactions with no flicker (RESEARCH Pitfall 2).
- Per-transaction rebuild gated: only on `tr.docChanged` OR `editorLivePreviewField` flip (RESEARCH Pattern 1).

### Widget Geometry (contract)

- Widget type: inline (`side: 1`, **NOT** `block: true`).
- Anchor position: `state.doc.line(fence.closerLine).to` â€” end-of-line immediately after the closing fence's backticks.
- DOM shape: `<div class="leetcode-code-actions"><button class="leetcode-code-action-run">Run</button><button class="leetcode-code-action-submit">Submit</button></div>`.
- Layout direction: flex row, `justify-content: flex-end` (right-aligned), `gap: 8px`, `margin-top: 4px`, `margin-bottom: 8px`.
- Button dimensions: `height: 24px`, `padding: 0 8px`, `border-radius: 4px`, `font-size: 12px`, `border: 1px solid var(--background-modifier-border)`.
- **No block-level vertical space reserved.** Widget flows inline in the line box of the row after the closing fence. Does **not** stack a new row onto the editor's block layout. This is the specific fix for the 05-UAT G1 failure.

## Out of Scope (documented to prevent drift)

The following are **explicitly NOT** part of Phase 5.1's design contract, even though downstream agents may be tempted to add them:

- **Floating / corner / status-bar toolbar.** User-locked out (D-01).
- **Block-level widget (`block: true`).** Locked out by 05-UAT G1 (D-03).
- **Run/Submit buttons under fences other than `## Code`** (e.g., example fences in `## Problem`, legacy `## Custom Tests` fences). Locked by D-02 / reading-mode parity.
- **New Notice copy.** All error surfaces reuse existing handler Notices (CF-19).
- **Spinner / loading indicator on buttons.** Matches Reading-Mode by deliberate non-decision (see "Loading / Pending State").
- **Disabled state on buttons during pending response.** Same â€” single-flight lives in handler layer.
- **Custom focus ring / outline overrides.** Native browser `:focus-visible` preserved.
- **Additional action buttons** (View Submissions, Reset, Pin-to-note, etc.). Out of scope â€” pin-to-note rejected in Phase 5 Wave 3; View-Past-Submissions already has a command; Reset belongs to Run modal.
- **Icons in button labels.** Two text-label buttons only. No icon library pulled in.
- **Any vault writes from this surface.** CF-06 locked â€” Phase 5.1 is read-only against the note.
- **Hotkey defaults** on any command. FND-03 / CF-05.
