  <how-to-verify>
    **Setup (one-time):**
    1. Open Obsidian with a **dedicated test vault** (NOT your primary vault). Create one at e.g. `~/obsidian-leetcode-test-vault/` if you don't have one.
    2. Install the Phase 4 build of the plugin into that vault's `.obsidian/plugins/obsidian-leetcode/` (copy main.js, manifest.json, styles.css). Enable the plugin.
    3. Log in to LeetCode via the plugin (ribbon icon → Open LeetCode browser → Log in via embedded window OR paste a valid LEETCODE_SESSION cookie via settings tab). Confirm authentication succeeds.
    4. Settings → Notes → Problems folder: `LeetCode` (default). Default language: `python3` (default).
    5. Pick a problem you have NOT yet solved and have NOT yet opened in the plugin. Easy tier recommended (e.g., Two Sum if you haven't solved it; else any unsolved Easy).

    **Section A — Fresh AC on a brand-new problem note (GRAPH-01..04):**

    1. Open the problem browser (ribbon). Search for your chosen problem. Click the row.
    2. ✅ Verify: a new note is created at `LeetCode/{id}-{slug}.md` with `## Problem`, `## Code` (empty or starter), `## Notes` headings.
    3. Type a known-correct solution inside the fenced block under `## Code`.
    4. Command palette → `LeetCode: Submit`. Verdict modal opens in pending state, transitions to "Accepted" with runtime + memory + percentile.
    5. Close the verdict modal.
    6. ✅ Verify (frontmatter — properties pane or source mode): note frontmatter now contains:
       - `lc-status: accepted`
       - `lc-solved-date: YYYY-MM-DDTHH:MM:SS±HH:MM` (your local tz)
       - `lc-runtime-ms: <number>` (same as LC's reported ms value)
       - `lc-memory-mb: <number>` (same as LC's reported MB value)
       - `lc-language: python3` (or whatever langSlug you submitted)
       - `tags:` includes `lc/easy` + one `lc/{topic-slug}` per LC topic tag on the problem
    7. ✅ Verify (body): note now has a `## Techniques` section BETWEEN `## Notes` and (if present) `## Custom Tests`, containing `- [[Topic Name]]` bullets — one per LC topic tag, in LC's natural order (not alphabetical).
    8. ✅ Verify (stub files): `LeetCode/Techniques/` folder exists. One `.md` file per topic tag, filename = topic tag name. Each file has frontmatter only: `lc-technique: <slug>`, `aliases: [<name>]`, `tags: [lc/technique/<slug>]`, empty body.
    9. ✅ Verify (GRAPH-01 revised — D-01): the note has NO `## Solution` heading anywhere. Also run `grep -rE "^## Solution" .` inside the vault — should return 0 matches.
    10. ✅ Verify (Graph View): open Obsidian's Graph View. The problem note shows edges to each technique stub note.
    11. ✅ Verify (user tag preservation): edit the note, add a line `- [[My Personal Technique]]` inside `## Techniques`, then re-run submit (same solution, will be AC again). After AC, the `[[My Personal Technique]]` line is STILL PRESENT (union-merge preserved it per D-13).

    **Section B — View past submissions picker (GRAPH-01 revised):**

    12. With the problem note still active, command palette → `LeetCode: View past submissions`.
    13. ✅ Verify: picker modal opens with title `Past submissions`. Shows at least 1 row (the AC you just did). Each row has:
        - Verdict chip (`AC` for the one you just submitted; colored green via reused `.leetcode-verdict-ac`)
        - Date formatted `YYYY-MM-DD HH:MM`
        - Runtime · Memory (e.g., `12 ms · 14.2 MB`)
        - Language chip on the right (e.g., `python3`)
    14. Click your AC row.
    15. ✅ Verify: detail modal opens with title `Accepted · {Problem Title}` (middle-dot `·` U+00B7, single space each side — per CONTEXT D-04). Metadata row shows `Runtime: X ms · Memory: Y MB · python3 · <ISO-8601 timestamp>`. Code block renders with **syntax highlighting** (via MarkdownRenderer).
    16. Close the detail modal (Escape or Close button). Verify: returns to the picker (picker stays open behind).
    17. Close the picker.

    **Section C — Copy-to-Code flow (GRAPH-01 revised):**

    18. In the note, modify the `## Code` fenced block — add a few lines of garbage so the block is non-empty but different from your AC.
    19. Open the picker again (`LeetCode: View past submissions`). Click your AC row.
    20. In the detail modal, click `Copy to ## Code`.
    21. ✅ Verify: `ConfirmOverwriteModal` appears with title `Overwrite current code?`, body `Your current ## Code block will be replaced with this submission.`, note `This can't be undone from the modal, but Obsidian's undo (Cmd/Ctrl+Z) works after closing.`, buttons `[Yes, overwrite]` (primary accent) and `[Cancel]` (neutral, default-focused).
    22. ✅ Verify (accessibility): press `Enter` without clicking — because Cancel has default focus, the modal dismisses; NO overwrite happens. Your garbage is preserved.
    23. Repeat steps 19-21. This time click `Yes, overwrite` explicitly.
    24. ✅ Verify: detail + confirm modals both close. Picker stays open. The note's `## Code` fenced block now contains the AC'd code, tagged with the submission's language (`python3`).
    25. Close the picker.

    **Section D — Opt-out flow (GRAPH-05):**

    26. Close Obsidian (preserves data.json state). Locate the plugin's data.json at `{testvault}/.obsidian/plugins/obsidian-leetcode/data.json`. Edit `autoBacklinksEnabled` from `true` to `false`. Save.
    27. Reopen Obsidian.
    28. Pick a SECOND unsolved problem. Open it. Submit a known-correct solution → AC.
    29. ✅ Verify (frontmatter still writes): the second note has `lc-status: accepted`, `lc-solved-date`, `lc-runtime-ms`, `lc-memory-mb`, `lc-language` populated.
    30. ✅ Verify (topic tags still write): the second note's `tags:` array includes `lc/{topic-slug}` for every LC topic tag on that problem (D-20 — opt-out keeps frontmatter tags).
    31. ✅ Verify (opt-out skips): the second note has NO `## Techniques` heading. Check `LeetCode/Techniques/` folder — NO new stub files created for the second problem's topic tags (assuming those tags don't overlap with problem 1's — if they DO overlap, the existing stubs from problem 1 are still there; that's fine and correct).
    32. Revert `autoBacklinksEnabled` to `true` in data.json; restart Obsidian for later use.

    **Section E — Session-expiry handling (D-06):**

    33. In settings, log out (or manually blank out `LEETCODE_SESSION` in data.json).
    34. Open a problem note that has `lc-slug` frontmatter. Command palette → `LeetCode: View past submissions`.
    35. ✅ Verify: picker briefly opens, then closes. Notice fires with EXACT copy `LeetCode session expired. Log in again.` (8s duration).
    36. Log back in.

    **Section F — Non-AC verdicts skip pipeline (D-23):**

    37. On a solved problem (where the note already has `## Techniques` written), open the note and temporarily BREAK your code (e.g., change a return value so it produces WA).
    38. Command palette → `LeetCode: Submit`. Wait for WA verdict.
    39. ✅ Verify (negative): the `## Techniques` section is UNCHANGED. Frontmatter `lc-status` remains `accepted` (not downgraded). No new stub files created.
    40. Fix the code back to the working version. (Resubmitting is not required to pass this section.)

    **Section G — Visual and accessibility spot checks:**

    41. ✅ Verify (chip colors — dark mode): open picker on a problem with mixed-verdict history. AC chips are green-tinted, WA/TLE/MLE/RE are red-tinted, CE is orange-tinted. Switch Obsidian to dark mode — chip contrast remains legible (Obsidian CSS variables adapt).
    42. ✅ Verify (keyboard nav): in the picker, Tab cycles through rows; Enter on a focused row opens the detail modal.
    43. ✅ Verify (reduced motion): if you can enable system-level reduced-motion (macOS: System Settings → Accessibility → Display → Reduce motion), the picker's loading spinner no longer animates. (If unable to test, SKIP this sub-check.)
    44. ✅ Verify (focus rings): Obsidian's default focus ring is visible on buttons when Tab-focused. No `outline: none` should be overriding.

    **Section H — Full regression (Phase 1-3 paths still work):**

    45. ✅ Verify: `LeetCode: Submit` still works on a non-AC submission (WA produces the verdict modal with Input/Output/Expected diff).
    46. ✅ Verify: `LeetCode: Run code (sample)` still works — opens verdict modal with sample-test results.
    47. ✅ Verify: `LeetCode: Run code (custom input)` still opens the custom-test tabbed modal.
    48. ✅ Verify: `LeetCode: Insert starter code` still rewrites the `## Code` fenced block.
    49. ✅ Verify: plugin can be disabled + re-enabled from Obsidian's community plugin settings without crash.
