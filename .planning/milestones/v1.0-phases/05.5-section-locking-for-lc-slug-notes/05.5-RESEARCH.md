# Phase 05.5: Section Locking for lc-slug Notes - Research

**Researched:** 2026-05-13
**Domain:** CodeMirror 6 read-only-range enforcement inside an Obsidian Edit-Mode plugin
**Confidence:** HIGH

## Summary

CONTEXT D-04 names `EditorState.transactionFilter` as the enforcement primitive, but verification against the locally-installed `@codemirror/state@6.6.0` `index.d.ts` and the canonical CodeMirror reference manual surfaced a **higher-precision API for this exact use case**: `EditorState.changeFilter`. `changeFilter` is documented as the granular position-based filter that "suppresses part of the transaction's changes" by returning an array of `[from, to, ...]` ranges. `transactionFilter` is the all-or-nothing transaction-level hook — usable but coarser. Both are stable production CM6 APIs (CM6.0 in 2022; current series 6.6.0 published 2026-04-13).

The recommended stack is therefore: `changeFilter` for the hard read-only enforcement (drop changes touching locked ranges, preserve selection moves and other annotations automatically), `EditorView.atomicRanges` for cursor-skip behavior on arrow-key navigation, and an optional `Decoration.mark` decoration for visual dimming (CONTEXT D-04 planner discretion). Locked-range computation is line-scan over `state.doc` per-call (cheap for problem notes <2000 lines) — no `StateField` cache needed at v1, but the StateField pattern is documented as the upgrade path. Heading literals come from `NoteTemplate.ts` exports (Phase 2 D-03 SSoT invariant). The `lc-slug` frontmatter gate reuses Phase 5.1's `editorInfoField` + `metadataCache.getFileCache` pattern verbatim.

Test infrastructure extends `tests/helpers/obsidian-stub.ts` with one new factory — a minimal `EditorState`-shape adapter that exposes `state.doc` (line scan), `state.facet(changeFilter)` (filter invocation), and `state.field(editorInfoField)` (file gate). Phase 5.1 + 5.3 test patterns establish the precedent. No new dependencies; both `@codemirror/state` and `@codemirror/view` are already transitive peers of `obsidian@1.12.3`, marked external in `esbuild.config.mjs`.

**Primary recommendation:** Use `EditorState.changeFilter` to drop changes that overlap locked ranges, paired with `EditorView.atomicRanges` for cursor-skip. Use `Decoration.mark` for optional visual dimming. Compute locked ranges via a pure `computeLockedRanges(state)` helper that line-scans `state.doc`. Gate on `lc-slug` frontmatter via the Phase 5.1 `editorInfoField` + `metadataCache` precedent. Single new file: `src/main/sectionLockExtension.ts`. One new test file: `tests/main/sectionLockExtension.test.ts`. Zero new dependencies.

### Pattern 1: Pure locked-range computation

**What:** A pure function `computeLockedRanges(state: EditorState): readonly number[]` that line-scans `state.doc` and emits flat `[from, to, from, to, ...]` pairs ready for `changeFilter` return.

**When to use:** Called from inside the `changeFilter` callback — every transaction. O(lines) per call; fine for problem notes <2000 lines. If profiling shows a problem, wrap in a `StateField<readonly number[]>` that recomputes only on `tr.docChanged` AND only when the changed lines intersect a heading or fence.

**Example:**

```typescript
// src/main/sectionLockExtension.ts (excerpt)
// Source: hand-rolled per CONTEXT D-03 + D-08 + D-09; cites Phase 5.1 findCodeFence.
// [VERIFIED: src/main/codeActionsEditorExtension.ts:168-203 for findCodeFence]
import type { EditorState } from '@codemirror/state';
import {
  PROBLEM_HEADING_LINE,
  CODE_HEADING_LINE,
  TECHNIQUES_HEADING_LINE,
  NOTES_HEADING_LINE,
} from '../notes/NoteTemplate';
import { findCodeFence } from './codeActionsEditorExtension';

/**
 * Compute the flat [from, to, from, to, ...] suppression ranges per CONTEXT D-03.
 * - ## Problem heading + ENTIRE BODY locked.
 * - ## Code heading + fence opener line + closing fence line locked. Body editable.
 * - ## Techniques heading locked. Body editable.
 * - ## Notes heading locked. Body editable.
 * - ## Custom Tests intentionally NOT locked (legacy section).
 *
 * D-08 nuance: heading-line lock includes the trailing newline so the user
 * cannot delete the heading by selecting the line + Backspace. We achieve this
 * by emitting `[lineFrom, nextLineFrom]` rather than `[lineFrom, lineEnd]` —
 * the trailing \n is between line.to and the next line's from.
 *
 * D-09 nuance: when ## Code's fence is malformed (no closer), only the heading
 * line is locked; body falls through (no fence opener/closer locked).
 */
export function computeLockedRanges(state: EditorState): readonly number[] {
  const out: number[] = [];
  if (state.doc.lines === 0) return out;

  // 1. Find each canonical heading line (literal-match per D-08).
  type Heading = 'problem' | 'code' | 'techniques' | 'notes';
  const headings: Array<{ kind: Heading; line: number }> = [];
  const total = state.doc.lines;

  for (let i = 1; i <= total; i++) {
    const text = state.doc.line(i).text;
    if (text === PROBLEM_HEADING_LINE) headings.push({ kind: 'problem', line: i });
    else if (text === CODE_HEADING_LINE) headings.push({ kind: 'code', line: i });
    else if (text === TECHNIQUES_HEADING_LINE) headings.push({ kind: 'techniques', line: i });
    else if (text === NOTES_HEADING_LINE) headings.push({ kind: 'notes', line: i });
  }

  // 2. For each found heading, compute its locked range.
  for (let h = 0; h < headings.length; h++) {
    const cur = headings[h];
    const headLine = state.doc.line(cur.line);
    // Inclusive heading-line lock: from start-of-line to start-of-NEXT-line.
    // This includes the trailing \n so backspace-on-line-start can't delete.
    const headFrom = headLine.from;
    const headTo = cur.line < total ? state.doc.line(cur.line + 1).from : headLine.to;

    if (cur.kind === 'problem') {
      // ## Problem locks the heading PLUS the entire body until the next H2
      // (or end-of-doc). Find the next heading line.
      const nextHeadingLine =
        h + 1 < headings.length
          ? headings[h + 1].line
          : total + 1; // sentinel = end-of-doc
      const bodyFrom = headFrom; // include heading
      const bodyTo =
        nextHeadingLine <= total
          ? state.doc.line(nextHeadingLine).from // up to (not including) next heading
          : state.doc.line(total).to; // end of doc
      out.push(bodyFrom, bodyTo);
    } else if (cur.kind === 'code') {
      // Heading line locked.
      out.push(headFrom, headTo);
      // Fence opener + closer (D-09). findCodeFence is the canonical detector.
      const fence = findCodeFence(state);
      if (fence) {
        const opener = state.doc.line(fence.openerLine);
        const closer = state.doc.line(fence.closerLine);
        const openerTo = fence.openerLine < total
          ? state.doc.line(fence.openerLine + 1).from
          : opener.to;
        out.push(opener.from, openerTo);
        const closerTo = fence.closerLine < total
          ? state.doc.line(fence.closerLine + 1).from
          : closer.to;
        out.push(closer.from, closerTo);
      }
      // Body BETWEEN opener and closer falls through (editable). Per D-09,
      // malformed-fence path also falls through.
    } else {
      // techniques | notes — heading line only.
      out.push(headFrom, headTo);
    }
  }

  return out;
}
```

### Pattern 2: changeFilter return shape

**What:** Register a function with `EditorState.changeFilter` that returns `true` (no-op) when the gate fails OR returns a `readonly number[]` of suppression ranges when the gate passes.

**When to use:** Always. This is the enforcement primitive.

**Example:**

```typescript
// src/main/sectionLockExtension.ts (excerpt)
// Source: [CITED: node_modules/@codemirror/state/dist/index.d.ts:1299-1312] +
// CONTEXT D-04, D-06, D-07 gating.
import { EditorState, type Extension } from '@codemirror/state';
import { editorInfoField, editorLivePreviewField, type Plugin } from 'obsidian';
import { EditorView } from '@codemirror/view';
import { computeLockedRanges } from './sectionLockExtension'; // self-import shown for clarity

export function buildSectionLockExtension(plugin: Plugin): Extension {
  return [
    EditorState.changeFilter.of((tr) => {
      // Gate 1: editorInfoField → file present?
      const file = tr.startState.field(editorInfoField)?.file;
      if (!file) return true;

      // Gate 2: D-07 — Edit Mode only (Live Preview + Source).
      // editorLivePreviewField is true in Live Preview, false in Source.
      // BOTH should lock; only Reading Mode is exempt — and Reading Mode
      // doesn't run CM6 transaction filters anyway, so this gate is mostly
      // a safety check. (Reading Mode renders via MarkdownPostProcessor.)
      // Practically: if editorInfoField has a file, we're in Edit Mode.

      // Gate 3: D-06 — lc-slug frontmatter.
      const fm = plugin.app.metadataCache.getFileCache(file)?.frontmatter as
        | Record<string, unknown>
        | undefined;
      const slug = fm?.['lc-slug'];
      if (typeof slug !== 'string' || slug.length === 0) return true;

      // Gate passes — compute and return locked ranges.
      // Returning [] (empty array) is documented as "no ranges suppressed"
      // (equivalent to true). Returning [from, to, from, to] suppresses changes
      // whose [fromA, toA] overlaps any [from, to] pair.
      const ranges = computeLockedRanges(tr.startState);
      if (ranges.length === 0) return true; // no headings yet; nothing to lock
      return ranges;
    }),
    // Pattern 3 — atomicRanges for cursor-skip on arrow keys.
    // (See Pattern 3 below for the RangeSet construction.)
    EditorView.atomicRanges.of((view) => buildAtomicRangeSet(view.state)),
    // OPTIONAL Pattern 4 — Decoration.mark for visual dim (planner discretion).
    // EditorView.decorations.of((view) => buildLockedDecorations(view.state)),
  ];
}
```

**Return-value semantics quoted from the canonical type definition:**

> "Such a function can return `true` to indicate that it doesn't want to do anything, `false` to completely stop the changes in the transaction, or a set of ranges in which changes should be suppressed. Such ranges are represented as an array of numbers, with each pair of two numbers indicating the start and end of a range. So for example `[10, 20, 100, 110]` suppresses changes between 10 and 20, and between 100 and 110."
> `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1299-1310 — JSDoc on EditorState.changeFilter]`

**Critical correctness note:** `tr.startState` (the state BEFORE the transaction) is the correct frame for computing locked ranges. The `changeFilter` runs against `startState`, NOT against `tr.state` (which would force creation of a state that's about to be filtered — explicitly discouraged by the JSDoc on `transactionFilter` line 1323-1326). `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1323-1326]`

### Pattern 3: atomicRanges for cursor-skip

**What:** Register `EditorView.atomicRanges.of((view) => RangeSet)` with the same locked ranges packaged as decoration ranges. CM6 calls this whenever cursor motion would land in such a region; the cursor skips to the boundary.

**When to use:** Always (paired with Pattern 2). Without atomicRanges, the user can move the cursor INTO a locked range with arrow keys; their typing then drops silently — works but feels broken. With atomicRanges, arrow-key motion glides past the locked region.

**Reference quote:**

> "Used to provide ranges that should be treated as atoms as far as cursor motion is concerned. This causes methods like `moveByChar` and `moveVertically` (and the commands built on top of them) to skip across such regions when a selection endpoint would enter them. This does _not_ prevent direct programmatic selection updates from moving into such regions."
> `[CITED: node_modules/@codemirror/view/dist/index.d.ts:1273-1284 — JSDoc on EditorView.atomicRanges]`

**Example:**

```typescript
// src/main/sectionLockExtension.ts (excerpt)
import { RangeSetBuilder, type EditorState, type RangeSet } from '@codemirror/state';
import { Decoration } from '@codemirror/view';

/**
 * Build a RangeSet over the same locked ranges so EditorView.atomicRanges can
 * skip cursor over them on arrow-key motion. The values inside the RangeSet are
 * required to be RangeValue subclasses; using a marker Decoration is the
 * idiomatic CM6 pattern (the decoration's actual styling is irrelevant here —
 * atomicRanges only consults the [from, to] bounds).
 */
function buildAtomicRangeSet(state: EditorState): RangeSet<Decoration> {
  const ranges = computeLockedRanges(state);
  const b = new RangeSetBuilder<Decoration>();
  for (let i = 0; i < ranges.length; i += 2) {
    const from = ranges[i];
    const to = ranges[i + 1];
    if (from < to) {
      b.add(from, to, Decoration.mark({ class: 'leetcode-section-locked-atomic' }));
    }
  }
  return b.finish();
}
```

**Click behavior caveat:** `atomicRanges` skips KEYBOARD cursor motion. Mouse clicks still position the cursor wherever the user clicks (a feature, per CONTEXT D-04 — "moving the cursor INTO a locked range is allowed (so user can click anywhere); typing while inside a locked range silently drops"). This matches the spec exactly. `[VERIFIED: node_modules/@codemirror/view/dist/index.d.ts:1280-1283]`

### Pattern 4 (OPTIONAL): Visual dim via Decoration.mark

**What:** Register `EditorView.decorations.of((view) => DecorationSet)` with `Decoration.mark` over each locked range. Add a single CSS rule that dims the marked range's background to `var(--background-secondary)`.

**When to use:** Planner discretion (CONTEXT D-04). Adding this is optional but improves discoverability — without it, the user only learns the lock exists by typing and seeing nothing happen.

**Decoration choice rationale (open question Q8 from upstream):**

| Option | Behavior | Pros | Cons |
|--------|----------|------|------|
| `Decoration.mark` | Adds a CSS class to existing text range; original text remains visible | Heading text stays readable; dim is purely visual; cursor can still click in (matches D-04) | None for this case |
| `Decoration.replace` | Replaces the range with a placeholder DOM widget; original text hidden | None for this case — would HIDE the heading text | User can't see what section they're in; breaks the "structural anchors are heading literals" model |

**Recommendation:** `Decoration.mark` if visual dim is added. `Decoration.replace` is the wrong tool. `[VERIFIED: node_modules/@codemirror/view/dist/index.d.ts:1247-1262 — decorations facet contract]`

**Example CSS rule (single rule; no hardcoded colors):**

### Pattern 5: Heading-detection nuance — D-08 trailing-newline lock

**What:** D-08 requires the heading-line lock to include the trailing `\n`. The naive lock `[line.from, line.to]` would let the user select the entire line + Backspace and the deletion would land at line.to → next-line.from (which is OUTSIDE the locked range). Solution: lock through the start of the NEXT line.

**Example calculation:**

```
Doc:                  L1: "## Problem\n"      from=0, to=10
                      L2: "body content"      from=11, to=23

Naive lock:           [0, 10]      → user can delete the \n
Correct (D-08) lock:  [0, 11]      → \n included; deletion blocked
```

The `computeLockedRanges` example above implements this correctly via `state.doc.line(cur.line + 1).from`.

### Pattern 6: ## Problem-body extent calculation

**What:** D-03 says ENTIRE `## Problem` body is locked. Compute it as `[problemHeading.from, nextHeading.from)` — which means everything from the heading line through the line BEFORE the next H2 is locked (regardless of whether that next H2 is `## Code`, `## Techniques`, `## Notes`, or `## Custom Tests`).

**Edge case:** if `## Problem` is the LAST heading in the doc (which shouldn't happen for a properly-templated note but is possible mid-edit), lock through end-of-doc. Implementation in `computeLockedRanges` above handles this via the `total + 1` sentinel.

### Anti-Patterns to Avoid

- **Returning `false` from `changeFilter` to drop changes.** This stops ALL changes in the transaction, including changes elsewhere that don't touch a locked range. Correct: return the suppression-range array; the change at locked positions drops, other changes pass through. `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1304-1310]`

- **Using `transactionFilter` and returning `[]` (empty array).** The empty-array case is documented as "combine zero specs" — effectively cancels the transaction entirely (no changes, no selection moves, no annotations). Loses selection-tracking behavior. Use `changeFilter` instead. `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1314-1330 — transactionFilter JSDoc]`

- **Computing locked ranges from `tr.state` instead of `tr.startState`.** The JSDoc on `transactionFilter` (and applies analogously to `changeFilter`) explicitly warns: "When possible, it is recommended to avoid accessing `Transaction.state` in a filter, since it will force creation of a state that will then be discarded again." Use `tr.startState`. `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1323-1326]`

- **Adding a Decoration with `block: true` for visual dim.** Phase 5.1 RESEARCH Pitfall 1 forbids this for content-line-anchored widgets. `Decoration.mark` is the correct primitive (range mark, not block widget) — does not have this problem. (Phase 5.1's specific block-widget bug was about line-anchored block widgets shifting layout; `Decoration.mark` is a range-mark in flow with text and does not perturb layout.)

- **Hardcoding heading literals.** Phase 2 D-03 SSoT invariant. Always import from `NoteTemplate.ts`. ESLint won't catch this; verify by grep.

- **Locking `## Custom Tests`.** D-03 explicitly excludes. Phase 5 D-08 ignores it on read/write. The plugin doesn't touch it; the user owns it. Adding it to `LOCKED_HEADINGS` is a bug.

- **Special-casing the `app.workspace.getActiveViewOfType(MarkdownView)` lookup inside the filter.** The filter has access to `state.field(editorInfoField)?.file` directly — same path Phase 5.1 uses. No need to round-trip through workspace.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Read-only ranges in CM6 | Custom `keydown` listener that intercepts `Enter` / `Backspace` / printable keys | `EditorState.changeFilter` | DOM-level keyboard interception misses paste, drop, IME composition, undo/redo, programmatic dispatches (chevron switch, copy-to-code). `changeFilter` runs on EVERY transaction. |
| Cursor skip over read-only ranges | Custom `keymap` extension for arrow keys | `EditorView.atomicRanges` | CM6's built-in atomicRanges skipping handles `moveByChar`, `moveVertically`, AND every command built on them — far more cases than a hand-rolled keymap. |
| Visual dim of read-only regions | DOM-side `MutationObserver` that injects spans | `Decoration.mark` | CM6 decorations are the supported primitive; survive viewport changes, selection updates, and re-renders. |
| Frontmatter gate | Re-parse YAML in the filter | `metadataCache.getFileCache(file)?.frontmatter` | Obsidian already parses + caches frontmatter. Phase 5.1 / 5.3 already use this — single source of truth. |
| Heading detection | Hardcode `'## Problem'` etc. inside the extension | Import from `NoteTemplate.ts` | Phase 2 D-03 SSoT invariant. Schema drift bug class. |
| Fence detection | Re-implement regex line-scan inside the lock extension | Import `findCodeFence` from `codeActionsEditorExtension.ts` | Already shipped + tested in Phase 5.1. Single source of truth (Phase 5.3 D-13). |
| Locked-range invalidation | Custom `MutationObserver` on the editor DOM | The pure-recompute `changeFilter` callback OR a `StateField<readonly number[]>` invalidated by `tr.docChanged` | CM6's facet system + StateField update predicate already handle this elegantly. |

**Key insight:** CM6 has a documented, granular API (`changeFilter` + `atomicRanges`) that exactly matches this use case. The hand-rolled alternatives (DOM keyboard interception, MutationObserver-based dim, custom keymap) are all measurably worse — they miss cases, fight CM6's own state management, or break under undo/redo. **The single most important research finding for this phase is: do not write custom keystroke logic; use the canonical facets.**

## Common Pitfalls

### Pitfall 1: Returning `false` from `changeFilter` for "no changes allowed inside locked ranges"

**What goes wrong:** The user types ANYWHERE in the document (locked or not), and ALL changes in the transaction are dropped. The whole editor effectively becomes read-only.

**Why it happens:** `false` means "completely stop the changes in the transaction" per the JSDoc — not "stop the changes inside locked ranges." The `[from, to, ...]` array form is the granular case.

**How to avoid:** Always return the suppression-range array. Return `true` (not `false`) for the "no locking applies" path (gate failure, no `## Code`, etc.).

**Warning signs:** User reports "the editor is completely read-only"; tests for editing inside `## Code` body fail. `[CITED: node_modules/@codemirror/state/dist/index.d.ts:1304-1306]`

### Pitfall 2: Computing ranges from `tr.state` (the post-transaction state)

**What goes wrong:** Subtle correctness bugs around boundary cases (e.g., user typing AT the end of the `## Code` heading line creates a transient state where the heading is no longer literally `## Code`), or worse — performance regression from the JSDoc-warned "force creation of a state that will then be discarded."

**Why it happens:** CM6 makes both `tr.startState` and `tr.state` available; the correct one for filters is `startState` (BEFORE the transaction is applied — that's the frame the filter is meant to evaluate against).

**How to avoid:** Always `tr.startState.doc.line(...)`. Never `tr.state.doc.line(...)` inside `changeFilter`.

**Warning signs:** Performance profiling shows a hot path through transaction creation; subtle off-by-one bugs at heading boundaries.

`[CITED: node_modules/@codemirror/state/dist/index.d.ts:1323-1326]`

### Pitfall 3: D-08 trailing-newline omission

**What goes wrong:** User selects the entire `## Problem` line + Backspace; the deletion succeeds because `[line.from, line.to]` does not cover the `\n` at position `line.to`. After deletion the heading is gone, the body promotes to the next preceding heading, and the `## Problem` write path's literal-string detection fails on the next refresh.

**Why it happens:** CM6 line offsets follow the convention where `line.to` is BEFORE the trailing newline; the newline is between this line's `to` and the next line's `from`.

**How to avoid:** Lock from `line.from` to `nextLine.from` (or to `line.to` if it's the last line of the doc). The `computeLockedRanges` example handles this.

**Warning signs:** Test "selecting heading line + Backspace deletes the heading" passes when it should fail.

### Pitfall 4: Treating `## Custom Tests` as a plugin section

**What goes wrong:** User reports inability to edit `## Custom Tests` content; complains that the lock breaks an old workflow.

**Why it happens:** Phase 5 D-08 made `## Custom Tests` a legacy section that the plugin no longer reads or writes. CONTEXT D-03 explicitly excludes it from the lock. Any code path that includes `CUSTOM_TESTS_HEADING_LINE` in the locked-headings list is buggy.

**How to avoid:** `LOCKED_HEADINGS = [PROBLEM_HEADING_LINE, CODE_HEADING_LINE, TECHNIQUES_HEADING_LINE, NOTES_HEADING_LINE]` — exactly four entries; `CUSTOM_TESTS_HEADING_LINE` is NOT in this array.

**Warning signs:** Test for "user can freely edit `## Custom Tests` body" fails.

### Pitfall 5: Programmatic dispatches (chevron switch, copy-to-code) are filtered

**What goes wrong:** User clicks the chevron to switch language; nothing happens because the chevron's `cm.dispatch({ changes: [openerChange, bodyChange], userEvent: 'leetcode.lang-switch' })` (`src/main.ts:799-805`) hits ranges inside the locked `## Code` heading + fence opener, and `changeFilter` drops the changes.

**Why it happens:** `changeFilter` runs on EVERY transaction regardless of source. Programmatic dispatches from plugin code go through the same filter as user keystrokes. This is the SAME root cause as one of the open questions raised in upstream Q3.

**How to avoid:** Use `userEvent` annotations as an escape hatch. CM6's `Transaction.userEvent` is the canonical way to mark provenance. The filter checks for plugin-owned events:

```typescript
EditorState.changeFilter.of((tr) => {
  // Plugin-owned dispatches bypass the lock. The chevron switch, copy-to-code,
  // background refresh, etc. set userEvent='leetcode.*' so they thread through.
  // Reading-mode writes use vault.process and don't hit the editor at all.
  const userEvent = tr.annotation(Transaction.userEvent);
  if (typeof userEvent === 'string' && userEvent.startsWith('leetcode.')) {
    return true; // pass plugin dispatches through unchanged
  }
  // … existing gate + range-compute logic …
});
```

**Critical Phase 5.3 dispatch already uses this pattern:** `src/main.ts:804` sets `userEvent: 'leetcode.lang-switch'`. The lock extension MUST honor this annotation, OR the planner MUST audit every plugin-side `cm.dispatch` and ensure each carries a `leetcode.*` userEvent.

**Audit list (current `cm.dispatch` callsites):**
- `src/main.ts:799` (`switchFenceLanguage`) — already sets `userEvent: 'leetcode.lang-switch'` ✓
- `src/main/codeActionsEditorExtension.ts:320` (`languageRefreshEffect` dispatch) — only dispatches an effect, NO `changes`; not gated by `changeFilter` ✓
- Any future programmatic edits MUST set a `leetcode.*` userEvent.

This is a **load-bearing planning requirement** — without it, the chevron switch and copy-to-code break.

**Warning signs:** Live-smoke shows chevron click does nothing; or test "chevron switch dispatches changes through filter" fails.

### Pitfall 6: Vault writes (background refresh / on-AC / copy-to-code) appearing to bypass the lock

**What goes wrong:** None — but a planner who hasn't internalized D-02 might worry. Phase 05.5's D-02 explicitly says these writes bypass the lock (because they're `vault.process()` writes, not editor transactions). The lock is editor-side only.

**Why it happens (correctly):** `vault.process` writes happen at the vault layer, BELOW the CM6 editor. The CM6 buffer reloads from disk after `vault.process` completes; the reload is a programmatic state replacement, not a `changeFilter`-eligible transaction.

**How to avoid:** N/A — this is the desired behavior. Document for the planner that this asymmetry is intentional. CONTEXT D-02 covers it.

**Warning signs:** Planner adds a "lock prevents background-refresh" requirement; PR reviewer flags it as scope creep (it would be — D-02 / D-04 explicit).

### Pitfall 7: Multiple `## Problem` headings in the same doc

**What goes wrong:** User pastes content that includes a literal `## Problem` line in the middle of, say, `## Notes`. The `computeLockedRanges` line-scan finds two `## Problem` matches and locks both regions.

**Why it happens:** Heading detection is literal-line-equality (D-08 / D-01 SSoT discipline). It can't distinguish "the canonical `## Problem` heading" from "a `## Problem` line inside `## Notes` body."

**How to avoid:** Two options:
1. **Accept the behavior.** This is the conservative path: the user pasting `## Problem` into `## Notes` is rare; if they do, the second `## Problem` and its body get locked too. Recoverable by deleting the offending line. CONTEXT D-09 has a similar "if detection fails, only heading is locked" failure-mode disclaimer. (Recommendation: ACCEPT.)
2. **Lock only the FIRST occurrence of each canonical heading.** Trivial in `computeLockedRanges` — track a `seen: Set<Heading>`. Costs three lines of code.

**Recommendation:** Pick (1) for v1 simplicity; document the edge case in `## Notes` of the plugin README so users understand it. Switch to (2) if dogfood reveals friction.

**Warning signs:** Test "second `## Problem` line in body of `## Notes` locks unintended range" fails — but this test isn't in the canonical coverage; it's an edge-case test the planner adds at discretion.

### Pitfall 8: Source Mode vs Live Preview asymmetry

**What goes wrong:** Lock works in Live Preview but not Source (or vice versa).

**Why it happens:** `editorLivePreviewField` is `true` in Live Preview, `false` in Source. CM6 transaction filters fire in BOTH modes — they're state-layer, not view-layer. So if the gate logic is `if (state.field(editorLivePreviewField)) { ... }`, Source Mode is incorrectly bypassed.

**How to avoid:** Per CONTEXT D-07, BOTH modes lock; only Reading Mode is exempt (and Reading Mode doesn't run CM6 anyway). The gate should NOT consult `editorLivePreviewField` — only `editorInfoField` (file present? = Edit Mode) + `metadataCache` (lc-slug?).

**Warning signs:** Source Mode keystroke-drop test passes but Live Preview fails (or vice versa).

### Pitfall 9: First-paint cold-cache (metadataCache not yet populated on file open)

**What goes wrong:** User opens an `lc-slug` note; for the first ~50–100 ms, `metadataCache.getFileCache(file)?.frontmatter` returns `undefined` because the cache hasn't populated yet. The gate `if (typeof slug !== 'string') return true` correctly fires `true` (no lock), but during that window the user can theoretically type into a "locked" region.

**Why it happens:** Same as Phase 5.1 RESEARCH Pitfall 4 — metadataCache is asynchronous from a file-open. Phase 5.1 documented this and accepted it as a transient behavior.

**How to avoid:** Accept the transient window. The user has to physically type within ~50 ms of file-open AND target a locked region — vanishingly rare. The next docChanged transaction (their 2nd keystroke, or any background refresh) re-fires the filter with the cache populated.

**Warning signs:** None practical. Edge case for tests if pursued.

## Open Questions

1. **Should plugin-side dispatches bypass the lock via `userEvent: 'leetcode.*'` annotation, or should the planner audit every dispatch to ensure it doesn't touch a locked range?**
   - What we know: Phase 5.3's chevron dispatch (`src/main.ts:799-805`) already sets `userEvent: 'leetcode.lang-switch'`. It edits the fence opener line — which IS in a locked range under D-09.
   - What's unclear: whether the planner prefers the "annotation as escape hatch" pattern (one filter check) or the "audit each dispatch" pattern (rewrite each call site to bypass via a different mechanism, e.g., `tr.filter = false` on the spec).
   - Recommendation: **Annotation as escape hatch** is simpler and more robust. Document the convention `'leetcode.*'` in CLAUDE.md §Conventions. The planner verifies in Wave 0 that all current `cm.dispatch` callsites already use `'leetcode.*'` userEvents (Pitfall 5 lists them).

2. **Should `LOCKED_HEADINGS` be a new export from `NoteTemplate.ts` or an internal constant in `sectionLockExtension.ts`?**
   - What we know: Phase 2 D-03 SSoT invariant says heading literals come from `NoteTemplate.ts`. The four heading constants ARE already exported.
   - What's unclear: whether the `LOCKED_HEADINGS` array (the convention "these 4 are locked, ## Custom Tests is not") belongs in `NoteTemplate.ts` (cross-cutting policy) or `sectionLockExtension.ts` (lock-specific).
   - Recommendation: export from `NoteTemplate.ts` so future readers find it next to the heading literals. The literal export does not add a new `lc-*` key (CF-12 preserved); it's just an array of strings.

3. **Should the lock's pure functions live in `sectionLockExtension.ts` or in a new pure module `src/notes/sectionLockRanges.ts`?**
   - What we know: Phase 5.3 split runArity helpers into `src/solve/runArity.ts` (a pure module) for testability. Same pattern would work here.
   - What's unclear: marginal benefit vs. one-file simplicity at this scope.
   - Recommendation: keep both in `sectionLockExtension.ts` for v1; export the pure helpers (`computeLockedRanges`, `LOCKED_HEADINGS`) so tests can hit them without driving the full filter. Split if a downstream phase needs them outside the lock.

4. **Visual dim — ship with the lock or ship in a follow-up?**
   - What we know: D-04 says optional / planner discretion. Discoverability concern (Specifics): users learn the lock exists by typing and seeing nothing happen.
   - What's unclear: whether dim noticeably helps discoverability without being intrusive in dogfood.
   - Recommendation: ship a single CSS rule with `Decoration.mark` enabled by default. Cost is one CSS rule + one decoration extension; benefit is "user sees the locked region is dimmer" before they try to type. If dogfood reveals it's distracting, remove in a follow-up (or add a settings toggle, deferred).

5. **One-shot debug log on first dropped keystroke per session — ship or omit?**
   - What we know: D-04 / Discretion mentions this as useful for dogfood, not user-facing.
   - What's unclear: whether the planner wants additional logger surface.
   - Recommendation: include via `logger.debug('section-lock: dropped change in locked range', { fromA, toA })` inside the filter callback when ranges are returned. `logger.debug` is gated by env (CLAUDE.md §logger pattern); zero-cost in production.
