## Decisions Made

- **Flat `[changeFilter.of(cb), atomicRanges.of(...)]` composition** rather than exporting a separate named filter callback. Plan 01's `extractChangeFilterCallback` test helper accepts either shape; flat composition is simpler and has no downstream callsite that needs the named-export form. If Plan 04's main.ts wiring or Plan 03's integration tests later need the callback in isolation, refactor to a named export at that time.
- **Visual-dim Decoration.mark deferred to Plan 04.** CONTEXT D-04 explicitly marks visual dimming "planner discretion." Plan 02 ships the lock without it; the atomic-ranges decoration class `'leetcode-section-locked-atomic'` is internal-only (not styled here). Plan 04 may add a CSS rule that targets it OR may add a separate `EditorView.decorations.of(...)` facet for visual dimming. This deferral matches the must_haves frontmatter: "Do NOT add the optional Decoration.mark visual-dim extension here — that lands in Plan 04."
- **Cost-ascending gate order in the changeFilter callback.** Pitfall 5 userEvent bypass is gate 1 (cheapest — single annotation lookup, no field access). File-presence gate is gate 2 (single field read). Frontmatter gate is gate 3 (metadataCache lookup, costlier). computeLockedRanges is gate 4 (line-scan, costliest). This ordering ensures plugin-internal dispatches (which fire frequently — every chevron click, every copy-to-code) pay only one annotation lookup, never the metadataCache + line-scan cost.
- **Removing the Wave 0 eslint-disable directive in Task 2** is anticipated cleanup, NOT a Plan 01 contract violation. Plan 01's header comment said verbatim: "When Wave 1 (Plan 02) creates the real module with typed exports, TypeScript will infer concrete types, the no-unsafe-* cascade will evaporate, and these disables can be removed." The test logic + the 16 it-blocks are byte-identical to Plan 01.
