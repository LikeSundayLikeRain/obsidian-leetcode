## Gaps

### G1 — Chevron dropdown stale after language switch (RESOLVED)
- discovered: 2026-05-13 during item #3 UAT
- symptom: After picking a new language from the chevron dropdown, the fence opener and `lc-language` frontmatter updated correctly, but the chevron LABEL and the dropdown's `is-current` highlight remained on the OLD language until the user typed in the fence body
- root cause: `buildDecorations` re-read `metadataCache.getFileCache(file).frontmatter['lc-language']`, but `processFrontMatter`'s vault write → metadataCache flush is async; the StateField rebuild raced ahead of the cache update and painted the old language
- fix: `languageRefreshEffect` now carries `string | undefined` payload; chevron switch passes `newSlug` directly through the effect, bypassing metadataCache entirely. The metadataCache `'changed'` listener also passes the freshly-flushed slug. `buildDecorations` accepts `overrideSlug` parameter that wins over the cache read. Combined with consolidating switchFenceLanguage's two dispatches into ONE (changes + effect in one transaction).
- status: RESOLVED — UAT confirmed 2026-05-13

### G2 — D-04 visual-dim rejected; marker-hide adopted instead (RESOLVED)
- discovered: 2026-05-13 during item #7 UAT
- symptom: User found the visual dim (`background: var(--background-secondary)`) noisy/unwanted. Original concern was the `## ` heading prefix being visible when cursor sat on a locked heading line, breaking parity with Reading Mode.
- decision: dropped the dim layer entirely; replaced with `Decoration.line` + CSS `display:none` on `.cm-formatting-header` tokens for locked heading lines. Locked headings now render parity with Reading Mode regardless of cursor position. Original RESEARCH Open Q4 recommendation ("ship visual dim default-on") superseded by UAT preference.
- status: RESOLVED — UAT confirmed 2026-05-13

### G3 — Cursor navigation into locked regions (RESOLVED)
- discovered: 2026-05-13 during item #6 UAT
- symptom: Cursor could land at column 0 of locked headings via click and arrow keys; `EditorView.atomicRanges` only covered some motions; column-step Down/Up snuck cursor into locked range. After early fixes, ping-pong between adjacent locked clusters; Up arrow from fence body line 1 teleported to the frontmatter pocket (rendered as "Problem col 0").
- root causes (layered):
  1. `EditorView.atomicRanges` doesn't cover click-to-position or column-step motion
  2. `changeFilter` is exclusive at boundaries — pure insertions at `lockFrom` weren't suppressed
  3. Adjacent locked ranges (Problem body end == Code heading start) caused snap to ping-pong; needed merge step
  4. Backward snap target landed in unreachable frontmatter pocket above first cluster
  5. Vault-sync transactions (no userEvent) were being suppressed, corrupting buffer when copyToCode wrote
- fix: replaced `EditorView.atomicRanges` with a `EditorState.transactionFilter` that snaps cursor out of locked ranges. Logic extracted into pure helpers `mergeLockedRanges` + `computeSnapTarget` for testability. Three-branch decision tree: forward motion → escape forward; backward motion with editable target → escape backward; backward motion into unreachable pocket OR click inside cluster → stay put / forward escape. `changeFilter` Gate 0 added: only suppress for known user-input userEvents (`input.*`, `delete.*`, `undo`, `redo`); programmatic dispatches (vault sync, etc.) pass through. Selections (head !== anchor) bypass snap so user can copy locked text.
- hardening: `mergeLockedRanges` and `computeSnapTarget` are exported pure helpers with 13 unit tests covering all branches.
- status: RESOLVED — UAT confirmed 2026-05-13
