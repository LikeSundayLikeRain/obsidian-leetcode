## Lock mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Heading-only convention (no markers, NoteTemplate stays the SSoT) | Same as today — heading text alone signals plugin ownership. No new markers. CM6 read-only treatment + write-path checks scan for the heading, walk to next H2. | ✓ |
| HTML-comment sentinels around plugin regions | `<!-- lc-lock:problem-start -->` ... `<!-- lc-lock:problem-end -->`. Survives heading renames; needs migration; visible in Source Mode. | |

**User's choice:** Heading-only convention (no markers, NoteTemplate stays the SSoT)
**Notes:** Captured as D-01 in CONTEXT.md. Existing notes work unchanged on first re-open after 05.5 ships — no migration. NoteTemplate.ts remains the SSoT for plugin-owned regions; the CM6 extension reads heading literals from there.

## Conflict policy

| Option | Description | Selected |
|--------|-------------|----------|
| Always overwrite plugin sections silently (status quo, no conflict detection) | Plugin sections are PLUGIN's — user edits to them are not protected. The 'lock' becomes purely a CM6 visual/read-only thing. No detection, no diff, no sidecar. | ✓ |
| Detect user edits; skip overwrite + Notice | Compare current region content to last plugin-written content. Skip on divergence + Notice 'use Reset to overwrite.' Reset command forces overwrite. | |
| Detect + auto-backup user edits to a sidecar before overwrite | Save user's modified region to `.lc-backup/0001-two-sum.problem.bak.md` then overwrite. Most preserves user data; biggest implementation surface. | |
| Hybrid: ## Code = always overwrite, ## Problem/## Techniques = detect + skip | Different sections have different volatility. ## Code is meant to be overwritten by chevron-switch / copy-to-code (user expects it). | |

**User's choice:** Always overwrite plugin sections silently (status quo, no conflict detection)
**Notes:** Captured as D-02 in CONTEXT.md. The lock's purpose is to PREVENT user edits from landing in those regions, not to PROTECT user edits if they somehow do. This pairs cleanly with hard read-only enforcement (D-04): the keystroke is blocked at CM6 entry → no edit reaches buffer → no edit can collide with a plugin write → no detection needed. Side benefit: zero new state in frontmatter or `data.json`.

## Editor visual treatment

| Option | Description | Selected |
|--------|-------------|----------|
| Hard read-only via CM6 — cursor can't enter, keystrokes blocked | EditorState.transactionFilter drops any change touching locked ranges. Cursor visually skips; keystrokes drop silently. Strictest; most predictable. | ✓ |
| Soft warn-on-edit — user CAN edit but Notice fires + edit gets reverted | User types; CM6 reverts on next tick + Notices 'plugin-managed.' More forgiving; same end-state as hard read-only. | |
| Visual-dim + lock-icon, but user CAN edit (signaling-only) | Dimmed background + 🔒 icon. User edits NOT blocked. Lightest touch; combined with D-02 'always overwrite,' user edits would silently disappear on next plugin write. | |
| Hard read-only on heading + fence boundaries; soft-warn on ## Problem body | Hardest enforcement on boundaries; soft-warn on ## Problem body. Most differentiated. | |

**User's choice:** Hard read-only via CM6 — cursor can't enter, keystrokes blocked
**Notes:** Captured as D-04 in CONTEXT.md. Implementation via `EditorState.transactionFilter` that reads each transaction's `changes` (using `ChangeSet.iterChanges`), checks for overlap with locked ranges, and drops the entire transaction if any overlap is found. Cursor moves are passed through. No Notice on dropped keystroke — silent failure is the right UX (Notices on every keystroke would be obnoxious).

## Migration policy

| Option | Description | Selected |
|--------|-------------|----------|
| Per D-02: silently overwrite on next background-refresh; no migration command | Consistent with D-02. Plugin doesn't notice or warn — next background-refresh re-renders ## Problem from cached HTML and prior user content vanishes. | ✓ |
| Run a one-time scan on plugin load: detect non-empty user content + offer 'Move to ## Notes' command | Friendly migration; substantial implementation surface (scanner + mover + UAT). | |
| Don't automate — just warn ONCE in Settings + README, then silently overwrite | One-time Notice + README docs the change. After warning, behavior matches option A. | |
| Lock applies to NEW notes only; existing notes remain unlocked until plugin-rewrites them | `lc-locked: true` frontmatter flag set on note creation post-05.5. Existing notes stay editable in plugin regions. Eventually phase out. | |

**User's choice:** Per D-02: silently overwrite on next background-refresh; no migration command
**Notes:** Captured as D-05 in CONTEXT.md. The phase ships with uniform behavior across all `lc-slug` notes from the moment 05.5 ships — no migration scanner, no opt-in flag, no "Audit notes" command, no Notice at first plugin load. Existing notes flow through 05.5 as: user opens note → CM6 lock applies → user can no longer add to plugin sections → eventually background-refresh / chevron-switch / on-AC overwrites the section and cleans up any legacy user content. Zero new data structures.
