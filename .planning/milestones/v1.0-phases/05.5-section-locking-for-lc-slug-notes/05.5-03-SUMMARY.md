## Audit Finding: src/graph/copyToCode.ts

**Question:** Does `src/graph/copyToCode.ts` use `cm.dispatch` anywhere?

**Answer:** **No.** The file uses `app.vault.process` (Step 1, fence body rewrite) and `app.fileManager.processFrontMatter` (Step 2, lc-language sync) exclusively. Verified by:

1. **Manual file read** — read all 174 lines of `src/graph/copyToCode.ts`. No `cm.dispatch` callsite anywhere.
2. **Static integration-test assertion** — Scenario 4 reads the file via `readFileSync` and asserts `content.includes('cm.dispatch') === false`. Test passes; assertion holds at HEAD = `55447e4`.
3. **Architectural rationale** — `vault.process` writes happen at the vault layer, BELOW CM6. The CM6 buffer reloads from disk after `vault.process` resolves, so the section lock (which is a CM6 transaction filter) has no transaction to filter for the copy-to-code path. This is intentional asymmetry — the lock is editor-side; copy-to-code is vault-side; they do not interact.

The Scenario 4 assertion locks this in: any future PR that refactors copy-to-code to dispatch CM6 transactions directly will fail the integration test at PR-time. That PR will then have to either (a) revert to the vault.process pattern, or (b) explicitly set `userEvent: 'leetcode.copy-to-code'` to use the D-04 escape hatch — and update Scenario 4 to expect the new architecture. Either way, the question is forced into review rather than slipping through silently.

## Decisions Made

- **Removed the file-header eslint-disable directive.** Plan 01's RED-state template carried a `/* eslint-disable @typescript-eslint/no-unsafe-call, ... */` block at the top of the test file. Plan 03's must_haves frontmatter said "copy Plan 01's pattern verbatim for safety." However, lint surfaced a `Unused eslint-disable directive` warning because the integration tests do not exercise the unsafe-* paths — `getChangeFilterCallback` returns a typed `(tr: unknown) => unknown`, and `lineIsLocked` accepts typed parameters. Removing the directive cleared the warning to zero errors / zero warnings. This is a Plan-02-style anticipated cleanup (matches Plan 02's own removal of the unused eslint-disable in `tests/main/sectionLockExtension.test.ts`). Test logic is unchanged.
- **Added a defensive `lineIsLocked(ranges, lineFrom, lineTo)` predicate** that returns false when `lineTo <= lineFrom`. Without this guard, the malformed-fence "body falls through to editable" loop would falsely classify zero-width trailing lines (e.g., the empty string after the final `\n` in `makeStateForLockTests`'s line adapter) as "locked" because every `[from, to]` pair where from === to satisfies `from <= zeroWidth && to >= zeroWidth`. The guard is correct because zero-width lines have no editable content to be either locked or unlocked.
- **Added a `missingOpener` option to canonicalNoteBody.** Plan 01's helper had only `unterminatedFence`. The plan's behavior spec required testing BOTH malformed cases: (a) `## Code` heading with NO fence at all (Scenario 2 first it-block — body fallthrough), and (b) opener exists but no closer (Scenario 2 second it-block — findCodeFence returns null). Adding `missingOpener` keeps the helper symmetric with Plan 01's API while supporting the new case.
- **Scenario 1 has two it-blocks both asserting filter(tr) === true under the same userEvent annotation.** The Plan 02 filter is invoked ONCE per transaction, not per change — so even though the chevron's dispatch carries TWO changes (opener + body), the filter sees one transaction with one userEvent annotation. The second it-block in Scenario 1 exists for documentation-of-intent: it makes the chevron's atomic two-change dispatch explicit in the test record. Both halves pass through together because the leetcode.* prefix gate fires first (cost-ascending gate order from Plan 02).
