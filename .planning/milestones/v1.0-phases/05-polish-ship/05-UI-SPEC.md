# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Polish & Ship.
> Generated by gsd-ui-researcher. All decisions pre-populated from CONTEXT.md, RESEARCH.md,
> and existing codebase scan. No design-system gate applied (Obsidian plugin — no React/shadcn).

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — Obsidian plugin DOM API (`createEl` / `createDiv` / `createSpan`) |
| Preset | not applicable |
| Component library | Obsidian `Setting` API + `Modal` subclasses (native plugin toolkit) |
| Icon library | Obsidian `setIcon` (Lucide subset bundled by Obsidian host) |
| Font | Inherited from Obsidian theme: `var(--font-text)` body, `var(--font-monospace)` code |

**Critical constraint:** All DOM must be constructed via `createEl()` / `createDiv()` / `createSpan()`.
`innerHTML`, `outerHTML`, and `insertAdjacentHTML` are permanently forbidden (CF-07, D-27 grep-gate).
Every color must reference an Obsidian CSS variable — no raw hex, no rgba literals
(existing `styles.css` rule, enforced by convention, never broken in Phases 1–4).

## Spacing Scale

Declared values (multiples of 4 — matches existing `styles.css` geometry throughout Phases 1–4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, chip padding |
| sm | 8px | Intra-element gaps, tab button padding, textarea padding |
| md | 12px | Section-internal margin-bottom, modal inner spacing |
| lg | 16px | Modal content padding, section spacing |
| xl | 20px | — (not used in Phase 5 surfaces) |
| 2xl | 24px | — (not used in Phase 5 surfaces) |

**Carry-forward exceptions** (from Phases 1-4 styles.css — not in standard set {4, 8, 16, 24, 32, 48, 64} but retained for visual continuity with shipped UI):
- `md: 12px` — used in RunModal footer `margin-top: 12px` and `.leetcode-run-modal-footer` internal spacing. Multiples of 4; predates Phase 5.
- `xl: 20px` — declared in the token table above as a carry-forward reserve; not introduced by Phase 5 surfaces. Present in earlier phases for section-level margin use.

These values remain multiples of 4 and are not introduced by Phase 5 — they are inherited from the existing codebase. Phase 5 adds no new off-standard spacing values.

**Note:** Obsidian's `Setting` API handles its own internal spacing automatically. These tokens
apply to custom-DOM surfaces only: `.leetcode-code-actions`, RunModal tab row, RunModal footer,
and `styles.css` additions for Phase 5.

Exceptions:
- Tab buttons in RunModal: height 28px (matches existing `.leetcode-custom-test-tab` from Phase 3 — carry forward unchanged).
- Footer button row gap: 8px (matches existing `.lc-fm__footer-right` gap).
- `.leetcode-code-actions` container: `margin-top: 4px` (tight coupling to the `<pre>` block above it).

## Typography

All sizes inherited from Obsidian host theme. Phase 5 adds no new font declarations.
The following are the concrete values already in `styles.css` — carried forward, not changed.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body / UI label | `var(--font-ui-small)` ≈ 13px | 400 | 1.5 | Setting descriptions, modal body text, picker row text |
| Small / meta | 12px | 400 | 1.6 | Monospace meta lines, verdict runtime, chip text, code action buttons |
| Tab label | 13px | 400 active → 600 | 1.0 | RunModal case tab labels (`.is-active` gets weight 600) |
| Section heading | Obsidian `setHeading()` size | 600 | Obsidian default | Settings section headings (Authentication / Notes / Knowledge Graph) |

**Note on 12px vs 13px proximity:** The 12px and 13px sizes are deliberately separated by only 1 pixel because they serve distinct semantic roles inherited from Phases 1-4: 12px is used exclusively for metadata, chips, and condensed rows where information density matters; 13px is used for all primary UI labels and body text. The proximity is intentional — both sizes were locked by earlier phases to match Obsidian's native theme conventions. Phase 5 adds no new font sizes.

## Color

All values are Obsidian CSS variable references. No raw hex is ever written to `styles.css`.

| Role | Token | Usage |
|------|-------|-------|
| Dominant surface (60%) | `var(--background-primary)` | Modal backgrounds, view backgrounds, settings tab background |
| Secondary surface (30%) | `var(--background-secondary)` | Card backgrounds, textarea backgrounds, code block backgrounds, tab inactive |
| Accent (10%) | `var(--interactive-accent)` | **Reserved — see list below** |
| Success | `var(--text-success)` / `var(--background-modifier-success)` | AC verdict, focus ring (alternate) |
| Error / destructive | `var(--text-error)` / `var(--background-modifier-error)` | WA/TLE/MLE/RE/CE chips, WA diff border |
| Muted | `var(--text-muted)` | Secondary labels, metadata, inactive tab text, placeholder text |
| Border | `var(--background-modifier-border)` | Dividers, textarea border, inactive chip background |
| Hover | `var(--background-modifier-hover)` | Button and row hover state |
| On-accent text | `var(--text-on-accent)` | Text on accent-colored surfaces |

**Accent (`var(--interactive-accent)`) is reserved for exactly:**
1. Primary `Log in via embedded window` button in `SettingsTab.ts` — the single `.setCta()` call (existing, unchanged)
2. The `Log in` action button inside the session-expired Notice — `mod-cta` class on the `<button>` in the DocumentFragment (D-21, RESEARCH Pattern 1). This is the same CTA semantic rendered in a different context.
3. `.lc-iconbtn__badge` background (existing, Phase 1 — unchanged)
4. `lc-progress__fill` progress bar (existing, Phase 1 — unchanged)
5. `.leetcode-submissions-picker .leetcode-submissions-row` focus outline (D-30 fix — see CSS section)

**Accent is explicitly NOT applied to:**
- Reset button in RunModal footer (neutral, no `.setCta()`)
- `×` delete buttons on tabs (neutral)
- Run / Submit buttons in `.leetcode-code-actions` (neutral, D-13)
- Any new button in the Knowledge Graph settings section

**CE verdict chip tint fix (D-29):**
Current: `.leetcode-submissions-chip--ce` uses `var(--background-modifier-error)` / `var(--text-error)` (red).
Required: Change to `var(--color-orange, #e67e22)` background tint + `var(--color-orange, #e67e22)` text.
Implementation: Add a specific override rule for `--ce` that separates it from the shared `--wa, --tle, --mle, --ole, --re, --ce` rule. CE is a user error (bad code syntax), not a judge rejection — orange signals "fixable" rather than "wrong answer."

```css
/* D-29: CE verdict — orange tint (separate from WA/TLE/MLE/RE which stay red) */
.leetcode-submissions .leetcode-submissions-chip--ce {
  background: color-mix(in srgb, var(--background-secondary) 80%, var(--color-orange, #e67e22) 20%);
  color: var(--color-orange, #e67e22);
}
```

**CE chip raw hex exception:** The CE chip uses `var(--color-orange, #e67e22)`. The `#e67e22` fallback is a deliberate safety net for themes that do not define `--color-orange`. Verified: `styles.css:145` already uses this variable and it resolves correctly in both shipped Obsidian light and dark themes. This is the ONLY raw hex value in the spec and is explicitly documented as an exception to the "no raw hex" rule — it exists to guarantee the CE chip never reverts to red on a non-standard theme.

Verified rationale: `var(--color-orange)` is already used in the codebase at `styles.css:145` for the paid/premium lock icon — confirming it resolves in both light and dark Obsidian themes. The `#e67e22` fallback is the standard amber-orange used by Obsidian for orange when the variable is absent.

### 4. Notice with Action Button — Session Expired (Wave 2)

**Context:** D-21 session-expired Notice gains a clickable `Log in` action.

**Implementation pattern (RESEARCH Pattern 1 — CORRECTED):**
`Notice.addAction()` does NOT exist in `obsidian@1.12.3`. Use DocumentFragment constructor.

```typescript
function showSessionExpiredNotice(login: () => void): void {
  const frag = document.createDocumentFragment();
  frag.createEl('span', {
    text: 'LeetCode session expired. Log in again.',  // CF-04 LOCKED
  });
  frag.createEl('span', { text: ' ' });
  const btn = frag.createEl('button', {
    cls: 'leetcode-notice-action mod-cta',
    text: 'Log in',
  });
  const notice = new Notice(frag, 0); // duration 0 = sticky until actioned (RESEARCH Pitfall 7)
  btn.addEventListener('click', () => {
    notice.hide();
    void login();
  });
}
```

**Button styling:**
- Class: `leetcode-notice-action mod-cta`
- `mod-cta` applies Obsidian's accent color — appropriate here because this IS the primary auth CTA, same semantic as the SettingsTab login button (RESEARCH §Open Question 2 resolved: treat as same CTA)
- No additional CSS required — `mod-cta` is provided by Obsidian's base stylesheet

**Notice duration:** `0` (sticky) — user must click `Log in` or manually dismiss. Rationale: session expiry is a blocking state; auto-dismissal would lose the action affordance (RESEARCH Pitfall 7).

### 5. Error Notices (Wave 2)

All four Notice strings are LOCKED (CF-19). Duration 8000ms except session-expired (0 / sticky).

| Condition | Copy (LOCKED) | Duration | Source |
|-----------|---------------|----------|--------|
| 429 rate-limited | `LeetCode is rate limiting us. Try again in a moment.` | 8000ms | D-18 |
| Network failure / offline | `Couldn't reach LeetCode. Check your connection.` | 8000ms | D-19 |
| Timeout (10s) | `LeetCode is slow to respond. Try again.` | 8000ms | D-20 |
| Session expired | `LeetCode session expired. Log in again.` + `Log in` action | 0 (sticky) | D-21 / CF-04 |

**Routing rule (D-22):**
- Command-palette-initiated flows (Run, Submit, Open problem, Refresh) → Notice
- Modal-initiated flows (Picker, Detail modal, Run modal) → inline error in the current modal
- Code-block action buttons → treated as command-palette (delegates via `executeCommandById`) → Notice

**429 Notice:** fires only on the second consecutive 429 (after the single silent auto-retry per D-18). First 429 is invisible to the user (5s backoff + retry).

### 7. Submission Picker Focus Ring Fix (Wave 4, D-30)

**File:** `styles.css`
**Change:** Strengthen `.leetcode-submission-row:focus` outline for light mode.

Current state: Phase 4 shipped `.leetcode-submissions-picker .leetcode-submissions-row:hover, :focus` with `background: var(--background-modifier-hover); outline: none`. The `outline: none` removes the default browser focus ring; in light mode the hover background change is too subtle to meet WCAG 2.1 AA 3:1 contrast ratio for focus indicators.

```css
/* D-30: strengthen focus ring on picker rows — both themes */
.leetcode-submissions-picker .leetcode-submissions-row:focus {
  background: var(--background-modifier-hover);
  outline: 2px solid var(--interactive-accent);
  outline-offset: -2px; /* inset so it doesn't overflow the row's border-radius */
}
```

Note: `outline-offset: -2px` (inset) is used instead of positive offset to keep the focus ring inside the row's 4px border-radius without clipping. Verified pattern used by Obsidian's own interactive elements.

### 8. SubmissionDetailModal — MarkdownRenderer Upgrade (Wave 4, D-31)

**File:** `src/graph/SubmissionDetailModal.ts`
**Change:** Replace `<pre><code class="language-*">` + `textContent` approach with `MarkdownRenderer.render` + Component lifecycle.

**Visual change from user perspective:** Syntax highlighting now works (CM6 fence rendering). Layout is otherwise identical — code block appears in the same position inside the modal.

**Component lifecycle (RESEARCH Pattern 3 — CRITICAL):**
```typescript
private readonly component = new Component();

onOpen(): void {
  this.component.load(); // MUST be called before render
  // … render chrome …
  const fenced = '```' + lang + '\n' + code + '\n```\n';
  await MarkdownRenderer.render(this.app, fenced, codeContainer, sourcePath, this.component);
}

onClose(): void {
  this.component.unload(); // disposes CM6 child components
  this.contentEl.empty();
}
```

**Pitfall guard (RESEARCH Pitfall 6):** `component.load()` MUST be called before `MarkdownRenderer.render()`. Without it, Obsidian logs "Child loaded before parent" and embedded elements (mermaid, transclusions) fail silently.
