## What Shipped

### D-09 — Chip contrast

`styles.css` lines 749-789: the base `.leetcode-submissions-chip` rule was rewritten from `font-weight: 600` + no `min-width` to `font-weight: 700`, `min-width: 60px`, `justify-content: center` so labels like "Accepted" and "Wrong Answer" render in full without truncation and read bolder against the darker pill backgrounds.

AC chip was changed from `background: var(--background-modifier-success); color: var(--text-success);` (which washed out against the green on both light and dark themes per user report) to `background: color-mix(in srgb, var(--background-modifier-success) 80%, var(--color-green, #22c55e) 20%); color: var(--text-on-accent, #fff);`. The 80/20 darkening ratio is the same pattern the Phase 5 CE orange override uses (SP-7 / PATTERNS §9), so the visual language stays consistent across chip variants.

WA/TLE/MLE/OLE/RE got the parallel treatment against `--color-red, #ef4444`. Critically, `--ce` was **removed** from the shared error rule's selector list so that CE's Phase 5 D-29 orange override (which lives at line 787, after the shared rule at line 780) now wins via cascade without needing `!important`. The cascade invariant is the main correctness risk here (T-05.2-14 mitigate): line-number check confirms CE > shared-error in the file order, which is the same invariant the CE override was originally positioned for.

### D-13 — python3 → python Reading-Mode alias

New file `src/main/python3Highlighter.ts` exports `registerPython3Highlighter(plugin: Plugin): void`. It registers a single `MarkdownPostProcessor` that selects `code.language-python3` from the rendered Reading-Mode DOM and swaps the class to `language-python`. Obsidian's Prism.js recognizes `python` but NOT `python3` (https://prismjs.com/#supported-languages), so before this plan, python3-tagged fences rendered as plain text. LeetCode's canonical langSlug for Python is `python3` — which is exactly what our starter-code inserter emits via `codeBlockFor` — so the mismatch broke syntax highlighting on every LC-generated problem note.

The post-processor is deliberately **global** (not gated on `lc-slug` frontmatter) per D-13 design. T-05.2-13 accepts this as low-risk: the processor only mutates class names, no data read, no code execution, no network. Users with `python3` fences in non-LC notes also benefit.

Synchronous implementation beats `loadPrism().then(p => p.languages.python3 = p.languages.python)` because the class swap is guaranteed to happen before Prism tokenizes (MarkdownPostProcessor runs synchronously with render); the async alternative has a render race (RESEARCH Pitfall 7) that would cause a flash of unstyled code on first paint.

`src/main.ts` imports the new function and calls it in onload Step 6h (after Step 6g file-open hook; order doesn't matter between these two — they're independent).
