# Phase 5.2 — UI Design Contract

> Visual and interaction contract for the Phase 5.2 pre-ship UX polish batch. This is a **documentation-only UI-SPEC**: Phase 5.2 reuses visual idioms already shipped in Phases 1–5.1. No new design tokens, no new layout primitives, no new components are introduced.
>
> Phase 5.2 is primarily *removal + re-arrangement* of existing DOM plus three targeted fixes: a flex-row merge (Item 1), a Prism post-processor class swap (Item 9), and a CSS contrast fix on existing chips (Item 6). Every other item reuses existing idioms verbatim.

### Item 6 — Chip contrast fix (the phase's one real color decision)

**Problem:** User reports AC/WA chip text is "still not visible" on shipped build. Investigation (RESEARCH A2 + Pitfall 4): `setText(chip, info.displayName)` IS called at `SubmissionPickerModal.ts:213`, so the text is in the DOM. The symptom is **low contrast**: `var(--text-success)` on `var(--background-modifier-success)` (AC) and `var(--text-error)` on `var(--background-modifier-error)` (WA) produce near-identical luminance on some Obsidian themes.

**Fix (LOCKED for Phase 5.2):**

- **AC chip:** change `color` from `var(--text-success)` → **`var(--text-on-accent)`** only when a `--text-on-accent` variable is defined by the theme; fall back to `#fff` (hard-coded white) for Obsidian's default-light theme where `--text-success` is a dim green. Use CSS `color-mix` to darken the green background 15% so white text has ≥ 4.5:1 contrast:

```css
.leetcode-submissions .leetcode-submissions-chip--ac {
  background: color-mix(in srgb, var(--background-modifier-success) 80%, var(--color-green, #22c55e) 20%);
  color: var(--text-on-accent, #fff);
}
```

- **WA / TLE / MLE / OLE / RE chips:** same treatment on the error side:

```css
.leetcode-submissions .leetcode-submissions-chip--wa,
.leetcode-submissions .leetcode-submissions-chip--tle,
.leetcode-submissions .leetcode-submissions-chip--mle,
.leetcode-submissions .leetcode-submissions-chip--ole,
.leetcode-submissions .leetcode-submissions-chip--re {
  background: color-mix(in srgb, var(--background-modifier-error) 80%, var(--color-red, #ef4444) 20%);
  color: var(--text-on-accent, #fff);
}
```

- **CE chip:** existing D-29 orange override stays (uses `color-mix` with `--color-orange`). Confirmed contrast via `#e67e22` on 80% secondary background. **No change in Phase 5.2.**

- **Shared base rule:** also bump `font-weight: 600 → 700` and add `min-width: 60px` so "Accepted" / "Wrong Answer" labels always render the full background pill (no text truncation).

**Rationale for `var(--text-on-accent, #fff)` instead of darker text on pale background:** Obsidian theme authors reliably define `--text-on-accent` as a high-contrast color against the accent palette. Using it with the `#fff` fallback covers themes that don't define it. This reads well in both light and dark mode because we also darken the pill background — the new rule doesn't depend on the theme's default `--text-success` / `--text-error` luminance being well-tuned.

**Alternative considered (rejected):** "just bump font-weight to 700 and keep the existing colors" — rejected because RESEARCH Pitfall 4 identified root cause as *chroma mismatch*, not weight mismatch. Weight alone on a low-chroma pair is still unreadable.

**Verification (manual):** Open Past Submissions picker on a LC note with mixed AC/WA history in (a) Obsidian default light, (b) Obsidian default dark, (c) Minimal theme (light + dark), (d) Things theme (light + dark). Chip text must be clearly readable in all 6 combinations.

## Layout Decisions (answers to the 3 open design questions)

### Question 1 — Item 1 Settings row merge: flex-row or absolute?

**Decision: native Obsidian `Setting` helper — ONE `new Setting(containerEl).setName('Status').setDesc(statusText).addButton(...)` call.** No custom CSS, no absolute positioning, no flexbox overrides.

**Rationale:**
- Obsidian's `Setting` helper already renders as `<div class="setting-item"><div class="setting-item-info">…left…</div><div class="setting-item-control">…right…</div></div>`. The control cell is right-aligned by the Obsidian theme's built-in CSS. Adding `.addButton()` places the button in the control cell — done.
- Custom flexbox (option: `.leetcode-settings-status { display: flex; justify-content: space-between }`) would fight Obsidian's built-in `.setting-item` layout and break in community themes that customize it.
- Absolute positioning would cause layout jumps as the status text width changes (e.g., `Logged in as jonsnow123` vs `Not logged in`).
- This is the same pattern Obsidian's sample-plugin + all first-party plugins use for Status + Action rows.

**Code sketch** (from RESEARCH §Pattern 1):

```typescript
new Setting(containerEl)
  .setName('Status')
  .setDesc(statusText)   // 'Logged in as {username}' or 'Not logged in'
  .addButton((b) => {
    if (loggedIn) {
      b.setButtonText('Logout')
        .setTooltip('Log out of LeetCode')  // inline disable for sentence-case: LeetCode is brand name
        .onClick(async () => { await this.plugin.auth.logout(); this.display(); });
    } else {
      b.setButtonText('Log in via embedded window')
        .setCta()  // preserves exactly-1-.setCta() grep gate
        .onClick(async () => { await this.plugin.auth.login(); this.display(); });
    }
  });
```

**Grep gate invariant:** `grep -c "\.setCta()" src/settings/SettingsTab.ts` must equal 1 after Phase 5.2 ships.

### Question 2 — Item 6 chip text color: which Obsidian CSS variable?

**Decision: `var(--text-on-accent, #fff)` for both AC and WA chips, with darker pill backgrounds via `color-mix`.** See `## Color` section above for the full CSS rule.

**Rationale:**
- `var(--text-on-accent)` is Obsidian's canonical high-contrast-against-colored-background variable, defined by all major themes. Fallback `#fff` covers themes that omit it.
- We also darken the background with `color-mix` (20% saturation boost toward pure green / red) so white text has strong contrast in both light and dark themes. This matches the treatment the CE chip already uses (D-29, Phase 5 Plan 05) — precedent is set.
- `var(--text-success)` (AC) and `var(--text-error)` (WA) are NOT used — the low-contrast problem is caused by their use in combination with `var(--background-modifier-success)` / `--error`, which are intentionally subtle in Obsidian themes.
- `var(--color-green)` and `var(--color-red)` are first-class Obsidian palette variables (defined in Obsidian 1.4+); we use them as the `color-mix` source. Fallback `#22c55e` / `#ef4444` covers older themes.

### Question 3 — Item 3 premium multi-select pattern: checkbox list or tag-style?

**Decision: reuse the existing `renderMultiSelect` + `openValuePopover` pattern already used by status / difficulty / topics.** Visual idiom: chevron-anchored pill showing "is N items" summary; clicking opens a popover with a vertical checkbox list.

**Rationale:**
- Already implemented at `FilterModal.ts:320` (`renderMultiSelect`) and `:357` (`openValuePopover`). Reusing it = zero new DOM, zero new CSS, zero new interaction affordances.
- `PREMIUM_OPTIONS` constant already defined at `FilterModal.ts:69-72` with exactly two values (`premium` / `non-premium`). Maps directly onto `renderMultiSelect`'s `options` parameter.
- Alternative (tag-style pills with × remove affordance) would introduce a new interaction idiom for a 2-option field — wasteful.
- Alternative (radio group) was the previous single-value behavior; D-03 explicitly rejects single-value.

**Code sketch** (from RESEARCH §Pattern 2):

```typescript
// src/browse/FilterModal.ts:311 — renderValueEditor premium branch
case 'premium':
  this.renderMultiSelect(cell, rule, PREMIUM_OPTIONS);
  break;

// Delete the entire renderPremiumEditor method (FilterModal.ts:425-476).
```
