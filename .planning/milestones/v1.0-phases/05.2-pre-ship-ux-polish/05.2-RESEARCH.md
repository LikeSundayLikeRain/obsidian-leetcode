**Researched:** 2026-05-11
**Domain:** Obsidian plugin UI polish (Settings layout, Filter modal, Command lifecycle, Markdown post-processing, Prism aliasing)
**Confidence:** HIGH

## Summary

Phase 5.2 is a grab-bag of 9 small UX polish items surfaced during 5.1 live smoke. Every item has a concrete, verified implementation path in the existing codebase — this is a stitch-together phase, not a design phase. Two items (4 filter-badge fix, 5 starter-code auto-insert + Reset) introduce one genuinely new decision each; the remaining 7 are mechanical edits (DOM re-arrangement, array edits, string-replace, fence-tag alias).

The phase should ship as **2–3 plans grouped by touched-area** (CONTEXT "Claude's Discretion"): Settings + Language list bundle (items 1, 8); Filter bundle (items 2, 3, 4); Starter-code + Copy-to-Code bundle (items 5, 7); Past Submissions (item 6); python3 alias (item 9). Items 1/8, 6, and 9 are each single-file edits; items 2/3/4 touch `FilterModal.ts` + `SettingsStore.ts` + `ProblemBrowserView.ts`; items 5/7 touch `main.ts` + `SubmissionDetailModal.ts` and reuse existing `forceInjectCodeSection` + `ConfirmOverwriteModal` primitives verbatim.

**Primary recommendation:** Bundle items by touched-area into 3 plans; reuse existing primitives (`forceInjectCodeSection`, `ConfirmOverwriteModal`, `classifyStatus`, `FENCE_TAG_ALIASES`) — no new primitives needed. For item 9 (python3 highlighting), register a `MarkdownPostProcessor` that rewrites `language-python3` class to `language-python` on rendered `<code>` elements — this is the least-invasive approach and does NOT require CM6 language-pack integration (which is Phase 5.3 scope).

### Locked Decisions

**Settings (Item 1)**
- **D-01:** Merge Status row and Logout button. The button moves to the right side of the existing `.leetcode-settings-status` block. No new Settings section. No settings rename. Likely a pure CSS + DOM tweak in SettingsTab.ts.

**Problem Filter (Items 2, 3, 4)**
- **D-02:** Remove `language` from the FilterModal field-selector's options list. Existing code paths that check for `language` filter rules still need to handle pre-existing user filters (graceful fallback: ignore the rule, don't crash). Planner decides migration (drop silently, or migrate on load).
- **D-03:** `premium` filter field becomes multi-value. Use Obsidian's existing multi-select affordance (or a checkbox group) matching the pattern the `difficulty` field uses today. Values: `non-premium`, `premium`.
- **D-04:** Filter badge counts **user-modified** rules only. Rules auto-populated by first-open logic (the `premium: non-premium` default applied when `isPremium === false`) should NOT count. Implementation idea: set a `__autoDefault: true` marker on auto-applied rules and exclude them from `updateFilterBadge`'s count, OR only count rules after the user has explicitly opened FilterModal once. Researcher + planner decide.

**Starter Code (Item 5)**
- **D-05:** `Insert starter code` command is removed. The command was added in Phase 2, but item 5's user insight is correct — users expect `## Code` to always exist on an LC note.
- **D-06:** Auto-insert trigger: when a user opens an `lc-slug` note (first time, or any time the `## Code` section is missing), the plugin appends the starter template silently. Likely hook: `file-open` event on MarkdownView, or processFrontMatter on-demand. Do NOT trigger on every cursor change — only on open/switch.
- **D-07:** `Reset code` command replaces `Insert starter code` in the command palette. When invoked on an `lc-slug` note, it overwrites the `## Code` fence with the starter template. KEEPS the existing ConfirmOverwriteModal pattern (unlike Copy-to-Code per D-11) because Reset is a destructive user-initiated action.
- **D-08:** Starter template source stays the same — LC's per-language starter code fetched from the problem detail. No change to the template itself.

**Past Submissions Picker (Item 6)**
- **D-09:** AC chip: render with text "Accepted". WA chip: render with "Wrong Answer". Other verdicts (TLE, MLE, RE, CE): render with their existing status-map labels from Phase 3 Plan 04's `classifyStatus` output. The chip currently has a colored background but empty text — just add the label inside it. CSS may need minor tweaks for readability (padding, min-width).

**Copy-to-Code (Item 7)**
- **D-10:** Remove `ConfirmOverwriteModal` from the Copy-to-Code click path. Copy-to-Code now overwrites the `## Code` fence silently. This matches Run/Submit click-through (both silent after Phase 5 Plan 05 retrofit).
- **D-11:** The `ConfirmOverwriteModal` class is NOT deleted from the codebase — it stays for `Reset code` command (per D-07). Only the Copy-to-Code call site is changed.

**Language List (Item 8)**

**Python3 Highlighting (Item 9)**

Simplest implementation: alias `python3` → `python` via Obsidian's codemirror language registry or via a custom syntax-highlight preprocessor. NOT a CM6 language pack — Phase 5.3 owns that scope. Scope here is minimal: one tag alias so blocks don't render unhighlighted.

### Pattern 1: Merging Obsidian Settings rows with `.setting-item-control`

**What:** A `new Setting(containerEl).setName('Status').setDesc(statusText)` row renders as `<div class="setting-item"><div class="setting-item-info">…</div><div class="setting-item-control"></div></div>`. The control cell is where buttons go. Appending a `.addButton()` on the SAME Setting call renders the button on the right of the same row.

**When to use:** Item 1 — the cleanest merge is a single `new Setting(containerEl).setName('Status').setDesc(statusText).addButton(...)` call, not a DOM surgery. This is the idiomatic Obsidian pattern and what `vault.getLeavesOfType('file-explorer')` views also use (sidebar subsection toggles).

**Example:**

```typescript
// Source: Obsidian sample-plugin SettingTab pattern (obsidian-developer-docs)
// Item 1 sketch — ONE Setting object, conditional branch inside addButton:
const loggedIn = this.plugin.auth.isLoggedIn();
const username = this.plugin.settings.getUsername();
const statusText = loggedIn ? `Logged in as ${username ?? '…'}` : 'Not logged in';

new Setting(containerEl)
  .setName('Status')
  .setDesc(statusText)
  .addButton((b) => {
    if (loggedIn) {
      b.setButtonText('Logout')
        // eslint-disable-next-line obsidianmd/ui/sentence-case -- "LeetCode" is LOCKED brand name
        .setTooltip('Log out of LeetCode')
        .onClick(async () => { await this.plugin.auth.logout(); this.display(); });
    } else {
      b.setButtonText('Log in via embedded window')
        .setCta()   // Accent modifier preserved — grep gate still passes with exactly 1 invocation.
        .onClick(async () => { await this.plugin.auth.login(); this.display(); });
    }
  });
```

**Why this pattern vs DOM surgery:** Obsidian's `Setting` helper owns the flex layout for `.setting-item-info` (left) and `.setting-item-control` (right). Trying to `.setCssStyles` the existing two-row layout to put Logout on the right of the Status cell would fight Obsidian's CSS and break in themes. Collapsing to one `Setting` call gets the layout for free.

### Pattern 2: Multi-value filter UI — reuse the existing `openValuePopover`

**What:** `FilterModal.ts:357` already has an `openValuePopover` helper that renders a checkbox popover with multi-select. It is used today by `renderMultiSelect` for status/difficulty/topics. The premium field uses a separate `renderPremiumEditor` (single-select radio-like picker).

**When to use:** Item 3 — delete `renderPremiumEditor`; wire `premium` through `renderMultiSelect` with `PREMIUM_OPTIONS` as the options list. Change the rule shape from `{ field: 'premium'; op: 'is'; value: 'premium' | 'non-premium' | null }` to `{ field: 'premium'; op: 'is'; values: string[] }`.

**Example:**

```typescript
// Source: src/browse/FilterModal.ts:287 renderValueEditor
// Item 3 sketch — change the premium case to use renderMultiSelect:
case 'premium':
  this.renderMultiSelect(cell, rule, PREMIUM_OPTIONS);
  break;

// Source: src/settings/SettingsStore.ts:90 — FilterRule type change:
export type FilterRule =
  | { field: 'status'; op: 'is' | 'is-not'; values: string[] }
  | { field: 'difficulty'; op: 'is' | 'is-not'; values: string[] }
  | { field: 'topics'; op: 'is' | 'is-not'; values: string[] }
  | { field: 'question-id'; op: 'range'; min: number | null; max: number | null }
  | { field: 'acceptance'; op: 'range'; min: number | null; max: number | null }
  | { field: 'premium'; op: 'is'; values: string[] };   // was: value: 'premium' | 'non-premium' | null
```

**Evaluator change (`src/browse/ProblemListService.ts:259`):**

```typescript
case 'premium': {
  if (r.values.length === 0) return undefined;   // no-op rule
  // row.paid === true maps to 'premium'; false maps to 'non-premium'
  const rowVal = p.paid ? 'premium' : 'non-premium';
  return r.values.includes(rowVal);
}
```

**Note:** When both values are selected, the rule effectively matches everything (`premium OR non-premium`) — correct semantics, no special case needed. When zero values are selected, the rule is a no-op (returns `undefined`), which matches the convention already used for status/difficulty/topics.

### Pattern 3: Auto-marker for first-open default rules

**What:** Item 4 — distinguish "user explicitly added this rule" from "plugin auto-populated this rule on first open so the user sees LC's starter layout".

**When to use:** Item 4 — the cleanest mechanism is a non-persisted `__autoDefault?: true` field on the rule. The badge-count reader filters it out; `isValidCompoundFilter` either allows the field or strips it on load (the rule STILL evaluates correctly either way — the field is metadata only).

**Example:**

```typescript
// Source: src/browse/ProblemBrowserView.ts:66-72 first-open default logic
// Item 4 sketch — mark the auto-default rule:
if (this.filter === null && this.plugin.settings.getIsPremium() === false) {
  this.filter = {
    match: 'all',
    rules: [{
      field: 'premium',
      op: 'is',
      values: ['non-premium'],
      __autoDefault: true,
    } as FilterRule & { __autoDefault?: boolean }],
  };
  await this.plugin.settings.setFilter(this.filter);
}

// Source: src/browse/ProblemBrowserView.ts:386 updateFilterBadge
private updateFilterBadge(): void {
  if (!this.filterBadgeEl) return;
  const userRules = (this.filter?.rules ?? []).filter(
    (r) => !(r as FilterRule & { __autoDefault?: boolean }).__autoDefault,
  );
  const count = userRules.length;
  // ...existing logic using `count`...
}
```

**Design choice rationale:** `__autoDefault` is less invasive than a separate "has user touched filter modal" flag because:
1. It's created in exactly ONE place (the 2 lines of first-open default logic) and consumed in exactly ONE place (the badge count).
2. The rule itself still evaluates identically — `applyCompoundFilter` doesn't care about the marker.
3. When the user opens the FilterModal and hits Apply, the `onApply` callback persists `draft.rules` — the draft is built from `initial.rules.map((r) => ({ ...r }))` at FilterModal.ts:94, which copies the marker. To clear the marker on user edit, either (a) strip it on modal Apply (simplest), or (b) strip it when the user explicitly interacts with the rule's value editor (more faithful; may be over-engineered). Recommend (a): on Apply, strip all `__autoDefault` markers so once the user acknowledges the filter, every rule counts as user-owned.
4. `isValidCompoundFilter` — the shape-guard at SettingsStore.ts:168 — doesn't reject unknown fields on a rule object; it only validates the fields it checks. So the marker survives a load/save round-trip without any shape-guard change.

### Pattern 4: Register `MarkdownPostProcessor` for language-tag alias

**What:** Item 9 — Obsidian renders fenced code blocks to `<pre><code class="language-{tag}">...</code></pre>`. Prism.js (Obsidian's highlighter — verified via `loadPrism()` in obsidian.d.ts:3744) reads the `language-*` class to pick its highlighter. Prism does NOT recognize `python3` as a language, only `python` and `py` [CITED: https://prismjs.com/#supported-languages]. Therefore, rewriting the class from `language-python3` to `language-python` on the rendered DOM makes Prism highlight the block.

**When to use:** Item 9 — this is the Phase 5.2 scope. Phase 5.3 will add CM6 language packs for Edit Mode; this post-processor is Reading-Mode-only and is orthogonal to the CM6 work.

**Example:**

```typescript
// NEW: src/main/python3Highlighter.ts

/**
 * Phase 5.2 Item 9 — python3 → python language-tag alias for rendered code blocks.
 *
 * Obsidian uses Prism.js for syntax highlighting (verified via `loadPrism` export
 * in obsidian.d.ts). Prism recognizes `python` and `py` but NOT `python3`
 * (https://prismjs.com/#supported-languages). LC's canonical langSlug for Python
 * is `python3`, which matches the fenced block tags our starter-code inserter
 * emits. Rewrite `language-python3` → `language-python` on the rendered <code>
 * elements so Prism picks the python highlighter.
 *
 * Edit-Mode CM6 highlighting is Phase 5.3 scope; this processor is Reading-Mode
 * only.
 */
export function registerPython3Highlighter(plugin: Plugin): void {
  plugin.registerMarkdownPostProcessor((element) => {
    const codes = element.querySelectorAll('code.language-python3');
    for (const code of Array.from(codes)) {
      code.classList.remove('language-python3');
      code.classList.add('language-python');
    }
  });
}
```

**Wiring (`src/main.ts` onload Step 6e area):**

```typescript

```

**Why NOT CM6 registry / loadPrism.languages.python3 = python:**
- CM6 language registry — that's Phase 5.3 scope; requires `@codemirror/lang-python` import and bundle growth.
- `loadPrism().then(p => p.languages.python3 = p.languages.python)` — mutates a shared Prism global at load time, which is async; a code block rendered before Prism resolves would still render unhighlighted. Class-swap-in-post-processor is synchronous with the render.

### Anti-Patterns to Avoid

- **Touching Phase 5.1's `codeActionsEditorExtension.ts`:** CONTEXT `<canonical_refs>` explicitly forbids changes here. The edit-mode Run/Submit button extension is out of scope — python3 highlighting must be implemented in a SEPARATE mechanism (MarkdownPostProcessor, not CM6 StateField).
- **Deleting `ConfirmOverwriteModal`:** D-11 explicitly keeps it for the Reset command. Only the Copy-to-Code call site changes.
- **Shape-guard rejecting unknown fields:** The current `isValidCompoundFilter` in SettingsStore.ts doesn't reject unknown fields on rule objects — it only validates the fields it needs. Item 4's `__autoDefault` marker rides on rules without triggering rejection. Do NOT tighten the shape-guard for this phase.
- **Auto-insert via `workspace.on('file-open')`:** The re-open retrofit ALREADY FIRES from `NoteWriter.openProblem` (confirmed at NoteWriter.ts:220, 290, 364, 395). Adding a parallel `workspace.on('file-open')` hook would double-fire the retrofit on the row-click path. Only add the hook if there's a gap where NoteWriter's retrofit doesn't run (see §Item 5 investigation — there IS such a gap for direct file-picker-open paths).
- **Migrating legacy `language` filter rules on load:** `isValidCompoundFilter` already drops unknown field rules silently (returns `false` for the whole filter). Existing user data.json entries with a `language` rule would be dropped entirely on load — which is fine, the filter persists without the unknown rule. Recommend: leave shape-guard as-is, which effectively means "drop silently" by rejecting the whole filter if a `language` rule is present. See §Item 2 investigation for the nuance.

### Pitfall 3: Shape-guard rejects legacy `language` filter rules and drops the entire filter

**What goes wrong:** Item 2 — a user who previously saved a filter containing a `language` rule will have their entire persisted filter silently dropped at load time, because `isValidCompoundFilter` returns `false` for any unrecognized field. They reopen Obsidian and see no filter active.

**Why it happens:** Shape-guard is whole-or-nothing: one bad rule → drop the whole filter.

**How to avoid:** CONTEXT D-02 explicitly sanctions this behavior ("graceful fallback: ignore the rule, don't crash"). A per-rule skip would be more graceful — loop through rules, keep only the ones that pass shape-guard, return the clean filter. Implementation at SettingsStore.ts:168:

```typescript
function isValidCompoundFilter(v: unknown): v is CompoundFilter {
  if (!v || typeof v !== 'object') return false;
  const f = v as Partial<CompoundFilter>;
  if (f.match !== 'all' && f.match !== 'any') return false;
  if (!Array.isArray(f.rules)) return false;
  return true;   // per-rule validity checked below at sanitize time
}

// And at load-time, ADD a sanitize pass:
function sanitizeCompoundFilter(f: CompoundFilter): CompoundFilter {
  return { match: f.match, rules: f.rules.filter(isValidFilterRule) };
}
```

This is Claude's discretion per D-02 — choose "skip unknown rules, keep known rules" vs "drop the whole filter". Recommend skip-unknown-rules (least user-hostile).

**Warning signs:** Returning dogfood users notice their saved filter (difficulty=Easy, status=attempted, etc.) disappeared after upgrading.

### Pitfall 5: Auto-insert on open path — NoteWriter's retrofit already fires; adding `workspace.on('file-open')` double-fires

**What goes wrong:** Item 5a — if a plan adds `workspace.on('file-open', (file) => retrofit(...))`, every row-click (which goes through `openProblem → retrofitStarterCode`) AND every other file-open invocation (picker modal, backlink click, random shuffle) fires retrofit. Idempotency should save us (injectCodeSection is idempotent when a recognized fence exists), but there's a tiny perf cost and a subtle risk: if the detail cache misses mid-flight, NoteWriter retries on the canonical-path recovery branch.

**Why it happens:** `NoteWriter.retrofitStarterCode` already fires at:
- NoteWriter.ts:220 — re-open path
- NoteWriter.ts:290 — recovered canonical-path (cache cleared but file exists)
- NoteWriter.ts:364 — new-note path (belt-and-suspenders)
- NoteWriter.ts:395 — backgroundRefresh path

**How to avoid:** DO NOT add `workspace.on('file-open')`. Instead, the D-06 auto-insert requirement is ALREADY SATISFIED by the existing retrofit wiring. The research for Item 5 reveals a subtler story:

1. Current state: on every row-click (`openProblem → NoteWriter.openProblem`), retrofit fires silently. If the user opens an `lc-slug` note by other means (daily-note backlink, file-explorer click, command palette "Open file"), NoteWriter.openProblem is NOT called — so retrofit does NOT fire.
2. D-06 says: "when a user opens an `lc-slug` note without a `## Code` section … auto-inserts the starter template silently on open." That's a SUPERSET of the current coverage.
3. CONTEXT D-06 sanctions `file-open` event. But it also sanctions "processFrontMatter on-demand" — a softer hook.

**Recommendation:** Add a `workspace.on('file-open')` hook that:
- Reads the file's frontmatter via `metadataCache.getFileCache`
- Gates on `lc-slug` being a valid string
- Calls `retrofitStarterCode(file, cachedDetail)` with the cached detail
- Swallows all errors silently (D-09 silent-on-failure posture)

Idempotency of `injectCodeSection` (already verified by `tests/solve/starterCodeInjector.test.ts`) makes double-firing safe on the row-click path — no user-visible effect.

**Warning signs:** Opening an existing lc-slug note via backlink does not get a starter block.

### Pitfall 6: `retrofitStarterCode` skips when both defaultLang AND starter snippets are missing

**What goes wrong:** Item 5a — `NoteWriter.retrofitStarterCode` (line 180) has a pre-guard: `if (!defaultLang && !hasAnyStarter) return;`. On a brand-new user who hasn't configured a default language AND the detail cache doesn't have snippets yet, the auto-insert is a no-op.

**Why it happens:** Empty language + empty starter → would write `\n\`\`\`\n\n\`\`\`\n` (blank fenced block, no tag). That's malformed — the pre-guard prevents it.

**How to avoid:** The default language is already `'python3'` on fresh install (SettingsStore.ts:104). So the guard condition falls back to: "skip only when the user explicitly cleared defaultLanguage AND the detail cache has no snippets." In practice this is exceedingly rare on a fresh install. The auto-insert will fire on the row-click path and insert at least a tagged empty fenced block.

**Warning signs:** Edge-case user report: "I cleared my default language to try another, opened a note via backlink, no starter block appeared." — known, acceptable per D-09.

### Pitfall 7: Prism doesn't highlight `python` blocks inside a reading-mode pane that hasn't triggered `loadPrism()` yet

**What goes wrong:** Item 9 — on first plugin load, if a note with a python code block renders before Obsidian has demand-loaded Prism, the block may briefly render unhighlighted. The class-swap fix is synchronous, but Prism itself is async-loaded.

**Why it happens:** `loadPrism()` is lazily invoked by Obsidian's renderer on first encounter. Before it resolves, `<code class="language-python">` is just text.

**How to avoid:** This is pre-existing behavior, not introduced by item 9. The class swap makes python3 blocks behave identically to python blocks — whatever Prism-load timing applies to `language-python` applies equally after the swap. No mitigation needed for phase 5.2.

**Warning signs:** Reading a freshly-opened note shows a flash of unhighlighted code on first load. Normal Obsidian behavior.

### Item 9: python3 → python MarkdownPostProcessor

```typescript
// NEW FILE: src/main/python3Highlighter.ts
import type { Plugin } from 'obsidian';

/**
 * Phase 5.2 Item 9 — python3 → python language-tag alias for Reading Mode
 * syntax highlighting. Obsidian uses Prism.js (verified via `loadPrism` in
 * obsidian.d.ts); Prism recognizes `python`/`py` but NOT `python3`
 * (https://prismjs.com/#supported-languages). Rewriting the class on the
 * rendered DOM is synchronous with the render and avoids the race that
 * `loadPrism().then(p => p.languages.python3 = p.languages.python)` would
 * introduce.
 *
 * Edit Mode (CM6) highlighting is Phase 5.3 scope (POLISH-09); this processor
 * only fixes Reading Mode. Do NOT touch codeActionsEditorExtension.ts.
 */
export function registerPython3Highlighter(plugin: Plugin): void {
  plugin.registerMarkdownPostProcessor((element) => {
    const codes = element.querySelectorAll('code.language-python3');
    for (const code of Array.from(codes)) {
      code.classList.remove('language-python3');
      code.classList.add('language-python');
    }
  });
}

// Wire in main.ts onload after registerCodeBlockActionProcessor:
registerPython3Highlighter(this);
```
