# Phase 5.2: Pre-ship UX Polish - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning
**Source:** Fast-path CONTEXT (user-supplied decision list during Phase 5.1 live-smoke feedback)

<domain>
## Phase Boundary

Phase 5.2 batches nine small UX fixes and polish items surfaced during Phase 5.1 live smoke into a single phase. Each item is independent; most are 1–2 file changes. Shipping them together before 05-07 avoids nine micro-phases and gives the 0.1.0 community-plugins release a coherent first-impression UX.

**In scope (locked by user during 05-11 feedback session):**

1. Settings layout: merge the Status row and Logout button into one row (logout button on the right side of the status block).
2. Problem filter: remove the `language` field from FilterModal's field-selection dropdown. Filter UI stays; language filtering is deferred to a future phase (no user-visible language filter).
3. Problem filter: `premium` field becomes multi-value — user can select any combination of `non-premium`, `premium` (or both) instead of a single-value radio.
4. Filter badge zero-count bug: `updateFilterBadge` currently shows "3" on a fresh install because it counts auto-populated default rules. Fix: either don't count auto-defaults, or only show the badge when the user has explicitly modified the filter. (Planner decides exact mechanism; acceptance: fresh-install vault shows no badge.)
5. Starter-code auto-insert + Reset command:
   - The `Insert starter code` command is removed from the command palette.
   - When a user opens an `lc-slug` note without a `## Code` section (or with an empty fence), the plugin auto-inserts the starter template silently on open.
   - A new `Reset code` command restores the starter template (overwrites existing `## Code` fence). Confirmation prompt sanctioned — this IS destructive. (Plan 05-05 removed the overwrite confirm for Run/Submit buttons; Reset is a different action and keeps the confirm.)
6. Past Submissions picker: AC chip and WA chip currently render as empty colored pills. Add visible text labels — "Accepted" for AC, "Wrong Answer" for WA (or whatever verdict slug LC returns — match existing status-map labels from Phase 3 Plan 04).
7. Copy-to-Code: remove the overwrite confirmation modal (`ConfirmOverwriteModal`). Copy-to-Code now overwrites the `## Code` fence silently, matching the Run/Submit click-through behavior shipped in Phase 5 Plan 05.
8. Default language list in Settings: the language dropdown ships with a curated list. Update that list to exactly match LC's official submission-language dropdown (user will confirm exact list during discuss-phase if planner's research surfaces ambiguity).
9. Python3 syntax highlighting: the `python3` language tag on fenced code blocks currently renders as plain text. Fix so `python3`-tagged blocks highlight identically to `python`-tagged blocks. (Implementation: probably a language-tag alias, or registering `python3` with Obsidian's syntax-highlight layer. Research phase decides.)

**Explicitly out of scope for Phase 5.2:**

- **Auto-indentation / IDE-grade editor features** — Phase 5.3 owns this. 5.2's python3 fix is a pure language-tag alias, not a CM6 language-pack integration.
- **Language filter in problem browser** — deferred; 5.2 just removes the field option.
- **Settings UX rework beyond item 1** — no new settings, no reorganization of other sections.
- **Localization / i18n** — English-only, like the rest of the plugin.
- **New commands beyond Reset code** — Reset code is the only new command this phase adds.

</domain>

<decisions>
## Implementation Decisions

### Settings (Item 1)

- **D-01:** Merge Status row and Logout button. The button moves to the right side of the existing `.leetcode-settings-status` block. No new Settings section. No settings rename. Likely a pure CSS + DOM tweak in SettingsTab.ts.

### Problem Filter (Items 2, 3, 4)

- **D-02:** Remove `language` from the FilterModal field-selector's options list. Existing code paths that check for `language` filter rules still need to handle pre-existing user filters (graceful fallback: ignore the rule, don't crash). Planner decides migration (drop silently, or migrate on load).
- **D-03:** `premium` filter field becomes multi-value. Use Obsidian's existing multi-select affordance (or a checkbox group) matching the pattern the `difficulty` field uses today. Values: `non-premium`, `premium`.
- **D-04:** Filter badge counts **user-modified** rules only. Rules auto-populated by first-open logic (the `premium: non-premium` default applied when `isPremium === false`) should NOT count. Implementation idea: set a `__autoDefault: true` marker on auto-applied rules and exclude them from `updateFilterBadge`'s count, OR only count rules after the user has explicitly opened FilterModal once. Researcher + planner decide.

### Starter Code (Item 5)

- **D-05:** `Insert starter code` command is removed. The command was added in Phase 2, but item 5's user insight is correct — users expect `## Code` to always exist on an LC note.
- **D-06:** Auto-insert trigger: when a user opens an `lc-slug` note (first time, or any time the `## Code` section is missing), the plugin appends the starter template silently. Likely hook: `file-open` event on MarkdownView, or processFrontMatter on-demand. Do NOT trigger on every cursor change — only on open/switch.
- **D-07:** `Reset code` command replaces `Insert starter code` in the command palette. When invoked on an `lc-slug` note, it overwrites the `## Code` fence with the starter template. KEEPS the existing ConfirmOverwriteModal pattern (unlike Copy-to-Code per D-11) because Reset is a destructive user-initiated action.
- **D-08:** Starter template source stays the same — LC's per-language starter code fetched from the problem detail. No change to the template itself.

### Past Submissions Picker (Item 6)

- **D-09:** AC chip: render with text "Accepted". WA chip: render with "Wrong Answer". Other verdicts (TLE, MLE, RE, CE): render with their existing status-map labels from Phase 3 Plan 04's `classifyStatus` output. The chip currently has a colored background but empty text — just add the label inside it. CSS may need minor tweaks for readability (padding, min-width).

### Copy-to-Code (Item 7)

- **D-10:** Remove `ConfirmOverwriteModal` from the Copy-to-Code click path. Copy-to-Code now overwrites the `## Code` fence silently. This matches Run/Submit click-through (both silent after Phase 5 Plan 05 retrofit).
- **D-11:** The `ConfirmOverwriteModal` class is NOT deleted from the codebase — it stays for `Reset code` command (per D-07). Only the Copy-to-Code call site is changed.

### Language List (Item 8)

- **D-12:** Settings default language list mirrors LC's official submission dropdown. Exact list locked during planning's research phase (planner fetches current LC language list and pins it). If LC's list changes in the future, this is a follow-up phase; we don't build a dynamic fetch for it.

### Python3 Highlighting (Item 9)

- **D-13:** `python3` language tag on fenced code blocks renders with Python syntax highlighting. Simplest implementation: alias `python3` → `python` via Obsidian's codemirror language registry or via a custom syntax-highlight preprocessor. NOT a CM6 language pack — Phase 5.3 owns that scope. Scope here is minimal: one tag alias so blocks don't render unhighlighted.

### Claude's Discretion

Planner has full latitude on:

- Exact mechanism for filter-badge fix (marker on auto-default rules vs. "has user opened FilterModal" flag vs. other — pick what's least invasive).
- Migration strategy for legacy `language` filter rules in user data.json (drop-silently vs. migrate).
- File splitting for the 9 items — may bundle them into 2–3 plans by touched-area (settings, filter, starter-code, picker, copy-path, language).
- Whether Python3 alias is implemented via Obsidian's `addExtension` on the markdown-post-processor side, a CM6 language registry entry, or a pure CSS + classname mapping. Whichever is least code and doesn't break Phase 5.1's edit-mode extension.

### Carried Forward (not re-asked)

- **CF-01:** All LC calls through `requestUrl` + `throttledRequestUrl`. No new network paths in 5.2.
- **CF-02:** `isDesktopOnly: true`. No new Electron APIs.
- **CF-05:** Zero Required `eslint-plugin-obsidianmd` violations. No `innerHTML`, no `fetch`, no default hotkeys on new commands.
- **CF-06:** `vault.process()` / `processFrontMatter()` only for note writes. Starter-code auto-insert (D-06) uses `vault.process()` to append the `## Code` section.
- **CF-07:** `createEl()` discipline for all DOM.
- **CF-11:** Feature-first folder layout. Changes stay within `src/settings/`, `src/browse/`, `src/solve/`, `src/graph/` as appropriate.
- **CF-13:** `lc-slug` frontmatter gates the affordance. Starter-code auto-insert (D-06) gates on `lc-slug`.

</decisions>
