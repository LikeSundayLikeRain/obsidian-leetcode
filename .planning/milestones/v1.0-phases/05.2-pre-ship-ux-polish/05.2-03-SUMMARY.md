# Phase 5.2 Plan 03: Filter UX Cluster (D-02 / D-03 / D-04) Summary

Three filter-UX fixes land together: removing the `Language` field from the FilterModal
dropdown (D-02), turning `premium` into a multi-value multi-select (D-03), and stopping
the filter badge from counting auto-applied first-open defaults (D-04). Ships the
FilterRule shape change + all four consumer sites in one coherent plan so the type
change and runtime behavior stay in lock-step.

## What Shipped

### D-02 — Remove Language field + graceful legacy drop

- `src/browse/FilterModal.ts`: `DEFERRED_STUB_FIELDS` is now an empty exported array
  (kept exported so Wave 0 test asserts its shape); the render-loop that called
  `renderDeferredStub(…)` and the `renderDeferredStub` method both deleted. The
  `Language` row no longer appears at the bottom of the filter modal.
- `src/settings/SettingsStore.ts`: `isValidCompoundFilter` now only validates the
  container (match + rules array). Per-rule validation lives in a new
  `isValidFilterRule` helper. Unknown field names (including legacy `'language'`)
  fail `isValidFilterRule`, and the new `sanitizeCompoundFilter` drops them silently
  when `loadSettings` runs. A legacy `data.json` holding a `language` rule loads
  without crashing — user simply sees one fewer filter rule on reload.

### D-03 — Premium field becomes multi-value multi-select

- `src/settings/SettingsStore.ts` FilterRule union: premium variant changed from
  `{ field:'premium', op:'is', value:'premium'|'non-premium'|null }` to
  `{ field:'premium', op:'is', values: string[] }`. Shape-guard accepts values that
  are arrays of `'premium'` or `'non-premium'` strings.
- `src/browse/FilterModal.ts`: `FIELD_DEFS` premium `blank()` now returns
  `{ field:'premium', op:'is', values: [] }`. The premium case in
  `renderValueEditor` routes through `renderMultiSelect(cell, rule, PREMIUM_OPTIONS)`
  — identical UX to status/difficulty/topics. The per-field `renderPremiumEditor`
  method was deleted entirely; `PREMIUM_OPTIONS` constant stays and is now the
  value list for the shared multi-select.
- `src/browse/ProblemListService.ts`: the `premium` case in `evaluateRule` is now
  multi-value semantics. `values=[]` returns `undefined` (no-op); otherwise the row's
  `'premium' | 'non-premium'` derived value must be present in `values`. Both-values
  selected is still effectively a no-op from the user's perspective.
- `ProblemListService.evaluateRule` is now an exported symbol so the Wave 0 shell
  can exercise it directly without standing up a full `CompoundFilter`.

### D-04 — Filter badge ignores auto-applied default rules

- `src/browse/ProblemBrowserView.ts` first-open default: the auto-applied
  `premium: non-premium` rule (applied when `isPremium === false`) now carries a
  `__autoDefault: true` marker on top of the multi-value shape. The marker is an
  extra property on the rule object; `isValidFilterRule` ignores unknown properties
  so it survives JSON round-trip through `data.json`.
- `updateFilterBadge` delegates to a new module-scope `computeFilterBadgeCount`
  helper that filters out rules carrying `__autoDefault`. Fresh-install vault:
  badge count is 0, DOM `is-visible` class not applied, user sees no badge.
- `FilterModal.stripAutoDefaults(rules)` helper is exported and invoked by the
  Apply-button click handler. Once the user opens the modal and hits Apply (even
  with no edits), the marker is stripped before the filter is persisted.
  Subsequent loads count the rule like any user-authored rule.

### Plan-guided behavior kept

- Plan's `grep -c "__autoDefault" == 2` criterion was aspirational; the test suite

  it outright, because the Wave 0 test explicitly probes `mod.DEFERRED_STUB_FIELDS`
  and accepts either `undefined` or `[]` — matching the test's declared acceptance
  set keeps both options viable.
