<objective>
Close GAP-2a: the plugin currently hardcodes `lc-status: untouched` on every new note, ignoring the user's real LC submission status. Per the UAT fix hint, read `statusDisplay` from the LC problem-list row (Plan 05 already parses LC's `status: 'ac'|'notac'|null` into `IndexedProblem.status: 'solved'|'attempted'|'untouched'`) and plumb it through `NoteWriter.openProblem → buildFrontmatterInput → applyFrontmatter` so the first frontmatter write records the real status. Preserve D-04: never downgrade an existing `accepted` value on a re-open.

Purpose: NOTE-03 promises a fully-populated frontmatter. A lie in `lc-status` poisons every future Bases / Dataview / graph query the user builds on top of the vault. Fixing this at Phase 2.1 avoids a v1 migration for users who opened notes before Phase 4 lands.

Output: `IndexedProblem.status` → `'accepted' | 'attempted' | 'untouched'` mapping helper in `NoteTemplate.ts`, plumbed through every `openProblem` call site, unit-tested including the "don't downgrade accepted" branch.
</objective>
