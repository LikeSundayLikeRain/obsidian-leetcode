### `src/main/languageChevronWidget.ts` (component, event-driven)

**Analog:** `src/main/codeBlockButtonRow.ts` (entire 36-line file).

**Why this analog:** Both modules are pure DOM builders that receive a `Document` + plugin reference, build elements via `doc.createElement` + `textContent` (CSP-safe; no `innerHTML`), and attach click handlers that call `e.preventDefault()` + `e.stopPropagation()` + invoke a plugin method. The chevron is structurally a "third button" in the same conceptual family.

**Imports pattern (from `codeBlockButtonRow.ts:1`):**
```typescript
import type { Plugin } from 'obsidian';
```
Chevron variant adds `TFile` and the new `languages.ts` constants:
```typescript
import type { Plugin, TFile } from 'obsidian';
import {
  LC_LANG_DISPLAY_LABELS,
  LC_CHEVRON_LANG_ORDER,
} from '../solve/languages';
import type { CodeBlockButtonRowHost } from './codeBlockButtonRow';
```

**Host-interface pattern (from `codeBlockButtonRow.ts:3-6`):**
```typescript
export interface CodeBlockButtonRowHost {
  runFromActive(): void | Promise<void>;
  submitFromActive(): void | Promise<void>;
}
```
Chevron extends with the new switch method:
```typescript
export interface LanguageChevronHost extends CodeBlockButtonRowHost {
  switchLanguage(file: TFile, newSlug: string): Promise<void>;
}
```

**Element-creation pattern (from `codeBlockButtonRow.ts:12-23`):**
```typescript
const row = doc.createElement('div');
row.className = 'leetcode-code-actions';

const runBtn = doc.createElement('button');
runBtn.className = 'leetcode-code-action-run';
runBtn.textContent = 'Run';
runBtn.addEventListener('click', (e) => {
  e.preventDefault();
  e.stopPropagation();
  void plugin.runFromActive();
});
row.appendChild(runBtn);
```
Chevron applies the same shape — `createElement` → set `className` → set `textContent` → `addEventListener('click', ...)` → `appendChild`. **Never `innerHTML`** (CLAUDE.md store-submission rule + RESEARCH `<system-reminder>` constraint).

**Click-handler discipline (load-bearing):**
- `e.preventDefault()` then `e.stopPropagation()` first — prevents CM6 selection bleed (Phase 5.1 RESEARCH Pitfall 3).
- `void plugin.method()` — discard the Promise so the listener stays sync-typed.

**Document selection (from `codeActionsEditorExtension.ts:62-66`):**
```typescript
toDOM(view: EditorView): HTMLElement {
  // Per RESEARCH Pitfall 10: use the editor's own Document (popout-window
  // safe) rather than the global `document`.
  return buildCodeBlockButtonRow(view.dom.ownerDocument, this.plugin);
}
```
Chevron MUST receive `view.dom.ownerDocument` from its caller — same popout-safety discipline. Never reach for global `document` (eslint rule `obsidianmd/prefer-active-doc`).

### `src/main/dispatchFenceRewrite.ts` (utility, transform — optional split)

**Analog:** `src/main/codeActionsEditorExtension.ts::findCodeFence` (lines 93–128).

**Why this analog:** Same shape — read `EditorState`, return a description object, no side effects on the helper function itself. The dispatch helper layers on `view.dispatch({ changes: [...] })` which has no codebase analog (this is the first multi-range atomic dispatch in the plugin).

**Reuse imperative (D-13 parity):** The helper MUST call `findCodeFence(view.state)` directly — do not re-implement fence detection. RESEARCH "Anti-Pattern: Hand-rolling fence detection" is explicit.

**State-read pattern (from `codeActionsEditorExtension.ts:106-127`):**
```typescript
for (let i = 1; i <= total; i++) {
  const text = state.doc.line(i).text;
  // … H2_CODE_RE / FENCE_RE matching …
  if (FENCE_RE.test(text)) {
    for (let j = i + 1; j <= total; j++) {
      if (FENCE_RE.test(state.doc.line(j).text)) {
        return { openerLine: i, closerLine: j };
      }
    }
    return null; // unterminated fence
  }
}
return null;
```
Lines are 1-indexed (CM6 convention). `state.doc.line(n).text` reads, `state.doc.line(n).from`/`.to` give offsets.

**Dispatch shape (NEW — no codebase analog; per RESEARCH Pattern 1):**
```typescript
view.dispatch({
  changes: [
    { from: opener.from, to: opener.to, insert: newOpenerText },
    { from: bodyStart, to: bodyEnd, insert: newStarterCode + '\n' },
  ],
  userEvent: 'leetcode.lang-switch',
});
```
Two ranges in one `dispatch` = one undo step. **No `setTimeout(0)` defer** — this runs from a click event, not from inside `ViewPlugin.update()`.

```typescript
// MarkdownView + EditorView imports (TFile is already imported elsewhere)
import { MarkdownView, type TFile } from 'obsidian';
// EditorView from CM6 — same eslint-disable transitive-peer comment as
// codeActionsEditorExtension.ts uses (line 33).
// eslint-disable-next-line import/no-extraneous-dependencies
import type { EditorView } from '@codemirror/view';

async switchFenceLanguage(file: TFile, newSlug: string): Promise<void> {
  const view = this.app.workspace.getActiveViewOfType(MarkdownView);
  if (!view || view.file !== file) return; // raced with leaf change

  // 1. Resolve starter code via existing cached client path.
  const slug = /* read lc-slug from frontmatter */;
  let snippet = '';
  try {
    const detail = await this.client.getProblemDetail(slug);
    snippet = detail?.codeSnippets?.find(s => s.langSlug === newSlug)?.code ?? '';
  } catch {
    // Network failure — keep current body (CONTEXT D-11 safer default).
    // Surface a single Notice (Pitfall 4) — copy parity with CF-04 / CF-19.
    new Notice(`Couldn't fetch starter code for ${LC_LANG_DISPLAY_LABELS[newSlug] ?? newSlug}.`);
    return;
  }

  // 2. CM6 dispatch FIRST — atomic fence rewrite (one undo step).
  const cm = view.editor.cm as EditorView; // Pitfall 6 — undocumented internal
  const ok = dispatchFenceRewrite(cm, newSlug, snippet);
  if (!ok) return;

  // 3. processFrontMatter SECOND — separate undo stack (Pitfall 1).
  await this.app.fileManager.processFrontMatter(file, (fm) => {
    fm['lc-language'] = newSlug;
  });
}
```

**Sequence rationale:** CM6 dispatch first (user-visible change is instant); `processFrontMatter` second (async vault write; would block visible rewrite by 5–20 ms otherwise). RESEARCH "Anti-Pattern: Calling `processFrontMatter` BEFORE `view.dispatch`" makes this load-bearing.

**Reusable existing patterns in main.ts:**
- `getProblemDetail` cache-then-network pattern is used by `NoteWriter.ts:509`, `starterCodeInjector.ts:220`, `resetCodeWithConfirm.ts:54` — `detail?.codeSnippets?.find(s => s.langSlug === targetSlug)?.code ?? ''`. Reuse verbatim.
