# Phase 5.3 — UI Design Contract (REPLAN)

> Visual and interaction contract for Phase 5.3 after the **2026-05-11 replan** that pivoted away from per-view CodeMirror `Compartment` LanguageSupport (empirically failed at D-14 live smoke — see CONTEXT historical note + RESEARCH "Historical: Why the original approach failed") to a **write-time fence-tag remap + language chevron dropdown**.
>
> The Phase 5.3 user-facing surface is now:
> 1. **Native markdown syntax highlighting** inside the `## Code` fence (Edit Mode AND Reading Mode) — achieved without registering any new language extensions, by ensuring the fence-opener tag is one Obsidian's bundled `lang-markdown` nested parser already recognizes (`python3 → python`, `golang → go`, `c → cpp`).
> 2. **A new language chevron dropdown** rendered LEFT of Phase 5.1's existing Run/Submit row in Edit Mode (Live Preview + Source). Clicking a different language **destructively replaces** the fence body with LC's canonical starter code for that language, rewrites the fence opener tag, and updates the `lc-language` frontmatter.
> 3. **Frontmatter cleanup:** drop never-read `lc-runtime-ms` and `lc-memory-mb` keys (D-01/D-02). `lc-language` is preserved as the canonical LC-API source of truth (D-03).
>
> Auto-indentation, bracket auto-close, and Lezer-grade indent rules — the original POLISH-09 ambition — are **deferred to Phase 6.x** per CONTEXT `<deferred>`. Phase 5.3 satisfies POLISH-09's "language support" half (recognized highlighting + language switching UX); the IDE-grade indent half is parked for a future phase with a 2–3 week budget.
>
> **Source of decisions:** CONTEXT.md D-01..D-13 + RESEARCH.md (chevron pattern + atomic dispatch) — both pre-lock the phase boundary, fence-tag remap table, dropdown UX model, layout direction, error posture, and revert plan. No user questions were asked — upstream artifacts fully cover this contract.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Obsidian plugin — not a React/Next/Vite app; shadcn not applicable) |
| Preset | not applicable |

## Spacing Scale

Phase 5.3 introduces no new spacing tokens. All spacing values reuse Phase 5.1's button-row scale, the 4-multiple discipline locked in CLAUDE.md, and Obsidian theme defaults.

| Token | Value | Usage in Phase 5.3 |
|-------|-------|---------------------|
| xs | 4px | Chevron-dropdown top margin (`margin-top: 4px` between chevron button and dropdown popover); button-row base margin-bottom; dropdown popover vertical inset (`padding: 4px 0`); dropdown popover `border-radius: 4px` (mirrors Phase 5.1 chevron button border-radius) |
| sm | 8px | Chevron horizontal padding (`padding: 0 8px` — mirrors `.leetcode-code-action-run`); button-row gap (`gap: 8px`); dropdown item vertical padding (`padding: 8px 12px` — both axes 4-multiples; vertical chosen to satisfy comfortable click-target height ~32px) |
| md | 12px | Dropdown item horizontal padding (right axis of `padding: 8px 12px`; 12 is a 4-multiple) |
| lg | 24px | Chevron + dropdown-item baseline height (`height: 24px` on chevron — mirrors `.leetcode-code-action-run`; dropdown item resolved height ~32px from 8px+8px padding + 12px line-height — also a 4-multiple) |

**Indent inside the fence:** Markdown's default fence-content indent applies once `lang-markdown` recognizes the fence tag. No per-language indent enforcement (deferred to Phase 6.x). Pressing Enter inherits the previous line's leading whitespace; `}` does NOT dedent. This is the LC-on-Obsidian baseline behavior and is accepted for v1.

**Exceptions: none.** All declared values are 4-multiples (4, 8, 12, 24, 32). Earlier draft proposed `padding: 6px 12px` on dropdown items and `border-radius: 6px` on the popover; both 6px values were normalized away during ui-checker pass — vertical padding bumped to 8px (roomier touch target, 4-multiple), border-radius dropped to 4px (matches chevron button radius for visual continuity).

The triangle and the label are space-separated; no other glyphs. |
| Dropdown item label | `{DisplayLabel}` (no triangle). One per supported LC language in `LC_CHEVRON_LANG_ORDER`: `Python`, `Java`, `C++`, `C`, `JavaScript`, `TypeScript`, `Go`, `Rust` — exactly 8 entries, in that order. |
| Primary CTA | not applicable — chevron is not a CTA; it's a picker. No primary action button is added by this phase. |
| Empty state heading | not applicable — chevron is rendered conditionally on the same gate as Phase 5.1's button row (lc-slug note + `## Code` fence). When the gate fails the chevron is absent, not empty. |
| Empty state body | not applicable |
| Error state — network failure on starter-code refetch | Notice copy: `Couldn't fetch starter code for {DisplayLabel}.` — single sentence, no period after `{DisplayLabel}` interpolation other than the trailing period shown. | Loading / pending state | **no spinner. No disabled chevron.** First language switch in a session pays a ~200–500 ms `LeetCodeClient.getProblemDetail` cache-miss latency; subsequent switches are instant (cache hit). RESEARCH Open Question 2 — trust the 7-day TTL; no progress indicator. |
| New commands | **zero** — no new command-palette entries. |
| New Notices (besides the one network-failure case above) | **zero** — no other Notices added. |
| New settings | **zero** — no new SettingsTab rows, toggles, or text inputs. |
| Tooltip / hover hint | **zero** — chevron button has no `title` attribute. The label `▼ Python` is self-evident; hovering does not surface additional copy. |

### Error surfacing rationale

- **Network failure on starter-code refetch:** Visible Notice (one of two phase-introduced strings). The fetch happens BEFORE the CM6 dispatch — when it fails, the user clicked the chevron and nothing visible happened to the fence. Without a Notice the chevron would feel broken (Pitfall 4). The Notice tells the user the operation failed and gestures at the cause (`Check your connection.` is implied but not added — keep copy short, single sentence).
- **`findCodeFence` returns null:** Silent no-op. The user is mid-edit and removed the closer fence — surfacing a Notice would be noise. The chevron resumes working when the fence is restored.
- **`view.editor.cm` reach-into-internals:** No copy needed — this is a runtime invariant, not a user-visible state. If Obsidian deprecates `editor.cm` in a future release, the chevron breaks loudly (TypeScript runtime error in console). Acceptable per Pitfall 6.

### Loading / pending posture (deliberate non-decision)

- **No spinner.** **No disabled chevron during fetch.** **No "Loading Java starter code..." placeholder.**
- Rationale:
  1. First switch latency is 200–500 ms; a spinner shown for sub-300 ms would itself be a spinner-flash UX regression.
  2. Subsequent switches are cache hits (instant).
  3. The dropdown closes immediately on selection (within the click handler tick); the user has tactile confirmation that their selection was registered. The fence rewrite arriving 200 ms later feels like one fluid action.
- **Contract:** If a future phase needs visible load-progress feedback (e.g., an ARIA live region announcing "Loading"), it MUST debounce to ≥500 ms before showing any indicator. Phase 5.3 ships no such surface.

### Carried-forward copy invariants

- **No `LeetCode:` command prefix** — no new commands in this phase, so N/A. The lint rule `eslint-plugin-obsidianmd/commands/no-leetcode-prefix` would catch any drift.
- **Sentence case** — the one new Notice (`Couldn't fetch starter code for {DisplayLabel}.`) is sentence case. Display labels (`Python`, `Java`, `C++`) are proper nouns (language names), not exempted from sentence case but rendered exactly as LC renders them in its own language picker. `JavaScript` / `TypeScript` are camel-cased proper nouns.
- **Notice copy locked (CF-19)** — the one new Notice copy is locked here. No paraphrasing in implementation.

### Why the chevron is Edit-Mode only (D-09 lock)

Reading Mode is a render-only view. Switching language there has no semantic meaning: the user cannot edit the rendered code, and Run/Submit dispatch the active note's current code unchanged. Adding a chevron in Reading Mode would either (a) be a no-op visual affordance (confusing) or (b) require Reading Mode to mutate the underlying file — which violates the read-only mental model. **Phase 5.1's "Edit + Reading parity" precedent breaks here only for the chevron** — Run/Submit still appear in both modes; chevron is Edit-only.

### Gate: when the chevron renders

The chevron renders **if and only if** Phase 5.1's button row renders — same gate, same source-of-truth (`buildDecorations` in `codeActionsEditorExtension.ts:146-189`). Specifically:

1. The active note has non-empty `lc-slug` frontmatter (verified via `state.field(editorInfoField)?.file` + `app.metadataCache.getFileCache(file)?.frontmatter['lc-slug']`).
2. The note body contains an `## Code` H2 heading followed (before the next H2) by a fenced code block with an opener and closer (`findCodeFence`).
3. The mode is Edit (Live Preview or Source) — the StateField is only registered for editor extensions; Reading Mode goes through the post-processor path which does not pass `opts.prefix`.

All three TRUE → chevron + Run + Submit render together. ANY false → no chevron, no Run, no Submit (parity with 5.1).

### Chevron click → dropdown open

- Single left-click on chevron button → dropdown popover toggles visible/hidden.
- `aria-expanded` toggles `"true"` / `"false"`.
- Outside-click anywhere else → dropdown closes (handler is added on chevron-button mount, removed on dropdown close — keep simple per RESEARCH).
- Re-clicking the chevron while dropdown is open → dropdown closes (toggle behavior).
- Esc key → dropdown closes (RESEARCH Open Question 4 — basic keyboard support is in scope).

### Dropdown item click → language switch

When the user clicks a dropdown item:

1. Dropdown closes (`display: none`, `aria-expanded="false"`) — instant.
2. **If the clicked item is the currently-selected language:** no-op. No network call, no dispatch, no Notice.
3. **Otherwise** (different language picked):
   - **Step A — Fetch starter code.** Call `plugin.client.getProblemDetail(slug)` (cache-then-network; existing `LeetCodeClient` path with 7-day TTL). Pull `codeSnippets.find(s => s.langSlug === newSlug)?.code`.
     - **On rejection (network failure):** Show Notice `Couldn't fetch starter code for {DisplayLabel}.` Return early — do NOT mutate fence or frontmatter. (CONTEXT D-11 + Pitfall 4.)
     - **On empty snippet (LC has no code for that language):** treat the same as success with empty body — fence body becomes empty (rare; LC consistently returns snippets for all 8 supported languages).
   - **Step B — Atomic CM6 dispatch.** Single `view.dispatch({ changes: [...] })` carrying TWO ranges:
     - Replace the opener-line text with `` ```{newFenceTag} `` where `newFenceTag = lcSlugToFenceTag(newSlug)`.
     - Replace the fence body (line-after-opener through start-of-closer) with the fetched starter code + `\n`.
     - Annotate `userEvent: 'leetcode.lang-switch'`. **One transaction = one editor undo step.** Cmd-Z reverts both fence opener AND body atomically.
   - **Step C — Frontmatter write.** `await app.fileManager.processFrontMatter(file, fm => { fm['lc-language'] = newSlug; })`. Lands on Obsidian's vault undo stack — separate from CM6's editor undo (Pitfall 1; accepted divergence).
   - **Order is load-bearing:** Step B (CM6 dispatch) BEFORE Step C (processFrontMatter). Reasoning in RESEARCH Pattern 4: Step C is async and would delay user-visible fence rewrite by 5–20 ms. Doing C first creates a window where `lc-language` says Java but fence still has Python — Run during that window dispatches mismatched language to LC.
4. Chevron label refreshes via Phase 5.1's StateField rebuild on docChanged — the next render reads `lc-language` from `metadataCache` and shows `▼ Java`.

### Coexistence with Phase 5.1 Run/Submit buttons

- **Layout:** chevron (LEFT) + spacer + Run + Submit (RIGHT) in the same `.leetcode-code-actions` flex row.
- **Phase 5.1 widget reuse:** `CodeActionsWidget.eq()` continues to return `true` for same-plugin instances — CM6 reuses DOM across transactions, no flicker on the button row when the StateField rebuilds. - **Untouched.** `src/main/python3Highlighter.ts` continues to alias Reading-Mode `python3` fences to Prism's `python` grammar.
- **Synergy:** When `NoteTemplate` emits `python` instead of `python3` at write time (Phase 5.3 D-04), the Reading-Mode alias becomes redundant for new notes — but it stays in place because (a) existing user notes still have `python3` fences pre-replan, (b) the chevron's destructive-replace overwrites only on user action, (c) keeping the alias avoids a regression for users who haven't switched their notes.

### Coexistence with Phase 5 (Reading-Mode buttons + post-processor)

- **Untouched** beyond `codeBlockButtonRow.ts` signature widening (third-arg optional; Reading-Mode caller passes only two args, fully backward-compatible).
- Reading-Mode Run + Submit buttons continue to render via `codeActionsPostProcessor.ts`.

### Idempotency

- **One chevron per fence.** Phase 5.1's StateField paints exactly one widget per `## Code` fence (gate enforces); the widget's prefix factory builds exactly one chevron.
- **Multi-view:** Multiple editor views of the same file have independent StateFields; each renders its own chevron. They read the same `lc-language` from `metadataCache`; updates flow to all chevrons via the metadata-cache change → StateField rebuild path.
- **Rapid clicks:** Successive chevron clicks toggle the dropdown open/closed (no debounce needed — happens in the click-event tick).
- **Rapid language switches:** Each switch is independent. If the user clicks Python → wait → click Java mid-fetch, the in-flight Python `getProblemDetail` resolves and is discarded by the time the Java handler has its own resolution. (No request-cancellation logic — accepted because (a) cache makes most calls free, (b) the LeetCodeClient throttle layer handles concurrent calls gracefully.)

## Bundle-Size Gate (ship-affecting constraint)

Bundle size is locked as a phase contract because it materially affects user experience (plugin install download size, Obsidian load time). Documented here for completeness:

| Gate | Limit | Enforcement |
|------|-------|-------------|
| `main.js` post-build size — Phase 5.3 cap | **250 KB hard / 200 KB warn** (RECOMMENDED — RESEARCH Q7; planner may overrule) | `scripts/check-bundle-size.sh` exit 0 (hard) / warning message (soft) after every `npm run build` AND as part of `prerelease-check.sh` Gate 12 |
| Pre-replan bundle | 520 KB (`main.js` with the 6 `@codemirror/lang-*` packs bundled) | Subject to the OLD 700/600 cap — passed by 180 KB headroom, but the cap was sized for the failed approach. |
| Post-replan bundle | ~155 KB target (149 KB pre-Phase-5.3 baseline + ~5 KB chevron + remap) | Validated by Wave 1 build + `bash scripts/check-bundle-size.sh`. |
| Escalation order if tripped | (a) inspect chevron CSS for redundant rules; (b) audit the dropdown DOM for over-construction; (c) consider raising cap by 25 KB increments with explicit ROADMAP justification | Documented in CONTEXT §D-13 lineage |

**Rationale for tightening the cap:** Post-revert bundle is ~155 KB; leaving the cap at 700 KB would mean any future regression up to 4.5× the current size would silently pass — the cap ceases to be a regression gate. 250 KB hard / 200 KB warn gives ~95 KB of headroom for Phase 5.3 polish + future minor phases while still catching ≥60% bloat regressions.

## Out of Scope (documented to prevent drift)

The following are **explicitly NOT** part of Phase 5.3 (replan)'s design contract:

- **Per-fence CodeMirror `LanguageSupport`** (full IDE-grade indent + brackets + Lezer highlight). Deferred to Phase 6.x — see CONTEXT `<deferred>` Path A (mutate Obsidian internal `codeLanguages` map) or Path B (nested EditorView per fence).
- **Auto-indentation rules per language** (Python `:` bumps in, C-family `{` bumps in, etc.). Markdown default applies once tag is recognized; no per-language indent enforcement.
- **Bracket auto-close** (`(`, `[`, `{`, `"`, `'`, etc.). Same — markdown default inside the fence; no `closeBrackets()` extension installed.
- **C# language support** (chevron entry or highlighting). LC slug `csharp` falls through to "unsupported" branch — fence renders plain monospace; chevron does not list C# in the dropdown. CONTEXT D-04 + Open Question 1 — out for v1.
- **Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/PostgreSQL/MSSQL/Oracle SQL, Elixir, Erlang, Racket.** All fall through to the "unsupported" branch — fence verbatim, no chevron entry.
- **Edit-Mode ↔ Reading-Mode highlighting theme parity.** Token colors come from two different highlighting engines (lang-markdown vs Prism) — minor token-color drift is ACCEPTED. No cross-mode color reconciliation.
- **Confirmation modal before destructive language switch.** D-08 LOCKS no-confirm. Cmd-Z is the revert affordance.
- **Reading-Mode chevron.** D-09 LOCKS Edit-Mode-only.
- **Per-note "favorite languages" preference.** Dropdown lists all 8 supported in fixed `LC_CHEVRON_LANG_ORDER`; no per-user reordering.
- **Loading spinner / progress indicator on chevron switch.** First switch ~200–500 ms latency is accepted as instant-feeling; subsequent switches are cache hits.
- **Visible "current language" indicator on the fence itself** (e.g., a small badge inside the code block). The chevron label IS the current-language indicator.
- **Mutating Phase 5.1's `codeActionsEditorExtension.ts` widget shape, Phase 5.2's `python3Highlighter.ts`, or Phase 5 reading-mode buttons.** Phase 5.1's `toDOM` is the only edit (passes prefix factory). Everything else is purely additive.
- **Bumping `manifest.json` `minAppVersion`.** Stays at `1.10.0`; Phase 5.3 (replan) does not move it.
- **Unifying editor + frontmatter undo into one Cmd-Z.** Public Obsidian APIs do not bridge the two stacks. Pitfall 1 — accepted divergence.
- **Custom dropdown animation / transition.** Instant show/hide.
- **Persistent dropdown state across editor focus changes** (mode flip, leaf change). Closes on any focus-change event — RESEARCH Open Question 3.
- **Status indicator showing which language Run/Submit will use.** The chevron label IS that indicator. No separate status text.
- **Fence-tag remap for `## Solution` fences** (Phase 4). Out of scope — chevron and remap apply only to the `## Code` fence (Phase 5.1 `findCodeFence` returns ONLY `## Code`).
- **Migrating existing user notes' fence tags from `python3` → `python`.** Existing notes are not retroactively rewritten. New notes (Phase 2's `NoteTemplate`) emit remapped tags; chevron switches rewrite them. Old notes continue to render with Phase 5.2's Prism alias — no broken state.
