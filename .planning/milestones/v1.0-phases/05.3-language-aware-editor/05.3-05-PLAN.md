The current item click handler (line 116) calls `e.preventDefault()` +
`e.stopPropagation()` on the `click` event, but CM6's caret-positioning
runs on `pointerdown` (or `mousedown`), which fires BEFORE `click`. The
caret has already moved by the time the click handler runs.

Fix: add a `pointerdown` listener on the `wrapper` element (parent of both
button and dropdown) that calls `e.preventDefault()` + `e.stopPropagation()`.
Place this OUTSIDE the existing per-item handlers so it covers the entire
chevron+dropdown surface. Use `pointerdown` (covers mouse + touch + pen);
no separate `mousedown` listener needed — Obsidian/CodeMirror v6 dispatches
pointerdown first.

Reference the gap IDs in code comments: `// C6 — Esc dismissal` and
`// G-CLICK-THROUGH — pointerdown stopPropagation`.

  <files>src/main/codeActionsEditorExtension.ts, tests/main/codeActionsEditorExtension.test.ts</files>
  <action>
The chevron label lags one click because the StateField in
`buildCodeActionsEditorExtension` (lines 263-283 of
`codeActionsEditorExtension.ts`) only rebuilds the DecorationSet when
`tr.docChanged` is true OR `editorLivePreviewField` flips. The
`switchFenceLanguage` flow in `src/main.ts:738` does:

  1. cm.dispatch({ changes: [...] })  → docChanged=true → rebuild fires
     BUT the rebuild reads `lc-language` from `metadataCache` BEFORE step C
     (processFrontMatter) lands.
  2. await app.fileManager.processFrontMatter(file, fmObj => fmObj['lc-language'] = newSlug)
     → vault write → metadataCache eventually updates → NO new transaction
     → no rebuild → chevron stays on old label.

The next user-typing transaction triggers a docChanged → rebuild, finally
showing the new label — that's the "one click stale" symptom.

**Fix:** Subscribe to `metadataCache.on('changed', file => ...)` from
`buildCodeActionsEditorExtension` and force a no-op `cm.dispatch({})` (no
changes, no effects) on the active EditorView when `file` matches the
StateField's current file. The empty dispatch is a transaction with
docChanged=false, so we MUST also widen the StateField's `update` predicate
to include "frontmatter changed" — pass a custom StateEffect.

Concretely (planner picks one of the two equivalent paths; lock the choice
in the implementation):

PATH A (preferred — minimal surface): Define a new `StateEffect<void>` —
`languageRefreshEffect` — exported from `codeActionsEditorExtension.ts`.
Have the StateField's `update(old, tr)` call `buildDecorations` whenever
`tr.effects.some(e => e.is(languageRefreshEffect))` is true (in addition
to the existing `tr.docChanged || modeFlipped` predicate). Then in
`buildCodeActionsEditorExtension(plugin)`, register a
`plugin.registerEvent(plugin.app.metadataCache.on('changed', file => { ... }))`
that finds the active editor view via `plugin.app.workspace.getActiveViewOfType(MarkdownView)`,
checks `view?.file === file`, and dispatches
`(view.editor as unknown as { cm: EditorView }).cm.dispatch({ effects: languageRefreshEffect.of(undefined) })`.
Wrap in a try/catch — if the editor isn't a MarkdownView (e.g., during
plugin teardown) silently skip.

PATH B (alternative — caller-driven): Have `switchFenceLanguage`
in `src/main.ts` call `cm.dispatch({ effects: languageRefreshEffect.of(undefined) })`
directly AFTER `processFrontMatter` resolves. Cleaner causality but couples
the editor extension to main.ts. Acceptable if the implementer prefers it.

**Implementer chooses PATH A** unless they discover a concrete reason
PATH B is safer; document the choice in a code comment that begins
`// G-LABEL-LAG fix:` and references the two paths.

Either path: import `StateEffect` from `@codemirror/state` (already imported
adjacent to `StateField`); the existing `eslint-disable-next-line
import/no-extraneous-dependencies -- transitive peer of obsidian; external in esbuild`
comment pattern at line 31 covers the new import.

If PATH A: add `MarkdownView` to the `obsidian` import block; the
`MarkdownView` value (not just type) must be available at runtime since
`getActiveViewOfType` is called with it as the constructor argument.

**Tests:** Add to `tests/main/codeActionsEditorExtension.test.ts` a new

(see `buildDecorations` lines 229-243): `state.doc.line(fence.closerLine + 1).from`

guides reflect the indent of THAT line's content. When the user types `Tab`

widget right.

**Fix:** Anchor the widget at the END of the closer line (the line containing
the closing `` ``` ``) with `side: 1`. That position is on the closer-fence
line itself — its indent is the fence's own indent (always 0 for a top-level
fence; the only time it has indent is in nested-list fences which are not
in scope for `## Code` blocks). The widget then renders BELOW the closer
fence regardless of the next line's content.

Concretely, in `buildDecorations` (lines 229-243):

REPLACE the `hasLineAfterCloser` ternary block with a single anchor:
- `const anchor = state.doc.line(fence.closerLine).to;`
- `const side = 1;`

REMOVE the `hasLineAfterCloser` variable and the `if/else` for fallback
(this was the end-of-doc fallback case — `line(fence.closerLine).to` works
in both cases now since we no longer index into the post-closer line).

Update the comment block above the anchor logic to read:
`// G-LAYOUT fix (gap-closure 05.3-05): anchor at end of closer-fence line
