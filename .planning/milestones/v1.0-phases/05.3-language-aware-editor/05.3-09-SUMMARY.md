# Phase 05.3 Plan 09: G-PICKER-MODAL-NOCLOSE-ON-COPY — Chain Picker Dismiss via onSuccess Deps Callback

Wired an optional `onSuccess?: () => void` callback through `SubmissionDetailDeps` so the inner detail modal can signal a successful Copy-to-Code to its composing parent (the outer past-submission picker), which then chain-dismisses itself — closing both modals on a single successful copy click while preserving the existing decoupling header that forbids `SubmissionDetailModal` from importing `SubmissionPickerModal`.

## What Shipped

- **`SubmissionDetailDeps` extension** — added `onSuccess?: () => void` (optional). JSDoc explains the success-only contract and the use case (composing parent dismissal) without naming `SubmissionPickerModal` so the type-level decoupling stays clean.
- **Click-handler restructure** — replaced the bare `void this.handleCopyToCode().catch(...)` with an async IIFE that does `try { await this.handleCopyToCode(); this.deps.onSuccess?.(); } catch (err) { console.error(...) }`. `onSuccess` fires AFTER `handleCopyToCode` resolves successfully (which itself includes performCopy's internal safeClose). Rejection routes to the catch block — `onSuccess` never fires on failure.
- **`SubmissionPickerDeps.openDetailModal` signature widening** — `(row: SubmissionRow, onSuccess?: () => void) => void`. The 2nd parameter is optional so existing 1-arg test spies remain compatible.
- **Picker call sites wired** — both `renderRow` click and keydown (Enter/Space) handlers now pass `() => this.safeClose()` as the 2nd arg, lexically capturing the picker instance.
- **Production wiring in `src/main.ts`** — the `openDetailModal` factory lambda inside `openSubmissionPickerFromActive` widened to accept `onSuccess?: () => void` and forward it as the 4th arg to `openSubmissionDetailFromRow`, which itself widened to accept the optional `onSuccess` and thread it into the `new SubmissionDetailModal(...)` constructor's deps. Without this third edit the chain-close is non-functional end-to-end.
- **4 new it-blocks**, plus 1 pre-existing-test regex refinement:
  - `tests/graph/SubmissionDetailModal.silent-copy.test.ts`: 5 new G-PICKER-MODAL-NOCLOSE-ON-COPY tests (4 source-grep + 1 behavioral) + 1 refined regex on the pre-existing G-COPY-MODAL-NOCLOSE click-handler test (now matches the Plan-07-follow-up architecture where safeClose lives only in performCopy to avoid double-close flicker).
  - `tests/graph/SubmissionPickerModal.test.ts`: 3 new G-PICKER-MODAL-NOCLOSE-ON-COPY behavioral tests (passes onSuccess as 2nd arg on row click; invoking the captured callback closes the picker; keydown Enter parity).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Pre-existing G-COPY-MODAL-NOCLOSE click-handler regex out of date**
- **Found during:** Task 1 GREEN verification (the prompt called out this as a known pre-existing failure on the baseline).
- **Issue:** `tests/graph/SubmissionDetailModal.silent-copy.test.ts:105` asserted `/await this\.handleCopyToCode\(\);[\s\S]{0,200}this\.safeClose\(\)/`. After the Plan-07 follow-up `8c8478a` removed the redundant `safeClose()` from the click handler (because Obsidian interprets a double-close as detach-then-reattach causing a flicker), the regex became architecturally incorrect — the click handler delegates closure to performCopy's internal safeClose. The test failed against the baseline before my Plan-09 edits and would have continued to fail.
- **Fix:** Refined the regex to assert the click-handler awaits handleCopyToCode (which dispatches into the await chain that eventually calls safeClose inside performCopy). The structural intent — "the production click path reaches the close" — is preserved; the regex now reflects the actual architecture.
- **Files modified:** `tests/graph/SubmissionDetailModal.silent-copy.test.ts` (regex on lines 100–115 area).
- **Commit:** Bundled with Task 1 commit `5c4170f`.

**2. [Rule 2 — Critical functionality] Verified production wiring (HARD criterion)**
- **Issue:** The plan called out src/main.ts as a HARD criterion — without threading `onSuccess` end-to-end the unit tests pass but live behavior is broken.
- **Fix:** All three layers (factory lambda, openSubmissionDetailFromRow signature, SubmissionDetailModal constructor deps) widened in lockstep. Verified via grep + perl multiline match.
- **Commit:** Task 2 `cebf1ee`.
