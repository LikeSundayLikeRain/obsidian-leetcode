# Phase 5.3: Language-Aware Editor — Context

**Gathered:** 2026-05-11 (initial)
**Updated:** 2026-05-11 (replan after live-smoke D-14 failure)
**Status:** Ready for research + planning (chevron approach)

> **HISTORICAL NOTE — empirical pivot:** This file was substantially rewritten after the live-smoke checkpoint (D-14) revealed that the original D-10/D-11 approach (per-view CM6 Compartment swap to install `@codemirror/lang-*` packs) **dispatches successfully but does not take effect inside markdown fences**. Obsidian's `lang-markdown` nested-parser owns the fence sub-tree and routes language data per syntax-tree node, not per outer compartment. Replacing the outer compartment with `python()` would need to tokenize the whole document as Python (wrong); applying it to the fence sub-region requires either mutating Obsidian's internal `codeLanguages` map (not publicly accessible) or mounting a nested EditorView per fence (2–3 weeks of bidirectional sync, cursor bridging, undo merging — too expensive for this milestone).
>
> The pivot: **make markdown's existing nested-parser do the work** by ensuring fence tags are markdown-recognized (write-time remap `python3 → python`, `golang → go`), and **add a chevron dropdown** that switches the LC language while keeping the fence tag markdown-recognizable. The 6 `@codemirror/lang-*` packs, `languagePackRegistry.ts`, `whitespaceCopyIndent.ts`, and `codeFenceLanguageExtension.ts` become deletable. Bundle drops 520 KB → ~149 KB.
>
> See **Deferred Ideas** for the per-fence-LanguageSupport work (full IDE-grade indent + bracket auto-close); this is parked for a future Phase 6.x with a proper 2–3 week budget.

<domain>
## Phase Boundary

Phase 5.3 makes Edit Mode of `## Code` fences on `lc-slug` notes **render with native markdown syntax highlighting for the LC language**, by ensuring fence tags are markdown-recognized at write time. It also surfaces a **language-chevron dropdown** inline with Phase 5.1's Run/Submit buttons that switches the LC language for the active note (writes `lc-language` frontmatter, refetches LC's starter code via GraphQL, replaces the fence body with the new language's template, and remaps the fence tag if needed).

Phase 5.2 D-13 shipped `python3 → python` Prism alias for **Reading Mode**. Phase 5.3 ships the same remap **at write time** so Obsidian's existing `lang-markdown` nested-parser handles Edit-Mode highlighting natively. No new CM6 language extensions are registered.

**In scope (locked by ROADMAP.md §Phase 5.3 success criteria + this discussion D-01..D-12):**

1. `## Code` fence content in Edit Mode (Live Preview + Source) renders with native markdown syntax highlighting for any LC language whose fence tag is markdown-recognized (after write-time remap).
2. Language chevron `[▼ Python]` appears inline with Phase 5.1's Run/Submit buttons (left-aligned, becomes a 3-element button row with chevron on the LEFT, Run/Submit on the right).
3. Chevron is **always visible** on lc-slug notes that have a `## Code` fence — same gating as Phase 5.1's button row (Phase 5.1 parity).
4. Chevron is **Edit-Mode only** — does NOT appear in Reading Mode (D-09 — Reading Mode is a render-only view; switching language there is meaningless).
5. Clicking the chevron opens a dropdown of the 8 supported LC languages; selecting one:
   - **Replaces the `## Code` fence body destructively** with LC's canonical starter code for the selected language (one-shot Cmd-Z reverts).
   - **Refetches starter code from LC's GraphQL `questionEditorData` endpoint** (same endpoint Phase 2 uses).
   - **Writes `lc-language: <slug>` frontmatter** via `app.fileManager.processFrontMatter`.
   - **Rewrites the fence opener tag** to the markdown-recognized form (e.g., `python3` LC slug → `python` fence tag; `golang` → `go`).
6. Frontmatter cleanup: drop `lc-runtime-ms` and `lc-memory-mb` (write-only, never read; display fetches fresh from LC GraphQL). Keep `lc-language` (now sole source of truth for Run/Submit API + chevron's persistent state).
7. Reverts the failed D-10/D-11 implementation: deletes `src/main/codeFenceLanguageExtension.ts`, `src/main/languagePackRegistry.ts`, `src/main/whitespaceCopyIndent.ts`, their tests, and uninstalls the 6 `@codemirror/lang-*` packs. Bundle returns from 520 KB → ~149 KB.
8. No regression in Phase 5.1 Run/Submit (Edit + Reading), Phase 5 Reading-Mode buttons, or Phase 5.2's `python3Highlighter` Prism alias.

**Explicitly out of scope for Phase 5.3:**

- **Per-fence LanguageSupport (full CM6 indent + brackets + Lezer highlighting)** — moved to Deferred Ideas. Requires either Obsidian internal API access (not public) or nested EditorView per fence (2–3 weeks). Phase 6.x territory.
- **Auto-indentation rules** — markdown's default indent applies inside fences once the tag is recognized; no per-language indent enforcement. Pressing Enter inherits previous line's whitespace; `}` does NOT dedent automatically.
- **Bracket auto-close** — same: relies on whatever `lang-markdown` provides for fence content (typically markdown defaults). No `closeBrackets()` extension installed.
- **Confirmation prompt before destructive switch** — D-04: chevron is destructive by design; one-shot Cmd-Z reverts. No confirm dialog.
- **Reading-Mode chevron** — D-09: chevron is Edit-Mode only. Reading Mode is render-only; language switching there has no meaning.
- **Caching starter code per language at note creation** — D-11: refetch on switch is fine; one network call per click is acceptable. Cache is implementation detail (planner's choice).
- **Reverse-mapping fence tag → LC slug** — D-12: with `lc-language` frontmatter as source of truth, plugin reads frontmatter, not fence tag, for the LC API call. Fence tag becomes display-only after the switch.

<decisions>
## Implementation Decisions

### Frontmatter Cleanup (D-01..D-03)

- **D-01:** **Drop `lc-runtime-ms` frontmatter.** Write-only, never read. `SubmissionDetailModal.ts` displays runtime from a fresh GraphQL fetch (`runtimeDisplay`), not frontmatter. Removing eliminates stale-data risk. `NoteTemplate.ts:356` (apply) and 358 (clear-on-N/A) writes are deleted.

- **D-02:** **Drop `lc-memory-mb` frontmatter.** Identical to D-01 — write-only, never read, display path uses fresh GraphQL. `NoteTemplate.ts:361` (apply) and 363 (clear-on-N/A) writes are deleted.

- **D-03:** **Keep `lc-language` frontmatter as the canonical LC-language source.** Read at note open by `NoteWriter` (already falls back to `settings.getDefaultLanguage()` if missing) and by Run/Submit dispatch. Chevron writes here on switch. Removing would require either reverse-mapping fence tags back to LC slugs (lossy: `cpp` could be `c` or `cpp`; `python` could be `python` or `python3`) or a parallel side-channel storage. Keeping it is the cheapest correct path.

### Fence Tag Remap Table (D-04, D-05)

- **D-04:** **Write-time fence-tag remap is the entire highlighting solution.** Two LC slugs need remapping for markdown-recognized fence tags; everything else works natively:

  | LC slug | Fence tag | Why |
  |---------|-----------|-----|
  | `python3` | `python` | Empirical: live-smoke confirmed Prism/Lezer don't recognize `python3`; Phase 5.2 already aliases `python3 → python` for Reading Mode |
  | `golang` | `go` | LC uses `golang`; markdown/Prism/Lezer all expect `go` |
  | `python` | `python` | Native |
  | `java` | `java` | Native |
  | `cpp` | `cpp` | Native |
  | `c` | `cpp` | Shared parser; map to `cpp` so highlighting works (LC uses `c` slug for plain C) |
  | `javascript` | `javascript` | Native |
  | `typescript` | `typescript` | Native |
  | `rust` | `rust` | Native |

  **For unsupported LC languages** (csharp, kotlin, ruby, sql variants, swift, scala, php, dart, elixir, erlang, racket): write the fence tag as the LC slug as-is (e.g., ` ```csharp `). The fence will render plain monospace (no highlighting) — same UX outcome as today's "no Edit-Mode highlight" baseline. No fallback markdown tag.

- **D-05:** **Chevron is destructive on switch (D-08 below) and uses `app.fileManager.processFrontMatter` for the frontmatter write + a CM6 dispatch for the fence tag/body rewrite.** Single-transaction so Cmd-Z reverts the whole switch atomically (frontmatter + fence body) — not two separate undo steps.

### Chevron Widget (D-06..D-10)

- **D-06:** **Chevron lives inline with Phase 5.1's Run/Submit button row, LEFT-aligned.** Layout: `[▼ Python]                                  [Run] [Submit]`. The chevron and the action buttons share the same flex row but the chevron has `margin-right: auto` (or the row uses `space-between`) so chevron clings to the left edge, Run/Submit cling to the right. Reuses `.leetcode-code-actions` flex container; new class `.leetcode-language-chevron` mirrors `.leetcode-code-action-*` button styling (24px height, 12px text, secondary background, hover uplift).

- **D-07:** **Always visible** (Phase 5.1 parity). Chevron renders on every lc-slug note that has a `## Code` fence regardless of cursor position. Same gating logic as Phase 5.1's button widget — if the buttons render, the chevron renders.

- **D-08:** **Destructive replace on switch.** Selecting a different language from the chevron immediately replaces the fence body with LC's starter code for that language. No confirm dialog. Cmd-Z reverts as a single atomic transaction (frontmatter + fence opener tag + fence body in one CM6 dispatch).

- **D-09:** **Edit-Mode only — no Reading-Mode chevron.** Reading Mode is render-only; language switching there has no semantic meaning (you can't edit the rendered code; Run/Submit dispatch the current submission's code unchanged). Phase 5.1's "Edit + Reading parity" precedent breaks here only for the chevron — Run/Submit still appear in both modes. Reading-Mode `codeActionsPostProcessor.ts` is unchanged.

- **D-10:** **Chevron text shows the current LC language label** (e.g., `▼ Python`, `▼ Java`, `▼ C++`). Use a friendly display name, not the raw slug. Mapping `slug → label` lives in `src/solve/languages.ts` (already has `LC_LANG_LABELS` or planner can extend `resolveLangSlug`).

### Default-Code Source (D-11)

- **D-11:** **Refetch starter code from LC GraphQL `questionEditorData` endpoint on every switch.** Same endpoint Phase 2 uses for initial note creation. ~200–500 ms cost on first switch per language; can be cached after (planner's discretion, non-blocking). Network failure → silent fallback to a generic stub OR keep current fence body (planner picks the safer default; recommend keep-current-body to avoid blanking the user's work on a transient network blip).

### Reverse-Mapping (D-12)

- **D-12:** **No reverse-mapping needed at runtime.** Plugin reads `lc-language` frontmatter for the Run/Submit API call (already does this — `NoteWriter`, `runFromActive`, `submitFromActive`). The fence tag becomes a display concern only; never consulted for LC API dispatch after Phase 5.3 ships.

### Phase 5.3 Cleanup (D-13)

- **D-13:** **Revert and supersede in place.** Phase 5.3 retains its phase number and ROADMAP slot. The 4 implementation commits (`fa0d0e7`, `4b39f08`, `40b48b3`, `9a13123`) plus the test commits (`a88e3ab`, `bb93602`) and the bundle-size gate plan 03 commits (`4f05515`, `0fdc3a8`, `6dde6ed`) are reverted (or superseded by deletions in the new implementation). The 6 `@codemirror/lang-*` packs are uninstalled. RESEARCH.md is preserved for the empirical learning. SUMMARY.md files for plans 01/02/03 are preserved as deferred-work breadcrumbs. The new chevron+remap implementation lands as new plans.

- Whether the chevron uses native `<select>`, a custom dropdown, or a Modal — UX-spec concern; UI-SPEC.md will lock.
- Specifics of the destructive-switch CM6 transaction shape (effects, annotations) — planner reads CM6 docs.
- How to integrate the fence-tag remap into `NoteTemplate.ts` (whether at template-render time or in the chevron handler) — planner picks based on minimum surface area.

## Specific Ideas

### Chevron interaction model
- Click chevron → opens dropdown listing 8 supported LC languages (Python, Python3, Java, C++, C, JavaScript, TypeScript, Go, Rust). Possibly also unsupported languages with a visual de-emphasis ("plain — no highlight") if the user is on an unsupported language.
- Selected language is shown next to the chevron triangle (`▼ Python`).
- Switch flow: fetch LC starter code → atomic CM6 dispatch (rewrite fence opener tag + replace fence body + write `lc-language` frontmatter) → re-render.

### Layout
```
┌────────────────────────────────────────────────────────────┐
│ ## Code                                                    │
│ ```python                                                  │  ← markdown-recognized; native highlight
│ class Solution:                                            │
│     def twoSum(self, nums: List[int], target: int):        │
│         pass                                               │
│ ```                                                        │
│                                                            │
│ [▼ Python]                              [Run] [Submit]     │  ← chevron LEFT, actions RIGHT
└────────────────────────────────────────────────────────────┘
```

### Frontmatter shape after Phase 5.3 ships
```yaml
---
lc-slug: two-sum
lc-language: python3       # canonical LC-API slug; chevron writes here
# lc-runtime-ms: dropped
# lc-memory-mb: dropped
---
```

<deferred>
## Deferred Ideas

### Per-fence LanguageSupport (full IDE-grade indent + brackets + Lezer highlight)
**Status:** Investigated and parked.

The original Phase 5.3 ambition was full CM6 `LanguageSupport` per fence — language-aware indent (Python `:` bumps in, `else`/`elif` dedent; C-family `{` bumps in, `}` dedents), bracket auto-close, and Lezer-grade syntax highlighting for the 8 supported LC languages.

Two viable paths exist for a future phase:

**Path A: Mutate Obsidian's internal `codeLanguages` map** (small change, big unknown)
- Obsidian's markdown editor uses `@codemirror/lang-markdown` with a `codeLanguages` callback that already routes fenced content to nested languages. If we can access this callback at runtime (via `(app as any).workspace.activeEditor.editor.cm` or a similar private path), we can extend it with `python()`, `java()`, etc.
- **Risk:** undocumented private API; breaks on Obsidian updates. RESEARCH §Pitfall 1 verified the public API doesn't expose this.
- **Effort:** 1–2 days to spike + harden if it works.

**Path B: Nested EditorView per fence** (well-understood pattern, large effort)
- `Decoration.replace({ widget: new FenceEditorWidget(code, lang) })` mounts a separate, smaller EditorView dedicated to the fence content with the desired LanguageSupport. Parent doc holds a placeholder; child view handles all typing, syncs back via dispatched changes.
- **Pattern precedents:** `handlebauer/codemirror-extensions` (read-only, has the architecture), `sunxvming/obsidian-vscode-editor` (modal, sidesteps the bidirectional-sync problem).
- **Real engineering:** bidirectional sync, cursor crossing widget boundaries, undo/redo grouping, Live-Preview re-render storms, copy/paste handlers, Phase 5.1 widget coexistence.
- **Effort:** 2–3 weeks for a robust ship.

Both paths are larger than Phase 5.3's milestone budget. **Recommend tackling as Phase 6.x** (or later) when:
- The user explicitly wants language-aware indent inside the editor (not just highlighting via fence tag remap)
- There's a 2–3 week budget to invest properly
- We've used Phase 5.3's chevron ship in real practice and confirmed which IDE features matter most

### Language-pack scaffolding to keep / discard
- All Phase 5.3 implementation source files (`codeFenceLanguageExtension.ts`, `languagePackRegistry.ts`, `whitespaceCopyIndent.ts`) and their tests are deleted by D-13.
- The 6 `@codemirror/lang-*` packs are uninstalled (reclaim ~370 KB of bundle).
- `scripts/check-bundle-size.sh` is PRESERVED — useful infrastructure regardless of approach.
- RESEARCH.md is preserved for the empirical evidence of why D-11 failed.

### Other deferrals carried forward
- **Auto-completion / IntelliSense / linting / LSP** — hard out per PROJECT.md.
- **C# first-party language support** — needs an official `@codemirror/lang-csharp` pack; if revived for Path A/B above, third-party `@replit/codemirror-lang-csharp` is the only option and was rejected for unvetted dependency risk.
- **Snippets / language-specific keymaps** — out of scope for any near-term phase.
