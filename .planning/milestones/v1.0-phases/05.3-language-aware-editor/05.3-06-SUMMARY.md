## Gap closed

### G-COPY-TO-CODE-LANG-DRIFT — Copy-to-Code language drift (MEDIUM severity)

**Symptom (from 05.3-UAT.md):** When the user opens a previous submission via the SubmissionDetailModal and clicks "Copy to ## Code", the fence body and fence opener tag are rewritten correctly, but the note's `lc-language` frontmatter and chevron label are NOT updated. Result: the note's stored language and the actual code in the fence drift apart, and a subsequent Run/Submit dispatches with the wrong slug to the LC API.

**Root cause:** `copyToCode` only ran `vault.process` for the fence body. There was no `processFrontMatter` write, and Plan 05's `languageRefreshEffect` listener only fires off `metadataCache.on('changed')` — which only fires when frontmatter is written.

**Fix (in `src/graph/copyToCode.ts`):**

```diff
+import { LC_LANG_SLUGS } from '../solve/languages';

 export async function copyToCode(app, file, code, langSlug) {
   await app.vault.process(file, (current) =>
     forceInjectCodeSection(current, { starterCode: code, langSlug }),
   );
+  if (!LC_LANG_SLUGS.has(langSlug)) return;
+  const fm = app.metadataCache.getFileCache(file)?.frontmatter as
+    | Record<string, unknown> | undefined;
+  if (fm?.['lc-language'] === langSlug) return;
+  await app.fileManager.processFrontMatter(file, (fmObj) => {
+    fmObj['lc-language'] = langSlug;
+  });
 }
```

The chevron label refresh comes for free because Plan 05's `languageRefreshEffect` listens to `metadataCache.on('changed')` — when this `processFrontMatter` write lands, the StateField rebuilds and the chevron re-renders to the submission's language.

## Decision rationale: why no shared helper

The init prompt suggested either reusing `switchFenceLanguage` or extracting a shared helper.

**Trace of the call shapes:**
- `switchFenceLanguage` (`src/main.ts:738`) does THREE things: (A) fetch starter code via `client.getProblemDetail`, (B) atomic CM6 dispatch rewriting opener tag + fence body, (C) write `lc-language`. It depends on `LeetCodeClient`, an active `MarkdownView`, and the CM6 EditorView.
- `copyToCode` does ZERO of (A) and (B) the same way — its body is supplied by the caller, and the rewrite goes through `vault.process` not a CM6 dispatch (different architectural tier — vault file mutation vs. live editor dispatch).
- The ONLY shared step is (C) — three lines of `processFrontMatter`.

A shared helper for three lines adds an indirection without saving real code, and would force `copyToCode` to depend on a helper file that conceptually belongs to the chevron-switch path. **Decision:** inline the three lines into `copyToCode` with a doc comment that references `switchFenceLanguage` Step C for the lineage. If a third call-site for the frontmatter-write step appears in a future phase, extract then.
