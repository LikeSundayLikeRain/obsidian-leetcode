<objective>
Close `G-PICKER-MODAL-NOCLOSE-ON-COPY` (queued in `05.3-UAT.md` `## Open follow-ups` and `05.3-07-SUMMARY.md`): after Plan 07's `3503255` commit, the inner `SubmissionDetailModal` auto-dismisses on successful Copy-to-Code, but the outer `SubmissionPickerModal` stays open. The user expected BOTH modals to dismiss so they land back on the note. The decoupling discipline at `src/graph/SubmissionDetailModal.ts:26-28` ("MUST NOT import from SubmissionPickerModal") forbids the detail modal from directly calling the picker's close method — wiring must flow through deps/callbacks.

Purpose: Add an optional `onSuccess?: () => void` callback to `SubmissionDetailDeps`. The detail modal invokes it after a successful copy + close. The picker modal supplies the callback when constructing the detail modal (via the existing `openDetailModal` injected factory) so the picker also dismisses. The production factory at `src/main.ts:openSubmissionPickerFromActive` is widened in lockstep so the callback is threaded all the way into `new SubmissionDetailModal(...).deps.onSuccess` — the chain-close is non-functional without this third edit.

Output: `SubmissionDetailDeps` extended with `onSuccess?: () => void`; click-handler success path invokes it after `safeClose()`; production wiring in `SubmissionPickerModal.renderRow` supplies `onSuccess: () => this.safeClose()` on each `openDetailModal(row, onSuccess)` call; production factory in `src/main.ts` (the `openDetailModal` lambda inside `openSubmissionPickerFromActive` AND its callee `openSubmissionDetailFromRow`) widens to accept and forward `onSuccess` into the `SubmissionDetailModal` constructor's deps; two new test it-blocks asserting both halves of the wiring.
</objective>
