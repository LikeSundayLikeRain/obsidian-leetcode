# Requirements: Obsidian LeetCode

**Defined:** 2026-05-07
**Core Value:** Every LeetCode problem you solve becomes a first-class note in your Obsidian vault — tagged, linked, and discoverable — so practice builds a knowledge graph instead of scattered code files.

## v1 Requirements

v1 scope targets desktop Obsidian (macOS/Windows/Linux) with leetcode.com only. Author dogfoods until stable, then submits to the Obsidian community plugin store.

### Foundation

- [ ] **FND-01**: Plugin installs and loads in Obsidian 1.5+ without errors (desktop)
- [ ] **FND-02**: Plugin declares `"isDesktopOnly": true` in `manifest.json` (required for Electron `BrowserWindow` usage)
- [ ] **FND-03**: Plugin bundles pass `eslint-plugin-obsidianmd` with zero Required violations
- [ ] **FND-04**: All HTTP calls to `leetcode.com` go through Obsidian's `requestUrl` API (not `fetch`/`axios`)
- [ ] **FND-05**: User can enable/disable plugin from Obsidian's community plugin settings without crashes on reload

### Authentication

- [ ] **AUTH-01**: User can log in to LeetCode via an embedded Electron `BrowserWindow` that opens `https://leetcode.com/accounts/login/`
- [ ] **AUTH-02**: Plugin captures `LEETCODE_SESSION` and `csrftoken` cookies after successful login and stores them in `data.json`
- [ ] **AUTH-03**: User can paste a session cookie manually as a fallback auth method (settings tab)
- [ ] **AUTH-04**: Plugin detects expired sessions (GraphQL `response.errors` check) and prompts the user to re-authenticate
- [ ] **AUTH-05**: User can log out from the settings tab, clearing stored credentials
- [ ] **AUTH-06**: Session cookie and CSRF token are never logged, never transmitted anywhere except leetcode.com

### Problem Browsing

- [ ] **BROWSE-01**: User can open a problem browser view (via ribbon icon and command palette) listing LeetCode problems
- [ ] **BROWSE-02**: Problem list loads lazily — plugin never bulk-downloads all 3,300+ problems into memory
- [ ] **BROWSE-03**: User can search problems by title or LC id in the browser
- [ ] **BROWSE-04**: User can filter problems by difficulty (Easy / Medium / Hard) and solved/attempted/untouched status
- [ ] **BROWSE-05**: Problem list respects LC rate limits (throttled under 20 req / 10 s, max 2 concurrent)

### Problems as Notes

- [ ] **NOTE-01**: User can open a problem to create/update a note at `{configured folder}/{id}-{slug}.md` (e.g. `LeetCode/0001-two-sum.md`)
- [ ] **NOTE-02**: Problem HTML statement is converted to Markdown via `turndown` and inserted under a `## Problem` heading (never via `innerHTML`)
- [ ] **NOTE-03**: Note frontmatter is populated via `fileManager.processFrontMatter()` with fields: `lc-id`, `lc-slug`, `lc-title`, `lc-difficulty`, `lc-url`, `lc-status`, `lc-language`
- [ ] **NOTE-04**: LC tags are imported as Obsidian tags with a `lc/` namespace prefix. Phase 2 scope = difficulty tag only (`lc/easy`, `lc/medium`, `lc/hard`). Phase 4 scope = topic tags (e.g. `lc/array`, `lc/dynamic-programming`) written on first Accepted submission (per D-05).
- [ ] **NOTE-05**: User-added personal tags (e.g. `#revisit`, `#tricky`, `#interview-asked`) are preserved across note regeneration
- [ ] **NOTE-06**: User-authored content under `## Notes` is preserved across plugin-triggered note updates (no overwrite of user edits)
- [ ] **NOTE-07**: Previously-fetched problem notes are readable offline without network access
- [ ] **NOTE-08**: User can configure the vault folder where problem notes are created (settings)
- [ ] **NOTE-09**: User can configure their default programming language (settings) for new problems

### Solving & Submission

- [ ] **SOLVE-01**: User can write solution code directly in the problem note's code block using the native editor
- [ ] **SOLVE-02**: Starter code snippet for the user's chosen language is inserted into the note on first open
- [ ] **SOLVE-03**: User can run code against LC's sample test cases via a command/button (hand-rolled call to `/interpret_solution/`)
- [ ] **SOLVE-04**: User can run code against custom input (user-provided test case)
- [ ] **SOLVE-05**: User can submit code to LC's judge via a command/button (hand-rolled call to `/submit/`)
- [ ] **SOLVE-06**: Plugin polls `/check/{id}` with exponential backoff until verdict is returned (AC / WA / TLE / MLE / CE / RE)
- [ ] **SOLVE-07**: Verdict, runtime, memory, and any error output are displayed in a result modal/pane
- [ ] **SOLVE-08**: All supported LC submission languages are available for selection per problem
- [ ] **SOLVE-09**: Code uses the user's current note content at submit time (no stale snapshots)

### Knowledge Graph Wiring

- [ ] **GRAPH-01**: On Accepted submission, plugin appends the accepted code to `## Solution` in the problem note using `vault.process()` (atomic)
- [ ] **GRAPH-02**: On Accepted submission, plugin updates frontmatter fields: `lc-status: accepted`, `lc-solved-date`, `lc-runtime-ms`, `lc-memory-mb`, `lc-language`
- [ ] **GRAPH-03**: On Accepted submission, plugin writes `[[Technique Name]]` wikilinks under a `## Techniques` section, one per LC topic tag on the problem
- [ ] **GRAPH-04**: Plugin creates stub technique notes in a configurable `Techniques/` folder on first reference; never overwrites existing technique notes
- [ ] **GRAPH-05**: User can opt out of auto-backlink creation in settings (still writes solution + frontmatter)

### Polish & Release

- [ ] **POLISH-01**: Settings UI provides: auth status/login/logout, vault folder, technique folder, default language, auto-backlink toggle, manual cookie paste field
- [ ] **POLISH-02**: Plugin surfaces graceful error messages (not silent failures) for: LC offline, 429 rate-limited, expired session, network error
- [ ] **POLISH-03**: Plugin has no telemetry, no remote code evaluation, no unauthorized network calls
- [ ] **POLISH-04**: README includes install instructions, screenshots, usage walkthrough, and network usage disclosure
- [ ] **POLISH-05**: Repository has a LICENSE file (MIT or similar OSS license)
- [ ] **POLISH-06**: Plugin submitted as a PR to `obsidianmd/obsidian-releases` following community plugin guidelines
- [x] **POLISH-07**: Run UX rework — replace `Run code (sample)` + `Run code (custom input)` with a single unified `Run` command. Tabs pre-fill from `exampleTestcases`; custom cases are ephemeral in-memory by default; `Pin to note` is an opt-in affordance that persists a single case under `## Custom Tests` without disturbing inter-case text. Existing persisted cases load as pinned tabs on modal re-open. Replaces deferred Phase 3 UAT Tests 14/15
- [x] **POLISH-08**: Pre-ship UX polish batch — settings layout (merge status + logout row), filter UX (remove language field, multi-value premium, fix badge zero-count bug), starter-code auto-insert + Reset command (replaces Insert starter code), Past Submissions chip labels for AC/WA, remove Copy-to-Code overwrite confirmation, default language list mirrors LC, python3 syntax highlighting
- [ ] **POLISH-09**: Language-aware editor — IDE-grade auto-indentation + bracket handling in `## Code` fences via CM6 language packs. Supports LC's core language set (Python/python3, Java, C++, JS, TS, Go, Rust, C, C#). Scoped to `lc-slug` notes in Live Preview; unsupported languages fall back to whitespace-copy

## v2 Requirements

Deferred to future milestones.

### Spaced Repetition

- **SR-01**: User can rate a solve after Accepted (easy / medium / hard / forgot)
- **SR-02**: Plugin surfaces a review queue of problems due for revisit
- **SR-03**: Integrate with Obsidian's Spaced Repetition plugin (or ship a built-in simple scheduler)

### leetcode.cn

- **CN-01**: User can choose leetcode.com or leetcode.cn in settings
- **CN-02**: All API calls route to the correct host and honor site-specific auth

### Mobile

- **MOB-01**: Core browse + read flows work on Obsidian mobile (read-only, paste-auth only)
- **MOB-02**: Drop the `isDesktopOnly` constraint once feasible

### AI Enhancements

- **AI-01**: AI-generated tags augment LC's topic tags (with user approval)
- **AI-02**: AI-assisted hint/explanation surface on a stuck problem

## Out of Scope

Permanent exclusions for v1 — re-adding requires a scope revision.

| Feature | Reason |
|---------|--------|
| Obsidian mobile support | Embedded `BrowserWindow` unavailable; fundamentally different auth path. Defer to v2. |
| leetcode.cn support | Different API surface, URLs, and auth flow. Deferred to v2. |
| Local code execution | LC's remote Run Code endpoint handles every language; local runtimes add scope without value. |
| IntelliSense / linting / debugger | Obsidian isn't an IDE; users keep their IDE for deep editing. |
| Telemetry / analytics | Obsidian community plugin norms + user privacy. |
| Custom rich-text editor | Obsidian's native CodeMirror 6 code block is sufficient. |
| Problem discussion / comments import | LC ToS risk, high scope, low value to knowledge graph. |
| Payment / Premium problem bypass | Respect LC's paywall; Premium content only if user has Premium access. |
