# Phase 07 — UI Design Contract: AI Provider Foundation

> Visual and interaction contract for the AI Provider Foundation phase. Generated by gsd-ui-researcher in `--auto` mode (no user prompts; decisions locked from CONTEXT.md C + D and inferred from existing Obsidian-native precedent in `src/settings/SettingsTab.ts` and `styles.css`).

**Phase delivers three frontend surfaces:**

1. New "AI" section inside the existing `LeetCodeSettingTab` (between Preview and Knowledge graph) — active-provider dropdown that swaps visible form fields, masked API-key input, base URL field, model field, "Test connection" button.
2. **Disclosure modal** ("Heads up: this will send data to {provider}") — fired once per provider switch, lists active provider name + base URL + Phase-07-stub data manifest, two actions (Cancel / I understand — continue).
3. **Notice copy** for: Test connection success/failure, "AI call cancelled" (after disclosure cancel), "Cleared AI key for {provider}".

**No third-party UI library is added.** All UI is Obsidian-native: `Setting`, `Modal`, `Notice`, `PluginSettingTab` from the `obsidian` package; styles consume Obsidian's CSS custom properties (`--text-muted`, `--background-modifier-error`, `--interactive-accent`, etc).

| Token | Value | Usage in Phase 07 |

| lg | 24px | Reserved — not used in Phase 07 |
| xl | 32px | Reserved — not used in Phase 07 |

**Exceptions:** None. Phase 07 inherits Obsidian's `Setting` row vertical rhythm (~12px between rows) and `Modal` chrome (~24px outer / 16px inner) untouched. **Custom CSS for `Setting` rows or modal chrome is forbidden in this phase** — the only new rule is a single `.leetcode-settings .lc-ai-input input` selector that extends the existing `.lc-cookie-input` pattern at `styles.css:194` for the masked password fields.

## Typography

The plugin inherits the active Obsidian theme's font stack. We declare exactly four roles, each mapped to an existing Obsidian CSS variable so the contract holds across light/dark themes and across user font overrides.

| Role | Size | Weight | Line Height | Implementation |
|------|------|--------|-------------|----------------|
| Body | `var(--font-ui-medium)` (≈14px theme default) | 400 | 1.5 (Obsidian `.modal-content` default) | Modal `<p>` body text; "Active provider" / "Base URL" / "what gets sent" lists |
| Label | `var(--font-ui-small)` (≈13px theme default) | 400 | 1.4 | `Setting.setName()` rows in the AI section; matches `.leetcode-browser { font-size: var(--font-ui-small) }` precedent |
| Heading | `var(--font-ui-medium)` (≈14px theme default) | 600 | 1.3 | `setHeading()` produces an Obsidian-native `.setting-item-heading` (already styled by host) — no override; modal `titleEl` |
| Mono | `var(--font-monospace)` | 400 | 1.4 | API key `<input type="password">` and base URL `<input>` (extends `.lc-cookie-input` precedent at `styles.css:194`) |

**Why only Obsidian variables, no raw px:** Theme overrides (Minimal, Things, Border) shift the base font size; pinning raw px would break visual harmony with surrounding settings rows. The same posture is enforced at `styles.css:1` ("Every color references an Obsidian CSS variable — no raw hex, no rgba literals."). Typography follows the same rule.

**Weight count:** 2 (regular 400 + semibold 600 — both supplied by Obsidian's font stack; no custom font face is added).

## Color

The plugin's color contract is **100% Obsidian CSS custom properties** — no raw hex, no rgba literals (enforced top-of-file at `styles.css:1`). Phase 07 adds zero new color tokens; it consumes existing Obsidian variables.

| Role | Obsidian CSS Variable | Phase 07 Usage |
|------|----------------------|----------------|
| Dominant (60%) | `var(--background-primary)` | Modal body background; settings tab body background — Obsidian-native, no override |
| Secondary (30%) | `var(--background-secondary)` | Masked input backgrounds (extends `.lc-cookie-input` precedent); modal title bar (Obsidian-native) |
| Accent (10%) | `var(--interactive-accent)` | **Reserved exclusively** for the modal's "I understand — continue" button (calls `Setting.addButton(b => b.setCta())`) — see grep gate below |
| Destructive | `var(--background-modifier-error)` (border) + `var(--text-error)` (text) | Test connection failure Notice border + text; "AI call cancelled" Notice (neutral, Obsidian default — NO destructive color, since cancel is a user-initiated abort, not an error) |
| Muted text | `var(--text-muted)` | Help/description text under each Setting row (`Setting.setDesc()`) |

### Accent reserved for (explicit list — never "all interactive elements")

1. **Modal "I understand — continue" button** — the only call-to-action in Phase 07's UI surface. Implemented via `Setting.addButton(b => b.setCta())`.

That is the **only** invocation of the accent modifier in any Phase 07 file. **Grep gate:** `grep -rn "setCta\(" src/ai/ src/settings/SettingsTab.ts | wc -l` MUST return exactly **1 net new** invocation introduced by this phase (the disclosure modal continue button), preserving the pre-existing single accent invocation in `SettingsTab.ts:86` (the "Log in via embedded window" button — Phase 1 D-09 lock, unchanged).

**Buttons that MUST stay neutral (NO `setCta`, NO destructive modifier):**

- "Test connection" button in the AI settings section (frequent action, low stakes — neutral mirrors v1.0 "Save cookies" precedent at `SettingsTab.ts:124`)
- "Cancel" button in the disclosure modal (Obsidian default neutral — destructive color would falsely signal "this destroys data" when it merely aborts a network call)
- All Setting rows' inline buttons except the explicit single accent call-out above

**Why not destructive color for "Clear AI key" command:** The command runs from the palette (no UI button in this phase) and its result surfaces as a Notice ("Cleared AI key for {provider}"). Settings UI does not expose a destructive button in Phase 07. **Decision deferred** — if a future phase adds a "Clear AI key" Setting button, that button should follow the v1.0 "Logout" precedent (neutral `addButton` with no `.setCta()` modifier per `SettingsTab.ts:75-82`) — destructive color in Obsidian is reserved for irreversible vault-data destruction, not API-key wipe.

### Error state copy posture

All error Notices follow the **`{provider name}: {vendor error verbatim}`** shape. The vendor error text passes through `extractFromJson(body)` (07-RESEARCH.md §Open Questions Q5) unchanged — we do NOT humanize, translate, or re-prefix it. This is a deliberate decision so that:

1. Users can search the error text on the provider's docs / Stack Overflow.
2. The plugin doesn't accumulate stale "friendly error" mappings as providers change wording.

The single transform applied is **truncation at 200 chars total** (provider prefix + vendor message combined) to avoid Notice-overflow per CONTEXT.md decision E.

### Destructive actions in Phase 07

| Action | Confirmation approach |
|--------|----------------------|
| Clear AI key (palette command `clear-ai-key`) | **No confirmation modal** — mirrors v1.0 D-22 / D-23 / Phase 1 AUTH-05 logout precedent (`SettingsTab.ts:75`). Result Notice provides feedback. **Rationale:** the user typed the command name explicitly into the palette; double-confirmation is friction. Re-pasting the key from the provider's dashboard is trivial recovery. |
| Reset AI provider disclosures (palette command `reset-ai-disclosures`) | **No confirmation modal** — same posture; result Notice confirms. The "destruction" is just clearing four boolean flags; the disclosure modal will simply re-fire on next call. Trivial recovery. |
| Switching active provider | **Not destructive** — prior provider's key is preserved in `providerConfigs[prior]` per CONTEXT.md decision C; no warning needed. |

## Interaction Contract

### State machine — Active AI provider

```
   ┌────────────────┐
   │ null           │  (initial — fresh install or after `reset-ai-disclosures` + manual `null` set)
   └──┬─────────────┘
      │ user picks provider X from dropdown
      ▼
   ┌──────────────────────────────────────┐
   │ activeAIProvider = X                 │
   │ providerConfigs[X].apiKey = ''       │  (sub-form fields render with defaults)
   │ providerConfigs[X].baseUrl = default │
   │ providerConfigs[X].model = default   │
   │ providerConfigs[X].disclosureAck = ? │  (preserved — false for new provider, true if previously acked)
   └──┬───────────────────────────────────┘
      │ user clicks "Test connection"
      ▼
   ┌──────────────────────────────────────┐
   │ if disclosureAck === false:          │
   │   open AIDisclosureModal             │
   │   ├─ Cancel  → Notice "AI call cancelled"
   │   └─ Continue → ack = true; persist; fall through to probe
   │ if disclosureAck === true:           │
   │   button label → "Testing…"; disabled │
   │   probe(X) → Notice success/failure  │
   │   button label → "Test connection"; enabled
   └──────────────────────────────────────┘
```

### Test connection — debouncing per CONTEXT.md decision E + 07-RESEARCH.md Open Question 9

- **Single in-flight probe per provider.** While a probe is running, the button is disabled and labelled `Testing…`. Subsequent clicks are no-ops.
- No global rate limiter, no token bucket. The cost of a 1-token Anthropic chat is ~$0.0001; users debugging their config may click Test connection repeatedly within seconds, and that is OK as long as serialization holds.

### Disclosure modal — once-per-provider semantics

Per CONTEXT.md decision D:

- Modal fires **once per provider** — `providerConfigs[X].disclosureAcknowledged` flips to `true` on Continue and persists in `data.json`.
- Switching `activeAIProvider` from X (acked) → Y (not yet acked) re-fires the modal on Y's next Test connection / invoke call.
- Switching back X (still acked) → no modal.
- Palette command `reset-ai-disclosures` clears all five `disclosureAcknowledged` flags simultaneously; modal re-fires on next call regardless of which provider is active.

### Keyboard navigation

| Surface | Tab order |
|---------|-----------|
| AI settings section | (Obsidian default) Active provider dropdown → API key input → Base URL input → Model input → Test connection button — all keyboard-reachable via Tab; no custom focus traps |
| AIDisclosureModal | (initial focus) Cancel button → Continue button. Esc = Cancel semantics. Enter on Continue = acknowledge; Enter on Cancel = cancel. |

### Accessibility (Obsidian-native, no overrides)

- All `<input>` elements have implicit `Setting` `<label>` association via Obsidian's `setName()` (rendered as `<div class="setting-item-name">`, programmatically associated with the input by Obsidian's runtime).
- Masked password inputs use `<input type="password">` — the user agent renders the appropriate masking behavior for the host OS (per AIPROV-02 and `SettingsTab.ts:109` precedent).
- Color contrast is inherited from the active Obsidian theme — Obsidian core themes meet WCAG AA in default light/dark modes; theme-specific contrast issues are out of scope.
- No `aria-*` overrides in this phase. The `Modal` class supplies `role="dialog"`, `aria-modal="true"`, and focus trap automatically.
