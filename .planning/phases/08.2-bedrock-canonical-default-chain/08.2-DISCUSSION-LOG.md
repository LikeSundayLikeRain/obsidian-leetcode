## A. Auth-mode dropdown — collapse or keep 4?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep 4 verbatim | sso-profile stays as-is per Pitfall 10 invariant; thin alias for "default-chain with explicit profile name override". Helper text updated. Zero migration. Backlog recommendation. | |
| Collapse to 3 | default-chain absorbs sso-profile; PluginData migration on load (`'sso-profile'` → `'default-chain'` with profile preserved). Cleaner mental model; breaks locked dropdown contract; adds migration code. | |
| Rename sso-profile → "Profile name" | Keep 4 options, rename UI label only. PluginData enum value stays `'sso-profile'` (no migration). Helper text clarifies. | ✓ (variant) |

**User's choice:** "the 4 drop down now should be default, profile name, explicit access id, key and session token, and bedrock api key"

**Notes:** User picked the rename variant AND added a new requirement: explicit-access-keys mode gains a `sessionToken` field for users with temporary STS credentials. This extends `BedrockProviderConfig` with a new optional field. Logger redaction already covers `sessionToken` (Phase 08.1 Plan 02 PHASE F) so no new redaction patterns needed. The Settings UI shows the new field as optional with helper text "Required only for temporary STS credentials". The PluginData enum value for the third dropdown option stays `'sso-profile'` to avoid migration; only the user-visible label flips to "Profile name".

## B. credential_process error UX

| Option | Description | Selected |
|--------|-------------|----------|
| Hard-fail with truncated stderr | Notice surfaces 'Bedrock: credential_process failed — <first 200 chars of stderr>'. probeBedrock returns `{ ok: false, errorMessage }`. No silent fall-through. Backlog recommendation. | ✓ |
| Silent fall-through to env vars / static keys | Matches AWS SDK chain-walk semantics; collapses to "env vars only when helper fails"; silently masks misconfigured helpers; defeats phase's purpose. | |
| Hybrid: fall-through only on 'helper not configured' | If `credential_process` directive absent → fall through to static keys (existing). If directive PRESENT and FAILS → hard-fail. AWS-aligned semantics. | |

**User's choice:** Hard-fail with truncated stderr (Recommended)

**Notes:** The chosen option is the strictest of the three. The hybrid option's distinction ("absent" vs "broken") is still preserved at the implementation level (D-07 in CONTEXT.md): if the directive is absent, the resolver falls through to static `aws_access_key_id` reads, matching Phase 08.1 behavior. If the directive is present and the helper fails (timeout / non-zero exit / malformed JSON / missing required field), the error hard-fails with truncated stderr. So the spirit of the hybrid is captured inside the hard-fail option, just framed as "absence is not failure".

## C. Concurrent refresh coalescing

| Option | Description | Selected |
|--------|-------------|----------|
| Coalesce on a single in-flight Promise | Cache holds either `{ status: 'fresh', creds, expiration }` or `{ status: 'refreshing', promise }`. First call kicks off spawn + stores Promise; concurrent callers await the same Promise. Prevents MFA-storm. ~10 LoC + one unit test. | ✓ |
| Allow concurrent spawns | Each call past skew window independently spawns the helper. Simpler; 5 concurrent AI calls = 5 helper spawns = 5 MFA prompts on aws-vault. Acceptable single-user; fragile in multi-call. | |

**User's choice:** Coalesce on a single in-flight Promise (Recommended)

**Notes:** Locked. The cache shape is a discriminated union per profile (D-09 in CONTEXT.md). On promise rejection, the entry is unset before propagating the error so subsequent calls re-spawn rather than awaiting a rejected Promise (D-11). A concurrent-refresh test asserts exactly 1 helper invocation across 5 simultaneous callers.

## D. Cache scope on profile/auth-method switch in Settings

| Option | Description | Selected |
|--------|-------------|----------|
| Persist until plugin reload | Cache key = profile name. Switching active Bedrock authMethod or profile-name in Settings does NOT touch the cache. Cache authority is helper's `Expiration`, not Settings UI lifecycle. Memory footprint trivial. Evicts naturally on plugin reload. | ✓ |
| Evict on profile/auth-method change | SettingsStore.setProviderConfig clears the cache. Tighter isolation; couples Settings persistence to credential subsystem; forces extra spawn on switch-back. | |

**User's choice:** Persist until plugin reload (Recommended)

**Notes:** Locked. Cache is module-level state, never invalidated by Settings writes (D-13). Switching authMethod between default-chain ↔ access-keys ↔ api-key does not interact with the cache because only `default-chain` and `sso-profile` modes consult it. Tests must guard the persistence invariant: switch authMethod → switch back → cached creds for the prior profile still served (within TTL).
