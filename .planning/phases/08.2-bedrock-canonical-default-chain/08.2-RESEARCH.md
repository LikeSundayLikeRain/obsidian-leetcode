**B. `credential_process` error UX — hard-fail with truncated stderr**
- **D-05:** Helper failures hard-fail; never silently fall through. Error messages:
  - Non-zero exit → `'credential_process exited <code>: <stderr first 200 chars>'`
  - Timeout > 30s → `'credential_process timed out after 30s'` (kill child)
  - Malformed JSON / wrong shape → `'credential_process returned invalid JSON: <first 200 chars>'`
  - Missing required field (`AccessKeyId` / `SecretAccessKey`) → `'credential_process missing required field: <name>'`
- **D-06:** `probeBedrock` contract preserved — probe NEVER throws; failures collapse to `{ ok: false, errorMessage }`.
- **D-07:** Helper-not-configured ≠ helper-broken. ABSENT directive → fall through to static keys (Phase 08.1 behavior preserved). PRESENT directive that fails → hard-fail per D-05; no silent regression to env vars.
- **D-08:** stderr capping at the `awsCredentials.ts` boundary (truncation at SOURCE, not downstream). `extractProviderError` receives already-truncated string. No double-truncation, no double-prefixing.

**C. Concurrent refresh — coalesce on a single in-flight Promise**
- **D-09:** Cache shape is a discriminated union per profile:
  ```ts
  type CacheEntry =
    | { status: 'fresh'; creds: ResolvedCredentials; expiresAt: number /* ms */ }
    | { status: 'refreshing'; promise: Promise<ResolvedCredentials> };
  ```
  Module-level `Map<profileName, CacheEntry>`. Plugin-reload evicts naturally (module reload).
- **D-10:** `getCachedOrRefresh(profile)` lookup logic:
  1. fresh && `Date.now() < expiresAt - 5min` → return cached creds (synchronously-resolved Promise).
  2. refreshing → return the in-flight Promise.
  3. otherwise → set `{ status: 'refreshing', promise }`, kick off `Promise.resolve().then(spawnSyncWrapper)`, on success store `{ status: 'fresh', creds, expiresAt }`, on failure delete the entry and re-throw.
- **D-11:** Failure does NOT leave cache in `refreshing` status — race-safe: rejection unsets entry before propagating.
- **D-12:** Concurrent-refresh test required — 5 simultaneous calls past skew window assert exactly 1 helper invocation. Uses `vi.useFakeTimers()` + stub spawnSync that counts calls.

**D. Cache scope — persist until plugin reload**
- **D-13:** Cache is module-level state, never invalidated by Settings writes. `SettingsStore.setProviderConfig` does NOT clear the credential cache. Cache authority is helper's `Expiration`, NOT Settings UI lifecycle.
- **D-14:** Cache key is profile name. `default-chain` and `'sso-profile'` (renamed `Profile name`) modes consult it. `access-keys` and `api-key` read from cfg directly.
- **D-15:** Eviction on plugin reload only. Memory footprint trivial.
- **D-16:** Tests must guard the persistence invariant: switch authMethod → switch back → cached creds still served (within TTL). Switch profile name → switch back → cached creds for original profile still served.

**E. Profile resolution canonical order (locked)**
- **D-17:** Resolution waterfall:
  1. `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` env vars (default-chain mode only — preserves Phase 08.1 behavior; Profile-name mode SKIPS this and goes straight to step 2).
  2. **Profile name resolution:** `AWS_PROFILE` env → `AWS_DEFAULT_PROFILE` env → literal `"default"`. Profile-name mode overrides this entirely with the user-typed profile name.
  3. **Read `~/.aws/credentials`** (or `AWS_SHARED_CREDENTIALS_FILE` if set). Section header: bare `[<name>]`.
  4. **Read `~/.aws/config`** (or `AWS_CONFIG_FILE` if set). Section header: `[profile <name>]` (with the special exception `[default]` for the default profile).
  5. **Merge per profile:** config wins on `region`, credentials wins on static keys. `credential_process` directive can appear in either file.
  6. **Per-profile resolution:** `credential_process` present → spawn helper, parse, cache, return. Else `aws_access_key_id` + `aws_secret_access_key` → return directly. Else → empty result.
- **D-18:** Empty-result invariant preserved (file-absence collapses to `{}`, not throw). Hard-fail (D-05) applies ONLY to credential_process invocation, not file-absence.

**F. spawnSync wiring (locked)**
- **D-19:** `nodeRequire('child_process')` via the existing renderer shim — same pattern as `src/auth/BrowserWindowLogin.ts:nodeRequire` and `src/ai/awsCredentials.ts:34-49`. No new shim, no new dep.
- **D-20:** spawnSync options: `timeout: 30_000`, `encoding: 'utf8'`, `windowsHide: true`, `shell: false` (no shell injection vector — command + args parsed via `parseCommandLine`).
- **D-21:** Command parsing — handles double-quoted args (`aws-vault exec "my profile" --json`). Single-quotes NOT supported (matches AWS CLI behavior). Backslash-escape for embedded quotes.
- **D-22:** Stdout NEVER logged. Stderr CAN be logged truncated (200 chars) for diagnostics.

## Summary

Phase 08.2 closes the credential-resolution drift between Phase 08.1's hand-rolled `awsCredentials.ts` (138 LoC; reads `~/.aws/credentials [default]` only) and the canonical AWS chain. Three concrete capabilities ship, all as pure JS extensions:

1. **Profile resolution waterfall** — `AWS_PROFILE` env → `AWS_DEFAULT_PROFILE` env → literal `"default"`, replacing the hardcoded `[default]` lookup. [CITED: docs.aws.amazon.com/sdkref/latest/guide/file-format.html — "AWS_PROFILE — environment variable ... When this environment variable is set to a named profile or 'default', all SDK code and AWS CLI commands use the settings in that profile."]
2. **`AWS_SHARED_CREDENTIALS_FILE` / `AWS_CONFIG_FILE` env-var overrides** — defaults preserved. [CITED: docs.aws.amazon.com/sdkref/latest/guide/file-location.html — "config file environment variable: AWS_CONFIG_FILE ... credentials file environment variable: AWS_SHARED_CREDENTIALS_FILE"]
3. **`~/.aws/config` reader for `[profile <name>]` syntax + `credential_process` resolver** — new module spawning the helper command via `child_process.spawnSync` (same renderer Node-require shim Phase 08.1 already established for `fs`/`path`/`os`), parsing AWS CLI v1 JSON, caching in-memory keyed by profile name, refreshing on `Expiration - 5min` skew, coalescing concurrent in-flight refreshes onto a single Promise.

Plus a Settings UI surface change: the auth-method dropdown's third option label flips from "SSO profile" → **"Profile name"** (the resolved auth mode now resolves any profile, not just SSO-backed ones). Explicit access keys gains a `sessionToken` field for users with temporary STS credentials.

**Bundle math (estimated):** Pure JS extension — INI parser already exists (Phase 08.1, ~30 LoC). Net additions: profile-resolution waterfall (~20 LoC), `AWS_*_FILE` override handling (~15 LoC), `~/.aws/config` reader (~25 LoC — reuses existing parser), `credential_process` runner (~50 LoC including parser), in-memory cache + concurrent-refresh coalescing (~50 LoC), error-message construction (~15 LoC), `sessionToken` UI input + shape-guard (~10 LoC). **Total: ~185 LoC source + ~250 LoC tests.** Bundle delta well under 1 KB minified — Phase 08.1 baseline 1097.8 KB stays comfortably under the 1.2 MB ceiling (102.2 KB headroom). [VERIFIED: scripts/check-bundle-size.mjs HARD_LIMIT=1_200_000, SOFT_WARN=1_080_000.]

**Critical pitfalls surfaced (full detail in `## Common Pitfalls`):**
- AWS CLI's `credential_process` value uses **shell-style splitting with double-quoted args** — single quotes NOT supported, embedded quotes via `\"`. The plugin's parser must match this convention exactly to avoid silent surprise on `aws-vault exec "my profile"` etc. [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html]
- `Expiration` is RFC3339 — JS `new Date()` parses both `Z` and `+00:00` suffixes to identical timestamps. [VERIFIED: Node `new Date('2026-05-17T15:30:00Z').getTime()` === `new Date('2026-05-17T15:30:00+00:00').getTime()` === 1779031800000, this research session.]
- `child_process.spawnSync` on Windows with `shell: false` cannot directly launch `.bat` / `.cmd` files — only `.exe`. [CITED: nodejs.org/api/child_process.html — "On Windows, however, .bat and .cmd files are not executable on their own without a terminal"] Users with `aws-vault.exe` are fine; users with a `.cmd`/`.bat` wrapper see "ENOENT" and a Notice. The plugin documents this in helper text rather than building a Windows-only fallback path (out of scope for v1.1).
- Concurrent-refresh coalescing has subtle race-window behavior — `Promise.resolve().then(...)` deferral matters because `spawnSync` is synchronous-blocking. The Map.set must complete BEFORE the spawnSync call, OR a second concurrent caller can set its own `refreshing` entry. Recommended pattern: synchronous Map.set in `getCachedOrRefresh`, defer the spawn to a microtask via `Promise.resolve().then`. Test with `vi.useFakeTimers()` and 5 concurrent `Promise.all([cb1, cb2, cb3, cb4, cb5])` calls; assert exactly one spawn invocation.
- **`process.env` is mutable per-test** — vitest 4.x test pattern requires explicit `clearAwsEnv()` + `vi.unstubAllGlobals()` (already established in `tests/ai/awsCredentials.test.ts`). The new tests for AWS_PROFILE / AWS_*_FILE waterfall extend this pattern; researcher confirmed the existing test harness covers all cases.

**Primary recommendation:** Plan 08.2 is naturally a **single-plan scope** (~half-day) with three commits: (1) profile-resolution waterfall + AWS_*_FILE overrides + `[profile X]` config-file reader (no new files; extends `awsCredentials.ts`), (2) `credential_process` runner + cache + coalescing (recommend new file `src/ai/credentialProcess.ts`), (3) UI rename + sessionToken field + helper text + AIPROV-FUT-01 traceability update. Tests split per file: extend `awsCredentials.test.ts` for waterfall/override/config-reader; new `credentialProcess.test.ts` for spawn/cache/coalesce. Planner picks final task split.

### Pattern 1: Profile-resolution waterfall (D-17)

**What:** Public entry point `resolveAwsCredentials` dispatches on a discriminated `source` field. The internal waterfall handles all 6 steps.

**When to use:** Called from `createBedrockModel`'s `'default-chain'` and `'sso-profile'` branches. Other auth modes don't consult it.

**Why:** The existing Phase 08.1 entry signature already accepts `{ source: 'env-or-default-profile' | 'profile' }`; 08.2 extends the internal logic without changing the call signature. Bedrock adapter stays untouched.

**Code shape (planner-ready):**

```typescript
// src/ai/awsCredentials.ts (rewritten)

interface ResolvedCredentials {
  accessKeyId?: string;
  secretAccessKey?: string;
  sessionToken?: string;
}

export function resolveAwsCredentials(opts:
  | { source: 'env-or-default-profile' }
  | { source: 'profile'; profile: string },
): ResolvedCredentials {
  // STEP 1: env vars first (default-chain mode only — D-17 step 1).
  if (opts.source === 'env-or-default-profile') {
    const envCreds = readEnvCredentials();
    if (envCreds) return envCreds;
  }

  // STEP 2: profile-name resolution (D-17 step 2).
  const profileName = opts.source === 'profile'
    ? opts.profile
    : resolveProfileName();  // AWS_PROFILE → AWS_DEFAULT_PROFILE → "default"

  // STEPS 3+4: read credentials file + config file (with AWS_*_FILE overrides).
  const credPath = resolveCredentialsPath();  // env override or default
  const cfgPath = resolveConfigPath();        // env override or default
  const credSection = parseIniProfile(safeReadFile(credPath), profileName);
  const cfgSection = parseIniProfile(safeReadFile(cfgPath), profileName);

  // STEP 5: merge per profile (config wins region; credentials wins keys).
  const merged = mergeProfileSettings(cfgSection, credSection);

  // STEP 6: per-profile resolution.
  if (merged.credential_process) {
    // STEP 6a: spawn helper, parse, cache (D-05 hard-fail per error).
    return getCachedOrRefresh(profileName, merged.credential_process);
  }
  if (merged.aws_access_key_id && merged.aws_secret_access_key) {
    // STEP 6b: static keys.
    return {
      accessKeyId: merged.aws_access_key_id,
      secretAccessKey: merged.aws_secret_access_key,
      sessionToken: merged.aws_session_token,
    };
  }
  // STEP 6c: empty result. SDK probe surfaces as connectivity error
  // (D-18 — empty-result invariant preserved).
  return {};
}

function readEnvCredentials(): ResolvedCredentials | null {
  const e = (typeof process !== 'undefined' ? process.env : {}) as Record<string, string | undefined>;
  if (e.AWS_ACCESS_KEY_ID && e.AWS_SECRET_ACCESS_KEY) {
    return {
      accessKeyId: e.AWS_ACCESS_KEY_ID,
      secretAccessKey: e.AWS_SECRET_ACCESS_KEY,
      sessionToken: e.AWS_SESSION_TOKEN,
    };
  }
  return null;
}

function resolveProfileName(): string {
  const e = (typeof process !== 'undefined' ? process.env : {}) as Record<string, string | undefined>;
  return e.AWS_PROFILE ?? e.AWS_DEFAULT_PROFILE ?? 'default';
}
```

### Pattern 3: `credential_process` command parser (D-21)

**What:** Pure JS string parser splitting an AWS CLI credential_process value into `{ command, args[] }`. Handles double-quoted args with backslash escapes; single quotes NOT supported (matches AWS CLI behavior).

**When to use:** Called from `getCachedOrRefresh` before spawning the helper.

**Why:** [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html] AWS docs say:
> "If the path or file name contains a space, surround the complete path and file name with double-quotation marks (' ')."
> "If a parameter name or a parameter value contains a space, surround that element with double-quotation marks (' '). Surround only the name or value, not the pair."

The parser must match this convention — using `shell: true` would expose an injection vector (the `credential_process` value lives in user-controlled `~/.aws/config`, but spawning `/bin/sh -c <value>` would honor every shell metachar like `$(...)`, `&&`, `>`, etc., which AWS doesn't intend).

**Code shape (~30 LoC):**

```typescript
/**
 * Parse an AWS credential_process value into {command, args}.
 * Handles double-quoted args + backslash-escape for embedded quotes.
 * Single quotes NOT supported (matches AWS CLI behavior per
 * [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html]).
 *
 * Examples:
 *   `aws-vault exec work --json` → ['aws-vault', 'exec', 'work', '--json']
 *   `aws-vault exec "my profile" --json` → ['aws-vault', 'exec', 'my profile', '--json']
 *   `"/path with space/helper" arg`     → ['/path with space/helper', 'arg']
 *   `helper "embedded \"quote\""`        → ['helper', 'embedded "quote"']
 */
export function parseCommandLine(s: string): { command: string; args: string[] } {
  const tokens: string[] = [];
  let current = '';
  let inQuote = false;
  let i = 0;
  while (i < s.length) {
    const c = s[i];
    if (inQuote) {
      if (c === '\\' && i + 1 < s.length && s[i + 1] === '"') {
        current += '"';
        i += 2;
        continue;
      }
      if (c === '"') {
        inQuote = false;
        i++;
        continue;
      }
      current += c;
      i++;
      continue;
    }
    if (c === '"') {
      inQuote = true;
      i++;
      continue;
    }
    if (c === ' ' || c === '\t') {
      if (current.length > 0) {
        tokens.push(current);
        current = '';
      }
      i++;
      continue;
    }
    current += c;
    i++;
  }
  if (current.length > 0) tokens.push(current);
  if (tokens.length === 0) {
    throw new Error('credential_process value is empty');
  }
  const [command, ...args] = tokens;
  return { command, args };
}
```

### Pattern 4: spawnSync hardening (D-19, D-20)

**What:** Wrap `spawnSync` with strict options + post-call validation.

**Why:** [CITED: nodejs.org/api/child_process.html]
- `timeout: 30_000` — child receives `killSignal` (default `SIGTERM`); function blocks until child fully exits. Returned `error` populated, `status === null`, `signal === 'SIGTERM'`.
- `encoding: 'utf8'` — returns `string` for `stdout` / `stderr` (not Buffer).
- `windowsHide: true` — suppresses console window flash on Windows (matters because Obsidian users running `aws-vault.exe` would otherwise see a black flash).
- `shell: false` — no shell injection vector; PATH resolution still works for `.exe` on Windows. **Limitation:** `.bat`/`.cmd` files cannot be launched directly with `shell: false`. Plugin documents this in helper text; users with `.bat`/`.cmd` wrappers either rename to `.exe` or set `credential_process` to an explicit `cmd.exe /c <wrapper.bat>`.

**Code shape:**

```typescript
// src/ai/credentialProcess.ts

import { parseCommandLine } from './awsCredentials';  // or co-located

interface SpawnSyncResult {
  status: number | null;
  signal: string | null;
  stdout: string;
  stderr: string;
  error?: Error;
}

interface ChildProcessLike {
  spawnSync(command: string, args: string[], opts: {
    timeout: number;
    encoding: BufferEncoding;
    windowsHide: boolean;
    shell: boolean;
    env: NodeJS.ProcessEnv;
  }): SpawnSyncResult;
}

function runCredentialProcess(value: string): { stdout: string; stderr: string } {
  const cp = nodeRequire<ChildProcessLike>('child_process');
  if (!cp) {
    throw new Error('credential_process: child_process unavailable from renderer');
  }
  const { command, args } = parseCommandLine(value);
  const result = cp.spawnSync(command, args, {
    timeout: 30_000,
    encoding: 'utf8',
    windowsHide: true,
    shell: false,
    env: process.env,
  });
  // D-05 error message construction (truncation at SOURCE per D-08).
  const stderrTruncated = (result.stderr ?? '').slice(0, 200);
  if (result.error && (result.error as NodeJS.ErrnoException).code === 'ETIMEDOUT') {
    throw new Error('credential_process timed out after 30s');
  }
  if (result.signal === 'SIGTERM') {
    // Timeout fired — Node may report via signal field instead of error.code.
    throw new Error('credential_process timed out after 30s');
  }
  if (result.error) {
    // ENOENT, EACCES, etc.
    throw new Error(`credential_process spawn failed: ${result.error.message.slice(0, 200)}`);
  }
  if (result.status !== 0) {
    throw new Error(`credential_process exited ${result.status}: ${stderrTruncated}`);
  }
  return { stdout: result.stdout, stderr: stderrTruncated };
}
```

### Pattern 5: Stdout shape validation (D-05)

**What:** Strict JSON parse + AWS CLI v1 shape check.

**Why:** [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html]
> "the command you specify, whether a script or binary program, must generate JSON output on STDOUT that matches the following syntax. ... { 'Version': 1, 'AccessKeyId': '...', 'SecretAccessKey': '...', 'SessionToken': '...', 'Expiration': '...' }"
> "the Version key must be set to 1"
> "The Expiration key is an RFC3339 formatted timestamp. If the Expiration key isn't present in the tool's output, the SDK assumes that the credentials are long-term credentials that don't refresh."

Required fields are `AccessKeyId` and `SecretAccessKey`. `SessionToken` is required only for temporary credentials. `Expiration` absence implies long-term credentials. Plugin's hard-fail (D-05) fires when the shape is invalid OR required fields missing.

**Code shape:**

```typescript
interface CredentialProcessOutput {
  Version: number;
  AccessKeyId: string;
  SecretAccessKey: string;
  SessionToken?: string;
  Expiration?: string;  // RFC3339 timestamp
}

function parseCredentialProcessOutput(stdout: string): {
  accessKeyId: string;
  secretAccessKey: string;
  sessionToken?: string;
  expiresAt?: number;  // ms since epoch
} {
  let parsed: unknown;
  try {
    parsed = JSON.parse(stdout);
  } catch {
    throw new Error(`credential_process returned invalid JSON: ${stdout.slice(0, 200)}`);
  }
  if (!parsed || typeof parsed !== 'object') {
    throw new Error('credential_process returned invalid JSON: not an object');
  }
  const p = parsed as Partial<CredentialProcessOutput>;
  if (typeof p.AccessKeyId !== 'string' || p.AccessKeyId.length === 0) {
    throw new Error('credential_process missing required field: AccessKeyId');
  }
  if (typeof p.SecretAccessKey !== 'string' || p.SecretAccessKey.length === 0) {
    throw new Error('credential_process missing required field: SecretAccessKey');
  }
  let expiresAt: number | undefined = undefined;
  if (typeof p.Expiration === 'string' && p.Expiration.length > 0) {
    const parsedDate = new Date(p.Expiration).getTime();
    if (!Number.isFinite(parsedDate)) {
      throw new Error(`credential_process returned invalid JSON: Expiration is not a valid ISO 8601 timestamp`);
    }
    expiresAt = parsedDate;
  }
  return {
    accessKeyId: p.AccessKeyId,
    secretAccessKey: p.SecretAccessKey,
    sessionToken: typeof p.SessionToken === 'string' ? p.SessionToken : undefined,
    expiresAt,
  };
}
```

Different timezone offsets yield different timestamps as expected (`-07:00` → `1779057000000`). Test case: parametric assertions across `Z`, `+00:00`, `.000Z`, `+05:30`, `-07:00`.

### Pattern 6: Discriminated-union cache + concurrent-refresh coalescing (D-09, D-10, D-11, D-12)

**What:** Module-level `Map<string, CacheEntry>` with `'fresh' | 'refreshing'` discriminant. Concurrent callers in `refreshing` state await the same Promise.

**Critical race window:** Map.set MUST happen synchronously before the async work starts, OR a second concurrent caller can observe the entry as missing and start its own spawn. Pattern: synchronous Map.set with `{ status: 'refreshing', promise }`; the promise body kicks off the spawn via `Promise.resolve().then(...)` so spawnSync runs in a microtask.

**Why this works:** JS event loop is single-threaded. From the moment `getCachedOrRefresh` enters until it returns, no other code runs. Setting the Map.set BEFORE returning ensures the next concurrent caller (running on the next microtask/task) sees the `refreshing` entry.

**Code shape:**

```typescript
// src/ai/credentialProcess.ts

interface FreshEntry {
  status: 'fresh';
  creds: ResolvedCredentials;
  expiresAt: number;
}
interface RefreshingEntry {
  status: 'refreshing';
  promise: Promise<ResolvedCredentials>;
}
type CacheEntry = FreshEntry | RefreshingEntry;

const SKEW_MS = 5 * 60 * 1000;  // 5 minutes per AWS SDK convention
const cache = new Map<string, CacheEntry>();

export async function getCachedOrRefresh(
  profileName: string,
  credentialProcessValue: string,
): Promise<ResolvedCredentials> {
  const entry = cache.get(profileName);
  if (entry?.status === 'fresh' && Date.now() < entry.expiresAt - SKEW_MS) {
    return entry.creds;
  }
  if (entry?.status === 'refreshing') {
    return entry.promise;
  }
  // Need to refresh. CRITICAL: set the cache entry BEFORE awaiting anything.
  const promise = (async () => {
    try {
      const { stdout } = runCredentialProcess(credentialProcessValue);
      const parsed = parseCredentialProcessOutput(stdout);
      const creds: ResolvedCredentials = {
        accessKeyId: parsed.accessKeyId,
        secretAccessKey: parsed.secretAccessKey,
        sessionToken: parsed.sessionToken,
      };
      const expiresAt = parsed.expiresAt ?? (Date.now() + 24 * 60 * 60 * 1000);
      // ^ When Expiration absent (long-term creds), cache for 24h as a sane upper bound.
      // Re-check on next call. Phase 08.2 docs note that long-term creds
      // technically never refresh, but caching for 24h matches common runtime intuition.
      cache.set(profileName, { status: 'fresh', creds, expiresAt });
      return creds;
    } catch (err) {
      // D-11: failure deletes the entry so next call retries (NOT cached as 'refreshing').
      cache.delete(profileName);
      throw err;
    }
  })();
  cache.set(profileName, { status: 'refreshing', promise });
  return promise;
}

// Optional: exported for tests / plugin reload diagnostics
export function clearCredentialProcessCache(): void {
  cache.clear();
}
```

### Anti-Patterns to Avoid

- **Don't import `@aws-sdk/credential-providers` to "do it properly".** Phase 08.1 RESEARCH §Pitfall 5 — does not bundle. 08.2 extends the in-plugin parser; do not revisit this dep.
- **Don't pass `shell: true` to spawnSync.** Even though the `credential_process` value lives in user-controlled `~/.aws/config`, AWS CLI parses it as space-separated args with double-quote support — NOT as a shell command. `shell: true` opens an injection vector via `$(...)`, `;`, `&&`, etc., that AWS doesn't intend.
- **Don't log stdout from `credential_process` (D-22).** Stdout contains live credentials. The resolver receives stdout as a string and passes it directly to `JSON.parse` and the shape validator — no log call ever sees stdout. Stderr CAN be logged truncated.
- **Don't truncate the error message at multiple layers.** D-08 specifies truncation at the `awsCredentials.ts` boundary ONLY. Downstream `extractProviderError` adds the `Bedrock: ` prefix and may add bracketing — but doesn't re-truncate. Tests must verify "credential_process exited <code>: <stderr-200>" is byte-equal at the Notice rendering layer.
- **Don't clear the cache on Settings UI writes (D-13).** The cache is about the helper's `Expiration` skew, not the Settings UI lifecycle. Switching authMethod or active provider does NOT clear the cache. Switching profile name in Settings doesn't clear the prior profile's cache entry — both round-trip through cache hits within TTL.
- **Don't move the `nodeRequire` shim to a new file.** Phase 08.1 placed it inside `awsCredentials.ts:34-49` as a private helper. 08.2 reuses by either co-locating new code in the same file or importing the helper indirectly (TypeScript can re-export it). Don't extract to `nodeRequireShim.ts` — that's scope creep.
- **Don't assume `~` expansion.** The user's shell expanded `~` before `export` set the env var; the env-var value is already absolute by read time. Plugin passes the value verbatim per [CITED: docs.aws.amazon.com/sdkref/latest/guide/file-format.html — "Don't specify the home folder as ~"]. (For default paths, plugin uses `os.homedir()` directly — not a `~` literal.)

## Common Pitfalls

### Pitfall 1: AWS CLI's command-line parsing is a STRICT subset of shell syntax

**What goes wrong:** A user with `credential_process = my-helper 'arg with spaces'` (single quotes — common reflex from bash) sees `parseCommandLine` produce `['my-helper', "'arg", "with", "spaces'"]` (4 tokens, all quote chars literal), and the helper fires with mangled args.

**Why it happens:** [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html] AWS docs say "surround that element with double-quotation marks" — only double quotes are supported. Single quotes are literal.

**How to avoid:** Pattern 3's parser handles double quotes only — exactly matches AWS CLI behavior. Document the limitation in commit message + Settings helper text: "Helper command parsed like AWS CLI: use \"double quotes\" for args containing spaces."

**Warning signs:** Helper exits with `argument list malformed` or similar. The truncated stderr message surfaces this verbatim — user sees their literal `'` characters in the Notice and realizes the issue.

### Pitfall 2: `child_process.spawnSync` on Windows + `.bat`/`.cmd` files

**What goes wrong:** A user with `credential_process = C:\Users\me\aws-helper.cmd` (or `.bat`) sees `ENOENT` from spawnSync because `shell: false` cannot launch script files directly on Windows.

**Why it happens:** [CITED: nodejs.org/api/child_process.html] Node docs:
> "On Windows, however, .bat and .cmd files are not executable on their own without a terminal, and therefore cannot be launched using child_process.execFile(). When running on Windows, .bat and .cmd files can be invoked by:
> - using child_process.spawn() with the shell option set (not recommended, see DEP0190),
> - using child_process.exec(),
> - spawning cmd.exe and passing the .bat or .cmd file as an argument."

**How to avoid:** Plugin documents the workaround in Settings helper text + README:
> "Windows users with `.cmd`/`.bat` helpers: set `credential_process = cmd.exe /c C:\\path\\to\\helper.cmd`"

This pushes the shell-invocation responsibility onto the user (who is in their own `~/.aws/config` config-edit context) rather than the plugin — matches AWS CLI v2 behavior on Windows. **Out-of-scope to add a Windows-specific shell-prefix auto-detect** (D-25 is not a locked decision; researcher recommends documenting only).

**Warning signs:** Bedrock probe fails with `'credential_process spawn failed: spawn ENOENT'`. Plugin's truncated message includes `'spawn ENOENT'` so the user grep-finds AWS docs.

### Pitfall 3: Missing `Expiration` field — long-term credentials assumption

**What goes wrong:** A naive caching strategy treats `Expiration: undefined` as "never cache" or "expire immediately" — leading to spawn-on-every-request (D-12 is supposed to coalesce concurrent calls, but back-to-back calls a few seconds apart still re-spawn).

**Why it happens:** [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html — "If the Expiration key isn't present in the tool's output, the SDK assumes that the credentials are long-term credentials that don't refresh."]

**How to avoid:** Pattern 6's `expiresAt = parsed.expiresAt ?? (Date.now() + 24 * 60 * 60 * 1000)` — when Expiration absent, cache for 24 hours. Test case: helper returns `{Version:1, AccessKeyId, SecretAccessKey}` (no Expiration); subsequent call within TTL hits cache; subsequent call after TTL re-spawns.

**Warning signs:** None functional — but if a user reports "Bedrock works fine" then "Bedrock probe takes 5s sometimes," it's the helper firing again on a cache miss. UAT covers this.

### Pitfall 4: Concurrent-refresh race window — Map.set BEFORE async work

**What goes wrong:** Naive impl writes:
```typescript
const promise = doSpawn();          // ❌ async work starts BEFORE Map.set
cache.set(profile, { status: 'refreshing', promise });
return promise;
```
A second concurrent caller observing the cache mid-`doSpawn` sees the entry as missing and starts its own spawn. D-12 coalescing fails.

**Why it happens:** Even though JS is single-threaded per task, `doSpawn()` returns a Promise immediately and the function continues to Map.set — but `getCachedOrRefresh` itself returns the promise, suspending the current task. The next concurrent caller runs in a separate task; if Map.set hasn't completed yet, they observe a fresh-miss.

Wait — actually this is incorrect. JS evaluates the right-hand side of a `const` synchronously. `doSpawn()` returns a Promise WITHOUT awaiting; `cache.set` runs synchronously next. So the issue is NOT race-window — it's that **the Promise body inside `(async () => {...})()` runs in a microtask**. If the body itself does the cache.delete on failure synchronously after a synchronous `runCredentialProcess` call, the cache.delete happens in a microtask AFTER the outer cache.set, which is correct.

**Real concern:** What if `runCredentialProcess` completes SYNCHRONOUSLY (it can — `spawnSync` blocks)? Then by the time the Promise body kicks off, `runCredentialProcess` returns sync; `parseCredentialProcessOutput` runs sync; `cache.set('fresh', ...)` runs sync — all WITHIN the same microtask. If a concurrent caller invokes `getCachedOrRefresh` BEFORE this microtask runs (i.e., the original caller has already returned the promise), they see `{ status: 'refreshing' }` and await the same Promise. ✅ correct.

**Net:** Pattern 6's structure is correct. Test: spawn 5 concurrent `getCachedOrRefresh` calls; assert spawnSync mock called exactly once.

**How to avoid:** Use Pattern 6 structure verbatim. The IIFE Promise wrapper + outer `cache.set` ordering is load-bearing.

**Warning signs:** Test "5 concurrent calls trigger 5 spawns" indicates race window. If this fails, the planner has reordered the `cache.set` after the spawn body's first await.

### Pitfall 5: `Promise.resolve().then(spawnSync)` deferral consideration

**What goes wrong (theoretical):** `spawnSync` blocks the event loop. If `getCachedOrRefresh` calls it INSIDE the Promise IIFE without explicit microtask deferral, the first call's spawn fires synchronously inside the Promise constructor's executor. By the time the IIFE returns the promise, the spawn has already run. Concurrent callers in the same synchronous frame still see the unmodified cache — but in practice no JS code in the same frame can interleave with spawnSync (event loop is single-threaded).

**Reality:** Pattern 6 is correct as written. The "Promise.resolve().then" wrapper mentioned in CONTEXT.md D-10 is to "match async signature" — it's stylistic, not load-bearing for correctness. Using `(async () => { ... })()` IIFE achieves the same effect.

**How to avoid:** Use the IIFE pattern. Add a comment documenting that spawnSync's blocking behavior is OK because we're inside an async function body (Promise resolution happens after all sync work in the IIFE completes).

**Warning signs:** N/A — but if a future maintainer worries about this, point them to the `vi.useFakeTimers()` test that asserts coalescing works.

### Pitfall 6: Stdout MUST never reach the logger (D-22)

**What goes wrong:** A debug-friendly `logger.debug('credential_process result', { stdout })` ships, then someone runs the plugin in dev mode with devtools open — credentials in `console.debug` output.

**Why it happens:** Stdout contains the live access key + secret + session token. There's no logger redaction pattern that catches arbitrary base64 strings. The only safe posture is "never log stdout."

**How to avoid:**
1. The implementation NEVER passes stdout to `logger.*`. The only sites stdout flows through are: `runCredentialProcess` returns it; `parseCredentialProcessOutput` parses it; `getCachedOrRefresh` consumes the parsed shape.
2. ESLint custom rule (optional, planner-discretion): forbid `logger\.[a-z]+\([^)]*stdout` regex match in `src/ai/credentialProcess.ts`. Plan-checker can catch this.
3. Test: run probeBedrock with a fake `credential_process` that emits a known token; assert the token does NOT appear in any captured logger output.

**Warning signs:** Devtools console contains an `AccessKeyId` literal. Immediate red flag.

### Pitfall 7: Stderr can ALSO contain secrets (rare but documented)

**What goes wrong:** A misconfigured helper writes diagnostic output to stderr that includes a token — common with verbose `aws-vault` flags or custom helpers that don't follow AWS guidance.

**Why it happens:** [CITED: docs.aws.amazon.com/sdkref/latest/guide/feature-process-credentials.html — "Confirm that your custom credential tool does not write any secret information to StdErr, because the SDKs and AWS CLI can capture and log such information, potentially exposing it to unauthorized users."] AWS docs explicitly call this out.

**How to avoid:** Truncate stderr to 200 chars at SOURCE (D-08). 200 chars is short enough that a typical secret (40+ chars) plus surrounding context likely gets truncated mid-token — but not a guarantee. Plugin documents this in README + helper text:
> "If your `credential_process` helper writes secrets to stderr, the truncated 200-char stderr in error messages may contain partial credential data. AWS recommends helpers write only diagnostic info to stderr."

This is a user-side responsibility — same as the AWS CLI's own documentation.

**Warning signs:** User reports "I see part of my token in the error Notice." Plugin docs link to the AWS recommendation.

### Pitfall 8: Plugin reload semantics — module-level cache vs Settings UI

**What goes wrong:** User changes `bedrock.modelId` in Settings, expecting the credential cache to invalidate (it doesn't). The user has a stale `aws-vault` session that's about to expire — they expected restart-on-Settings-change.

**Why it happens:** D-13: cache authority is `Expiration`, NOT Settings UI lifecycle. The user's mental model differs from the implementation contract.

**How to avoid:**
- D-13 is a deliberate decision; document in commit message + helper text.
- Plugin reload (Cmd-P → "Reload app without saving") clears the cache via module re-evaluation. Helper text under "Profile name" mentions: "Cached AWS credentials persist until plugin reload — change profile, reload Obsidian."
- Researcher recommends NO settings-side cache invalidation. Adding it pulls into a slippery slope: which Settings changes invalidate? authMethod alone? model? Locking to "no invalidation" is the simplest contract.

**Warning signs:** User reports "I changed my profile and got the wrong credentials." They likely needed to reload Obsidian. README + helper text covers this.

### Pitfall 9: `Date.parse` of `Expiration` returning NaN

**What goes wrong:** Helper returns `Expiration: "2026-05-17 15:30:00"` (space instead of `T` — common shell-printf mistake). `new Date(s).getTime()` returns `NaN`. Naive impl treats this as 0 ms, immediate expiry, infinite re-spawn.

**Why it happens:** RFC3339 requires `T` or `t` between date and time. Lax parsers accept space; strict parsers (and ECMA spec) require `T`.

**How to avoid:** Pattern 5's `Number.isFinite(parsedDate)` check throws hard-fail with truncated stderr. User sees "credential_process returned invalid JSON: Expiration is not a valid ISO 8601 timestamp" — immediately fixes their helper.

**Warning signs:** Continuous re-spawn observed in DevTools network panel. Fail-loudly is the prevention.

### Pitfall 10: `process` global availability in Electron renderer

**What goes wrong:** Naive `process.env.AWS_PROFILE` access throws `ReferenceError: process is not defined` in some contextIsolation:true setups.

**Why it (mostly) doesn't happen:** Phase 07/08.1 verified `process.env` works in the Obsidian renderer — `tests/ai/awsCredentials.test.ts:74-76` (shipped) accesses `process.env.AWS_ACCESS_KEY_ID` and the test green-passes. Electron exposes `process` to renderer-context plugins via the Node-integration sub-shim.

**How to avoid:** Use the existing guard `(typeof process !== 'undefined' ? process.env : {})` (verified in Phase 08.1 source `src/ai/awsCredentials.ts:75`). 08.2 mirrors this guard verbatim across all new env-reading paths.

**Warning signs:** `ReferenceError: process is not defined` in user-visible logs. None observed in Phase 08.1 dogfood; pattern is hardened.

### Pitfall 11: Settings UI dropdown enum value vs label confusion (D-02)

**What goes wrong:** Planner naively renames the enum value `'sso-profile'` → `'profile-name'`. Existing users' data.json files contain `authMethod: 'sso-profile'`; the load-time shape-guard `VALID_BEDROCK_AUTH_METHODS` rejects the unknown value, collapsing to `'default-chain'`. Users lose their setting.

**Why it happens:** D-02 says "PluginData enum value stays `'sso-profile'` (UI-label-only change)". The label `"Profile name"` is what the user sees; the persisted enum is unchanged.

**How to avoid:** Settings UI code:
```typescript
.addOption('sso-profile', 'Profile name')  // ← value stays, label flips
```
NOT:
```typescript
.addOption('profile-name', 'Profile name')  // ❌ would break existing data.json
```

Test: load a data.json with `authMethod: 'sso-profile'`; verify the UI renders `'Profile name'` selected; verify `getProviderConfig('bedrock').authMethod` returns `'sso-profile'`.

**Warning signs:** Existing users report "my Bedrock auth mode reset after upgrade." Planner missed D-02's locked invariant.

## Open Questions

1. **Where should `credentialProcess.ts` live — separate file or inside `awsCredentials.ts`?**
   - What we know: Researcher recommends split for unit-test isolation; CONTEXT.md flags "Claude's Discretion."
   - What's unclear: How to keep the `nodeRequire` shim DRY across files. One option: extract `nodeRequire` to `src/ai/nodeRequire.ts` (~15 LoC) imported by both files. Phase 08.1 deliberately INLINED the shim in each consumer (BrowserWindowLogin, awsCredentials, obsidianFetch); 08.2 may follow that precedent or DRY.
   - Recommendation: **Co-locate in single `awsCredentials.ts`** — keeps the file at ~330 LoC (still small), inherits Phase 08.1's "shim is private to module" pattern, no new file. Planner has final say.

2. **Should A14 (5-minute skew) be discussed with user before lock?**
   - What we know: Phase 08.1 backlog note locked 5 min; CONTEXT.md D-09 lists `expiresAt - 5min`. AWS boto3 uses 10/15min.
   - What's unclear: Whether to match boto3 (looser) or stay tight at 5min (more aggressive refresh). Researcher leaned tight on the principle "false positive cache miss = re-spawn (free); false negative cache hit = expired creds (signing failure)."
   - Recommendation: **Keep 5 min as locked.** Document the tradeoff in commit message. Confirm with user if they want to discuss.

3. **Should the planner add a Windows-helper-text mentioning `cmd.exe /c` for `.bat`/`.cmd`?**
   - What we know: Pitfall 2 documents the limitation. Helper text in Settings UI under the "Profile name" auth-mode is a natural place.
   - What's unclear: Whether to add a sentence to the Settings helper text or document only in commit + README.
   - Recommendation: **Add to README "Network use" section** (or a new "Platform notes" subsection). Keeps Settings UI helper text short.

4. **Cache key — profile name only, or `(profile name, helper command)` tuple?**
   - What we know: D-14 says "cache key is profile name." If two users have the same profile name pointing to different helpers (rare — they're modifying their own ~/.aws/config), cache hit by name returns stale creds.
   - What's unclear: Whether the cache should include the helper command in the key, OR detect helper-command changes and invalidate.
   - Recommendation: **Profile name only.** Helper command changes happen via `~/.aws/config` edits; user is expected to reload Obsidian (D-15). Documented as an explicit limitation.

5. **Test split — extend `awsCredentials.test.ts` or split into per-concern files?**
   - What we know: CONTEXT.md flags this as "Claude's Discretion."
   - What's unclear: 08.2 adds ~250 LoC of tests on top of the existing ~215 LoC `awsCredentials.test.ts`. Hitting ~500 LoC is borderline-fine but starts to feel monolithic.
   - Recommendation: **Split into 3 files** — `awsCredentials.test.ts` (waterfall + AWS_*_FILE override + INI extension), `credentialProcess.test.ts` (spawn / parse / hard-fail / cache / coalesce — the meat of 08.2), `bedrock.sessionToken.test.ts` (D-03 UI + shape-guard). Planner picks final split.

6. **Should we add a `clearCredentialProcessCache()` exported helper?**
   - What we know: Pattern 6 sketches it. D-15 says cache evicts on plugin reload only.
   - What's unclear: Whether Settings UI gains a "Clear cached credentials" button or this remains a code-only export for tests.
   - Recommendation: **Export but no UI button.** Tests use it; users use plugin reload. UI button is v1.2 polish. Document the export as testing-only via JSDoc.

7. **Long-term creds (no `Expiration`) — 24h cache vs never-cache?**
   - What we know: Pattern 6 picks 24h. Boto3 caches indefinitely until process restart.
   - What's unclear: 24h could surface a long-term-cred user surprise — they updated their helper and wait up to 24h for it to take effect (but not if they reload Obsidian).
   - Recommendation: **24h cap.** Reload Obsidian invalidates anyway. Documented explicitly in helper text.

8. **Disclosure modal text — does the new "Profile name" mode require new copy?**
   - What we know: Phase 08.1 disclosure modal substitutes the configured region into the base URL. The `Active provider: AWS Bedrock — https://bedrock-runtime.us-east-1.amazonaws.com` line stays unchanged.
   - What's unclear: Whether to add a new bullet to `DISCLOSURE_BASE_COPY.willSend` mentioning that the plugin reads `~/.aws/credentials` and `~/.aws/config` (could be a privacy concern for some users).
   - Recommendation: **Don't change the disclosure copy.** The data flow is identical to Phase 08.1 (only credential resolution mechanics differ; user data sent to Bedrock is unchanged). Plus `DISCLOSURE_BASE_COPY` is locked per Phase 07-08 WR-02 freeze invariant — mutation is forbidden.

```typescript
      signal: string | null;
      stdout: string;
      stderr: string;
      error?: Error;

        signal: null,

          SecretAccessKey: 'secret-from-helper',
```
