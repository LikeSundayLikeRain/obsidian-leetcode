# Phase 08.2: Bedrock canonical default-chain + credential_process auto-refresh - Context

**Gathered:** 2026-05-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the credential-resolution drift between Phase 08.1's hand-rolled `awsCredentials.ts` and the canonical AWS chain. Three concrete capabilities ship:

1. **Profile resolution honoring `AWS_PROFILE` / `AWS_DEFAULT_PROFILE`** — currently hardcoded to `[default]`. Falls through canonically: `AWS_PROFILE` env → `AWS_DEFAULT_PROFILE` env → literal `"default"`.
2. **`AWS_SHARED_CREDENTIALS_FILE` / `AWS_CONFIG_FILE` env-var overrides** — currently always reads `~/.aws/credentials`. Defaults preserved when env vars unset.
3. **`~/.aws/config` reader for `[profile X]` syntax + `credential_process` resolver** — spawn the helper command via `child_process.spawnSync` (renderer Node-require shim — same pattern as `obsidianFetch.ts:loadElectronNet`), parse AWS CLI v1 JSON, cache in memory keyed by profile name, refresh on `Expiration - 5min` skew. Coalesce concurrent in-flight refreshes into a single Promise.

Plus one Settings UI surface change: the auth-method dropdown's third option label flips from "SSO profile" → **"Profile name"** (it now resolves any profile, not just SSO-backed ones), and explicit access keys grow a `sessionToken` field for users with temporary credentials.

**Out of scope (locked):** `role_arn` + `source_profile` (assume-role) chains, inline `sso_session` blocks / `aws sso login` orchestration, EC2/ECS instance-metadata service, container-credential-helper IPC. All require additional npm deps (~150KB for `@aws-sdk/client-sts`) or `child_process.spawn` for browser-flow auth — both flagged for v1.2.

</domain>

<decisions>
## Implementation Decisions

### A. Auth-method dropdown — keep 4 options, rename sso-profile

- **D-01: Dropdown stays 4 options.** Phase 08.1's locked dropdown contract is preserved per Pitfall 10 invariant (preserve-on-mode-switch). Order: **Default credential chain / Profile name / Explicit access keys / Bedrock API key**.
- **D-02: Third option renamed `sso-profile` → "Profile name"** in the UI label. The PluginData enum value stays `'sso-profile'` (no migration needed — UI-label-only change). Helper text under the field updates: "Any profile name from `~/.aws/credentials` or `~/.aws/config`. Use this if you want a specific profile without exporting `AWS_PROFILE`. Supports `credential_process` helpers (e.g. `aws-vault`, `awsume`)."
- **D-03: Explicit access keys gains a `sessionToken` field.** Plugin-data shape extended:
  ```
  bedrock.sessionToken?: string;  // when authMethod === 'access-keys'
  ```
  Logger redaction already covers `sessionToken` (Phase 08.1 Plan 02 PHASE F). Settings UI shows the field as optional with helper text "Required only for temporary STS credentials".
- **D-04: Default-chain helper text updated.** Reflects expanded support: "Plugin reads AWS credentials from `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` env vars first, then falls back to `AWS_PROFILE` (or `AWS_DEFAULT_PROFILE`, or `[default]`) in `~/.aws/credentials` and `~/.aws/config`. Supports `credential_process` helpers and honors `AWS_SHARED_CREDENTIALS_FILE` / `AWS_CONFIG_FILE` overrides. Run `aws sso login` (or your usual helper) before launching Obsidian if your profile uses SSO."

### B. `credential_process` error UX — hard-fail with truncated stderr

- **D-05: Helper failures hard-fail; never silently fall through.** Failure modes covered:
  - Non-zero exit → `errorMessage = 'credential_process exited <code>: <stderr first 200 chars>'`
  - Timeout > 30s → `errorMessage = 'credential_process timed out after 30s'` (kill the child)
  - Malformed JSON / wrong shape → `errorMessage = 'credential_process returned invalid JSON: <first 200 chars>'`
  - Missing required field (`AccessKeyId` / `SecretAccessKey`) → `errorMessage = 'credential_process missing required field: <name>'`
- **D-06: probeBedrock contract preserved.** Existing Phase 08.1 contract: probe NEVER throws — failures collapse to `{ ok: false, errorMessage }`. Live AI Debug calls surface the same string via the existing Notice path in `AIClient.invokeStream` failure handling.
- **D-07: Helper-not-configured ≠ helper-broken.** If `credential_process` directive is ABSENT from the resolved profile, fall through to static `aws_access_key_id` + `aws_secret_access_key` reads (existing Phase 08.1 behavior preserved). If the directive IS present, any failure hard-fails with truncated stderr — no silent regression to env vars.
- **D-08: stderr capping is at the `awsCredentials.ts` boundary.** Truncation to 200 chars happens once, where stderr is read. Downstream `extractProviderError` in `providers/index.ts` receives the already-truncated string. No double-truncation, no double-prefixing.

### C. Concurrent refresh — coalesce on a single in-flight Promise

- **D-09: Cache shape is a discriminated union per profile.**
  ```
  type CacheEntry =
    | { status: 'fresh'; creds: ResolvedCredentials; expiresAt: number /* ms */ }
    | { status: 'refreshing'; promise: Promise<ResolvedCredentials> };
  ```
  Module-level `Map<profileName, CacheEntry>`. Plugin-reload evicts naturally (module reload).
- **D-10: getCachedOrRefresh(profile) lookup logic:**
  1. If entry exists and `status === 'fresh'` and `Date.now() < expiresAt - 5min`: return cached creds synchronously-resolved Promise.
  2. If entry exists and `status === 'refreshing'`: return the in-flight Promise (caller awaits the same resolution).
  3. Otherwise (no entry / stale fresh / past skew): set `{ status: 'refreshing', promise }`, kick off `spawnSync` (wrapped in `Promise.resolve().then(...)` to match async signature), on success store `{ status: 'fresh', creds, expiresAt }`, on failure delete the entry (so the next call retries) and re-throw.
- **D-11: Failure does NOT leave the cache in `refreshing` status.** Race-safe: any rejection unsets the entry before propagating the error. Subsequent calls re-spawn rather than awaiting a rejected Promise.
- **D-12: Concurrent-refresh test required.** Spawn 5 simultaneous `getCachedOrRefresh('work')` calls past the skew window; assert exactly 1 helper invocation and all 5 callers receive the same creds. Uses `vi.useFakeTimers()` + a stub spawnSync that counts calls.

### D. Cache scope — persist until plugin reload

- **D-13: Cache is module-level state, never invalidated by Settings writes.** SettingsStore.setProviderConfig does NOT clear the credential cache. Cache authority is the helper's `Expiration` timestamp, NOT the Settings UI lifecycle.
- **D-14: Cache key is profile name.** Switching authMethod (default-chain ↔ access-keys ↔ api-key) does not interact with the cache — only `default-chain` and `sso-profile` (renamed `Profile name`) modes consult it. access-keys reads from cfg directly; api-key reads from cfg directly.
- **D-15: Eviction on plugin reload only.** Module-level `Map` is GC'd when `main.ts` re-evaluates. Memory footprint trivial — at most a handful of profiles per user, ~few hundred bytes each (3 base64 fields + numeric expiry).
- **D-16: Tests must guard the persistence invariant.** Test: switch authMethod → switch back → cached creds for the prior profile still served (within TTL). Test: switch profile name → switch back → cached creds for the original profile still served.

### E. Profile resolution canonical order (locked)

- **D-17: Resolution waterfall** — when no env vars satisfy step 1:
  1. `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` env vars (default-chain mode only — preserves Phase 08.1 behavior; Profile-name mode skips this and goes straight to step 2).
  2. **Profile name resolution:** `AWS_PROFILE` env → `AWS_DEFAULT_PROFILE` env → literal `"default"`. Profile-name mode overrides this entirely with the user-typed profile name.
  3. **Read `~/.aws/credentials`** (or `AWS_SHARED_CREDENTIALS_FILE` if set). Section header: bare `[<name>]`.
  4. **Read `~/.aws/config`** (or `AWS_CONFIG_FILE` if set). Section header: `[profile <name>]` (with the special exception `[default]` for the default profile).
  5. **Merge per profile:** config wins on `region`, credentials wins on static keys (matches AWS CLI v2 behavior). `credential_process` directive can appear in either file; either wins (typically lives in `~/.aws/config`).
  6. **Per-profile resolution:**
     - `credential_process` directive present → spawn helper, parse, cache, return.
     - else `aws_access_key_id` + `aws_secret_access_key` present → return directly.
     - else → empty result (collapses to "no credentials" at SDK call site).
- **D-18: Empty-result invariant preserved.** Existing Phase 08.1 behavior: missing files / missing fields / Node-require unavailable → empty `{}` object, NEVER throw. The bedrock adapter treats empty as "no creds" and the SDK probe surfaces it as a connectivity error. Hard-fail (D-05) applies ONLY to credential_process invocation, not file-absence.

### F. spawnSync wiring (locked)

- **D-19: Use `nodeRequire('child_process')` via the existing renderer shim.** Mirrors `src/auth/BrowserWindowLogin.ts:nodeRequire` and `src/ai/awsCredentials.ts` lines 22-49 (the same pattern this phase extends). No new shim, no new dep.
- **D-20: spawnSync options:** `timeout: 30_000`, `encoding: 'utf8'`, `windowsHide: true` (suppress console window flash on Windows), `shell: false` (no shell injection vector — command + args parsed via `parseCommandLine` helper that handles quoted args).
- **D-21: Command parsing.** AWS CLI uses simple shell-style splitting on `credential_process` values. Implement a small parser that handles double-quoted args (allows `credential_process = aws-vault exec "my profile" --json`). Single-quotes NOT supported (matches AWS CLI behavior). Backslash-escape for embedded quotes.
- **D-22: Stdout NEVER logged.** Logger redaction has no path to stdout; the resolver simply parses the JSON and returns the object. Stderr CAN be logged truncated (200 chars) for diagnostics.

### Claude's Discretion

- **Module structure**: Single file `awsCredentials.ts` grows from 138 LoC to ~330 LoC, OR split into `awsCredentials.ts` (resolver entry) + `awsConfigFiles.ts` (INI parser + file lookup) + `credentialProcess.ts` (spawn + cache). Planner picks based on test ergonomics; recommendation is split for unit-test isolation but not load-bearing.
- **Test organization**: Either expand `tests/ai/awsCredentials.test.ts` or split into per-concern files. Planner picks.
- **Notice prefix for credential_process errors**: Planner picks between `'Bedrock: '` (matches existing provider-prefix pattern in `extractProviderError`) and `'credential_process: '` (more specific, but breaks the per-provider Notice convention). Recommendation: `'Bedrock: '` to match `prettyName(provider)` precedent from Plan 07-04.

</decisions>
