# Phase 08.2: Code Review Report

**Reviewed:** 2026-05-17T12:00:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Phase 08.2 implements a canonical AWS credential chain (env vars, profile resolution, file reads, credential_process delegation) for the Bedrock provider. The implementation is generally solid with good defensive coding (safe file reads, truncated stderr, timeout enforcement, cache with TTL). However, there is a behavioral inconsistency between the exported test-utility INI parser and the production INI parser that creates a false sense of test coverage, a credential leakage path in error messages, and a precedence bug where credential_process overrides explicit static keys.

## Critical Issues

### CR-01: credential_process overrides static keys when both present in same profile

**File:** `src/ai/awsCredentials.ts:222-231`
**Issue:** When a profile has BOTH static keys (`aws_access_key_id`/`aws_secret_access_key`) AND a `credential_process` directive, the resolver unconditionally delegates to `credential_process` (line 222: `if (merged.credential_process)`). Per the [AWS CLI documentation](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-files.html#cli-configure-files-using), static credentials take precedence over `credential_process` when both are present in the same profile. The current behavior causes unnecessary process spawns and potential failures when users have both configured (a common pattern for fallback/migration scenarios).

This is particularly impactful because `credential_process` failures HARD-FAIL (D-05) -- so a profile with valid static keys but a broken credential_process helper will crash instead of using the working static keys.

**Fix:**
```typescript
// Line 222 — check static keys FIRST, only fall through to credential_process
// when static keys are absent.
if (merged.aws_access_key_id && merged.aws_secret_access_key) {
  // Static keys present — use them (AWS CLI precedence).
  return {
    accessKeyId: merged.aws_access_key_id,
    secretAccessKey: merged.aws_secret_access_key,
    sessionToken: merged.aws_session_token,
  };
}

if (merged.credential_process) {
  // credential_process present -> delegate to sync cache+spawn helper.
  const result = getCachedOrRefreshSync(profileName, merged.credential_process);
  return {
    accessKeyId: result.accessKeyId,
    secretAccessKey: result.secretAccessKey,
    sessionToken: result.sessionToken,
  };
}

// Neither present — return empty.
return {
  accessKeyId: merged.aws_access_key_id,
  secretAccessKey: merged.aws_secret_access_key,
  sessionToken: merged.aws_session_token,
};
```

## Warnings

### WR-01: Exported parseIniProfile has different semantics from production parseIniSection — false test coverage

**File:** `src/ai/awsCredentials.ts:294-323`
**Issue:** The exported `parseIniProfile` function (used in unit tests at line 222 of awsCredentials.test.ts) matches sections using `name === profile || name === 'profile ${profile}'` (line 306) -- meaning it treats BOTH bare `[work]` and `[profile work]` as matches regardless of file type. However, the production `parseIniSection` (line 248) discriminates by `fileType` parameter: credentials files use bare `[name]`, config files use `[profile name]` (with special `[default]`/`[profile default]` handling). Tests exercising `parseIniProfile` pass but do NOT validate the actual production behavior. A bug in `parseIniSection`'s config-file logic would not be caught.

**Fix:** Either remove `parseIniProfile` (dead code in production) or align its semantics with `parseIniSection` by accepting a `fileType` parameter. Better yet, export `parseIniSection` directly for testing:
```typescript
// Replace the export with the actual production function
export { parseIniSection as parseIniSectionForTesting };
```

### WR-02: credential_process stdout leaked in error message violates D-22 spirit

**File:** `src/ai/credentialProcess.ts:195-198`
**Issue:** When `credential_process` returns invalid JSON, up to 200 characters of raw stdout are embedded in the error message: `credential_process returned invalid JSON: ${stdout.slice(0, 200)}`. This error propagates through `probeBedrock` -> `extractProviderError` -> `Notice` (shown to user in main.ts:916). If the helper outputs a partial JSON blob containing credential material (e.g., it printed `{"AccessKeyId":"AKIA...","SecretAccessKey":"wJalr..."` but failed mid-stream), those secrets appear in the Obsidian Notice. While the D-22 invariant says "stdout NEVER passed to any logger call", this path surfaces stdout in error messages that reach the UI and could be captured by future logging.

**Fix:**
```typescript
throw new Error(
  'credential_process returned invalid JSON (output not shown for security)',
);
```

### WR-03: getCachedOrRefresh async coalescing is ineffective — inflight promise deleted before second caller can observe it

**File:** `src/ai/credentialProcess.ts:295-313`
**Issue:** The async coalescing logic stores a promise in `inflightPromises` (line 311), but because `spawnSync` is blocking and the promise body runs synchronously via `Promise.resolve().then(...)`, the `finally` block (line 307) deletes the inflight entry before any subsequent microtask-queued `.then()` from concurrent callers can observe it. In practice:
1. Caller A calls `getCachedOrRefresh` -> creates promise, stores in map.
2. `Promise.resolve().then(...)` schedules the sync work as a microtask.
3. Caller B calls `getCachedOrRefresh` in the same synchronous frame -> finds the promise in the map -> returns it. (This path works.)
4. But if Caller B arrives after the microtask fires (the spawnSync already ran and `finally` deleted the entry), Caller B creates a NEW promise and spawns again.

The test at line 551 of `credentialProcess.test.ts` passes only because all 5 `Promise.all` calls are queued before any microtask runs. In real usage with event-loop interleaving, coalescing may fail. The sync cache (`getCachedOrRefreshSync`) prevents duplicate spawns anyway (the cache entry becomes `'fresh'` after the first spawn), so this is not a correctness bug but the claimed D-12 guarantee is weaker than documented.

**Fix:** Acknowledge in comments that true coalescing relies on the sync cache, not the async promise dedup. Alternatively, keep the inflight promise alive until the NEXT event loop tick:
```typescript
const promise = Promise.resolve().then(() => {
  return getCachedOrRefreshSync(profileName, credentialProcessValue);
}).finally(() => {
  // Defer deletion to next microtask so concurrent awaits resolve first
  Promise.resolve().then(() => inflightPromises.delete(profileName));
});
```

## Info

### IN-01: Dead exported function parseIniProfile is unused in production code

**File:** `src/ai/awsCredentials.ts:294-323`
**Issue:** `parseIniProfile` is exported and has doc comments saying "Exported for unit testing — production callers go through resolveAwsCredentials." However, it is never imported by any production source file (only tests). It has divergent semantics from the actual production parser (see WR-01). This is dead code that increases maintenance surface area.

**Fix:** Remove or mark as `@internal`/`@deprecated`. If kept for test convenience, add a comment warning that its behavior differs from the production `parseIniSection`.
