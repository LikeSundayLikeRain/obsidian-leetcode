# Phase 5.1: Edit-mode Inline Run/Submit Buttons — Research

**Researched:** 2026-05-10
**Domain:** Obsidian CodeMirror 6 editor extensions — widget decorations anchored to a specific Markdown fence inside Live Preview + Source Mode
**Confidence:** HIGH (Context7 docs + installed-package type inspection + working community-plugin prior art all converged on a single pattern)

## Summary

Phase 5.1's ask — *inline Run/Submit buttons directly below the `## Code` fenced block, in both Live Preview and Source Mode, without block-widget layout corruption* — maps cleanly onto a **line-level non-block widget decoration** attached to the **line immediately after the closing fence**, provided by a `StateField<DecorationSet>` that rebuilds on every transaction. This is the same CM6 primitive stack (`registerEditorExtension` + `StateField.define<DecorationSet>` + `Decoration.widget` + `RangeSetBuilder` + `WidgetType.toDOM`) that ships in Obsidian's own developer docs and in at least two production community plugins (Strudel plugin, Code-Styler).

The 05-UAT G1 failure was **not** "CM6 widgets are wrong for this" — it was specifically `block: true`, which creates a full-line vertical-block structure that fights Obsidian's Live Preview widget layer (Obsidian's own `livePreviewState` ViewPlugin renders fenced blocks as formatted block widgets; plugin-side block widgets stacking on those reserve extra line height). The fix is **inline (non-block) widgets** attached at the end of a line — these flow in the existing line box and do not reserve vertical space. The Strudel plugin confirms the field-based pattern works at production scale; the key deviation from Strudel is that we use `block: false` (the default) and attach at the line **after** the closing fence rather than the line **before** the opening fence.

Detection of the `## Code` fence is solvable two ways — Lezer syntaxTree traversal (`syntaxTree(state).iterate()` looking for `FencedCode` nodes under an H2 whose text is `Code`) or line-text regex over `state.doc` (Strudel's approach: regex for fence opener lines, track from→to, walk forward for matching closer). We recommend **line-text regex** because (a) it's materially simpler to unit-test, (b) the Strudel prior art proves it scales, (c) it doesn't need `@codemirror/language` as an explicit dep, and (d) the `## Code` detection is a trivial backward scan from the fence opener line.

**Primary recommendation:** Ship one `StateField<DecorationSet>` extension registered via `plugin.registerEditorExtension()`. Detection = regex line-scan for fenced blocks preceded by a `## Code` H2. Widget = `Decoration.widget({ widget, side: 1 })` (inline, no `block: true`) attached at the start-of-line position **immediately after** the closing fence's line-end. Widget's `toDOM()` returns the exact DOM produced by the existing `buildCodeBlockButtonRow(doc, plugin)` helper — reusing CF-07 createEl discipline and the locked `.leetcode-code-actions` class contract. Gate on `lc-slug` frontmatter via `editorInfoField` + `metadataCache`. Live-smoke checkpoint (D-10) is non-negotiable — layout-corruption regressions are invisible to unit tests.

Data flow summary: user opens an `lc-slug` note → CM6 creates an EditorView → our extension's StateField fires `create()` → scans `state.doc` for a `## Code` fence → builds a DecorationSet with one `Decoration.widget` at the position right after the closing fence → EditorView renders the widget DOM inline → user sees Run + Submit buttons inline below the fence → click dispatches to `plugin.runFromActive()` / `plugin.submitFromActive()`. On every transaction (typing, scrolling, toggling Source↔Live-Preview), `update()` fires, re-runs `buildDecorations(state)`, and hands a fresh DecorationSet to the view. CM6 diffs the sets and keeps the DOM stable if the widget's position + `eq()` result are unchanged (Pitfall 3 idempotency: solved by returning `true` from `CodeActionsWidget.eq(other)` for equivalent plugin references).

### Pattern 1: StateField-based decoration extension (canonical)

**What:** A CM6 extension that derives a `DecorationSet` from editor state, provided via `EditorView.decorations.from(field)`. Rebuilds deterministically on every transaction.

**When to use:** Any time you need to paint inline widgets based on document text OR external state (frontmatter, plugin settings). This is the Obsidian-endorsed pattern — see obsidian-developer-docs `en/Plugins/Editor/Decorations.md`. **[CITED: https://github.com/obsidianmd/obsidian-developer-docs/blob/main/en/Plugins/Editor/Decorations.md]**

**Example (near-verbatim from Obsidian docs + Strudel prior art):**
```typescript
// Source: /obsidianmd/obsidian-developer-docs "Decorations" page + Strudel plugin pattern [VERIFIED: ctx7 fetch 2026-05-10]
import { StateField, RangeSetBuilder, type EditorState, type Transaction } from '@codemirror/state';
import { Decoration, DecorationSet, EditorView, WidgetType } from '@codemirror/view';
import { editorInfoField, editorLivePreviewField } from 'obsidian';

function buildDecorations(state: EditorState, plugin: PluginWithHostMethods): DecorationSet {
  const builder = new RangeSetBuilder<Decoration>();

  // Step 1: frontmatter gate (D-06 / CF-13)
  const file = state.field(editorInfoField)?.file;
  if (!file) return builder.finish();
  const fm = plugin.app.metadataCache.getFileCache(file)?.frontmatter as
    Record<string, unknown> | undefined;
  const slug = fm?.['lc-slug'];
  if (typeof slug !== 'string' || slug.length === 0) return builder.finish();

  // Step 2: locate the `## Code` fence
  const fence = findCodeFence(state);   // returns { openerLine, closerLine } | null
  if (!fence) return builder.finish();

  // Step 3: anchor widget at end of closer line (inline, not block)
  const anchor = state.doc.line(fence.closerLine).to;
  builder.add(
    anchor,
    anchor,
    Decoration.widget({
      widget: new CodeActionsWidget(plugin),
      side: 1,   // after the position; NOT block:true
    }),
  );

  return builder.finish();
}

export function buildCodeActionsEditorExtension(plugin: PluginWithHostMethods) {
  return StateField.define<DecorationSet>({
    create(state) {
      return buildDecorations(state, plugin);
    },
    update(old, tr) {
      // Rebuild only when it can matter — docChanged OR Source↔Live-Preview flip.
      if (
        tr.docChanged ||
        tr.state.field(editorLivePreviewField) !== tr.startState.field(editorLivePreviewField)
      ) {
        return buildDecorations(tr.state, plugin);
      }
      return old.map(tr.changes);
    },
    provide(field) {
      return EditorView.decorations.from(field);
    },
  });
}

class CodeActionsWidget extends WidgetType {
  constructor(private readonly plugin: PluginWithHostMethods) { super(); }

  toDOM(view: EditorView): HTMLElement {
    // Reuse the reading-mode DOM builder (CF-07 createEl discipline,
    // class-contract centralized, zero CSS drift risk).
    return buildCodeBlockButtonRow(view.dom.ownerDocument, this.plugin);
  }

  eq(other: CodeActionsWidget): boolean {
    // Pitfall 3 idempotency — widgets with same plugin reference are equivalent.
    // CM6 will reuse the existing DOM instead of re-rendering. NO flicker.
    return other.plugin === this.plugin;
  }

  ignoreEvent(): boolean {
    // false = let click events propagate through to the row's button handlers.
    return false;
  }
}
```

### Pattern 2: Fence detection via doc line-scan (recommended over Lezer)

**What:** Walk `state.doc.iterLines()` (or equivalent `state.doc.line(n)` loop), regex-match fence openers + H2 headers, track the `## Code` → nearest-following fence pair.

**When to use:** When you need cheap, test-friendly fence detection without pulling in `@codemirror/language` as an explicit dep. The Strudel plugin ships this exact approach in production. **[VERIFIED: github.com/dudaanton/obsidian-strudel-plugin/blob/main/src/editor/StrudelPlugin.ts]**

**Example:**
```typescript
// Given a state, find the `## Code` fence (opener + closer lines).
// Returns { openerLine, closerLine } — both 1-indexed per CM6's state.doc.line() API.
// Returns null if no ## Code section exists or the fence is unterminated.
function findCodeFence(state: EditorState): { openerLine: number; closerLine: number } | null {
  const FENCE_RE = /^\s*```/;
  const H2_CODE_RE = /^\s*##\s+Code\s*$/;
  const H2_ANY_RE  = /^\s*##\s+.+$/;

  let inCodeSection = false;
  const total = state.doc.lines;

  for (let i = 1; i <= total; i++) {
    const line = state.doc.line(i);
    const text = line.text;

    if (H2_CODE_RE.test(text)) { inCodeSection = true; continue; }
    if (H2_ANY_RE.test(text))  { inCodeSection = false; continue; }

    if (inCodeSection && FENCE_RE.test(text)) {
      // Found opener — walk forward for closer.
      for (let j = i + 1; j <= total; j++) {
        const closer = state.doc.line(j);
        if (FENCE_RE.test(closer.text)) return { openerLine: i, closerLine: j };
      }
      return null;    // unterminated fence
    }
  }
  return null;
}
```

**Why not Lezer:** Lezer detection would look like `syntaxTree(state).iterate({ enter(node) { if (node.name === 'FencedCode') {...} } })` and then backward-walk for an `ATXHeading2`-with-`Code`-text ancestor. It works, but: (a) it requires `@codemirror/language` as a dep, (b) FencedCode node naming isn't guaranteed stable across Lezer markdown versions, (c) the unit test surface expands because you have to stub syntaxTree. Regex is simpler and has a production exemplar.

### Anti-Patterns to Avoid

- **`Decoration.widget({ block: true })`** — 05-UAT G1 confirmed layout corruption. Block widgets create vertical structure that fights Obsidian's `livePreviewState` ViewPlugin rendering of fences as formatted block widgets. Inline widgets (default, or explicit `block: false`) flow in the existing line box and do not reserve height.
- **`Decoration.replace` to replace the closing fence** — would hide the fence backticks, breaking Source Mode user expectation.
- **ViewPlugin with eventHandlers.click on `view.dom`** — works but puts routing logic in the extension instead of the DOM button; also risks double-firing in combination with the widget's own click handlers. Stick with the widget's inner button handlers.
- **Building DOM directly in `toDOM()` without the helper** — duplicates the `.leetcode-code-actions` class contract and invites drift from reading mode's styling.
- **Querying `app.workspace.getActiveViewOfType(MarkdownView)` inside the decoration builder** — wrong level; the editor state already knows its file via `editorInfoField`. The workspace-active-view read is for command-palette handlers, not for per-view extensions.
- **Caching the frontmatter lookup result across transactions** — the metadataCache is already in-memory; the per-build lookup cost is negligible, and caching adds invalidation complexity (file rename, external edit).
- **`side: -1`** — places widget before the anchor position. We want it after → `side: 1`. Strudel uses `side: -1` because it anchors at the start of the fence block to paint a header widget BEFORE the block; we anchor at the end-of-closer-line to paint below. Opposite direction, opposite sign.
- **Forgetting `eq()` on the WidgetType** — default `eq()` returns false, causing CM6 to rebuild the DOM on every decoration-set swap. This creates visible flicker and churns event-listener attachments. `eq(other) { return other.plugin === this.plugin; }` is sufficient because the plugin reference is stable for the lifetime of an editor view.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DOM builder for Run/Submit row | A new CM6-specific DOM emitter | `buildCodeBlockButtonRow(doc, plugin)` from `src/main/codeBlockButtonRow.ts` | Helper already exists, already test-covered, already satisfies CF-07 + class-contract |

**Key insight:** CM6 + Obsidian together provide every primitive needed for this feature. The temptation in 05-UAT was to reach for block widgets or absolute-positioned overlays. The canonical CM6 answer is "inline widget with `side: 1` at the line-end position" — Strudel, Code-Styler, and the Obsidian docs all converge on StateField + Decoration.widget. Stay on the well-lit path.

## Common Pitfalls

### Pitfall 1: Block widgets corrupt Live Preview layout (05-UAT G1)
**What goes wrong:** `Decoration.widget({ block: true })` reserves a full-line block in the editor's vertical structure. In Live Preview, Obsidian's own `livePreviewState` ViewPlugin renders fenced blocks as formatted block widgets. Stacking a plugin-side block widget onto those creates extra line height — the "large blank rectangle" from the 05-UAT G1 screenshot.
**Why it happens:** CM6 block widgets occupy a row in the editor's visual block layout. That row is composed with Obsidian's own block-rendering of fences, and the two layers don't coordinate.
**How to avoid:** Use inline widgets (default, or `block: false`). Anchor at `state.doc.line(closerLine).to` with `side: 1` — this attaches the widget to the end of the closing fence's line, rendered in-flow at the line's right edge (styled with `justify-content: flex-end` via the existing `.leetcode-code-actions` CSS).
**Warning signs:** Any visible blank rectangle inserted mid-document → immediately suspect a block widget. Toggling to Source Mode and back is a fast discriminator.
**[CITED: .planning/phases/05-polish-ship/05-UAT.md Test 4 failed-approach notes]**

### Pitfall 2: Widget DOM flicker on every transaction
**What goes wrong:** If `WidgetType.eq()` returns false (the default), CM6 rebuilds the widget DOM on every decoration-set swap. Users see flicker; click handlers re-attach; transient focus loss on keyboard-nav.
**Why it happens:** CM6's decoration differ uses `eq()` to decide if a rendered DOM can be reused. Default `WidgetType.eq` compares instances (always false for new instances).
**How to avoid:** Override `eq(other)` to return true when the rendered output would be identical. For Phase 5.1: `eq(other) { return other instanceof CodeActionsWidget && other.plugin === this.plugin; }` — plugin reference is stable for the view's lifetime.
**Warning signs:** Buttons visibly redraw on every keystroke. Chrome DevTools' "Paint flashing" flag lights up over the button row.

### Pitfall 3: Widget fires click on drag-select
**What goes wrong:** Click handlers attached to the button DOM fire on any pointer event CM6 interprets as a click — including the release of a text-selection drag that passes over the button.
**Why it happens:** `WidgetType.ignoreEvent()` default is true for non-standard events but false for clicks. Combined with a button's own click handler, this works fine — CM6 suppresses its own selection behavior when a widget swallows the event. But the existing `buildCodeBlockButtonRow` uses `e.preventDefault(); e.stopPropagation();` which is sufficient for this case.
**How to avoid:** Keep the helper's `e.preventDefault(); e.stopPropagation()` in click handlers. Let `WidgetType.ignoreEvent()` return false (default for clicks) so the handlers run; the button's own `stopPropagation` prevents CM6's selection from advancing.
**Warning signs:** Shift-click or drag-across-button triggers Run/Submit.

### Pitfall 4: Frontmatter gate misses freshly-opened notes
**What goes wrong:** On file-open, the metadataCache may not have populated frontmatter for the new file before the StateField's `create()` fires. The slug check sees `undefined`, the decoration set is empty, buttons never appear.
**Why it happens:** `metadataCache.getFileCache(file)` is synchronous but reads a cache that's populated asynchronously on file-change events. For a brand-new file, there's a window where the entry is null.
**How to avoid:** Rely on `update()` re-running on any `docChanged` transaction — Obsidian fires dummy transactions when it parses frontmatter, so the buttons appear with ≤100 ms of lag on first file open. If this turns out to be visibly lagging in live smoke, register `metadataCache` on the plugin events (`plugin.registerEvent(app.metadataCache.on('changed', ...))`) and dispatch a no-op transaction via `view.dispatch({})` to force a StateField re-evaluation. **Skip this complexity unless live smoke proves it's needed.**
**Warning signs:** First open of an `lc-slug` note shows no buttons; they appear after any keystroke.

### Pitfall 5: Toggling Source↔Live-Preview loses decoration
**What goes wrong:** The StateField's `update()` maps the old DecorationSet through `tr.changes` (identity map for view-mode flips, because doc didn't change). If the widget position is still valid, this is fine — but if the `update()` path has a bug and returns stale `old` when editorLivePreviewField flipped, the widget may vanish or appear in the wrong place.
**Why it happens:** Some extensions branch on `editorLivePreviewField` and hide decorations in Source Mode (Strudel does this). Phase 5.1 does NOT branch — both modes must show buttons. The `update()` must rebuild on editorLivePreviewField flip to avoid stale state.
**How to avoid:** `if (tr.docChanged || tr.state.field(editorLivePreviewField) !== tr.startState.field(editorLivePreviewField))` — rebuild on either condition. Phase 5.1's implementation always produces decorations regardless of mode, but rebuilding ensures consistency (e.g., if the code path ever reads Live Preview state).
**Warning signs:** Buttons vanish after toggling mode with Cmd-E; appear after one keystroke.

### Pitfall 6: `editorCheckCallback` gate regression (from 05-05)
**What goes wrong:** If the click handlers dispatch via `executeCommandById('leetcode:run')` instead of calling `plugin.runFromActive()` directly, Obsidian's `editorCheckCallback` gate runs first. In Reading Mode it gated out (because there's no editor). In Edit Mode it should pass — but the gate still fires, and any bug in it silently no-ops the button.
**Why it happens:** `editorCheckCallback` guards the command palette entry; a command dispatched via `executeCommandById` walks through the same gate.
**How to avoid:** **Call `plugin.runFromActive()` / `plugin.submitFromActive()` DIRECTLY** (D-05). This is what 05-05's live-smoke fix converged on. The helper already does this — no change needed. Just make sure the new CM6 widget also uses the helper (not a custom executeCommandById wiring).
**Warning signs:** Palette Run works; inline-button Run silently no-ops.

### Pitfall 7: Unit tests pass while layout is corrupted (05-UAT G1 lesson)
**What goes wrong:** Every unit test for decoration logic, frontmatter gate, and click dispatch passes green. Live Preview is visibly broken anyway — because the corruption is a visual composition between the plugin's widget and Obsidian's own Live Preview layer, which has no test double.
**Why it happens:** CM6's real rendering pipeline is not reproduced by happy-dom. Widgets, block structure, and Obsidian's livePreviewState ViewPlugin interaction are all runtime-only.
**How to avoid:** **Mandatory live-smoke checkpoint between waves (D-10).** Before closing any wave that touches the CM6 extension, dev-vault Cmd-R reload → open an `lc-slug` note → Live Preview → scroll → Source Mode → scroll → Reading Mode regression → eyes-on verify. Unit tests build confidence in logic; live smoke is the only gate for layout correctness.
**Warning signs:** High unit-test coverage with no live-smoke evidence → treat as unverified.

### Pitfall 8: `state.doc.line(n)` on an empty doc throws
**What goes wrong:** If `state.doc.lines === 0`, `state.doc.line(1)` throws. Brand-new empty notes trigger this.
**Why it happens:** CM6 documents can be empty; `line(n)` requires `1 ≤ n ≤ doc.lines`.
**How to avoid:** Guard `findCodeFence` with an early return when `state.doc.lines === 0`. Alternatively, `state.doc.length === 0` also works.
**Warning signs:** Console error "RangeError: Line 1 is out of range (doc has 0 lines)" on opening a blank note.

### Pitfall 9: Multiple fences under `## Code` — which one gets buttons?
**What goes wrong:** A user puts two fenced blocks under `## Code` (e.g., reference implementation + their own attempt). Our code attaches buttons after the first fence. Is that right?
**Why it happens:** Reading-mode `findCodeSectionPre` has the same behavior — first `<pre>` under `## Code` gets the buttons.
**How to avoid:** Match reading mode's behavior exactly: attach to the FIRST fence whose opener line follows a `## Code` H2 and precedes the next H2. Document this explicitly in the module docstring. If users have multiple fences under `## Code`, buttons appear below the first one (consistent with 05-05 `codeActionsPostProcessor.ts`). This is the current reading-mode contract — Phase 5.1 mirrors, doesn't innovate.
**Warning signs:** User reports "buttons on the wrong code block" → verify the fence ordering matches reading-mode UX.

### Pitfall 10: `view.dom.ownerDocument` vs `document`
**What goes wrong:** In Obsidian, each window (main + any popout panes) has its own `document`. Using the global `document` when the editor is in a popout window creates DOM in the wrong context, which fails silently or renders invisibly.
**Why it happens:** Obsidian supports popout windows (File → Open in new window). Each has its own DOMParser / document.
**How to avoid:** In `WidgetType.toDOM(view)`, use `view.dom.ownerDocument` — the Document that owns the editor's DOM node. The helper already takes a `Document` parameter, so just thread it through.
**Warning signs:** Popout-window problem notes show no buttons; main-window ones work fine.

## Code Examples

### Full editor extension (complete reference implementation)
```typescript
// src/main/codeActionsEditorExtension.ts — Phase 5.1 canonical implementation
// Source: Obsidian docs "Decorations" page + Strudel plugin prior art [VERIFIED: ctx7 + gh api fetch 2026-05-10]

import { StateField, RangeSetBuilder, type EditorState } from '@codemirror/state';
import { Decoration, DecorationSet, EditorView, WidgetType } from '@codemirror/view';
import { editorInfoField, editorLivePreviewField, type Plugin } from 'obsidian';
import { buildCodeBlockButtonRow, type CodeBlockButtonRowHost } from './codeBlockButtonRow';

type PluginHost = Plugin & CodeBlockButtonRowHost;

class CodeActionsWidget extends WidgetType {
  constructor(private readonly plugin: PluginHost) { super(); }

  toDOM(view: EditorView): HTMLElement {
    return buildCodeBlockButtonRow(view.dom.ownerDocument, this.plugin);
  }

  eq(other: CodeActionsWidget): boolean {
    return other instanceof CodeActionsWidget && other.plugin === this.plugin;
  }

  ignoreEvent(): boolean {
    return false;  // allow click events to propagate to the button handlers
  }
}

export function findCodeFence(state: EditorState): { openerLine: number; closerLine: number } | null {
  if (state.doc.lines === 0) return null;
  const FENCE_RE = /^\s*```/;
  const H2_CODE_RE = /^\s*##\s+Code\s*$/;
  const H2_ANY_RE  = /^\s*##\s+.+$/;
  let inCodeSection = false;
  const total = state.doc.lines;
  for (let i = 1; i <= total; i++) {
    const text = state.doc.line(i).text;
    if (H2_CODE_RE.test(text)) { inCodeSection = true; continue; }
    if (H2_ANY_RE.test(text))  { inCodeSection = false; continue; }
    if (inCodeSection && FENCE_RE.test(text)) {
      for (let j = i + 1; j <= total; j++) {
        if (FENCE_RE.test(state.doc.line(j).text)) return { openerLine: i, closerLine: j };
      }
      return null;
    }
  }
  return null;
}

function buildDecorations(state: EditorState, plugin: PluginHost): DecorationSet {
  const builder = new RangeSetBuilder<Decoration>();
  const file = state.field(editorInfoField)?.file;
  if (!file) return builder.finish();
  const fm = plugin.app.metadataCache.getFileCache(file)?.frontmatter as
    Record<string, unknown> | undefined;
  const slug = fm?.['lc-slug'];
  if (typeof slug !== 'string' || slug.length === 0) return builder.finish();
  const fence = findCodeFence(state);
  if (!fence) return builder.finish();
  const anchor = state.doc.line(fence.closerLine).to;
  builder.add(anchor, anchor, Decoration.widget({
    widget: new CodeActionsWidget(plugin),
    side: 1,
  }));
  return builder.finish();
}

export function buildCodeActionsEditorExtension(plugin: PluginHost) {
  return StateField.define<DecorationSet>({
    create(state) { return buildDecorations(state, plugin); },
    update(old, tr) {
      if (
        tr.docChanged ||
        tr.state.field(editorLivePreviewField) !== tr.startState.field(editorLivePreviewField)
      ) {
        return buildDecorations(tr.state, plugin);
      }
      return old.map(tr.changes);
    },
    provide(field) { return EditorView.decorations.from(field); },
  });
}
```

## Open Questions

1. **Should the `## Code` fence detection match reading-mode's `findCodeSectionPre` exactly, or relax to any fence under `## Code`?**
   - What we know: Reading-mode attaches to the first fence under `## Code`.
   - What's unclear: If the user has multiple fences under `## Code` (rare but possible — reference implementation + their attempt), should edit-mode buttons also attach to the first?
   - Recommendation: Match reading-mode exactly — first fence only. If a user reports confusion, a future phase can add "buttons below every fence under `## Code`" as an opt-in. Phase 5.1 prioritizes reading-mode parity.

2. **Should we rebuild decorations on `active-leaf-change` or `metadata-change` Obsidian events, or rely solely on CM6 transactions?**
   - What we know: CM6 fires transactions on doc edits + mode flips. Metadata-change events (Obsidian-level, frontmatter parse completion) are not CM6 transactions.
   - What's unclear: On first file open, how long is the window where frontmatter isn't yet populated in metadataCache? If >500 ms, users will see a visible "buttons appear late" flash.
   - Recommendation: Start with CM6-only rebuild. If live smoke (D-10) shows visible lag, add `plugin.registerEvent(app.metadataCache.on('changed', (file) => this.forceRebuildForFile(file)))` that dispatches a no-op transaction on the affected view. Don't add this complexity preemptively.

3. **Does `buildCodeBlockButtonRow(doc, plugin)` need any Edit-Mode-specific tweaks (spacing, padding) to line up visually with the CM6 fence?**
   - What we know: Reading-mode HTML rendering produces a `<pre>` element with its own margin. CM6 Live Preview produces a different DOM (editor line elements, not `<pre>`).
   - What's unclear: Whether the existing `.leetcode-code-actions` `margin-top: 4px` lines up visually in both contexts.
   - Recommendation: Ship with existing CSS. Live smoke (D-10) will surface misalignment. If visible, add a `.cm-editor .leetcode-code-actions { margin-top: 8px; }` override in `styles.css`. Budget for this in planning.
