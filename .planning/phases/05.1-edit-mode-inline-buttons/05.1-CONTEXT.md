# Phase 5.1: Edit-mode Inline Run/Submit Buttons - Context

**Gathered:** 2026-05-10
**Status:** Ready for research + planning

<domain>
## Phase Boundary

Phase 5.1 ships Run/Submit buttons anchored **inline directly below the `## Code` fenced block** in Edit Mode (Live Preview + Source Mode) on an `lc-slug` note — closing ship-blocker G1 from 05-UAT. Reading-mode buttons already shipped in 05-05 via `src/main/codeActionsPostProcessor.ts` + `src/main/codeBlockButtonRow.ts`; this phase mirrors that affordance for Edit Mode so users never have to toggle to Reading Mode during normal coding flow.

**In scope (locked by ROADMAP.md §Phase 5.1 success criteria):**

1. In Live Preview and Source Mode on an `lc-slug` note, `Run` + `Submit` buttons appear inline directly below the closing fence of the `## Code` block — anchored to that specific block, scrolling with note content.
2. Buttons are NOT a floating/corner toolbar. They live inline with the document.
3. Click dispatches `runFromActive` / `submitFromActive` (same command handlers as the reading-mode path).
4. No layout corruption in either edit mode — no inserted whitespace, no duplicate widgets, no shifted heading positions, no large blank rectangles.
5. Reading-Mode buttons shipped in 05-05 continue to work unchanged — zero regression.
6. Renders correctly in both light and dark themes.

Requirement: POLISH-07 (gap-closure from 05-UAT G1).

**Explicitly out of scope for Phase 5.1:**

- **Buttons under code blocks OTHER than `## Code`** — reading-mode parity: only the solution fence gets the affordance. Example blocks in `## Problem` stay bare (matches 05-UAT-1).
- **Floating / corner toolbar.** User explicitly rejected this during 05-05 live smoke: "not really what I want, sry, i need something that anchor to the code block." Do not re-propose a floating approach.
- **Block-widget Decoration with `block: true`.** First 05-05 attempt failed — reserved inappropriate line height and fought Obsidian's own Live Preview widget layer that renders fences as formatted blocks. User captured a screenshot showing a large blank rectangle mid-document. Do not re-attempt this specific CM6 pattern.
- **New Notice copy / settings UI.** Phase 5.1 is purely the anchored button surface.
- **Non-LC notes.** Frontmatter gate on `lc-slug` carries forward from 05-05 D-12 — notes without that key show nothing.
- **Hotkeys on any new command.** Phase 5.1 adds zero new registered commands; reuses existing `runFromActive` / `submitFromActive` exposed on `main.ts`.
- **Live Preview + Source Mode parity beyond the two success criteria.** If one mode is harder to anchor cleanly, acceptable fallback patterns are researcher's call — see D-03.

</domain>

<decisions>
## Implementation Decisions

### Button Placement & Anchoring

- **D-01:** **Buttons must anchor inline directly below the `## Code` fenced block's closing fence, scrolling with document content.** Not a floating/corner toolbar. Not a status-bar indicator. Not a ribbon button. The affordance lives in the document flow, visually and spatially tied to the solution fence. This is LOCKED from ROADMAP Phase 5.1 success criterion 1 and user's 05-UAT override quote.

- **D-02:** **Target only the `## Code` section's fence — not every fence in the note.** Reading-mode parity: `src/main/codeActionsPostProcessor.ts::findCodeSectionPre()` walks upward looking for an `## Code` H2 and only attaches to a fence immediately following it. Phase 5.1's Edit-Mode path MUST implement the equivalent guard — example fences in `## Problem`, custom-input fences in `## Custom Tests` (if legacy), any user-added fences anywhere else, all stay bare. Implementation will likely need CM6 Lezer-tree traversal or regex over `EditorState.doc` to find the closest preceding `## Code` H2 relative to each fence.

- **D-03:** **CM6 anchoring strategy — researcher's call, subject to hard constraints.** User explicitly delegated ("these are all just implementation details, you should figure it out"). But the following constraints are LOCKED by 05-UAT G1 notes:
  - **Block `Decoration.widget` with `block: true` is OUT.** First 05-05 attempt; caused visible blank-rectangle layout corruption in Live Preview.
  - **Floating bottom-right toolbar is OUT.** User explicitly rejected during 05-05 live smoke.
  - Acceptable starting points for research: `Decoration.widget` with `side: 1` (inline, not block) at end-of-line after closing fence; `EditorView.decorations` compartment on a per-viewport basis; `ViewPlugin` with DOM-injected overlay that uses `EditorView.contentDOM` geometry; a CSS-only approach that paints into an existing rendered node via `::after`. Researcher enumerates options + picks with rationale.
  - **Live-verify each iteration.** 05-UAT G1 explicit learning: "live-verify each iteration rather than relying on unit tests alone" — unit tests passed while Live Preview was visibly corrupted. Planner MUST include a live-smoke checkpoint gate between waves.

- **D-04:** **Source Mode vs Live Preview parity.** Both modes must show the buttons anchored below the `## Code` fence per success criteria 1 & 4. A single CM6 editor extension covering both is preferred (they share the same underlying `EditorView`). If one mode requires a distinct code path, researcher documents why + both paths must pass the "no layout corruption" bar independently. Reading Mode stays on the shipped `MarkdownPostProcessor` — untouched.

### Click Handler Wiring

- **D-05:** **Click handlers call `plugin.runFromActive()` / `plugin.submitFromActive()` directly.** This matches the 05-05 live-smoke fix: reading-mode buttons originally dispatched via `executeCommandById('leetcode:run' / ':submit')`, but `editorCheckCallback` gated out Reading Mode so the commands no-op'd. The fix was to call the public methods on the plugin instance directly (`src/main.ts::runFromActive` / `submitFromActive` at approximately lines 506 / 641). Phase 5.1 uses the same direct-call pattern for consistency and to avoid re-introducing the `editorCheckCallback` gate issue (in Edit Mode it SHOULD pass, but consistency with the shipped fix beats symmetry with the original D-11 intent).

- **D-06:** **`lc-slug` frontmatter gate applies.** No buttons on non-LC notes. Follows 05-05 D-12. Implementation may cache the frontmatter-check result per-editor to avoid re-querying `metadataCache` on every transaction — researcher's call.

- **D-07:** **Single `Run` + single `Submit` button, neutral styling — no `mod-cta`.** Mirrors reading-mode D-13. Reuse `src/main/codeBlockButtonRow.ts::buildCodeBlockButtonRow` if the DOM shape can be adapted; otherwise replicate the styling via a new helper that emits the same `.leetcode-code-actions` container + `.leetcode-code-action-run` / `.leetcode-code-action-submit` button classes so the existing `styles.css` rules apply without duplication.

### Idempotency & Rebuild Discipline

- **D-08:** **Widget must be idempotent across CM6 transactions.** Pitfall 3 from 05-05 RESEARCH — re-render must not duplicate. CM6's decoration model is declarative (rebuilt per view update), so the natural pattern is "rebuild decorations from scratch each update, guard against flicker with `RangeSet.diff` / equality on the decoration set." Researcher picks the precise mechanism (`ViewPlugin` with `update()` handler, `StateField<RangeSet>`, or compartment-based).

- **D-09:** **Active-file change handling — state wipes naturally.** The editor extension is per-`EditorView`, so switching notes creates a fresh view with fresh decorations. Nothing to do explicitly; no cross-note state leakage. If a long-lived cache is added (e.g., frontmatter-slug cache), researcher documents how it's invalidated on file rename / delete.

### Testing & Verification

- **D-10:** **Live-verify gate between waves — LOCKED lesson from 05-UAT G1.** Planner MUST include a manual live-smoke checkpoint after each wave that touches CM6 layout. Unit tests on decoration logic, frontmatter gate, and idempotency are still required (build confidence in the logic), but they are NOT sufficient to verify "no layout corruption in Live Preview" — that requires eyes on the actual rendered editor. The checkpoint is: dev vault open → Cmd-R reload → toggle to Live Preview on an `lc-slug` note → scroll through → toggle to Source Mode → scroll through → toggle to Reading Mode (regression check) → verify no blank rectangles, no shifted headings, no duplicates, no missing buttons.

- **D-11:** **Test the Reading-Mode regression.** 05-05's shipped `registerCodeBlockActionProcessor` and `SubmissionDetailModal` D-31 upgrade stay untouched. Phase 5.1 adds an Edit-Mode path alongside, not instead. The existing `tests/main/codeActionsPostProcessor.test.ts` must still pass post-Phase-5.1. If the shared `codeBlockButtonRow.ts` helper is refactored, its current reading-mode contract (the 4 tests exercising frontmatter gate, idempotency, button classes, executeCommandById dispatch) stays green.

- **D-12:** **Click-dispatch equivalence test.** Unit-test that the Edit-Mode button's click handler calls `plugin.runFromActive` / `plugin.submitFromActive` — the same methods the reading-mode path was retrofitted to call in 05-05 live smoke. Mock plugin, spy on the methods, assert invocation. This test catches the `editorCheckCallback` regression that bit 05-05.

### Claude's Discretion

User delegated all low-level CM6 implementation choices: "i don't have much context on the low level details. and seems these are all just implementation details, you should figure it out." Researcher + planner have full latitude within the D-01..D-04 guardrails above. Specifically open:

- Precise CM6 API choice for the anchoring widget (line-widget with `side: 1`, post-widget decoration, `ViewPlugin` with absolute-positioned DOM overlay via `posAtCoords` geometry, or another pattern).
- Lezer-tree-based `## Code` detection vs regex-over-`doc.text` detection.
- Whether Source Mode and Live Preview share one extension or have separate code paths.
- Exact rebuild trigger (on every `ViewUpdate`, only on `update.docChanged`, only on heading-structure changes).
- DOM-reuse strategy for the button row (reuse `buildCodeBlockButtonRow` helper vs a CM6-native builder that emits the same class contract).

### Carried Forward (not re-asked)

- **CF-01:** All LC calls through `api/throttle.ts → throttledRequestUrl`. Phase 5.1 adds no new network paths — click handlers delegate to existing `runFromActive` / `submitFromActive` which already route through the throttle pipe.
- **CF-02:** `isDesktopOnly: true`. No new Electron APIs. (CM6 is browser-layer; Electron-neutral.)
- **CF-05:** Zero Required `eslint-plugin-obsidianmd` violations. No `innerHTML`, no `fetch`, no default hotkeys on any command.
- **CF-06:** `vault.process()` + `processFrontMatter()` only. Phase 5.1 adds zero vault writes (pure UI surface).
- **CF-07:** `createEl()` discipline for all DOM. New Edit-Mode button row uses `createEl()` / `createDiv()` — never `innerHTML` / `insertAdjacentHTML`. Reading-mode shipped helper already follows this.
- **CF-11:** Feature-first folder layout. Phase 5.1 extends `src/main/` (peer of `codeActionsPostProcessor.ts` + `codeBlockButtonRow.ts`). Planner may name the new module `src/main/codeActionsEditorExtension.ts` or equivalent; no new top-level `src/` folder.
- **CF-13:** `lc-slug` frontmatter drives LC-note detection. Same key used in reading-mode D-12.
- **CF-19:** Notice copy locked. Phase 5.1 surfaces NO new Notices (button clicks delegate to existing handlers whose Notice copy is already locked).
