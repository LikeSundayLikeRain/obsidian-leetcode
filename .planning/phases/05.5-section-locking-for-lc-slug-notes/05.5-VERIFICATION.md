### Data-Flow Trace (Level 4)

The sectionLockExtension is a CM6 filter/decoration extension — it does not render dynamic data from a store or API. Data flow is:

1. `buildSectionLockExtension(plugin)` receives the Plugin instance for `app.metadataCache` lookups.
2. On every transaction: `tr.startState.field(editorInfoField)` reads the current file from CM6 state (live, not cached).
3. `plugin.app.metadataCache.getFileCache(file)?.frontmatter['lc-slug']` reads live frontmatter (live metadata, not stale).
4. `computeLockedRanges(tr.startState)` scans `state.doc` directly (live document state).

No static fallbacks, no empty-array returns masking real data. Filter correctly returns the flat number[] or `true`. Data flow is FLOWING.

### Human Verification Required

The following behaviors require a real Obsidian instance with a dev vault containing an lc-slug note.

#### 1. Keystroke drop in locked regions (D-04)

**Test:** Open an lc-slug note in Edit Mode (Live Preview). Click inside the `## Problem` body. Type any character.
**Expected:** The character does not appear in the buffer. No Notice appears. No toast, no alert.
**Why human:** `EditorState.changeFilter` side-effects require a real EditorView + Obsidian runtime to confirm the buffer mutation is actually prevented.

#### 2. Editable region writeable (D-03)

**Test:** In the same note, click between the fence opener and closer in `## Code`. Type a character. Also type in `## Techniques` body and `## Notes` body.
**Expected:** All three keystrokes land normally in their respective positions.
**Why human:** Requires real CM6 runtime to confirm the filter passes through these transactions.

#### 3. Chevron language switch bypasses lock (D-04 + Pitfall 5)

**Test:** With an lc-slug note open, click the chevron dropdown next to Run/Submit, pick a different language.
**Expected:** The fence opener line rewrites atomically (e.g., ` ```python` → ` ```java`), starter code loads, `lc-language` frontmatter updates. Single Cmd-Z reverts both the opener and body change together.
**Why human:** The `userEvent: 'leetcode.lang-switch'` bypass must thread through the live changeFilter in the real Obsidian + CM6 context.

#### 4. Non-lc-slug note unaffected (D-06)

**Test:** Open any regular (non-LeetCode) note in Edit Mode. Type freely, including headings.
**Expected:** All keystrokes land normally. The lock is entirely inactive.
**Why human:** metadataCache.getFileCache gate requires live Obsidian plugin runtime.

#### 5. Reading Mode unaffected (D-07)

**Test:** Open an lc-slug note, switch to Reading Mode (Cmd-E or Reading Mode toggle). Confirm the rendered note looks normal.
**Expected:** Reading Mode renders the note normally. No lock artifact, no visual anomaly.
**Why human:** Reading Mode uses MarkdownPostProcessor path — verified architecturally by absence, but visual regression check requires human.

#### 6. Arrow-key cursor skip over locked ranges (atomicRanges — D-04)

**Test:** In an lc-slug note in Edit Mode (Live Preview), place cursor at the last character of the `## Code` body (the line before the closing fence). Press the Down arrow key.
**Expected:** Cursor jumps to the start of the `## Techniques` body, skipping the closing fence line and `## Techniques` heading line.
**Why human:** `EditorView.atomicRanges` cursor-skip behavior requires a real EditorView with rendered DOM; not exercisable in vitest unit tests.

#### 7. Visual dim of locked regions (D-04 + CSS)

**Test:** Open an lc-slug note in Edit Mode in both Light and Dark Obsidian themes.
**Expected:** The following regions have a noticeably dimmer background (var(--background-secondary)): `## Problem` heading and body, `## Code` heading line, fence opener line, closing fence line, `## Techniques` heading line, `## Notes` heading line. Editable regions (fence body, ## Techniques body, ## Notes body) have the normal background.
**Why human:** CSS rendering of `Decoration.mark` class `leetcode-section-locked` requires visual inspection; both theme variants should be checked.
