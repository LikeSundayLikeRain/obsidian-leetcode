# Phase 05.5: Section Locking for lc-slug Notes - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 05.5 makes plugin-owned regions of `lc-slug` notes structurally inviolable in Edit Mode (Live Preview + Source). The lock is a **read-only enforcement** in the editor — not a divergence-detection layer on the write path. The plugin keeps overwriting its own sections exactly as it does today; the lock prevents the user from putting edits into those regions in the first place.

Coverage by section:

| Section | Heading line | Body |
|---|---|---|
| `## Problem` | locked (read-only) | locked (read-only) |
| `## Code` | locked | fence opener line + closing fence locked; **fence body editable** |
| `## Techniques` | locked | editable (future AI-driven insertion) |
| `## Notes` | locked | editable (always — user-owned content surface) |

**In scope:**
- CM6 read-only enforcement via `EditorState.transactionFilter` that rejects any change touching a locked range. Cursor visually skips locked ranges; keystrokes drop silently.
- Range computation: scan `state.doc` for the four canonical headings (from `NoteTemplate.ts`), compute the locked ranges per the table above.
- Lock applies only to notes with `lc-slug` frontmatter (matches Phase 5.1 / 5.3 gating pattern).
- Lock applies only in Edit Mode (Live Preview + Source). Reading Mode is unaffected — content is already non-editable there.
- Heading lines (`## Problem`, `## Code`, `## Techniques`, `## Notes`) are ALL locked verbatim — user cannot rename them or delete them via keystroke. This eliminates the D-09 (Phase 2) "rename → re-insert" recovery branch from happening as a side effect of typing.
- `## Code` fence-tag opener line (e.g., ` ```python `) and closing fence ` ``` ` are locked so the user cannot break the fence by typing or deleting at the boundary; language switches still go through the chevron (Phase 5.3 D-04).
- ` ## Code` fenced body (between opener and closer) remains freely editable — this is THE active editing surface (SOLVE-01).
- ` ## Techniques` body remains freely editable (user reserves the right to add manual `[[Wikilinks]]`; future AI-driven `## Techniques` analysis will write here too).
- ` ## Notes` body remains freely editable (the user-owned surface, unchanged from Phase 2 D-08).

**Explicitly out of scope:**
- **Divergence detection / conflict resolution** — D-02 locked: plugin always overwrites its own regions silently on background-refresh / chevron-switch / on-AC writes. No hashes, no diffs, no sidecar backups, no Notice on overwrite.
- **Migration of existing user content from plugin sections to ## Notes** — D-05 locked: no scanner, no migration command, no per-note opt-in flag. Behavior is uniform across all `lc-slug` notes from the moment 05.5 ships.
- **Reading Mode lock** — Reading Mode renders, doesn't edit. The CM6 extension is Edit-Mode only (parallels Phase 5.3 D-09 chevron lock).
- **Frontmatter lock** — `lc-*` frontmatter keys keep the union-merge semantics from Phase 2 D-10. User can still see and edit frontmatter via the property panel; plugin still overwrites `lc-*` keys per CF-12.
- **HTML-comment sentinels in note bodies** (`<!-- lc-lock:start -->` etc.) — D-01 rejected this approach. Keep notes visually clean; no body markers.
- **Section-rename recovery rewrites** — D-09 from Phase 2 stays as the fallback (if a note ends up with a renamed `## Problem` heading via some out-of-band path, NoteWriter's `rewriteProblemSection` still re-inserts a fresh `## Problem` block). The lock prevents the rename from happening through normal user typing in the first place.
- **AI insertion into ## Techniques** — explicitly deferred (user mentioned planned AI-driven Techniques analysis as the reason ## Techniques body must stay editable; that AI feature is its own future phase).
- **Lock state visible to user as a chip / sidebar / icon** — no UI surface beyond CM6 cursor behavior. Skipping cursor + dropped keystrokes IS the UX signal.
- **Per-section enable/disable settings toggle** — ship the lock for everyone, no opt-out. If the lock causes friction in dogfood, revisit post-v1.

<decisions>
## Implementation Decisions

### Lock expression in the file (D-01)

- **D-01:** **Heading-only convention. No new markers in the note body.** The four canonical heading literals (`## Problem`, `## Code`, `## Techniques`, `## Notes`, all already in `NoteTemplate.ts`) are the lock anchors. Existing notes work unchanged on first re-open after 05.5 ships — no migration. `NoteTemplate.ts` remains the SSoT for plugin-owned regions; the CM6 extension reads heading literals from there (no duplicated string constants).
  - **Rejected:** HTML-comment sentinels (`<!-- lc-lock:problem-start -->`) — would pollute Source Mode and require migrating hundreds of existing notes.
  - **Rejected:** Frontmatter checksum per section (`lc-section-hashes: { problem: 'abc123' }`) — pairs with divergence detection, but D-02 below rejects detection entirely, so the checksum is useless.
  - **Rejected:** Hybrid heading-convention + frontmatter hash — D-02 rejects detection, hash adds no value.

### Conflict policy (D-02)

- **D-02:** **Plugin always overwrites its own regions silently. No divergence detection, no Notice, no backup.** Status quo: `NoteWriter.backgroundRefresh` rewrites `## Problem` whenever cache is stale (D-11/D-12 from Phase 2); chevron-switch destructively replaces `## Code` body (Phase 5.3 D-04 — atomic CM6 dispatch); copy-to-code overwrites `## Code` body (Phase 5.2 D-10 + Phase 5.3 D-06); on-AC writes update `## Techniques` and frontmatter (Phase 4 D-08/D-09). All these write paths stay byte-identical. **The lock's purpose is to PREVENT user edits from landing in those regions, not to PROTECT user edits if they somehow do.**
  - User-edits-to-plugin-section flow under D-02: lock blocks the keystroke at CM6 entry point → no edit reaches the buffer → no edit can collide with a plugin write → no detection needed.
  - Side benefit: zero new state in frontmatter or `data.json`. Existing notes don't need a one-time migration write.
  - **Rejected:** Detect + skip overwrite — adds CM6/buffer compare logic, requires caching last-plugin-write per section, complicates the write path; does not match the strict-lock posture below.
  - **Rejected:** Detect + sidecar backup — biggest implementation surface for the smallest expected user benefit (lock prevents the divergence; backup serves a case that should never occur).
  - **Rejected:** Hybrid (## Code overwrite-always, ## Problem detect-and-skip) — D-04 below makes ## Code body editable, so chevron-switch doesn't conflict with user edits to fence-tag-line content (which is locked); detect-and-skip on ## Problem is unnecessary because the lock prevents user edits there in the first place.

### Per-section lock scope (D-03)

- **D-03:** **Heading lines locked across the board; body protection varies by section.** Concrete table (re-stating the domain table):
  - **`## Problem`:** heading + entire body locked (read-only). User cannot type inside.
  - **`## Code`:** heading locked. Fence opener line (` ```python `, etc.) locked. Closing fence ` ``` ` locked. **Body BETWEEN opener and closer is freely editable** — this is the SOLVE-01 editing surface; chevron-switch is the only path to change the fence-tag.
  - **`## Techniques`:** heading locked. Body editable (user may add manual `[[Wikilinks]]`; future AI feature writes here).
  - **`## Notes`:** heading locked. Body editable (Phase 2 D-08 surface; user-owned content).
  - **`## Custom Tests`** (legacy, Phase 5 D-08 ignored on read/write): NOT locked. The plugin doesn't touch it; the user can do whatever they want with it. If a planner wants to extend the lock to it later, that's a follow-up.
  - **Rationale for locking ALL plugin headings (not just `## Problem` / `## Code` / `## Techniques`):** locking `## Notes`'s heading line prevents the user from accidentally deleting it (which would orphan their content under `## Techniques` or break Phase 5.4 RunModal lookups that scan for `## Code`'s closer). Heading-line locks are cheap; they keep the entire note structure stable.

### Editor visual treatment (D-04)

- **D-04:** **Hard read-only via CM6 `EditorState.transactionFilter`.** The extension installs a transaction filter that:
  1. Reads each transaction's `changes` (using `ChangeSet.iterChanges`) and computes the affected ranges.
  2. For each affected range, checks whether ANY part of it overlaps a locked range (computed lazily from `state.doc` using `NoteTemplate.ts`'s heading constants + a fence-line scan inside `## Code`).
  3. If overlap is found → drop the entire transaction's changes (return `[]` for `changes`) but pass through selection moves and other annotations. The cursor visually skips over locked ranges via standard CM6 selection semantics.
  4. If no overlap → pass the transaction through unchanged.
- **Cursor behavior:** moving the cursor INTO a locked range is allowed (so user can click anywhere); typing while inside a locked range silently drops. This matches the LSP / "atomic ranges" pattern used by CodeMirror's own read-only ranges.
- **No Notice / no toast on dropped keystroke** — silent failure is the right UX (Notices on every typed character would be obnoxious). User figures it out immediately ("I'm typing and nothing's appearing → this section is locked").
- **Combinator stack (Edit-Mode CM6 extensions, in load order):**
  1. Phase 5.1 `codeActionsEditorExtension` (Run + Submit + chevron block widget) — unchanged.
  2. Phase 5.3 chevron `languageRefreshEffect` listener — unchanged.
  3. **NEW (05.5):** `sectionLockExtension` — transaction filter + (optionally) a CM6 decoration that visually dims locked-region backgrounds.
- **Rejected:** Soft warn-on-edit (let user type, revert + Notice) — same end-state as hard read-only but adds Notice noise.
- **Rejected:** Dim-only / icon-only (no enforcement, just signaling) — combined with D-02's "always overwrite," user edits would silently disappear on next plugin write. Strict enforcement is consistent.
- **Rejected:** Hard on boundaries + soft on `## Problem` body — adds complexity for no clear win.

### Migration policy (D-05)

- **D-05:** **No migration. Existing user-content-in-plugin-sections gets silently overwritten on next plugin write.** The lock applies uniformly to all `lc-slug` notes from the moment 05.5 ships. There is no scanner, no opt-in flag (`lc-locked: true`), no "Audit notes" command, no Notice at first plugin load.
  - Rationale: D-02 already says "plugin always overwrites; user edits in plugin sections were never officially supported." Adding a migration step is at odds with that posture (it implies the plugin SHOULD have been protecting that content, which D-02 explicitly rejects).
  - Existing notes flow through 05.5 as: user opens note → CM6 lock applies → user can no longer add to plugin sections (so the divergence stops growing). Eventually background-refresh / chevron-switch / on-AC overwrites the section and cleans up any legacy user content there. No data structures change.
  - **Rejected:** Scan-on-load Notice + Move-to-Notes command — adds substantial implementation surface (scanner, mover, UAT) for a one-time benefit.
  - **Rejected:** One-time README-and-Notice warning — README will already document the lock behavior in Phase 5 D-25's Configuration / Troubleshooting sections; per-load Notice is overkill.
  - **Rejected:** `lc-locked: true` per-note flag — adds frontmatter state, complicates the CM6 lock-detection, and creates an "old vs new note" UX cliff.

### Lock activation gate

- **D-06:** **Lock applies only to notes with `lc-slug` frontmatter.** Same gating used by Phase 5.1's `codeActionsEditorExtension` (`editorInfoField` + `metadataCache` frontmatter check). Non-LC notes are unaffected — user keeps full editor freedom on any note that isn't an LC problem.
- **D-07:** **Lock applies only in Edit Mode (Live Preview + Source).** Reading Mode renders only — no edits possible there anyway, so no lock needed. Matches Phase 5.3 D-09 (chevron edit-mode-only).

### Heading-line lock granularity

- **D-08:** **Heading lines lock the entire line including the trailing newline.** User cannot:
  - Edit characters in `## Problem` etc. (would let them rename to `## My Problem`).
  - Delete the heading by selecting the line + pressing Backspace (would orphan the content below).
  - Insert characters before or after the heading text (would break literal-string heading detection in `NoteTemplate.ts` callers).
- **D-09:** **`## Code` fence opener line is locked end-to-end including the language tag.** User cannot edit ` ```python ` to ` ```javascript ` directly — must use the chevron (Phase 5.3 D-04 atomic switch). The closing ` ``` ` is locked likewise. The lock detection finds the opener by scanning from the `## Code` heading line to the next ` ``` ` line; closer is the matching ` ``` ` line that ends the fence. If detection fails (malformed note, no fence) → only the heading line is locked, the body falls through to "everything else editable" — prevents the lock from breaking notes that pre-date Phase 3.

### Claude's Discretion

- Exact CM6 extension file path: `src/main/sectionLockExtension.ts` is the suggested location (parallels `codeActionsEditorExtension.ts`, `languageChevronWidget.ts`). Planner picks based on directory cohesion.
- Whether to add a CM6 `Decoration.mark` for locked ranges (visual dimming of background) on top of the hard transaction filter. Strictly optional; the cursor-skip + keystroke-drop is functional. If added, it should use `var(--background-secondary)` (no hardcoded colors per Phase 5.4 D-06).
- Algorithm for the locked-range computation cache. Naive: re-scan `state.doc` per transaction (cheap for problem notes <2000 lines). Smarter: cache the locked ranges in a `StateField` invalidated by document changes that touch heading lines or fence delimiters. Planner decides based on profiling — simple is fine until proven otherwise.
- Test mock strategy for `transactionFilter`. Phase 5.3 already established a CM6 obsidian-stub pattern (`tests/helpers/obsidian-stub.ts` has `editorInfoField`, `editorLivePreviewField`); this phase extends with whatever transaction-state shape the filter needs to drive in tests.
- Whether to emit a one-shot debug log on first dropped keystroke per session (`logger.debug('section-lock: dropped change in locked range')`). Useful for dogfood; not user-facing.
- Whether the lock should consider an empty `## Problem` (no body, just heading) editable. Per D-03 it's locked unconditionally — user shouldn't be filling in an empty `## Problem` (the cache fetch should populate it). Planner verifies this matches the intent or flags as an edge case during planning.
