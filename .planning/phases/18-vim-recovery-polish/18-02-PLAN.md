<objective>
Close UAT Test 23 / REPAIR-02 partial finding (backlog 999.3) — two compounding regressions revealed by the user's 2026-05-24 reproduction:

**Bug 1 — vim `dd` bypasses parent repair listener:** Plan 17-13's `createParentRepairExtension` is a CM6 `EditorView.updateListener` that observes only CM6 transactions. When the user deletes the fence closer line via vim's `dd` in Normal mode, Obsidian's app-level vim handler edits the doc directly via Obsidian's commands, NOT through CM6 dispatch. Result: `repairFenceStructure` never fires, the closer stays missing until manual repair or app reload.

**Bug 2 — Stale child render after reload on broken-fence note:** After app reload on a broken-fence note where parent doc + lc-language frontmatter both say Java, the child editor renders Python content (`class Solution:` + `def canMeasureWater(...)`). Hypothesis: stale chevron/registry state OR a chevron switch happened during vim-driven editing that got cached and replayed on reload (likely interaction with Bug 1 — vim writes flipping things outside CM6).

Per CONTEXT D-33 + D-34, the fix path is two-fold: (a) add `this.app.vault.on('modify', file => ...)` listener at plugin onload, gated on lc-slug frontmatter, that calls `repairFenceStructure` when `findCodeFence(parentView.state) === null` (Bug 1 closure); (b) at child mount AND on each `metadataCache.changed` for the active LC note, compare the child's tracked language slug (childLanguageTracker WeakMap) against parent fence opener tag AND lc-language frontmatter — on disagreement, force registry invalidation so the child re-mounts with the correct language (Bug 2 closure).

Plan 17-13's CM6-side trigger (`createParentRepairExtension`) is preserved verbatim — both triggers coexist; one observes CM6 transactions, the other observes vault writes from any source. The Plan 17-13 idempotency guard (skip when transactions carry `'leetcode.fence-repair'` userEvent) continues to protect against re-entry.

Purpose: REPAIR-02 is a v1.2 ship blocker — vim users currently see their dd-deleted fence closer stay broken until reload, and on reload the child editor renders the wrong language entirely. Both bugs corrupt the user's solution flow and must close before v1.2 ships.

Output: extended `src/main/childEditorSync.ts` with a small wrapper for vault.on('modify')-path repair (or no source change there if the existing `repairFenceStructure` export is sufficient); extended `src/main.ts` with the vault.on('modify') registration in onload AND the stale-child invalidation check in the existing `metadataCache.on('changed')` callback at line 920 + at the child mount call site; extended `tests/main/childEditorSync.repair.test.ts` with at least 2 new it blocks; round-3 findings appended to `.planning/debug/fence-auto-recovery-regression-round2.md`.

export function createParentRepairExtension(): Extension {
  return EditorView.updateListener.of((update) => {
    if (!update.docChanged) return;
    if (update.transactions.some(

    )) return;
    if (findCodeFence(update.state) !== null) return;
    const activeSlug = readLcLanguageFromDoc(update.state) ?? 'python3';

}
