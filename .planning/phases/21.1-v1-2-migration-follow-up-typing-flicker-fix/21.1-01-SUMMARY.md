# Plan 21.1-01 Summary — Typing-flicker fix + R6 fresh-create regression closure

## Outcome

MIGRATE-FLICKER-01 resolved. Plan 21.1-01 was scoped to fix UAT R10 (typing flicker on v1.3 LC widgets in Live Preview when `autoMigrateOnOpen=ON`). The TDD GREEN landed cleanly on the first cycle. UAT human-verify surfaced an R6 fresh-create regression that took four iterations to close — the final fix targeted the root cause rather than the symptom.

## What changed

### R10 fix (the planned scope)

**The bug:** `autoMigrateOnOpen=ON` gated two distinct fire-and-forget side-effect paths that re-entered on EVERY parent CM6 docChange (not just on file-open):

1. `src/widget/codeBlockProcessor.ts` — Reading-mode + LP rendered code-block post-processor. On every LP body flush, the handler awaited `migrateLegacyFenceIfNeeded` AND `repairFrontmatterIfNeeded` before constructing the widget. Each function performed `app.vault.read(file)` (~30-100ms on macOS APFS). For a fully-migrated note both calls returned false, but the disk I/O still serialized the widget render — the user saw a brief unmount/remount cycle synchronized with the DebouncedWriter ~500ms cadence.

2. `src/widget/liveModeBannerStateField.ts` — LP StateFields rebuilt on every `tr.docChanged`. The `buildLegacyBannerDecorations` and `buildLeetCodeWidgetDecorations` builders fired migrate / repair as fire-and-forget side-effects gated only on `migrateInFlight` / `repairInFlight` Sets. Those Sets cleared in `.finally()`, so on EVERY docChange after the previous fire completed, the gate was OPEN AGAIN.

**The fix:** Per-path attempt-once-this-session Sets (`migrateAttempted` / `repairAttempted` on the plugin instance, `codeBlockProcessorMigrateAttempted` at module scope in codeBlockProcessor.ts) that survive across `.finally()`. Cleared on `vault.on('rename')` (per-path), Plugin.onunload (all paths), and the "Migrate this note" command-palette entry (per-path). NOT cleared on `workspace.on('file-open')` because that event fires on tab focus and clearing there would reintroduce flicker on tab-switch.

### R6 fresh-create regression (discovered during UAT)

**The bug:** Opening a fresh problem from the problem browser left the LC widget unmounted — the user saw a read-only Java-syntax-highlighted fence with Obsidian's native "Edit this block" hover chrome instead of the v1.3 widget. Workarounds: hover→Edit-this-block, close+reopen tab, or sometimes the next problem opened correctly (alternating pattern observed).

**Root cause (the user's framing crystallized this):** Fundamental difference between opening an existing note vs a fresh problem —

- **Existing note:** metadataCache is populated when CM6 constructs the EditorState. `leetCodeWidgetStateField.create()` sees populated frontmatter on the first call, emits `Decoration.replace`, CM6 mounts the widget. Single clean pass.
- **Fresh problem:** `NoteWriter.openProblem` does `vault.create` → `applyFrontmatter` → `openLinkText`. `applyFrontmatter` resolves BEFORE `metadataCache.changed` fires. `openLinkText` constructs the EditorView while metadataCache still returns null. `leetCodeWidgetStateField` returns `Decoration.none` on first call (no slug visible), and Obsidian's built-in markdown CodeBlockWidget claims the fence range with its own `Decoration.replace`. The later `metadataCache.changed` dispatch can't reliably evict Obsidian's widget from the rendered viewport.

**The fix (commit `1a52f38`):** Wait for metadataCache to actually index the new file's `lc-slug` BEFORE calling `openLinkText`. Bounded poll: 16 ticks × 50ms = ~800ms ceiling. The fresh-create flow now matches the existing-note flow — `EditorState.create` runs against populated frontmatter, single-pass clean mount.

**Belt-and-suspenders fixes that landed in earlier commits:**
- `2bf274e`: When `repairFrontmatterIfNeeded` returns true on the codeBlockProcessor path, mirror the migrate path's `rerenderReadingModePanes + renderStaticFallback + return` shape instead of falling through to `addChild` against a polluted/stale `el`.
- `8330cb9`: Wire `metadataCache.on('changed')` to dispatch `leetcodeRefreshAnnotation` to LP leaves so `leetCodeWidgetStateField` recomputes against the now-indexed frontmatter.
- `81dcfa0`: Pair the annotation with a no-op selection set on the same transaction so CM6's selection-change pathway forces a viewport rebuild that picks up the freshly-emitted `Decoration.replace`.

The four belt-and-suspenders fixes reduce the race window but were insufficient as the primary fix (CM6 viewport recommit on annotation+selection works intermittently — depends on Obsidian's internal scheduler). The root-cause fix at `1a52f38` eliminates the race entirely.

## Lessons for future work

1. **Race conditions: prefer eliminating the race over patching its symptoms.** The first three R6 fixes (rerender hand-off, metadataCache.changed dispatch, annotation+selection pairing) all addressed visible symptoms of the underlying race. Each reduced but didn't eliminate the issue. The user's reframe — "what's fundamentally different between fresh-create and existing-note open?" — pointed directly at the race itself, which `1a52f38` eliminates by waiting for metadataCache before constructing the EditorView.

2. **Probe before fixing when CM6 internals are involved.** Five iterations of console probes (StateField update logs, decoration enumeration, widget DOM count, selection state, view mode) were essential to ruling out hypotheses. Each "obvious" fix (Prec.highest, block:true, requestMeasure) was disproven by the data before it could waste a deploy cycle.

3. **TDD test design must use insertion points that don't break the predicate cascade.** The flickerFix.test.ts cycle initially had test inserts at position 50 (inside the fence body), which disrupted the fence opener regex match in `findCodeFence`. Inserting at `state.doc.length - 1` keeps the fence structure intact across N test iterations.
