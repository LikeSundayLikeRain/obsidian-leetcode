<objective>
Eliminate the typing flicker on v1.3 LC widgets in Live Preview when `autoMigrateOnOpen=ON` (UAT R10).

Root cause (confirmed via static analysis of cycle-2 closures + diagnosis hints in 21-HUMAN-UAT.md R10):

The `autoMigrateOnOpen=ON` setting gates TWO distinct fire-and-forget side-effect paths that re-enter on EVERY parent CM6 docChange (not just on file-open):

1. **`src/widget/codeBlockProcessor.ts:166-247`** — the Reading-mode + Live-Preview rendered code-block post-processor. On every Live-Preview body flush, Obsidian re-fires this handler against the changed fence. The handler `await`s BOTH `migrateLegacyFenceIfNeeded(...)` AND `repairFrontmatterIfNeeded(...)` before constructing the widget. Each function performs `app.vault.read(file)` (~30-100ms on macOS APFS) plus predicate evaluation. For a fully-migrated note (`leetcode-solve` opener + `lc-language` present), both functions return false (idempotency / repair-predicate fail) — but the disk I/O still serializes the post-processor render. During those ~30-100ms the previous widget DOM is detached / replaced; the user sees a brief unmount/remount cycle synchronized with the DebouncedWriter ~500ms flush cadence.

2. **`src/widget/liveModeBannerStateField.ts:281-433`** — the LP StateFields rebuild on every `tr.docChanged` (every keystroke flush echoed back through the parent CM6 from disk). The `buildLegacyBannerDecorations` and `buildLeetCodeWidgetDecorations` builders fire `migrate` / `repair` as fire-and-forget side-effects gated only on `migrateInFlight` / `repairInFlight` Sets. Those Sets are cleared in the `.finally()` when the await resolves — so on EVERY docChange after the previous fire completes, the gate is OPEN AGAIN and a fresh `migrate` / `repair` Promise is dispatched even though it would no-op. For a fully-migrated v1.3 note this is wasted I/O; for a fully-repaired v1.3 note this is also wasted I/O. The async chain in `buildLeetCodeWidgetDecorations` post-repair `.then()` ALSO dispatches a `leetcodeRefreshAnnotation` transaction on success, which itself triggers another StateField rebuild on the same path — a second-order amplification.

Both paths are gated on `autoMigrateOnOpen=ON`, so disabling that setting eliminates the flicker (matches user observation). Neither path was added by Phase 21 cycle-2 plans 21-14..17 — the `migrateAttempted` / `repairAttempted` semantics are absent throughout. The Plan 21-14 `dispatchLeetCodeRefresh` introduced a polite-recompute mechanism but did not dedupe the underlying migrate/repair fire-and-forget.

Fix: Add per-path **attempt-once-this-session** Sets at every fire site so that:
  - The first parent docChange after file-open MAY fire migrate/repair (preserving R2 frontmatter-injection on a v1.3-shaped note with missing `lc-language`).
  - All subsequent docChanges on the same path SHORT-CIRCUIT before vault.read.
  - The Sets are cleared on `vault.on('rename')` (per-path) and on Plugin.onunload (all paths) — NOT on `workspace.on('file-open')` because that event fires on tab focus and clearing there would reintroduce flicker on tab-switch.
  - The "Migrate this note" command-palette entry clears all three sentinels for the active path BEFORE invoking the migrator, preserving manual retry after a failed auto-attempt.

Purpose: keeps the user's typing visually stable (no visible widget remount on flush cadence) while preserving every autoMigrate / repair pathway for the actual once-per-open trigger.
Output: 3 source files modified + 2 new vitest regression tests + human-verify checkpoint in dev vault.
</objective>
