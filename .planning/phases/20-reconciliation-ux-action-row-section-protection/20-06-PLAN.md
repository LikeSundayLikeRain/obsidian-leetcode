<objective>
Fix Cluster B (self-write remount cycle).

Symptom (`20-HUMAN-UAT.md` Test 6, blocker, carry-over from Phase 19 Test 1): after the 400ms
debouncedWriter flush, the widget loses focus + cursor + vim mode in BOTH vim and non-vim
configurations. Confirmed root cause (`.planning/debug/self-write-remount-cycle.md`):
`liveModeViewPlugin.ts:110-116` `update(update)` rebuilds the DecorationSet on EVERY
`update.docChanged` with no provenance check. After `debouncedWriter.flush()` →
`vault.process` writes the new fence body to disk → Obsidian dispatches a `docChange`
transaction onto the parent CM6 → ViewPlugin recomputes `sourceHash = djb2(NEW body)` → on-
screen widget's `sourceHash = djb2(OLD body)` → `LeetCodeFenceWidget.eq()` returns false →
CM6 destroys+remounts the widget DOM → focus/cursor/vim lost.

The vault-layer `selfWriteSuppression` (`src/widget/selfWriteSuppression.ts`) already arms a
per-path expected hash before `vault.process` and consumes it in `vault.on('modify')`. That
map is correct for the modify event but is invisible to the parent-CM6 transaction stream.

This plan plumbs the SAME suppression primitive into the CM6 ViewPlugin via a peek-only
accessor (`peekExpectedHash(path) → string | null`) so the ViewPlugin's `update()` can:
  - Compute the new sourceHash for the current build.
  - Peek the armed hash for this file (no consume — that stays the modify-handler's job).
  - If `peeked === newSourceHash`, REUSE the existing DecorationSet for this transaction
    (skip rebuild). The on-screen widget instance under `Decoration.replace` stays stable
    across the echo because `this.decorations` reference equality is preserved. **Plan 20-06
    revision Blocker #1**: do NOT mutate the on-screen widget's `sourceHash` in place — the
    `LeetCodeFenceWidget.sourceHash` field is `public readonly` (`LeetCodeFenceWidget.ts:40`)
    and its identity contract is content-hash-based per CONTEXT D-01 + RESEARCH Pitfall 19-F.
    The eq()-hardening in Task 3 makes the next-build mismatch a non-issue: when the next
    real keystroke triggers a rebuild, peek matches either widget's hash AND the on-screen
    widget instance is the one under the existing Decoration.replace (its hash is the
    pre-flush value), so eq() returns true via the suppression-map clause and CM6 reuses the
    DOM.

This is path (a) from the UAT.md `missing` field. Path (b) (userEvent annotation) was
considered and rejected for two reasons: (1) `vault.process` writes through Obsidian's vault
layer, not a direct CM6 dispatch on the parent — there's no place to attach an annotation
without also wrapping `vault.process` itself, which would pollute the public API; (2) a
peek-only suppression-map accessor is symmetrical with the existing arm/consume shape and
keeps a single source of truth for self-write provenance.

**Plan 20-06 revision Warning #9 — CONTEXT.md citation**: read of
`20-CONTEXT.md` (`.planning/phases/20-reconciliation-ux-action-row-section-protection/20-CONTEXT.md`)
confirms NO decision constrains the self-write provenance approach — Phase 20 CONTEXT covers
section protection (D-protect-01..04), action row (D-action-01..04), conflict modal
(D-conflict-01..04), and plan structure (D-plan-01..02), but does not surface the self-write
remount cycle as a discrete problem. The carry-over remount cycle from Phase 19 Test 1 was
recorded under Phase 19 SUMMARY's "Open Items Carrying Forward" without prescribing a
solution shape. This plan therefore selects path (a) on the engineering grounds above; the
selection is documented here so the executor (and any future revisor) can verify the absence
of a relevant D-XX rather than re-deriving it.

The CLAUDE.md `'leetcode.*'` userEvent convention is unaffected: this plan adds a NEW
provenance channel (the suppression-map peek), doesn't remove or modify the existing
convention. The canonical write-path pattern (Phase 17 D-05) is unaffected: writes still go
through `vault.process` (Plan 19-02 design).

Bonus hardening (per UAT.md missing item 3): `LeetCodeFenceWidget.eq()` is also extended to
peek the suppression map. When two widget instances differ only by sourceHash AND the new
sourceHash matches the armed expected hash for the file, return `true` (treat as same DOM —
let CM6 reuse). This decouples widget identity from any future echo source and is robust
against the multi-pane echo loop (Cluster A) re-emerging through a different path. **In the
revised plan (Blocker #1 fix), this clause IS the primary mechanism — not defense-in-depth
— because the in-place sourceHash mutation has been dropped.**

Purpose: Phase 20 Test 6 (blocker UAT) passes — focus + cursor + vim mode survive flush.
Output: ViewPlugin provenance check; widget eq() consults suppression map; new automated
tests pin both behaviors.
