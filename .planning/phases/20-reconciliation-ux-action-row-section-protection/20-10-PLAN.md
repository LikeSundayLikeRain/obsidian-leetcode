Close the five UAT-discovered gaps in Phase 20 (T3, T7, T8, T9, T10) with a single coordinated plan. The diagnosis is complete in three debug session files; this plan is implementation, not investigation.

The five gaps fall into three architectural clusters:

1. **Widget→plugin handoff cluster (T3, T7).** The `*FromWidget` surface from Plan 20-02 (D-action-04) feeds the widget's fence-only body (raw code, no ```fences) into orchestrator helpers (`extractFirstFencedBlock` at `src/solve/codeExtractor.ts:46`) that require a markdown-wrapped body. T7 is "Run/Submit can't find the code block." T3 is "language switch silent" — `switchLanguageFromWidget` (`src/main.ts:2910-2987`) dispatches new starter into the PARENT CM6 even though the v1.3 architecture (per 20-09 post-mortem key-decision) makes the registered child editor the canonical write target.

2. **Fence-kind-blind starterCodeInjector triad (T9 underlying layer + T10 — same root cause).** `stripFirstRecognizedCodeBlock` (`src/solve/starterCodeInjector.ts:217-220`) gates fence-opener detection on `LC_LANG_SLUGS` membership; `leetcode-solve` is not a langSlug, so a v1.3 fence is invisible to the strip helper. `codeBlockFor` (`src/notes/NoteTemplate.ts:107-110`) unconditionally emits `\`\`\`<lcSlugToFenceTag(langSlug)>` openers. Reset corrupts the v1.3 fence (T10). Retrieve grafts a sibling `\`\`\`python` fence (T9 underlying). 20-09 introduced the v1.3 fence kind without auditing this triad.

3. **Submission-history empty-cache (T9 surface layer).** `SubmissionHistoryStore` (`src/graph/SubmissionHistoryStore.ts:107`) caches an empty `rows[]` for 60s after the D-02 prefetch race at `main.ts:449-453`. The user sees "No past submissions found for this problem." even though LC has submissions.

4. **Action-row DOM hierarchy (T8 — cosmetic).** `mountLeetCodeWidget` (`src/widget/WidgetController.ts:929-1002`) appends both `.cm-editor` and `.leetcode-code-actions` to the same `.lc-nested-editor` container; the grey background on `.lc-nested-editor` (`styles.css:1934`) paints both. Target: introduce inner `.leetcode-widget-codeblock` wrapper that owns the grey paint; the action row sits as a sibling on the transparent outer shell.

The fixes do NOT compose into a single primitive — but the kind-blind starterCodeInjector cluster is shared by T9 underlying layer and T10, so we audit/fix it once. T3 / T7 are both in the `*FromWidget` surface from Plan 20-02; they share the architectural seam (D-action-04) but their failure modes are independent.

**SSoT reuse discipline (revision 2):** The kind-aware body-replace primitive already exists — `rewriteFenceBody` in `src/widget/fenceSerialization.ts:141`. This plan REUSES it; do NOT introduce `replaceLeetCodeSolveFenceBody` (would duplicate scan logic and risk regressing the Plan 19-04 nested-triple-backticks property tests). Likewise, `countLeetCodeSolveFenceOpeners` (`src/widget/fenceLocator.ts:118-138`) already gates v1.3-vs-legacy detection — REUSE it; do NOT inline a private detector in copyToCode.

**Active-view anti-pattern avoidance (revision 2):** The fence-kind resolver MUST NOT use `getActiveViewOfType(MarkdownView)` — that is the exact silent-bail pattern that root-caused T3. Use `vault.read(file)` + `countLeetCodeSolveFenceOpeners(text) > 0`, OR `childEditorRegistry.get(file.path)` first with text-scan fallback. Both shapes work in popout / non-active-pane / null-active-view scenarios.

The chosen Run/Submit fix shape (Task 5) MUST leave the post-Phase-22 path clean — refactoring `runWithCode`/`submitWithCode` to accept already-extracted code is preferred over synthetic-fence wrapping, because the synthetic wrap is dead code in Phase 22 once `*FromActive` is deleted.
