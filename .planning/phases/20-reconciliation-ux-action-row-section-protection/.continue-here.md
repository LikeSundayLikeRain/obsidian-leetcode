## What's Blocked

**Per-keystroke widget remount** (UAT, single-pane). Console trace from
last UAT iteration:

```
[lc-uat] child.update {docChanged: true, transactionCount: 1}
[lc-uat] child.dispatch->parent {changes: 1, bodyStart: 862, bodyEnd: 965}
[lc-uat] parent->child decide {isChildSync: true, ...}     ← gate works
[lc-uat] mountLeetCodeWidget.start {readOnly: false, hasParentView: true}
[lc-uat] sync.wiring.installed {parentDocLength: 981}
[lc-uat] child.dispatch.success {parentLength: 981}
```

The `mountLeetCodeWidget.start` line firing AFTER each child dispatch
means RenderChild is being unmounted+remounted by Obsidian per keystroke.

**Root cause hypothesis (unconfirmed):** `registerMarkdownCodeBlockProcessor`
re-runs whenever the codeblock content changes. Each child→parent dispatch
mutates the parent's fence body → Obsidian re-fires the post-processor →
new RenderChild instance → new `mountLeetCodeWidget` → user loses focus.

## Architecture Notes for Next Session

### Why we ended up rendering through RenderChild instead of ViewPlugin

- We tried `Decoration.replace({widget, block: true})` in
  `liveModeViewPlugin.ts` — `viewplugin.constructed` reported `rangeSize:1`
  but `toDOM` was NEVER called by CM6, leaving a visible empty gap.
- We tried bailing out of the RenderChild path in Live Preview — widget
  did not appear at all.
- Final compromise: RenderChild renders the visible widget in BOTH
  reading mode and Live Preview; ViewPlugin's `Decoration.replace` is
  retained ONLY for `EditorView.atomicRanges` contribution (not for
  visible rendering — `block: true` reverted to plain inline).

### Why this triggers the per-keystroke remount

Obsidian's `MarkdownPostProcessor` lifecycle re-fires on every
content change in the codeblock — there's no stable identity. When
the child writes through to the parent CM6, the codeblock content
changed, so Obsidian re-renders → fresh `LeetCodeWidgetRenderChild`
instance → `mountLeetCodeWidget` runs from scratch → focus lost.

### Possible next-session paths

1. **Make ViewPlugin's `Decoration.replace` actually render**
   (proper architecture). Investigate why `block: true` produced empty
   DOM. Possible cause: a competing extension or Obsidian's built-in
   codeblock fold interfering.
2. **Hoist the child editor out of `RenderChild.containerEl`** so it
   survives re-renders. RenderChild becomes a thin shim that just
   ensures the floating editor stays anchored to the right fence range
   in the DOM.
3. **Detect that the codeblock processor re-fire was caused by our own
   child→parent dispatch** and skip the re-mount. The newly-created
   RenderChild could check the `widgetRegistry` for an existing
   controller with matching `(file.path, fenceIndex, leafId)` and reuse
   it instead of constructing a new one.

Option 3 is probably the smallest change. The existing per-pane registry
key (Plan 20-05) already supports this — `widgetRegistry.get(registryKey)`
should return the prior controller; if present, the new mount could
re-use its DOM by NOT reconstructing the embedded EditorView.

## Critical Anti-Patterns Discovered (don't repeat)

| Anti-Pattern | Severity | Why it bit us |
|---|---|---|
| `Transaction.addToHistory.of(false)` on programmatic CM6 dispatches | blocking | Obsidian's auto-save heuristic is keyed off CM6's history integration. Setting addToHistory.of(false) bypasses history → MarkdownView never sees the change as a user edit → save scheduler never fires. The Tasks plugin canonical pattern dispatches with NO annotations and saves correctly ~250ms later. |
| Trusting `useNestedEditor`-only gates for v1.3 widget paths | blocking | The pre-v1.3 `retrofitStarterCode` on file-open is gated by neither `useNestedEditor` nor `useInlineWidget` — it always fires for any lc-slug note. Under useInlineWidget=ON, it injects a duplicate ```javascript fence on every file open because injectCodeSection doesn't recognize `leetcode-solve` as a code fence. |
| `Decoration.replace` competing with `registerMarkdownCodeBlockProcessor` | blocking | Both try to render the same DOM range in Live Preview. With the post-processor active, CM6's Decoration.replace's `toDOM` is never called even when the RangeSet contains the decoration (`rangeSize: 1`) — visible state is empty. Either disable the post-processor or accept that RenderChild owns the visible DOM. |
