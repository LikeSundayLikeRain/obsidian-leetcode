<objective>
Plan 21-05 closes the Phase 21 Reading-mode trigger gap (CR-01 BLOCKER) and the Live Preview dedupe-set leak (WR-01 WARNING) identified in 21-VERIFICATION.md.

Purpose: The verifier observed that `Plugin.registerMarkdownCodeBlockProcessor('leetcode-solve', ...)` at `src/main.ts:1041` binds the Reading-mode post-processor to the tag `'leetcode-solve'` only — legacy v1.2 notes carry ` ```python `, ` ```java `, etc., so the migration gate inside `leetCodeBlockProcessor` is never invoked on a legacy v1.2 note in Reading mode. Live Preview is fine (`liveModeViewPlugin.ts:158` correctly branches on `fence.kind === 'legacy'`). Reading mode is the user-visible default for many setups; the phase goal "every v1.2 note migrates lazily on first open" is FALSE in Reading mode without this fix. Per VERIFICATION.md and REVIEW.md CR-01 the recommended fix is L5-compliant: a `workspace.on('file-open', ...)` hook in `Plugin.onload()` that calls `migrateLegacyFenceIfNeeded` for any TFile whose frontmatter has `lc-slug` AND `useInlineWidget=ON` AND `autoMigrateOnOpen=ON`.

WR-01 is the natural sibling: the `migrateInFlight: Set<string>` declared at module scope in `liveModeViewPlugin.ts:67` is never cleared on plugin unload. With a new Reading-mode trigger added, the Set must be SHARED across both triggers (per-file lock, not per-mode lock) AND must reset on plugin unload. Hoisting it to a Plugin-instance field naturally resets on reload and lets both triggers consume the same dedupe surface.

Output: A new `workspace.on('file-open')` hook in `Plugin.onload()`; a Plugin-instance `migrateInFlight: Set<string>` field replacing the module-level constant; an updated banner-mount path that fires from the file-open hook when `autoMigrateOnOpen=OFF` AND the strict-match predicate accepts; a Reading-mode integration test that opens a v1.2 fixture in Reading mode and verifies migration ran exactly once; updated tests covering the master gate, the per-note gate, and the cross-mode dedupe.
</objective>
