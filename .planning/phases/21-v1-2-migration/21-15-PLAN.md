Per the diagnosis recorded in 21-HUMAN-UAT.md:

> AutoMigratingBannerWidget host element, once wrapped by CM6's block-decoration container, inherits surrounding code-block visual styling. Banner UI doesn't have a top-level CSS scope that isolates it from CM6's fence-block style.

The Plan 21-11 fix migrated the line-break-spanning Decoration.replace from the ViewPlugin's `decorations` field into a StateField — that closed the CM6 RangeError (the actual blocker). What remained: when CM6 wraps the StateField's `Decoration.replace({widget, block: true})` over a multi-line range, it places the widget DOM inside an internal block-decoration wrapper that inherits styles from the surrounding `.cm-content` / `.cm-line` / fence-block CSS. The `mountLegacyFenceBanner` helper (in `src/widget/legacyFenceBanner.ts`) was originally written for the Reading-mode `registerMarkdownCodeBlockProcessor` pathway, where the host element is a fresh DOM node not wrapped by CM6 — so the styling fits there. In LP, the same helper produces a host that inherits CM6 fence styling, blurring the visual boundary between banner UI and the read-only fence body preview.

**Investigation grep gate (per planning audit):** `grep -ic 'leetcode-migration-banner\|lc-legacy' /Users/moxu/projects/obsidian-leetcode/styles.css` returned **0**. There are NO existing CSS rules for the banner — currently the banner DOM relies entirely on inherited styling from the surrounding container. In Reading mode the inherited container is the markdown post-processor's `.markdown-source-view` / `.markdown-rendered` wrapper, which has neutral styling. In Live Preview the inherited container is CM6's block-decoration wrapper inside the `.cm-content` element, which carries the fence-block visual treatment. This is the structural reason Reading mode looks correct and Live Preview looks broken: same DOM, different ambient CSS.

**Fix shape:**

1. **Top-level CSS scope class on the LP banner host.** In `src/widget/liveModeBannerStateField.ts`, both `AutoMigratingBannerWidget.toDOM` and `ManualPromptBannerWidget.toDOM` construct the host with:

   ```ts
   const host = document.createElement('div');
   host.classList.add('leetcode-migration-banner-host');
   mountLegacyFenceBanner(host, this.source, this.file, this.plugin, 'auto-migrating' /* or 'manual-prompt' */);
   return host;
   ```

   ADD a second class — `lc-legacy-banner--livepreview` — so styles can target the LP shape distinctly:

   ```ts
   host.classList.add('leetcode-migration-banner-host', 'lc-legacy-banner--livepreview');
   ```

   The Reading-mode caller of `mountLegacyFenceBanner` (in `src/widget/codeBlockProcessor.ts:223-230`) passes the post-processor's `el` directly — this Plan does NOT add the LP class to that path. Reading mode keeps its existing styling (which the user reports is correct).

2. **CSS rules in styles.css.** The current styles.css has ZERO banner rules (verified). Add a new section with three concerns:

   a. **Reset inherited code-block visual treatment on the LP banner host.** CM6's block-decoration wrapper applies background tint + rounded corners + monospace font. The reset:

   ```css
   .lc-legacy-banner--livepreview {
     /* CM6's block-decoration wrapper inherits fence-block styling. Reset
        on the LP banner host so the banner reads as UI chrome, not code. */
     background: var(--background-primary);
     border: 1px solid var(--background-modifier-border);
     border-radius: var(--radius-s);
     padding: var(--size-4-3);
     margin: var(--size-4-2) 0;
     font-family: var(--font-text);
   }
   ```

   b. **Header (copy + CTA) uses UI font; visually separated from the body.** The `mountLegacyFenceBanner` helper at `src/widget/legacyFenceBanner.ts` builds:
      - An outer `.leetcode-migration-banner.leetcode-migration-banner--{mode}` div
      - A `.leetcode-migration-banner__copy` paragraph (the explanatory text)
      - For 'manual-prompt' mode: a `.leetcode-migration-banner__cta` button (the [Migrate now] CTA)
      - For 'manual-prompt' AND for 'auto-migrating' followed by `renderReadOnly`: a `<pre><code>` of the source body

   Style the header explicitly:

   ```css
   .lc-legacy-banner--livepreview .leetcode-migration-banner__copy {
     font-family: var(--font-text);
     font-size: var(--font-ui-small);
     margin: 0 0 var(--size-4-2) 0;
     color: var(--text-normal);
   }

   .lc-legacy-banner--livepreview .leetcode-migration-banner__cta {
     font-family: var(--font-text);
     font-size: var(--font-ui-small);
     padding: var(--size-4-1) var(--size-4-3);
     border-radius: var(--radius-s);
     background: var(--interactive-accent);
     color: var(--text-on-accent);
     border: none;
     cursor: pointer;
     margin-bottom: var(--size-4-3);  /* visually separates header from body */
   }

   .lc-legacy-banner--livepreview .leetcode-migration-banner__cta:hover {
     background: var(--interactive-accent-hover);
   }
   ```

   c. **Body (`<pre><code>`) keeps code-block styling but is visually separated from the header.** A border-top on the `<pre>` element provides the explicit separator:

   ```css
   .lc-legacy-banner--livepreview pre {
     margin: 0;
     padding: var(--size-4-2);
     background: var(--code-background);
     border: 1px solid var(--background-modifier-border);
     border-radius: var(--radius-s);
     font-family: var(--font-monospace);
     font-size: var(--font-ui-smaller);
     overflow-x: auto;
   }

   .lc-legacy-banner--livepreview pre code {
     background: transparent;
     padding: 0;
   }
   ```

   d. **Auto-migrating mode is single-line + no body.** The 'auto-migrating' branch in `mountLegacyFenceBanner` only renders the copy paragraph (no CTA, no `<pre>`). Style it more compactly so the migration window doesn't leave a large empty card — the 'auto-migrating' mode lasts ~50-200ms before the migrator + Plan 21-08's rerender remount the v1.3 widget in its place.

   ```css
   .lc-legacy-banner--livepreview .leetcode-migration-banner--auto-migrating {
     padding: var(--size-4-1) 0;
   }
   ```

   The 'auto-migrating' mode does NOT have a CTA or pre body, so the existing border + padding from the host class is sufficient — no need to suppress the body styles for this branch.

3. **NO change to `src/widget/legacyFenceBanner.ts`'s `mountLegacyFenceBanner` API or rendering logic.** The helper continues to attach to whatever host the caller passes. Reading-mode caller passes the post-processor's `el` directly (no LP class) — Reading-mode banner styling unchanged. Live-Preview callers (the two `*BannerWidget.toDOM` methods) pass a fresh `host` div with both classes — LP-scoped CSS applies only here.

   **WHY the helper itself does NOT need to add the LP class:** the LP class encodes the FACT that the host is going to be wrapped by CM6's block decoration container. The helper is mode-agnostic; the caller knows the mode. Pushing the LP class into the helper would either (a) add a third "mode" parameter (overloading the existing 'auto-migrating' / 'manual-prompt' / 'read-only-legacy' enum that means something else), or (b) hard-code the LP class in the helper which would also apply it to Reading-mode callers, defeating the isolation.
