<objective>
Close UAT Gap 3 (Test 4a, severity=major): "Reading-mode banner UX is silently broken on v1.2-shaped notes when `autoMigrateOnOpen=OFF`." Per 21-HUMAN-UAT.md lines 140-174 (root cause A), the existing `registerMarkdownCodeBlockProcessor('leetcode-solve', leetCodeBlockProcessor(this))` at `src/main.ts:1060-1063` only fires for fences whose tag IS LITERALLY ` ```leetcode-solve `. v1.2 notes carrying ` ```java `, ` ```python `, etc. are rendered by Obsidian's default markdown processor and never reach `leetCodeBlockProcessor`, so `mountLegacyFenceBanner('manual-prompt')` at `codeBlockProcessor.ts:198` is structurally unreachable in Reading mode. The user has no in-note way to discover or trigger migration when `autoMigrateOnOpen=OFF` — they must use the command palette.

Purpose: Add a `registerMarkdownPostProcessor` (NOT a code-block processor — those are tag-bound) that walks the rendered DOM, gates on the parent note's frontmatter + plugin settings + `isMigrationCandidate`, locates the rendered code-block element under `## Code` whose language class corresponds to a recognized LC `langSlug`, and replaces it with `mountLegacyFenceBanner(host, source, file, plugin, 'manual-prompt')`.

Output: New module `src/main/readingModeLegacyBannerPostProcessor.ts` exporting a `registerLegacyBannerPostProcessor(plugin)` factory. Wired in `src/main.ts` next to the existing `registerMarkdownCodeBlockProcessor('leetcode-solve', ...)` call.

PHASE 22 RETIREMENT MARKER: This module is v1.2-only scaffolding — it discovers v1.2-shaped fences (`java`/`python`/etc.) for the legacy banner. Phase 22 retires the entire v1.2 read path; this module must be deleted along with it. The module file MUST start with a header block comment containing the literal token `PHASE_22_DELETE_WITH_V1_2_PATH` (see Task 1 action) so the Phase 22 cleanup can grep for the marker and mechanically remove all such files.

    DOM walker logic (executed for every Reading-mode rendered markdown block):
    - Step 1: Resolve TFile via `plugin.app.vault.getAbstractFileByPath(ctx.sourcePath)`. Bail (no-op) if not a TFile.
    - Step 2: Read frontmatter via `plugin.app.metadataCache.getFileCache(file)?.frontmatter`. Bail if `lc-slug` missing/empty.
    - Step 3: Master gate — bail if `settings.getUseInlineWidget() === false`.
    - Step 4: Mode gate — bail if `settings.getAutoMigrateOnOpen() === true` (auto-path takes over).
    - Step 5: Read full note text via `await plugin.app.vault.cachedRead(file)` — `cachedRead` is the safe Reading-mode-context read (does NOT trigger metadata refresh; mirrors codeBlockProcessor.ts pattern).
    - Step 6: Predicate gate — bail if `isMigrationCandidate(text, fm) === false`.
    - Step 7: Locate target `<pre><code>` under the rendered DOM. Two-stage: (a) for each `pre > code` child of `element`, read `code.className` (Obsidian writes `language-{slug}` class); (b) extract slug; (c) if `resolveLangSlug(slug, '__sentinel__') !== '__sentinel__'` AND the resolved slug is in `LC_LANG_SLUGS`, this is a candidate code block. Take the FIRST match only (matches `isMigrationCandidate`'s "first fence inside ## Code" semantic; if rendering yielded multiple langSlug code blocks, only the first inside ## Code is the migration target).
    - Step 8: Verify the matched code block is logically inside ## Code. Strategy: iterate `element.previousElementSibling`-walk (or alternatively `ctx.getSectionInfo(element)?.text` — preferred when available since it gives the exact source span) to confirm the nearest preceding `<h2>` text is "Code". When `getSectionInfo` is null, fall back to ALWAYS RENDER the banner only when there's exactly one matched langSlug code block in `element` (single-fence common case). Document this fallback in the module header.
    - Step 9: Extract the source via `code.textContent ?? ''`. Build a host wrapper `<div class="leetcode-migration-banner-host">`. Replace the matched `<pre>` with the host using `pre.replaceWith(host)`. Then call `mountLegacyFenceBanner(host, source, file, plugin, 'manual-prompt')`.
    - Step 10: Wrap entire body in try/catch — on any throw, logger.debug and leave the rendered DOM untouched. (The user sees plain Obsidian rendering; no crash.)
