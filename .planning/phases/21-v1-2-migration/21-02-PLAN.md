<objective>
Plan 21-02 wires the Plan 21-01 migration foundation into the user-facing mount lifecycle. Both widget mount paths (Reading mode `codeBlockProcessor.ts` and Live Preview `liveModeViewPlugin.ts`) consult `migrateLegacyFenceIfNeeded` BEFORE constructing the widget, atomically rewriting any legacy fence in the same render cycle (D-trigger-01: migration IS the first edit). The new `autoMigrateOnOpen` setting (default ON) gates the silent-rewrite path; OFF surfaces a `legacyFenceBanner` with a [Migrate now] button. A command palette entry `LeetCode: Migrate current note` ships as a keyboard escape hatch, registered unconditionally per D-auto-03.

Purpose: Without mount integration, Plan 21-01's migrator is dead code. This plan makes migration user-visible — the only plan in Phase 21 that flips actual user behavior. The banner UX (D-auto-02) is the user-facing surface; the silent path (D-auto-01) is the majority-case happy path. The dev-vault smoke during this plan empirically answers RESEARCH Open Question §1 (`vault.process`+`processFrontMatter` single-frame vs. two-frame ordering) and lands the Phase 19 C-04 hash-arm fallback if needed (Pitfall 1). Implements MIGRATE-06.

Output: `autoMigrateOnOpen` settings field + UI; `src/widget/legacyFenceBanner.ts` (~80 LOC) banner with three render modes (`auto-migrating`, `manual-prompt`, `read-only-legacy`) + read-only display; modifications to `codeBlockProcessor.ts` (Reading mount gate), `liveModeViewPlugin.ts` (Live Preview synchronous banner mount during migration window), and `src/main.ts` (command palette entry); unit tests covering settings persistence, banner DOM, and mount-path integration.

**D-trigger-01 constraint (locked):** Per CONTEXT.md D-trigger-01, the user must NEVER see a 'no widget mounted on a legacy fence' transitional state during the migration window. Both Reading mode (sync await) and Live Preview (synchronous banner mount + async migration) honor this contract. The Live Preview path mounts the `auto-migrating` mode of `legacyFenceBanner` synchronously on the legacy fence range while the migration runs asynchronously; the banner unmounts when the post-migration `update()` cycle mounts the v1.3 widget on the rewritten fence. There is no render frame in which the user sees raw markdown or 'no widget' on a legacy fence.

**D-trigger-02 cost framing (locked):** Per CONTEXT.md D-trigger-02, the synchronous wait (~10–50ms) is acceptable because it is part of the same render frame the user is already waiting on; async background migration was rejected because it creates a transitional widget-mount-on-legacy-then-remount-on-v1.3 state with state-loss and decoration thrash.
</objective>
