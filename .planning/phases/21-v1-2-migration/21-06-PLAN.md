Plan 21-06 closes three Phase 21 backup-invariant gaps from 21-VERIFICATION.md / 21-REVIEW.md: CR-02 (double-backup on retry — BLOCKER), CR-03 (greedy slug regex permits TTL deletion of foreign folders — BLOCKER), and WR-05 (no concurrency guard on the GC sweep — WARNING). Purpose: Phase 22 cleanup logic may rely on D-backup-02's "one backup per note" invariant for safe deletion; CR-02's retry-path violation must be closed before Phase 22 can ship. CR-03's greedy regex weakens T-21-gc — the verifier accepts the realistic risk as low (no other plugin code creates folders with that shape today) but the contract guarantees in D-backup-03 must be tightened to match the implementation strength claimed. WR-05 is non-data-corrupting (rmdir is idempotent on missing folders) but adds noise during plugin reload races; closing it cleans up the log surface and makes the GC safe to re-invoke.

Output: A tightened `BACKUP_FOLDER_RE` exported from `migrationBackupGc.ts` (LC-shape slug character class); a pre-existence backup check in `fenceMigrator.ts` reusing the exported regex; a `gcRunning` module-level lock in `migrationBackupGc.ts`; new tests covering all three closures; no regression to the existing 8-test GC suite or 58-test fenceMigrator suite.
