<objective>
Plan 21-03 closes the language-derivation invariant for v1.3: codeExtractor reads `lc-language` frontmatter as the source of truth when the located fence is `leetcode-solve` (D-extract-01); new notes emit ` ```leetcode-solve ` directly via `codeBlockForV13` and the `injectCodeSection` `fenceKind` short-circuit (D-emit-01..02). After this plan, a freshly-migrated note's Run/Submit dispatch uses frontmatter (not the deleted fence tag), and v1.3-created notes never roundtrip through the legacy emitter.

Purpose: Without this refactor, MIGRATE-09 is broken — Run/Submit on a migrated note would call `resolveLangSlug('leetcode-solve', defaultLang)` and fall back to defaultLang, which is wrong for any user whose default doesn't match their actual language. And without MIGRATE-08, brand-new v1.3 notes ship with a `langSlug`-tagged fence; opening them later would re-trigger migration (creating spurious backups for notes that were never legacy). The plan threads `frontmatter` through 5 production consumers + 2 test files using TypeScript strict-mode as a forcing function (Pitfall 8) and gates new-note emission on `useInlineWidget=ON` so Phase 22's cleanup is purely mechanical (delete the gates, rename codeBlockForV13 → codeBlockFor, delete the legacy fence-tag branch in extractFirstFencedBlock).

Output: Refactored `src/solve/codeExtractor.ts` (signature widening + dual-path dispatch); 5 production consumer call-site updates (`src/main.ts`, `src/solve/submissionOrchestrator.ts`, `src/graph/KnowledgeGraphWriter.ts`, `src/ai/buildDebugPrompt.ts`, plus the 5th identified by grep); `codeBlockForV13` emitter in `src/notes/NoteTemplate.ts`; `fenceKind` arg added to `injectCodeSection` in `src/solve/starterCodeInjector.ts`; transition gates at fence-emit call sites; updated tests.
</objective>
