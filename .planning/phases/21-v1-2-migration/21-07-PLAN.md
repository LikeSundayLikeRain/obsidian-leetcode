<objective>
Plan 21-07 closes the Phase 21 banner robustness BLOCKER (CR-04) and three correctness WARNINGs (WR-02, WR-03, WR-07) identified in 21-VERIFICATION.md / 21-REVIEW.md. All four sit in user-facing UX / correctness paths; none are architectural — they're defensive hardening + helper hygiene.

Purpose: CR-04 is the most user-visible: a TypeError in the banner DOM construction can break the editor render cycle on a v1.2 note when running in any environment where Obsidian's HTMLElement.prototype patches haven't fired (test envs, popup windows, iframes). The fix is layered: defensive check on the chained `createEl` with a textContent fallback, plus a top-level try/catch around `mountLegacyFenceBanner` so the editor never crashes on a banner failure. WR-02 cleans up the misleading outer `needsLang` guard in `migrateLegacyFenceIfNeeded` — the inner re-check inside `processFrontMatter` is authoritative and any future code that relies on the outer check is at risk; the simplest closure is to remove the outer guard and let the inner check be the single source of truth. WR-03 is the multi-section scoping fix: `countLeetCodeSolveFenceOpeners` over-scopes to the whole note, so a user-authored ` ```leetcode-solve ` reference in `## Notes` (e.g., documenting the migration) aborts migration of the actual `## Code` fence. WR-07 is the same shape on the inject side: `injectCodeSection`'s v1.3 short-circuit replaces the FIRST `leetcode-solve` fence in the file regardless of section; the fix mirrors `forceInjectCodeSection`'s ## Code-scoped discipline.

Output: Defensive DOM in `legacyFenceBanner.ts` + top-level try/catch; outer `needsLang` removed in `fenceMigrator.ts`; new `countLeetCodeSolveFenceOpenersInCodeSection` helper consumed by both `isMigrationCandidate` and `injectCodeSection`; updated tests covering all four closures.
</objective>
