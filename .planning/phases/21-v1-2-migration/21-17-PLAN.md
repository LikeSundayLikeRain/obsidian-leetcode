<objective>
Close UAT cycle-2 post-finding R9 (`severity: minor`): "When the same LC note is open in two split panes, editing in one pane causes the cursor in the OTHER pane's widget to jump to position 0 (beginning of the widget). Cursor preservation is broken across split-pane sync."

Per `21-HUMAN-UAT.md` Gaps section (R9 entry, lines 473-490), the root cause is in the `vault.on('modify')` fan-out at `src/main.ts:1226-1341`: when pane A's `DebouncedWriter` flushes via `vault.process`, the resulting `modify` event reaches BOTH pane A's controller AND pane B's controller. The current handler treats them identically: it picks the FIRST matching widget at `src/main.ts:1264-1273` (which may be A or B — the registry walk is non-deterministic in iteration order), calls `reloadFromDisk('silent')` which dispatches a FULL-DOC REPLACEMENT transaction at `src/widget/WidgetController.ts:511-515`. That dispatch's selection is `EditorSelection.cursor(restoredHead)` computed from line/col clamp — fine for an EXTERNAL edit (the user expects their own caret on the line they were on), but WRONG for a sibling-pane edit in pane B's case: pane B's caret was at, say, line 8 col 12, and pane A's edit happened at line 3 — pane B's caret should map to line 8 col 12 in the new doc (offset +Δ from the prepended chars), NOT relocate to wherever the line/col clamp lands. The line/col clamp only happens to "preserve" the caret when pane A's edit happened to be downstream of pane B's caret; any upstream edit shifts pane B's logical cursor and the line/col clamp resets it to col 0 of the same line number — which the user perceives as "cursor jumped to the beginning of the widget".

This plan introduces a per-pane peer-sync path that is structurally separate from `reloadFromDisk('silent')`:

1. **Track the originating registryKey on every self-write.** `DebouncedWriter.arm()` (the only call site that arms `selfWriteSuppression`) is extended to pass the writing controller's `registryKey` (already constructed at `WidgetController.ts:1032`). `selfWriteSuppression.arm()` stores it alongside the expected hash. A new `peekOriginator(path)` accessor returns the `registryKey` of the most-recent self-write entry for `path`, OR `null` if no entry is armed.

2. **Add `WidgetController.applyPeerSync(newBody)` — incremental dispatch with mapped selection.** Computes a `ChangeSpec` describing the minimal edit from `this.view.state.doc.toString()` to `newBody` (longest common prefix + suffix; the middle becomes `{from, to, insert}`). Constructs a `ChangeSet.of(spec, oldDocLength)`; uses it to map the current selection's `head` and `anchor` through the change set (`changes.mapPos(pos, 1)` for forward bias). Dispatches a single transaction with: `changes` = the spec, `selection` = mapped selection, `annotations` = `[Transaction.userEvent.of('leetcode.peer-sync'), Transaction.addToHistory.of(false)]`. The userEvent annotation is per CLAUDE.md Phase 17 D-03 / D-05 convention (any plugin dispatch into a section-locked range MUST carry a `'leetcode.*'` userEvent so the section-lock filter passes it through). The `addToHistory.of(false)` ensures pane B's Cmd-Z still undoes only pane B's own typing.

3. **Modify-handler fan-out branch.** In `src/main.ts:1226-1341`, after the existing tryConsume + childDoc backup checks complete (i.e., the event is a confirmed self-write echo from SOME pane on this file), capture the `originatingRegistryKey = selfWriteSuppression.peekOriginator(file.path)` BEFORE the consume drops the entry. Then iterate `widgetRegistry.values()` filtered by `ctl.file.path === file.path AND ctl.registryKey !== originatingRegistryKey AND !ctl.isEmbed AND !ctl.readOnly`; for each peer, call `ctl.applyPeerSync(observedBody)`. The originating pane is skipped (its caret is already correct — its own typing produced the new state). External edits (no armed suppression entry — `peekOriginator` returns `null`) fall through to the existing `reloadFromDisk('silent')` path UNCHANGED for the single matching widget — preserving R8 (external-edit reload) and R7 (two-pane peer overlay) byte-identically.

Purpose: Without per-pane peer-sync the user perceives split-pane editing as broken — typing in one pane appears to "reset" the sibling pane's cursor. With this fix, split-pane editing matches what the user expects from any other CM6-backed editor (Obsidian's own native split panes, VS Code, etc.): the sibling caret stays put or maps cleanly through the change.

Output: `src/widget/selfWriteSuppression.ts` records originator on `arm`. `src/widget/DebouncedWriter.ts` threads `registryKey` to `arm`. `src/widget/WidgetController.ts` exposes `applyPeerSync(newBody)`. `src/main.ts` adds the peer fan-out branch. `tests/widget/splitPaneCursorPreservation.test.ts` covers (a) two-pane same-file: edit in A maps B's selection forward through the change, (b) originator skip: A's own caret is unchanged, (c) external-edit single-pane fallback: still uses `reloadFromDisk('silent')`, (d) `'leetcode.peer-sync'` userEvent annotation present on the dispatch, (e) `addToHistory.of(false)` annotation present, (f) StateField recompute path on B is not suppressed (a synthetic StateField reads the change and updates).

This is permanent v1.3 code (NOT v1.2 scaffolding) — `applyPeerSync` is a multi-pane primitive that persists past Phase 22 and seeds MULTI-01/MULTI-02 (deferred to v1.4+).
