<objective>
Close UAT post-finding Gap A (`severity: major`, deterministic across three triggers): "On every remount of an LC note (close tab → reopen, switch notes → switch back, close all → reopen), the widget mounts in a 'pre-takeover-handover' state — the 'Click to take over' overlay is rendered AND its click handler is dead. The user has to navigate via the file-nav pane to recover."

Per `21-HUMAN-UAT.md` Post-UAT Findings (lines 88-122) the debug agent diagnosed TWO compounding bugs:

1. **PRIMARY — `src/widget/multiPaneCoordinator.ts:140-145`**: the `reconcileFocus` else branch flips a controller to `'peer'` whenever `findLeafEl(ctl.container)` returns `null`. That null return does NOT mean "in a different leaf" — it means "no leaf ancestor at all" (parked, pre-attach, mid-teardown). The mid-mount-attach window (Open #2) is the trigger: `widgetRegistry.set` fires synchronously inside `mountLeetCodeWidget` at `WidgetController.ts:1176-1178` BEFORE the host element has been attached to the leaf; `active-leaf-change` fires DURING this attach window; `reconcileFocus` sees `registered controller + detached container` and flips it to `'peer'`. That mounts the overlay. The overlay's click handler calls `promoteThisPane`, which `getLeavesOfType('markdown')` walks — but at the moment of click, the leaf IS the active leaf, so `setActiveLeaf` is a no-op AND Obsidian dedupes the focus event AND `reconcileFocus` is never re-run AND the overlay never disappears. Recovery via file-nav click works because clicking the navigation pane fires a different focus event that DOES survive dedup.

2. **SECONDARY — `src/widget/WidgetController.ts:720-742` (`promoteThisPane`)**: even if a future bug recreates the same divergent state (visual `'peer'`, FSM thinks active), `promoteThisPane` cannot self-recover when the click target leaf is already the active leaf — `setActiveLeaf` short-circuits and `active-leaf-change` does not refire. The defense-in-depth fix: have `promoteThisPane` ALSO call `this.setPaneState('active')` directly so the overlay clears regardless of whether Obsidian dispatches the focus event.

Purpose: Without BOTH fixes, the bug is latent — fixing only #1 leaves #2 as a future-bug landmine; fixing only #2 plasters over #1 and leaves the spurious overlay flicker on every Open #2. The plan addresses both in one task because they share the same code surface (`reconcileFocus` ↔ `setPaneState`) and the regression test corpus needs to assert BOTH paths simultaneously to be meaningful.

Output: `src/widget/multiPaneCoordinator.ts:reconcileFocus` is hardened so a null `ctlLeafEl` defaults to `'active'`. `src/widget/WidgetController.ts:promoteThisPane` explicitly calls `this.setPaneState('active')` AFTER the `setActiveLeaf` attempt so the overlay clears even when Obsidian dedupes the focus event. Regression tests in `tests/widget/multiPaneCoordinator.test.ts` cover both the unit-level reconcileFocus null-leaf path AND the integration-level Open #1 / Open #2 / Open #3 sequence the UAT reproduction step describes.

This is permanent v1.3 code (NOT v1.2 scaffolding) — no `PHASE_22_DELETE_WITH_V1_2_PATH` marker is required.
