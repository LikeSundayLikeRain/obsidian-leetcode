**Three entry points, three fixes:**

A. **`src/main/readingModeMigrationHook.ts`** — VERIFY-ONLY. The existing implementation at lines 151-200 already captures `repairedFlag` and fires `rerenderPreviewLeaves(file.path)` when `migratedFlag || repairedFlag` is true (per the Plan 21-09 GREEN commit `da8513f`). B. **`src/widget/codeBlockProcessor.ts:188-201`** (Plan 21-09 GREEN repair gate). Currently:

  ```ts
  const repaired = await repairFrontmatterIfNeeded(plugin.app, file, { autoMigrateOnOpen: true, defaultLanguage });
  if (repaired) {
    // metadataCache.changed will refire the post-processor with the
    // repaired frontmatter — the widget mounts on the next cycle.
    renderStaticFallback(el, source);
    return;
  }
  ```

  The "metadataCache.changed will refire" assumption is the bug source. In live Obsidian, `metadataCache.changed` fires on the next macrotask AFTER `processFrontMatter` resolves, but Reading-mode post-processor invocations are not auto-rebound to metadataCache events for non-`leetcode-solve` fences (the registration tag is `'leetcode-solve'`-only) and the lc-language read in `resolveLanguageSlug` happens at MOUNT time on the FIRST post-processor call — by the time `metadataCache.changed` fires, the widget has already mounted in a stale state. **Fix**: after `repaired === true`, explicitly call `rerenderReadingModePanes(plugin.app, file.path)` — the same helper Plan 21-08 added for the migrate path. This forces preview-mode panes to re-run post-processors so the MOUNT call is repeated against the post-repair frontmatter.

C. **`src/widget/liveModeBannerStateField.ts:buildLeetCodeWidgetDecorations:289-316`** (Plan 21-09 GREEN repair fire-and-forget). The current shape:

  ```ts
  if (repairInFlight && !repairInFlight.has(file.path)) {
    repairInFlight.add(file.path);
    void repairFrontmatterIfNeeded(plugin.app, file, opts)
      .catch(() => { /* S-05 silent */ })
      .finally(() => { repairInFlight.delete(file.path); });
  }
  ```

  Side effect inside StateField builder; pure return value is independent. After the repair completes, NO mechanism asks the parent EditorView to re-evaluate the leetCodeWidgetStateField — and since the StateField only rebuilds on `tr.docChanged`, a frontmatter-only write does NOT trigger rebuild. The `LeetCodeFenceWidget` widget passed to `Decoration.replace` was constructed with the STALE language; even when `metadataCache.changed` fires, the widget-side language reactivity (Plan 20-02 metadataCache.on('changed') hook inside WidgetController) only swaps the language Compartment — it does NOT rerun `resolveLanguageSlug`'s Notice gate that fires at mount time. **Fix**: chain a post-repair scheduled refresh onto the .then BEFORE the .finally clears repairInFlight — schedule a microtask that walks the host plugin's known EditorView instances for this file and dispatches an empty/no-op transaction with a sentinel `userEvent: 'leetcode.repair-refresh'` annotation so the StateField's update() gets a chance to recompute against the new frontmatter (StateField update only fires on docChanged, so the transaction must include either a doc-touching no-op OR we must call `view.requestMeasure()` which triggers a fresh `state.field()` read on the next layout).

  Alternative for (C): instead of dispatching a no-op transaction, directly walk the EditorView leaves for this file (via `plugin.app.workspace.getLeavesOfType('markdown')`) and call `view.dispatch({ effects: <NEW_EFFECT> })` where `NEW_EFFECT` is a `StateEffect.define<void>()` we add to liveModeBannerStateField — but a sentinel effect adds API surface for a one-shot use. Simplest correct path: **dispatch a transaction with `changes: []` and `annotations: Transaction.userEvent.of('leetcode.repair-refresh')`** — CM6 considers any dispatch with empty changes as `tr.docChanged === false`, which is correct (frontmatter is not in the doc). To force the StateField to recompute, we widen its update predicate to ALSO recompute when the transaction carries our sentinel annotation. This is the surgical change.

**The mechanism — annotation-driven StateField recompute:**

In `src/widget/liveModeBannerStateField.ts`, both StateField update functions currently bail out on `!tr.docChanged`:

```ts
update: (value, tr) => {
  if (!tr.docChanged) return value;
  return buildLeetCodeWidgetDecorations(tr.state);
},
```

Add a sentinel `Annotation.define<true>()` exported as `leetcodeRefreshAnnotation`, and widen the predicate:

```ts
update: (value, tr) => {
  if (!tr.docChanged && !tr.annotation(leetcodeRefreshAnnotation)) return value;
  return buildLeetCodeWidgetDecorations(tr.state);
},
```

Then the post-repair .then dispatches a `view.dispatch({ annotations: leetcodeRefreshAnnotation.of(true) })` against each known EditorView for the file path. CM6 treats this as a non-mutating transaction; the section lock filter at `src/main/sectionLockExtension.ts` is moot (no doc changes); the StateField recomputes; the LeetCodeFenceWidget is re-instantiated with the post-repair language; the widget remounts in a working state.

NOTE: this plan does NOT need to use the Phase 05.5 `'leetcode.*'` userEvent convention because the section-lock filter only fires on transactions that change the doc (`tr.changes.length > 0`); a doc-empty annotation-only transaction never reaches the filter. We DO use the `'leetcode.'` prefix on the userEvent purely for log-traceability + future-proofing per CLAUDE.md's documented convention; the project skill rule says "Any future plugin dispatch that targets a locked range MUST set `userEvent: 'leetcode.<verb>'`" — this dispatch does NOT target a locked range (no changes), so the rule technically doesn't apply, but consistency keeps the convention legible.
