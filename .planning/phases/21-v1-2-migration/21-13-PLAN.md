<objective>
Close UAT post-finding Gap B (`severity: major`, deterministic on every fresh problem open from the browser): "Opening a new problem from the problem browser writes the new note's source on disk with TWO fences under `## Code` — a top ` ```<langSlug> ` (e.g. ` ```java `) plain Obsidian code block AND a bottom ` ```leetcode-solve ` widget fence — both containing the same starter code. Renderer is innocent; the writer is the bug."

Per `21-HUMAN-UAT.md` Post-UAT Findings (lines 124-164) the debug agent diagnosed:

**Root cause** — `src/solve/starterCodeInjector.ts:256-271` (the `retrofit()` wrapper) calls `injectCodeSection(current, { starterCode, langSlug })` WITHOUT a `fenceKind` argument. The v1.3 short-circuit at `injectCodeSection.ts:106-112` (which would route the v1.3 path through `rewriteFenceBody` to leave the existing ` ```leetcode-solve ` fence untouched) is therefore skipped. Execution falls through to the legacy code path at lines 114-146 which prepends a fresh ` ```<defaultLanguage> ` (in the user-reported case, ` ```java `) fence ahead of the existing v1.3 fence. Net result on disk: TWO fences in ## Code.

**The four latent call sites** in `src/notes/NoteWriter.ts` all share the `retrofitStarterCode` wrapper (lines 230-241):
1. **Line 272** — re-open path with cached detail (D-11 reveal-first).
2. **Line 343** — cache-cleared recovery (`existingAtCanonical` branch when settings cache is wiped but the file remains on disk).
3. **Line 419** — new-note belt-and-suspenders retrofit AFTER `vault.create` writes the canonical body via `buildNoteBody`. **This is the one that produces the user-reported duplicate fence** because `buildNoteBody({ useInlineWidget: true })` already emits a v1.3 ` ```leetcode-solve ` fence (NoteTemplate.ts:241-243), then the retrofit's fenceKind-less call to `injectCodeSection` graft a sibling ` ```<defaultLanguage> ` fence on top.
4. **Line 453** — `backgroundRefresh` after the 7-day TTL expires (D-11 / D-12).

Without the fix, ALL four call sites are vulnerable to the same data corruption pattern; the user happens to hit (3) most often because it fires on every fresh `openProblem` for a problem not yet in the vault.

**Fix direction** — there are two equivalent shapes; pick (a) for the smallest diff:

- **(a) PREFERRED — extend `retrofit()`'s settings parameter with an optional `getUseInlineWidget?(): boolean` getter (NO new positional argument).** Inside retrofit, derive `fenceKind: settings.getUseInlineWidget?.() ? 'leetcode-solve' : 'legacy'` and pass it into `injectCodeSection`'s `InjectOptions`. NoteWriter's `this.settings` already exposes `getUseInlineWidget?(): boolean` (NoteWriterSettings interface, NoteWriter.ts:97-105) — it was wired in Phase 21 Plan 21-03 for `buildNoteBody`'s gate. ZERO call-site change required at the four NoteWriter sites — the existing `retrofitStarterCode(file, detail)` wrapper already passes `this.settings` into the raw `retrofit` invocation at NoteWriter.ts:240. The diff is contained to `starterCodeInjector.ts` (parameter widening + one-line dispatch) plus the test files.
- **(b) ALTERNATIVE (rejected for Plan 21-13)** — add an explicit `fenceKind?: 'leetcode-solve' | 'legacy'` positional/named parameter to `retrofit()` and have NoteWriter compute the value at each call site. This requires touching all four NoteWriter call sites + the wrapper + the consumer, and the test surface explodes proportionally. Save for Phase 22 if the call sites diverge; for the gap closure the minimal-diff option (a) closes all four simultaneously through the shared wrapper.

**Defense-in-depth (mirrors Plan 20-09 / main.ts:1421-1429 file-open gate)** — additionally, gate ALL FOUR `retrofitStarterCode` call sites in NoteWriter on `if (!useInlineWidget)` so that under v1.3 (the post-Phase-22 default) the retrofit call is structurally skipped entirely. Phase 20 Plan 20-09 used this exact pattern in `main.ts:1421-1429` to close a near-identical corruption pattern on the file-open hook; the rationale carries: the v1.3 widget owns its own fence body via `vault.process` writes, and the retrofit's purpose (legacy starter-code injection on cache-recovery) is structurally meaningless when the inline widget is in charge.

Combining (a) + the defense-in-depth gate gives BELT-AND-SUSPENDERS coverage:
  - Belt: the fenceKind plumbing means even if the gate is bypassed, the v1.3 short-circuit handles the call correctly.
  - Suspenders: the gate means the retrofit code path is skipped on v1.3 entirely, so the fenceKind plumbing is exercised only on the legacy path where it's a no-op.

Decision recorded for executor (per the planning_context request to document the option-(a)-vs-option-(b) choice): **option (a) is the plan; the rationale is "smallest diff that closes all four call sites simultaneously through the shared wrapper". The defense-in-depth gate at the four NoteWriter call sites is added in addition because it mirrors the Phase 20 Plan 20-09 main.ts:1429 pattern that closed a near-identical bug.**

Output: `src/solve/starterCodeInjector.ts:retrofit()` accepts a settings shape with optional `getUseInlineWidget?(): boolean` and threads `fenceKind` into `injectCodeSection`. `src/notes/NoteWriter.ts:retrofitStarterCode` (the wrapper at lines 230-241) gates on `useInlineWidget` — when true, the wrapper short-circuits before even calling raw retrofit (defense-in-depth).
