## Decisions Made

(See `key-decisions` frontmatter array above for the full list — 10 decisions.)

Highlights:
1. **`setProviderConfig` signature widened to `ProviderConfig | BedrockProviderConfig`** instead of casting at every call site. Runtime dispatch routes `p === 'bedrock'` to `sanitizeBedrockProviderConfig`. The 6 existing providers continue to use the inherited `sanitizeProviderConfig`. This is more ergonomic than forcing every settings-tab onChange handler to cast, and the runtime dispatch is a single boolean check.

2. **Tasks 1+2 combined commit.** Widening AIProvider with `'bedrock'` triggers the S4 exhaustiveness check at 6 sites; splitting the union widening from the per-site case additions would leave the build broken mid-commit. The two tasks form a coherent atomic unit; Task 3 (UI surface + docs + lint) is the natural second commit.

3. **`@aws-sdk/credential-providers` REJECTED.** Verified at install time — esbuild fails with 9 unresolved `node:fs/crypto/os/path/child_process` imports because the package targets pure Node (spawns child processes for `aws sts assume-role`, etc.). The plugin owns credential resolution via a ~30-line INI parser, which covers default-chain (env vars + `[default]`) and sso-profile (named `[profile <name>]`) modes. Real `aws sts assume-role` flows are out of scope per CONTEXT decision E.

4. **Logger REDACT regex extended in BOTH camelCase and snake_case forms** — the case-insensitive `/i` flag handles intra-word case but NOT inter-segment underscores. So `aws_access_key_id` and `accessKeyId` are both alternates in the regex. Caught during initial test run (3 failures); fix landed in same commit.
