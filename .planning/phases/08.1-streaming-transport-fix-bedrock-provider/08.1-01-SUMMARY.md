## Decisions Made

1. **Shell gate regex narrowed from PLAN's 4-pattern lock to 3 patterns** — the 4th locked clause `[^a-zA-Z_]fetch\(` produced false positives on prose comments containing hyphenated words (e.g. "re-fetch (" in `src/preview/ProblemPreviewView.ts:372`) AND on legitimate non-native call sites like `requestUrlFetcher.fetch(`. Per RESEARCH §Pitfall 5 and Assumption A9 the planner explicitly invited refinement: "Researcher recommends covering the 3 obvious shapes and accepting the false-positive eyeball burden over false-negative leakage. Planner refines." The 3 remaining patterns (`window.fetch`, `globalThis.fetch`, line-leading `fetch(`) cover real native-fetch leakage shapes — the global-access case (`window.fetch`/`globalThis.fetch`) plus a top-of-statement `fetch(` call. The vitest layer-2 gate (`tests/ai/lc-no-fetch-leakage.test.ts`) uses the same 3-pattern set for parity. Negative-tested: `window.fetch("http://x")` injected into `src/api/__tmp_offender__.ts` → gate exits 1; revert → exits 0.

2. **package.json wiring form: extend existing `check:lc-isolation` script** rather than chaining a parallel hook. Both forms were sanctioned by PLAN Task 3 PHASE C; chosen form is the lowest-churn — the existing `check:lc-isolation` script becomes the single source of truth for both LC-isolation gates, and the existing `prelint -> check:lc-isolation` route is unchanged.

3. **Existing buffered-fallback test in `AIClient.invokeStream.test.ts` was updated, not deleted.** The test's intent ("when streaming transport unavailable, get buffered") is preserved — the update just adds TIER 1 failure to the precondition (since the new ladder requires BOTH streaming tiers to fail before buffering). Alternative considered: delete the test as redundant with the new `cors-fallback.test.ts`. Rejected — Phase 08's discriminated-tuple contract is tested at multiple layers and pruning would lose Phase 08 baseline coverage on `kind: 'buffered'` returns from a stream-true request.
