## Area 1 — Streaming transport strategy

**Q: Which streaming transport should be the primary path?**

Options presented:
1. Native `fetch()` with obsidianFetch fallback (Recommended)
2. `electron.net.request` (older API)
3. `@electron/remote` shim
4. Settings-driven transport selector

**User selected:** Option 1 — Native `fetch()` with obsidianFetch fallback.

Decision implications captured in CONTEXT.md decision A:
- Three-tier fallback: native `fetch()` → `obsidianFetch('stream')` → `requestUrl`-buffered.
- CORS detection at first call; per-provider posture documented by researcher.
- Eager probe in `obsidianFetch.ts` STAYS — it's the correct safety net for the secondary tier.
- No settings UI for transport selection — auto-routing only.
- `consumeStream` branch in AIClient.invokeStream becomes live code.

## Area 2 — Bedrock auth model

**Q: How should Bedrock authenticate?**

Options presented:
1. Default AWS credential chain only (Recommended)
2. Default chain + explicit access-key fallback
3. Named SSO profile
4. All three, user picks via dropdown

**User selected:** "4 + bedrock api key" — option 4 with a 4th choice added (Bedrock API key, the long-term AWS Bedrock bearer token introduced in 2025).

Decision implications captured in CONTEXT.md decision B:
- Settings dropdown exposes 4 auth methods: default-chain / access-keys / SSO-profile / Bedrock API key.
- Settings UI swaps fields per choice; only the relevant secret field is visible.
- PluginData shape extended with bedrock-specific fields (region, modelId, authMethod, plus per-auth secrets).
- All secrets in `data.json` (same plain-text posture as existing API keys); logger redaction extended.
- Researcher MUST probe Electron renderer access to `~/.aws/credentials` and confirm AWS SDK credential resolver works from renderer Node context.
- Bedrock API key env-var name + SDK wiring TBD by researcher.

## Area 3 — Bedrock SDK packaging

**Q: How should Bedrock be packaged?**

Options presented:
1. Bundle `@ai-sdk/amazon-bedrock` + measure (Recommended)
2. Mark `@aws-sdk/*` external + dynamic require()
3. Hand-rolled REST against Bedrock InvokeModelWithResponseStream
4. Lazy-import Bedrock adapter only when active

**User selected:** Option 1 — Bundle + measure.

Decision implications captured in CONTEXT.md decision C:
- Add `@ai-sdk/amazon-bedrock` (researcher pins exact version; backlog suggests 4.0.107).
- Run `npm run check:bundle-size` after install.
- If under 1.2 MB ceiling: ship as-is.
- If over: planner gates with Rule 3 ceiling bump (precedent Phase 07-03, Phase 08-02).
- If excessive (e.g. >1.5 MB): escalate to lazy-import (Option C-fallback-1) or hand-rolled SigV4 (Option C-fallback-2).
- Researcher MUST measure before planner finalizes.
