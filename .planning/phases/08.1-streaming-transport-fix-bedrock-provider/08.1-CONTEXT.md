### A. Streaming transport — native `fetch()` primary, obsidianFetch fallback

- **Pick:** native `window.fetch()` as the streaming primary; AI SDK's `streamText` iterates the real `ReadableStream` natively. CORS detection at first call (or a probe at provider activation) — if any provider's preflight rejects, fall back to `obsidianFetch('stream')` (which itself eager-probes `electron.net.fetch` and falls through to `requestUrl`-buffered when unavailable).
- **Why over electron.net.request (older API):** The backlog note flags `contextIsolation: true` likely blocks BOTH `electron.net.fetch` AND `electron.net.request`. Researcher confirms; if confirmed, this option is dead. Native `fetch()` has no Electron-internal coupling and survives any future Obsidian renderer hardening.
- **Why over `@electron/remote` shim:** Deprecated dep, plugin-store-reviewer surface area, future Electron major may remove. Not worth the maintenance for a path native `fetch()` already covers.
- **Why over a settings-driven transport selector:** Pushes complexity to users; doesn't pick a default; "auto" still has to resolve to one of the same options. We can add a settings escape hatch later if dogfood demands.
- **Three-tier fallback chain (locked):**
  1. **Primary:** native `fetch(url, { signal, ...init })` → AI SDK consumes the real `ReadableStream`. Live token-by-token streaming. Real `AbortController` propagation.
  2. **Secondary:** `obsidianFetch('stream')` → `electron.net.fetch` (where available; eager probe currently always throws on Obsidian 1.10+ contextIsolation:true, but kept as a future-proofing seam — see "Side note" in `streaming-transport-fix.md`).
  3. **Tertiary:** `requestUrl`-buffered → "Thinking…" indicator + mm:ss counter, single-chunk response (today's working state). The `requestUrl` path STAYS — it's the universal floor.
- **CORS-aware dispatch:** Researcher MUST confirm CORS posture for all 5 providers (4 existing + Bedrock). Document each provider's outcome:
  - **localhost (Ollama)** — no preflight expected; native `fetch()` always works.
  - **Anthropic / OpenAI / OpenRouter** — researcher verifies via a real Obsidian renderer test; may require setting `Access-Control-Allow-Origin` headers on the provider side (which we don't control). If preflight blocks, that provider falls to obsidianFetch.
  - **Bedrock** — likely NO native `fetch()` (AWS doesn't send CORS-permissive headers to browser-style clients). If confirmed, Bedrock always uses `obsidianFetch('stream')` or `requestUrl`-buffered. Acceptable — Bedrock will still exercise the buffered path, which is what Phase 08 dogfood confirmed works.
- **Eager probe lives** — `loadElectronNet()` in `obsidianFetch.ts` STAYS as-is. Native `fetch()` lives at a HIGHER tier in `AIClient.invokeStream`; obsidianFetch is consulted ONLY after `fetch()` fails. The eager probe is correct and is now correctly insulated by the new primary.
- **No settings UI for transport selection** — auto-routing only. If dogfood surfaces the need (e.g. CORS works for some providers but breaks for others on certain user setups), Phase 09+ can revisit.
- **Test contract:**
  - New unit tests cover the CORS-fallback decision tree (mock fetch rejecting, fall through to obsidianFetch).
  - New integration test exercises the LIVE `consumeStream` branch in `AIClient.invokeStream` (currently dead code) via a mock `fetch()` returning a real `ReadableStream` that emits multi-chunk text.
  - Real-Obsidian UAT covers native-`fetch()` against at least Anthropic + Ollama. (Bedrock UAT is its own thing in plan 08.1-02.)

### B. Bedrock auth — 4-option dropdown

- **Pick:** Settings UI exposes an Auth method dropdown with 4 options. User picks one; settings UI swaps fields per choice.
  1. **Default credential chain** (recommended in copy) — Plugin stores nothing. AWS SDK reads `~/.aws/credentials`, env vars, EC2 instance role at request time. Works for users running `aws sso login` or with `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` env vars set when launching Obsidian. **Risk:** Electron renderer may not have filesystem access to `~/.aws/credentials`; researcher MUST probe this from a real Obsidian renderer. If filesystem access is blocked, this option falls back to env-var-only and copy reflects that.
  2. **Explicit access keys** — Plugin stores `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` in `data.json` (same plain-text posture as existing API keys; logger redaction extended to mask both). Worse security posture; lower friction for users without `aws` CLI installed.
  3. **SSO profile** — User types a profile name; plugin reads `~/.aws/credentials` for that profile via the AWS SDK credential resolver at request time. Best for corporate users with SSO. **Researcher confirms** whether `aws sts assume-role` flow works from Electron Node context; if not, this option degrades to a Notice telling the user to `aws sso login` and ensure env vars are exported (collapses functionally to option 1).
  4. **Bedrock API key** — AWS introduced long-term Bedrock API keys in 2025. Plugin stores the bearer token in `data.json` (logger redaction added). Researcher pins the exact env-var name + SDK wiring. Simplest UX for users who don't have AWS CLI infrastructure.
- **PluginData shape extension** (illustrative; planner / researcher confirms exact field set):
  ```
  bedrock: {
    region: string;             // default 'us-east-1'
    modelId: string;            // e.g. 'anthropic.claude-3-5-sonnet-20241022-v2:0'
    authMethod: 'default-chain' | 'access-keys' | 'sso-profile' | 'api-key';
    accessKeyId?: string;       // when authMethod === 'access-keys'
    secretAccessKey?: string;   // when authMethod === 'access-keys'
    ssoProfile?: string;        // when authMethod === 'sso-profile'
    bedrockApiKey?: string;     // when authMethod === 'api-key'
    disclosureAcknowledged: boolean;  // mirrors other providers
  }
  ```
- **Settings UI per auth choice** — only the relevant secret field is visible; the rest are hidden (reduces leak risk if user forgets which path they picked). Each shown secret is `<input type="password">` per AIPROV-02.
- **Test connection probe** — Bedrock has no public models-list endpoint accessible via the SDK without credentials; probe is a 1-token chat completion via SDK with `max_tokens: 1`, prompt `"ping"`, against the configured model ID. Cost: ~$0.0001. Mirrors Anthropic's probe shape (decision E in 07-CONTEXT).
- **Why over key-only:** AWS-native users (most Bedrock users) want SSO; locking them to plugin-stored access keys is a security regression and a friction point.
- **Why over chain-only:** Users without AWS CLI installed need a path; access-keys + Bedrock-API-key cover them.
- **Disclosure modal text** — Bedrock's per-provider disclosure (when user activates Bedrock) lists base URL as `https://bedrock-runtime.{region}.amazonaws.com` with the user's configured region substituted. The "exactly what gets sent" copy is identical to the other providers (Phase 07 D + Phase 08 disclosure-extension factory).

### C. Bedrock packaging — bundle the AI SDK + measure

- **Pick:** Add `@ai-sdk/amazon-bedrock` (researcher pins exact version, backlog note suggests `4.0.107`). Bundle alongside the other AI SDK packages. After install, run `npm run build && npm run check:bundle-size`.
- **Bundle ceiling logic:**
  - Current measurement: 996 KB. Ceiling: 1.2 MB. Headroom: ~200 KB.
  - **If the new bundle is under 1.2 MB:** ship as-is. No ceiling change.
  - **If the new bundle exceeds 1.2 MB:** planner gates with a Rule 3 ceiling bump (precedent: Phase 07-03, Phase 08-02). Bump value derived from measured size + comfortable headroom (e.g. measured 1.28 MB → ceiling 1.4 MB). Documented in `08.1-VERIFICATION.md` and the bundle-size config.
  - **If the new bundle exceeds 1.5 MB or otherwise feels excessive:** planner escalates to a fallback strategy:
    - **Option C-fallback-1:** Lazy-import the Bedrock adapter via dynamic `import('./providers/bedrock')` only when `activeAIProvider === 'bedrock'`. AWS SDK doesn't load for users on Anthropic/OpenAI/etc. Adds resolveAdapter complexity but keeps cold-start budget tight.
    - **Option C-fallback-2:** Hand-rolled REST against Bedrock's `InvokeModelWithResponseStream` endpoint via `requestUrl` + custom SigV4 signing. Smallest bundle (~10 KB). Most code to write and maintain. Treated as v1.2 work pulled forward — only adopted if both bundling and lazy-import are unacceptable.
- **Why not start with lazy-import:** Adds resolveAdapter complexity preemptively; if static bundle is fine, this is YAGNI. Mirrors Phase 07's "ship static, lazy-import only if dogfood demands" stance.
- **Why not start with hand-rolled SigV4:** Way more code; SigV4 is non-trivial; deferred to v1.2 (AIPROV-FUT-01) for a reason. Only adopt if ceiling-bump is unacceptable.
- **Researcher MUST measure** the bundle delta BEFORE planner finalizes. The measurement gates the choice between static-bundle / lazy-import / hand-roll.
- **esbuild config:** Researcher confirms `@aws-sdk/*` tree-shakes acceptably with esbuild; if not, planner gates. **No `external` markers** unless absolutely required (Obsidian doesn't provide AWS SDK; marking external = runtime require failures).

### D. Plan ordering — streaming transport first, then Bedrock

- **Pick:** Two plans, sequential.
  - **Plan 08.1-01: Streaming transport — native `fetch()` primary + CORS fallback.** Lands the new transport against the existing 4 providers. Verifies live streaming works in real Obsidian. Updates `obsidianFetch.ts` factory + `AIClient.invokeStream` decision tree. Tests: CORS-fallback decision tree, live-stream consumption against mock `fetch()`, no-regression on buffered fallback. UAT: real Obsidian, Anthropic + Ollama. **Gates 08.1-02.**
  - **Plan 08.1-02: Bedrock provider adapter.** Adds `src/ai/providers/bedrock.ts`, extends `resolveAdapter`, widens `AIProvider` union, adds settings UI branch with 4-option auth dropdown, extends PluginData shape + shape-guards + log redaction, runs bundle measurement, README "Network use" update. Tests: per-auth-mode adapter behavior, settings shape-guard collapses, bundle ceiling. UAT: real Obsidian, Bedrock with at least 2 of the 4 auth methods (default-chain + 1 explicit choice).
- **Why over Bedrock first:** Plan 08.1-02 inherits the new transport for free; UAT validates streaming over a NEW transport on a NEW provider in one shot. If transport ships first, Bedrock UAT is the natural live-fire test.
- **Why over one merged plan:** Smaller PRs, easier rollback if dogfood surfaces an issue, easier to bisect.
- **Why over "decide later":** The dependency is one-way (transport doesn't need Bedrock; Bedrock benefits from transport). Lock now.
- **Test/UAT separation:**
  - **Plan 08.1-01 UAT** is "AI Debug streams live token-by-token against an existing provider (Anthropic or Ollama)." If the user's only working Anthropic access is via Bedrock, the user ALSO has an OpenAI key OR runs Ollama for this UAT — researcher / planner confirms with the user at planning time which provider to use.
  - **Plan 08.1-02 UAT** is "AI Debug works against Bedrock" — the new provider, exercising the new transport (or the obsidianFetch fallback if CORS blocks).
