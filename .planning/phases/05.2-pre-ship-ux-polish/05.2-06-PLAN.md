  <action>
    Execute the live-smoke checklist below in the dev vault and record results in `.planning/phases/05.2-pre-ship-ux-polish/05.2-06-SUMMARY.md`. Do NOT make any code changes during this task — if a checklist item fails, stop the smoke, log the fail in the SUMMARY, and return to the user for routing (gap-closure plan vs. defer vs. accept-as-shipped).

    **Pre-flight commands (in repo root):**
    ```
    npm run lint          # expect: zero Required violations
    npm test              # expect: full suite GREEN
    npm run build         # expect: clean main.js + styles.css
    ```

    Copy `main.js`, `manifest.json`, `styles.css` into your dev vault's `.obsidian/plugins/obsidian-leetcode/` (or rely on hot-reload plugin if installed), then reload Obsidian (Ctrl+R / Cmd+R).

    **Checklist (walk each item; record status in SUMMARY):**

    [ ] 1 (D-01) Settings row merge
    Open Settings → LeetCode. Observe the Status row.
    Expected: Status + description + (Login OR Logout) button all on ONE row. Button sits on the right side.
    - If logged in: description reads "Logged in as {username}"; Logout button visible on right.
    - If logged out: description reads "Not logged in"; Log-in button (accent-styled) visible on right.
    Click the button. Expected: auth action executes (login opens BrowserWindow OR logout clears state). UI re-renders.

    [ ] 2 (D-02) Filter `Language` option removed
    Open command palette → run `LeetCode: Open problem browser`. Click the filter chip / open FilterModal.
    Click `+ Add filter`.
    Expected: the popup menu lists Status, Difficulty, Topics, Question ID, Acceptance, Premium. NO `Language` entry.

    [ ] 3 (D-03) Premium filter multi-value
    In the same FilterModal, add a `Premium` rule via `+ Add filter`.
    Expected: the value cell shows a picker (chevron-anchored) identical in shape to Status/Difficulty pickers. Clicking opens a popover with TWO checkboxes: `Premium Content` and `Non-Premium Content`.
    Check both; hit Apply; reopen modal. Expected: rule persists as two-checkbox selection.

    [ ] 4 (D-04) Fresh-install badge
    (Requires a vault where the plugin was never configured, OR clear plugin data and reload.)
    Open the problem browser view. Before opening FilterModal, observe the filter-chip badge.
    Expected: NO badge visible (the `isPremium=false` auto-default rule is present under the hood but not counted).
    Now open FilterModal + hit Apply without changes. Reopen the browser. Expected: badge still absent for isPremium=true accounts; for free accounts the auto-default rule becomes marker-less after Apply so it DOES count as 1 → badge shows 1. Record which behavior you observed.

    [ ] 5a (D-06) Auto-insert on file-open
    Open a problem note that already exists in the vault via a path OTHER than the row-click (e.g., file-explorer click, backlink, `Cmd+O` quick-switcher).
    If the note currently lacks a `## Code` section, expected: starter block auto-inserted silently on open, no modal, no Notice.
    If the note already has a `## Code` section, expected: no change (idempotent).

    [ ] 5b (D-05) `Insert starter code` command absent
    Open command palette and type "insert starter".
    Expected: NO entry for `LeetCode: Insert starter code`.

    [ ] 5c (D-07) `Reset code` command present + confirm gate
    On an lc-slug note with some user-modified code in the `## Code` fence, open command palette → run `LeetCode: Reset code`.
    Expected: ConfirmOverwriteModal appears with Cancel default-focused.
    Hit Cancel. Expected: no change to note.
    Re-run `Reset code`. Hit Overwrite. Expected: fence replaced with starter code; Notice "Code reset to starter." appears.
    On an lc-slug note with empty `## Code` fence, run `Reset code` → expected: no confirm; proceeds silently to insert starter + shows Notice.
    On a non-lc-slug note (any random note), open command palette. Expected: `Reset code` command is NOT offered.

    [ ] 6a (D-09) AC chip — light theme
    Switch Obsidian to default light theme. Open `LeetCode: View past submissions` on a problem with AC history.
    Expected: each AC row's status chip reads "Accepted" in visibly readable white text on a darker green pill.

    [ ] 6b (D-09) AC chip — dark theme
    Switch to default dark theme. Repeat. Expected: same readability.

    [ ] 6c (D-09) WA chip — light theme
    In the same picker, find a row with WA (or trigger a WA submission first). Expected: chip reads "Wrong Answer" in white on darker red pill.

    [ ] 6d (D-09) WA chip — dark theme
    Switch dark theme. Expected: same readability.

    [ ] 6e (D-09) Other chips
    If history includes TLE/MLE/RE/CE: verify labels "Time Limit Exceeded", "Memory Limit Exceeded", "Runtime Error", "Compile Error" render readably. CE specifically must show the Phase 5 D-29 ORANGE background — NOT the new red background (cascade invariant).

    [ ] 7 (D-10) Silent Copy-to-Code
    Open a submission detail from the picker. Click "Copy to Code".
    Expected: NO modal appears. Vault updates silently. The submission window may auto-close or stay open per existing behavior — no regression.
    Repeat on a note with a non-empty `## Code` fence: still no modal; fence is overwritten silently.

    [ ] 8 (D-12) Language dropdown list
    Settings → LeetCode → Default language dropdown. Expected: 19 entries: Python3, Python, Java, C++, C, C#, JavaScript, TypeScript, PHP, Swift, Kotlin, Dart, Go, Ruby, Scala, Rust, Racket, Erlang, Elixir.
    Select one unfamiliar language (e.g. Elixir). Expected: selection persists on reload.

    [ ] 9 (D-13) python3 syntax highlighting
    Create (or open) a markdown note (can be non-LC) with a fenced block tagged `python3` (triple-backtick open, `python3` tag, `def hello(): print("hi")` body, triple-backtick close).
    Switch to Reading Mode. Expected: the block highlights like Python (keyword `def` in color, string `"hi"` in another color, indentation preserved).
    Switch to a note with a `python`-tagged block to compare — they should look identical.

    **Regression checks:**

    [ ] R1 Phase 5.1 edit-mode Run/Submit buttons
    On an lc-slug note in Live Preview mode, below the `## Code` fence, Run + Submit buttons should appear inline and function (clicking Submit triggers the submit flow). No duplicates, no visual drift from Phase 5.1 Plan 03 baseline.

    [ ] R2 Phase 5 reading-mode Run/Submit buttons
    Switch to Reading Mode on the same note. Run + Submit buttons should appear below the code block. Clicking Submit triggers submit. No regression.

    [ ] R3 Full `npm test` suite still GREEN
    One last local run before sign-off.
