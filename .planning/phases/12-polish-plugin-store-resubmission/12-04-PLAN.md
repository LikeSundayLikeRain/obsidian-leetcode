**D-12: Wikilink-to-preview for unresolved problem links.**

Per RESEARCH Open Question #1, the safest approach is: register a `file-open` handler that detects newly-created empty files matching the problem slug pattern, deletes the blank file, and opens preview instead.

Implementation in `src/main.ts` onload:

1. Register a workspace `file-open` event handler (via `this.registerEvent`) that fires after Obsidian creates and opens a file from a wikilink click.

2. In the handler:
   a. Get the opened file: `const file = this.app.workspace.getActiveFile();`
   b. Check if the file is in the problems folder: `file.path.startsWith(this.settings.getProblemsFolder())`
   c. Check if the file is empty (just created from a wikilink): read the file content via `await this.app.vault.read(file)` and verify it is empty string or only whitespace.
   d. Extract the slug from the filename: the filename convention is `{id}-{slug}.md` or just `{slug}.md`. Parse the basename (strip `.md` extension).
   e. Verify the slug exists in the problem index: `this.settings.getProblemDetail(slug)` returns non-null. If slug is not found, let Obsidian's default behavior stand (user created a non-LC note).
   f. If all checks pass: delete the blank file via `await this.app.vault.delete(file)`, then call `openOrReusePreview(this, slug)` to show the preview tab.

3. Guard against infinite loops: the handler must NOT fire on files opened by the preview-to-note flow or by NoteWriter. Add a flag `this._suppressWikilinkIntercept = true` before any legitimate note creation, and reset it after. OR check that the file has zero bytes BEFORE the handler runs (NoteWriter always writes content immediately via `vault.create` with a non-empty body).

4. The handler must be debounced or guarded so it doesn't fire on normal file opens (existing notes with content). The empty-content check is the gate — only truly blank files (0-length content) trigger the intercept.

Place the registration near the existing `file-open` handler at lines 786-795 (or integrate into the existing `makeFileOpenHandler` if architecturally appropriate). If integrating, extend the handler's options with a `wikilinkIntercept` check.

**D-10: Cold-start performance — deferred AI construction.**

Per RESEARCH Pitfall 6, dynamic `import()` is ineffective in esbuild CJS. The lever is deferred CONSTRUCTION — keep the static import but delay `new AIClient(...)` until the first AI call.

In `src/main.ts`:

1. Change `this.aiClient` from eager construction at Step 5.9 to a lazy getter pattern:
   - Declare the backing field: `private _aiClient: AIClient | null = null;`
   - Add a getter method: `get aiClient(): AIClient { if (!this._aiClient) { this._aiClient = new AIClient(this.settings, (provider, cfg) => this.requireAIDisclosure(provider, cfg)); } return this._aiClient; }`
   - Remove the direct `this.aiClient = new AIClient(...)` line at Step 5.9.

2. The downstream consumers that use `this.aiClient` (PatternClusterEngine, testActiveAIConnection, openAIDebug, startAutoReview) will trigger construction on first access. This is safe because all AI operations are user-initiated (first click on AI Debug, first AC with AI review enabled, test connection button).

3. PatternClusterEngine at line 356 takes `aiClient: this.aiClient` in its constructor options. With the lazy getter, this will trigger construction at engine creation time (line 356 runs during onload). To truly defer, change the PatternClusterEngine constructor to accept a getter function instead: `aiClient: () => this.aiClient` (typed as `() => AIClient`). Then in PatternClusterEngine, access the client via the getter on each classify call rather than storing a reference.

   **Alternative (simpler):** If the main cold-start cost is AI SDK module evaluation (which happens at import time regardless), the lazy construction may not help much. In that case, simply wrap the AI-related imports in a deferred pattern. However since esbuild CJS bundles all imports statically, the real win may be minimal. Implement the lazy getter pattern as described — it defers constructor-time work (which may include SettingsStore reads and adapter resolution) even if module evaluation is unavoidable.

4. Ensure `this.hubWriter` construction (line 352) is NOT deferred — it has no AI SDK dependency and is needed for the 1-hour reconcile interval.
