# Phase 12 — UI Design Contract

> Visual and interaction contract for the polish + release phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Obsidian plugin — pure DOM) |
| Preset | not applicable |
| Component library | none (Obsidian `createEl` / `createDiv` API) |
| Icon library | Lucide (via Obsidian `setIcon()`) |
| Font | Obsidian theme-inherited (`--font-interface`, `--font-monospace`) |

**Rationale:** This project is an Obsidian plugin. It uses Obsidian's native DOM helpers and scoped CSS with Obsidian CSS variables. No React, no Svelte, no shadcn. All UI is built with `document.createElement` wrappers and styled via a single `styles.css` file at the project root.

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline chip padding, badge internal padding |

| 2xl | 32px | Not used in Phase 12 |
| 3xl | 64px | Not used in Phase 12 |

Exceptions: none — Phase 12 reuses the existing spacing vocabulary established in Phases 3-10.

## Typography

| Role | Size | Weight | Line Height | CSS Variable |
|------|------|--------|-------------|--------------|
| Body | 13px | 400 | 1.5 | `font-size: 13px` (existing modal body convention) |
| Label | 12px | 600 | 1.0 | `font-size: 12px` (existing `leetcode-verdict-section-label`) |

**Note:** All font sizes inherit from existing codebase conventions. Phase 12 introduces no new typographic roles. The pattern chip uses the established chip typography (11px, weight 600) from the contest verdict chip pattern.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background-primary)` | Modal backgrounds, view surfaces |
| Secondary (30%) | `var(--background-secondary)` | Code blocks, input sections, chip backgrounds |
| Accent (10%) | `var(--interactive-accent)` | Pattern chip clickable state on hover only |
| Destructive | `var(--text-error)` | Not used in Phase 12 (no destructive actions) |
| Success | `var(--text-success)` / `var(--color-green)` | AC verdict title, pattern chip tint background |

Accent reserved for:
- Pattern chip hover state (border-color transition to accent)
- Pattern chip text color uses `var(--text-accent)` to signal clickability (matches `.leetcode-language-chevron-item.is-current` precedent)

**Color rule (enforced):** No raw hex values. No `rgba()` literals. All colors via Obsidian CSS variables. `color-mix()` permitted for tinted backgrounds (established precedent in `.leetcode-contest__verdict-chip--accepted`).

## Component Inventory

### 1. Pattern Chip (NEW — Verdict Modal AC state)

**Location:** Rendered between the "Accepted" display and the AI review stream area inside VerdictModal.

**Visual spec:**
```
.leetcode-verdict .leetcode-verdict-pattern-chip {
  display: inline-flex;
  align-items: center;
  height: 22px;
  padding: 0 8px;
  border-radius: 11px;
  font-size: 11px;
  font-weight: 600;
  background: color-mix(in srgb, var(--background-secondary) 85%, var(--interactive-accent) 15%);
  color: var(--text-accent);
  cursor: pointer;
  transition: background 120ms ease, border-color 120ms ease;
  border: 1px solid var(--background-modifier-border);
  margin: 8px auto 0;
  text-align: center;
}
.leetcode-verdict .leetcode-verdict-pattern-chip:hover {
  background: color-mix(in srgb, var(--background-secondary) 70%, var(--interactive-accent) 30%);
  border-color: var(--interactive-accent);
}
```

**Interaction:**
- Click closes the verdict modal, then opens `LeetCode/Patterns/{Pattern}.md` hub note via `app.workspace.openLinkText()`.
- Same close-then-navigate pattern as the AI Debug button (established in Phase 08-05).
- Chip text is the pattern display name from `lc-pattern` frontmatter (e.g., "Two Pointers", "Dynamic Programming").
- Only renders when `lc-pattern` is available (i.e., Phase 11 classification completed). Hidden otherwise (no empty state).

**DOM structure:**
```
<div class="leetcode-verdict-pattern-chip" data-lc-role="pattern-chip">
  {patternDisplayName}
</div>
```

**Placement in AC body (after Phase 12 removal of Close button):**
```
[Title: "Accepted -- {problemTitle}"]
[Runtime: X ms . Memory: Y MB]
[Body:]
  [.leetcode-verdict-ac-display: "Accepted"]
  [.leetcode-verdict-percentile: "Beats X% (runtime) . Y% (memory)"]
  [.leetcode-verdict-pattern-chip: "Two Pointers"]   <-- NEW
[AI Review stream area]                               <-- streams below
```

### 2. Verdict Modal Layout (MODIFIED — Remove Close Button)

**Change:** Remove ALL `data-lc-role="close"` button elements from `verdictModalRenderer.ts`. This affects 5 render paths:
1. `renderTimeout` footer
2. `renderRunResult` footer
3. `renderRunErrorBlock` footer
4. `renderSubmitVerdict` footer
5. `renderUnknownVerdict` footer

**Rationale (D-01):** Obsidian's standard X button in the modal title bar handles dismissal universally. Removing the Close button eliminates the layout problem where AI review content renders below it on AC.

**After removal:** The footer (`leetcode-verdict-footer`) still renders for action buttons (AI Debug, Copy failing testcase, Copy error, Copy payload). If no action buttons are needed (e.g., AC state), the footer div is not rendered at all — the AI review streams directly below the AC body.

**CSS change:**
```css
/* Remove margin-top from footer when it contains only action buttons (no Close) */
.leetcode-verdict .leetcode-verdict-footer:empty {
  display: none;
}
```

### 3. AI Review Stream Area (MODIFIED — Layout Adjustment)

**Change:** With Close button removed, the AI review stream area (`leetcode-ai-review-loading` + rendered markdown) naturally fills the modal body below the Accepted banner. No sticky footer needed.

**Existing classes remain unchanged:**
- `.leetcode-ai-review-loading` — pulsing "Reviewing..." indicator
- `.leetcode-ai-stream-body` — rendered markdown area with `user-select: text`

### 4. H1 Title in Problem Notes (NEW — NoteTemplate Change)

**Change:** Problem notes now include a `# {Title}` H1 heading at the top of the body, before `## Problem`.

**Template output:**
```markdown
---
lc-slug: two-sum
lc-difficulty: Easy
...
---

# Two Sum

## Problem

{problem content}

## Code
...
```

**Typography:** Standard H1 rendering by Obsidian — no custom CSS needed. Obsidian renders `# ` headings at the theme's `--h1-size` (typically 32px) with `--h1-weight` (typically 700).

**Interaction:** When "Show inline title" is OFF in Obsidian settings, this H1 provides the visible title. When "Show inline title" is ON, users see the title twice (once inline from filename, once from H1) — this is acceptable and follows standard Obsidian note conventions.

## Interaction Contracts

### Pattern Chip Click (New)

1. User sees "Two Pointers" chip in verdict modal after AC
2. User clicks the chip
3. Modal closes (same pattern as AI Debug button: close-then-navigate)
4. Hub note `LeetCode/Patterns/Two Pointers.md` opens in the workspace
5. If hub note does not exist yet (edge case: classification ran but hub writer hasn't executed), open via `openLinkText` which creates a dangling link — acceptable per Obsidian conventions

### Wikilink-to-Preview Navigation (New)

1. User clicks a `[[123-two-sum|Two Sum]]` wikilink in a hub note or Related Variants section
2. If a note exists at that path, Obsidian's default behavior applies (opens the note) — no interception
3. If NO note exists at that path, intercept the navigation event
4. Route through `previewRouter.openOrReusePreview(slug)` to show the problem preview tab
5. Preview tab shows difficulty chip + "Start Problem" button (existing Phase 06 chrome)

### Verdict Modal Close (Modified)

1. Obsidian's built-in X button in the modal title bar is the sole dismissal mechanism
2. Pressing Escape also closes the modal (Obsidian default Modal behavior)
3. No explicit "Close" button rendered in the modal body/footer
4. If AI review is streaming, closing the modal triggers the existing AbortController cancel (Phase 08 contract preserved)

### Contest Tab Idempotency (Bug Fix)

1. User clicks a contest problem that already has an open `ContestSolveView` tab
2. System scans workspace leaves for matching view type + slug (same pattern as `openOrReusePreview`)
3. If found: refocus existing leaf (no new tab created)
4. If not found: create new tab (existing behavior)

### Contest Sidebar AC Update (Bug Fix)

1. User submits a solution in `ContestSolveView`
2. Verdict comes back as Accepted
3. Submit handler calls `ContestSessionManager.onVerdictChange(slug, 'ac')`
4. Session manager updates internal state
5. Timer header UI re-renders badge for that problem (green checkmark)
6. No manual refresh needed

### AI Review During Contest (Bug Fix)

1. User ACs a problem during an active timed contest
2. `onStartReviewStream` callback is NOT passed to the VerdictModal constructor when originating from `ContestSolveView`
3. AI review is deferred — fires during finalization after notes are written
4. User experiences no delay between AC and returning to the next problem

### Contest Finish Lifecycle (Bug Fix)

1. User clicks "Finish" button in contest timer header
2. System awaits the full finalization pipeline:
   a. Write canonical problem notes to vault
   b. Write summary note to `LeetCode/Contests/{date}-{id}.md`
   c. Trigger AI contest analysis (if enabled)
3. ONLY after all steps complete: return to contest browser view
4. If any step fails: surface partial results via Notice, still return to browser
