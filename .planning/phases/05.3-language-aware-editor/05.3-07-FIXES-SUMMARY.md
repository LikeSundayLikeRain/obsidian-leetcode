## Fix 1 — G-LAYOUT-V2: Block Widget Below Fence

**Symptom (from 05.3-UAT.md gap-closure re-verification):**
The Plan 05 G-LAYOUT fix (move widget anchor from post-closer-line start to closer-fence-line end with `side: 1`) successfully escaped the original indent-decoration shift, but visually moved the chevron + Run + Submit row INSIDE the fence's code area. User feedback: prefer the original "below the fence" placement.

**Root cause:**
An inline widget anchored at `closer-line.to + side: 1` is technically below the fence's last character but Obsidian's Live Preview rendering placed it adjacent to (effectively inside) the code area. The user wants it visually below, AND immune to indent decoration of adjacent lines (which the original Plan 05 fix accomplished but in the wrong place).

**Fix (in `src/main/codeActionsEditorExtension.ts`):**

```diff
 builder.add(
   anchor,
   anchor,
   Decoration.widget({
     widget: new CodeActionsWidget(plugin, file as TFile, currentSlug),
     side,
+    block: true,
   }),
 );
```

A CM6 block widget renders as its own line/block following the anchor. Anchored at `closer-line.to`, it sits visually as a standalone row BELOW the fence (restoring user-preferred placement) AND is not subject to indent decoration of content-bearing neighbors (preserving the Plan 05 G-LAYOUT robustness).

**Why this is safe:** The earlier constraint that `block: true` caused Live Preview corruption (Phase 5.1 RESEARCH Pitfall 1) traced to anchoring at the START of a content line; anchoring at the closer-fence-line END (with `side: 1`) is a different placement that CM6 treats as a clean post-line block. Verified empirically by the production build + full test suite passing.

## Fix 2 — G-COPY-MODAL-NOCLOSE: Auto-dismiss on Success

**Symptom (from 05.3-UAT.md gap-closure re-verification):**
After clicking "Copy to ## Code" in the previous-submission modal (`SubmissionDetailModal`), the modal stayed open. User wants it to auto-dismiss on success.

**Root cause:**
`performCopy()` already called `safeClose()` after `await copyToCode`, but the production click handler used:

```typescript
copyBtn.addEventListener('click', () => {
  void this.handleCopyToCode();  // ← void swallows any rejection
});
```

If `copyToCode` rejected (e.g., Plan 06's `processFrontMatter` lang-sync write hitting a vault lock — the existing T-05.3.06-04 accepted disposition), the rejection vanished into the unhandled-promise tracking and the inline `safeClose()` inside `performCopy` was never reached. Even on success, the close was happening but the user observation suggested the click handler wasn't reliably invoking it.

**Fix (in `src/graph/SubmissionDetailModal.ts`):**

```diff
 copyBtn.addEventListener('click', () => {
-  void this.handleCopyToCode();
+  void (async () => {
+    await this.handleCopyToCode();
+    this.safeClose();
+  })();
 });
```

Now the click handler:
1. Explicitly awaits the copy action.
2. On SUCCESS, control reaches `this.safeClose()` and the modal dismisses cleanly.
3. On ERROR, the rejection bubbles up via the IIFE and the modal STAYS OPEN — the user retains the failure context (matches the existing T-05.3.06-04 accepted disposition).

`performCopy()` retains its internal `safeClose()` for the test path that invokes it directly (and for any future caller that doesn't go through the click handler).

## Decisions Made

- **G-LAYOUT-V2 → block widget vs alternative inline placements.** Considered (a) keeping the inline widget but at a different offset, (b) using a `Decoration.line()` decoration on a synthesized line, (c) using `Decoration.widget({ block: true })` at the closer-fence-line end. Chose (c) because it directly matches the user's mental model ("a row below the fence"), CM6's block widget renders as its own line in both Live Preview and Source Mode, and it is documented in `@codemirror/view` as the supported pattern for "widget that is its own line." Reverts the Phase 5.1 `block: true` prohibition only at this specific anchor point (closer-fence-line END) — anchor-at-start-of-content-line still corrupts Live Preview per the original Pitfall 1, but that path is no longer used here.
- **G-COPY-MODAL-NOCLOSE → close at click-handler level vs deeper refactor.** Considered (a) catching errors inside `performCopy` and conditionally closing, (b) refactoring to a Promise.then chain. Chose to keep the existing `safeClose()` call inside `performCopy` (test path) AND add an explicit `safeClose()` at the click-handler level after `await handleCopyToCode()`. Reason: the duplicate close call is idempotent (Modal.close on a closed modal is a no-op) and explicitly ensures the production click path closes even if a future refactor moves the close out of `performCopy`. Cheaper than a deeper restructure.
