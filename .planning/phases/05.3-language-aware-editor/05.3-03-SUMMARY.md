`▼ {Language}` LEFT of Phase 5.1's Run/Submit row; clicking a different language

transaction = one Cmd-Z) followed by `processFrontMatter` for `lc-language`.
Reading-Mode is byte-identical (D-09 lock). Bundle is **151,577 bytes (~148 KB)**,
+2,842 bytes vs Wave 1 baseline.

## src/main/languageChevronWidget.ts (NEW)

**157 lines**, exports:

```typescript
export interface LanguageChevronHost extends CodeBlockButtonRowHost {
  switchLanguage(file: TFile, newSlug: string): Promise<void>;
}

export function buildLanguageChevron(
  doc: Document,
  plugin: Plugin & LanguageChevronHost,
  file: TFile,
  currentSlug: string,
): HTMLElement;
```

Behavior contract (matches UI-SPEC §"Chevron click → dropdown open" + §"Dropdown
item click → language switch"):

- Renders `<span.leetcode-language-chevron-wrapper>` containing
  `<button.leetcode-language-chevron>` (label `▼ {DisplayLabel}`) +
  `<div.leetcode-language-chevron-dropdown>` (`role="listbox"`, initially
  `style.display: none`) with 8 `<button.leetcode-language-chevron-item>`
  children in `LC_CHEVRON_LANG_ORDER`.
- Current language item gets `.is-current`.
- `aria-haspopup="listbox"` + `aria-expanded` toggles on chevron click.
- Item click closes the dropdown FIRST (instant tactile feedback per UI-SPEC
  §"State machine"), then calls `plugin.switchLanguage(file, slug)` if the
  slug differs; current-language click is a no-op.
- Outside-click dismissal: capture-phase listener attached to the `doc`
  parameter only while the dropdown is open; self-detaches on close.
- Esc dismisses the dropdown.
- No `title` attribute (UI-SPEC §Copywriting "zero hover tooltip").
- CSP-safe: `createElement` + `textContent` exclusively; never reaches for
  the global Document.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] cwd drift wrote files into the main repo on first attempt**

- **Found during:** Task 1 — `npx vitest` reported "No test files found"
  immediately after the Write tool created the new files.
- **Issue:** The Write tool's first invocation used absolute paths starting
  with `/Users/moxu/projects/obsidian-leetcode/...` rather than the worktree
  root. macOS path resolution placed the new files in the **main repo**, not
  the worktree, so vitest (running with the worktree as cwd) couldn't find
  them. The orchestrator prompt's `<worktree_branch_check>` flagged exactly
  this hazard.
- **Fix:** Moved `src/main/languageChevronWidget.ts`,
  `tests/main/codeBlockButtonRow.test.ts`,
  `tests/main/languageChevronWidget.test.ts` from the main repo into the
  worktree via `mv`; copied `src/main/codeBlockButtonRow.ts` (which had
  Wave 2 edits) into the worktree, then `git checkout -- src/main/codeBlockButtonRow.ts`
  in the main repo to restore it. From this point on, every Write/Edit was
  rooted at the worktree's `git rev-parse --show-toplevel` path.
- **Files affected:** main repo working tree was returned to its pre-execution
  state (only pre-existing modifications to `.planning/REQUIREMENTS.md`
  remained — those exist outside this plan's scope).
- **Outcome:** No commits landed on the main repo; all 4 plan commits live
  on the worktree branch as expected.

**2. [Rule 1 — Bug] Single-cast `view.editor.cm as EditorView` failed TS2339**

- **Found during:** Task 3 verification (`npx tsc --noEmit`).
- **Issue:** `Editor` interface in obsidian.d.ts does not expose a `cm`
  property; the original RESEARCH §Pitfall 6 example used a single cast which
  passes at runtime but fails TS2339 at compile time.
- **Fix:** Double-cast through `unknown`:
  `(view.editor as unknown as { cm: EditorView }).cm`. Same pattern used in
  the codebase for other documented-internal Obsidian property access.
- **Files modified:** `src/main.ts` (single-line fix; no semantic change).

**3. [Rule 1 — Bug] codeBlockButtonRow.test.ts hit TS2532 strict-null-check failures**

- **Found during:** Task 2 verification (`npx tsc --noEmit`).
- **Issue:** `row.children[0].classList.contains(...)` — TypeScript strict
  null checks treat `HTMLCollection[i]` as possibly undefined.
- **Fix:** Added `!` non-null assertions on the indexed children references.
  Same pattern used in `codeActionsEditorExtension.test.ts` (`runBtn!.click()`)
  and elsewhere in the test suite.
- **Files modified:** `tests/main/codeBlockButtonRow.test.ts` (4 occurrences).
