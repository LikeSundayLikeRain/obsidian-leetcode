## Feature depth

### Q1 — What level of language behavior should Phase 5.3 ship?

| Option | Description | Selected |
|--------|-------------|----------|
| Full LanguageSupport | Use @codemirror/lang-* packs as-is: indent rules + bracket matching + auto-close brackets + syntax highlighting inside the fence. Closest to "IDE-grade" | ✓ |
| Indent rules only | Ship just auto-indent; no bracket auto-close, no highlighting. Smaller surface; cheaper bundle. | |
| Indent + bracket auto-close, no highlighting | Middle path; no color in Edit Mode; still feels smart | |
| Full pack but gate highlighting off | Full LanguageSupport but disable its highlighter extension | |

**User's choice:** Full LanguageSupport → **D-01**
**Notes:** Recommended option; closest to ROADMAP's "IDE-grade" promise. Trade-off is bundle size, addressed in Bundle strategy area.

---

### Q2 — Highlighting tension: Reading (Prism) vs Edit (Lezer) grammars won't match exactly. Acceptable?

| Option | Description | Selected |
|--------|-------------|----------|
| Accept mismatch | Different grammars in different modes; users rarely side-by-side compare | ✓ |
| Strip CM6 highlighting, keep only indent+brackets | Avoid mismatch by shipping no Edit-Mode colors; partial regression on 5.2 | |
| Investigate later | Ship full pack now; revisit theming parity if users complain | |

**User's choice:** Accept mismatch → **D-02**
**Notes:** Consistent with how most Obsidian plugins end up. No theme parity work in 5.3.

---

### Q3 — Auto-close brackets globally in Live Preview, or scoped to fence?

| Option | Description | Selected |
|--------|-------------|----------|
| Only inside supported-language fence | Auto-close is part of LanguageSupport for that fence's language | ✓ |
| Globally whenever LC note has any supported fence | Enable closeBrackets() at whole-editor level; prose typing affected | |

**User's choice:** Scoped to fence → **D-03**
**Notes:** Parity with 5.1 scoping discipline — plugin does not modify prose-typing behavior.

## Bundle strategy

### Q7 — How should the 6 language packs (~380kB total) ship?

| Option | Description | Selected |
|--------|-------------|----------|
| Lazy-load via dynamic import on first fence | Zero startup cost; +N kB per language actually used; async boundary adds complexity | ✓ |
| Eager-bundle all 6 packs | Simplest; guaranteed first-keystroke behavior; bundle ~680kB | |
| Conditional eager — only settings.defaultLanguage's pack | Saves bundle; inconsistent across languages | |

**User's choice:** Lazy-load → **D-07**
**Notes:** Recommended for startup-latency impact. Follow-up Q8 clarified this does NOT reduce bundle size (see D-08).

---

### Q8 — Async-resolution gap: esbuild chunking vs single main.js requirement?

| Option | Description | Selected |
|--------|-------------|----------|
| Inline dynamic imports via splitting:false + async wrapper | Keep CJS single-file; pack init deferred; net = startup-latency win, not size win | ✓ |
| True code-splitting via ESM + runtime loader shim | Real bundle win; breaks plugin-store single-main.js requirement; forbids eval | |
| Drop lazy-load; eager-bundle (revisit D-07) | Simplest; review-safe; accept ~680kB | |

**User's choice:** Inline-bundle with async wrappers → **D-08**
**Notes:** Store policy requires single main.js; ESM chunking is a ship-blocker. Clarified that lazy-load is a startup-latency deferral, not a size-reduction technique.

---

### Q9 — Bundle-size cap?

| Option | Description | Selected |
|--------|-------------|----------|
| Accept up to ~700kB | Realistic for 6 packs + inline bundling; enforced by scripts/check-bundle-size.sh | ✓ |
| Tight cap ~500kB | Forces dropping 1–2 larger packs; risks watering down D-01 | |
| No hard cap, measure and document | Risk of accidentally shipping 1.5MB | |

**User's choice:** 700kB cap → **D-09**
**Notes:** Enforced by new build gate; escalation path if tripped locked in D-09 body.

## Scoping

### Q10 — CM6 scoping pattern?

| Option | Description | Selected |
|--------|-------------|----------|
| Augment Obsidian's markdown codeLanguages map | Cleanest CM6-idiomatic path; Obsidian's parser routes fenced content to nested language; indent+brackets+highlight apply inside fence by construction | ✓ |
| Whole-editor language swap when caret in Code fence | Compartment swap; caret tracking adds complexity; rapid focus changes could flicker | |
| Build range-scoped indent + bracket extensions ourselves | No full LanguageSupport; contradicts D-01 | |
| Research decision — flag for planner | Accept all 3 plausible; let researcher prototype | |

**User's choice:** Augment markdown codeLanguages map → **D-10**
**Notes:** Preferred Plan A. Researcher verifies exact API access during research; fallback defined in next question.

## Claude's Discretion

- Registry module naming and file layout within `src/main/`.
- Sync vs async return contract of the per-language lazy-init helper.
- First-render behavior while pack is loading (plain markdown vs whitespace-copy).
- Pack-init caching strategy (per-language vs per-note).
- Exact `codeLanguages` augmentation mechanism (Facet vs Compartment vs override).
- Whether to preload user's `settings.defaultLanguage` pack at onload.
- Source Mode vs Live Preview parity if they diverge.

## Deferred Ideas

- C# support (no first-party pack).
- Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket (whitespace-copy fallback for v1).
- Prism ↔ Lezer theme parity (accept mismatch per D-02).
- Code-splitting / ESM chunking for true bundle reduction (store policy forbids).
- LSP / IntelliSense / linter / debugger (PROJECT.md hard exclusion).
- Snippets / code-complete / per-language indent-width overrides.
- Dynamic language-pack install at runtime (CF-05 forbids).
- Edit-mode status indicator (pack-loading / fallback-active).
