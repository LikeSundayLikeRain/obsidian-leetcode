# Phase 5.3 — UI Design Contract

> Visual and interaction contract for the Phase 5.3 language-aware editor. This is a **behavior-only UI-SPEC**: Phase 5.3 ships **zero new DOM, zero new CSS, zero new copy, zero new commands, zero new settings, zero new icons**.
>
> The entire user-visible surface is **keystroke behavior inside the existing `## Code` fence** on `lc-slug` notes in Live Preview + Source Mode — auto-indentation, bracket matching/auto-close, and Lezer-driven syntax highlighting provided verbatim by unmodified upstream `@codemirror/lang-*` packs. Every static visual dimension (color, typography, spacing, copywriting) is therefore **no change from Phase 5.1 / 5.2**; the interesting dimensions are interaction behavior per language, supported-vs-unsupported fallback states, and scoping parity.
>
> **Source of most decisions:** CONTEXT.md D-01..D-16 + RESEARCH.md (both pre-lock the feature depth, language set, bundle strategy, scoping mechanism, and test gates). No user questions were asked — upstream artifacts fully covered the design contract for a behavior-only phase.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Obsidian plugin — not a React/Next/Vite app; shadcn not applicable) |
| Preset | not applicable |
| Component library | CodeMirror 6 built-ins (`Compartment`, `StateField`, `ViewPlugin`, `LanguageSupport`, `indentService`) + Obsidian's `registerEditorExtension` host — no external UI component library |
| Icon library | none (no icons in this phase — behavior-only) |
| Font | inherited — Obsidian's user-theme monospace (`var(--font-monospace)`) inside the fence; Lezer grammar tokens colored via CM6's active `HighlightStyle` (sourced from Obsidian's theme) |
| Styling approach | **Zero new CSS rules.** Syntax colors come from Obsidian's built-in CM6 `HighlightStyle` applied to Lezer-produced token tags. `@codemirror/lang-*` packs ship grammar definitions only — no pack-specific stylesheets |
| Theme support | Automatic light/dark via Obsidian's bundled `HighlightStyle` — no theme-specific rules; same mechanism highlights every other code block in Obsidian |

**Rationale:** Phase 5.3 adds ZERO new CSS classes, ZERO new DOM elements, and ZERO new color rules. It registers one additional CM6 editor extension (`buildCodeFenceLanguageExtension`) alongside Phase 5.1's existing `buildCodeActionsEditorExtension` on every EditorView. Visual output inside the fence is entirely produced by upstream `@codemirror/lang-*` packs driving the `HighlightStyle` Obsidian already applies to all CM6 code blocks.

## Spacing Scale

Phase 5.3 introduces **no new spacing values**. The only line-geometry implication is that CM6's nested LanguageSupport does not change line heights or fence gutter margins — the existing `## Code` fence block-widget geometry (rendered by Obsidian's markdown editor) is preserved.

Declared spacing used by this phase:

| Token | Value | Usage in Phase 5.3 |
|-------|-------|---------------------|
| xs | 4px | not used — no new layout |
| sm | 8px | not used — no new layout |
| md | 16px | not used — no new layout |
| lg | 24px | not used — no new layout |
| xl | 32px | not used — no new layout |

Exceptions: **none.**

**Indent width (produced by the language packs at runtime):** Each `@codemirror/lang-*` pack provides its canonical indent unit via the grammar's `indentNodeProp`. Obsidian's per-vault `Editor → Tab indent` setting and the `indentUnit` facet determine the rendered tab/space substitution. Phase 5.3 does NOT override either — users who prefer 2-space Python or 4-space Go continue to get their Obsidian-configured indent width. The pack tells CM6 *how much* to indent (e.g., "one level after `:`"); Obsidian tells CM6 *how wide one level is*.

**Widget coexistence with Phase 5.1:** Phase 5.1's `CodeActionsWidget` is anchored at `state.doc.line(closerLine).to` with `side: 1` (inline, non-block). The Phase 5.3 language compartment is scoped to the *content* between opener and closer lines — it does not emit any decoration at the closer-line boundary. The two extensions do not fight for layout space (verified in RESEARCH Pitfall 12 + D-16 regression gate).

## Typography

Phase 5.3 does not declare new typography tokens. The fence content inherits:

| Role | Size | Weight | Line Height | Phase 5.3 Touchpoint |
|------|------|--------|-------------|----------------------|
| Fence monospace body | theme default via `var(--font-monospace)` (~14px in Obsidian default) | theme default | theme default | Obsidian renders; pack highlights tokens via `HighlightStyle` — no size/weight override |
| Lezer syntax tokens (keywords, strings, numbers, comments) | inherits fence monospace size | `HighlightStyle` may bump to bold for keywords (theme-dependent) | inherits | CM6 applies tags (`tags.keyword`, `tags.string`, `tags.comment`, etc.) via Obsidian's theme-provided `HighlightStyle` — identical to highlighting already active on every code block |

**Declared sizes: 0 new.** Every typographic dimension inside the fence comes from Obsidian's active theme via CM6's built-in `HighlightStyle` applied to Lezer tokens.

**Why no typography additions:** Phase 5.3's entire "output" is (a) caret position effects when the user presses Enter, (b) bracket-match decorations that pulse briefly (CM6 default), (c) token color changes driven by the pack's grammar. None of these introduce new text content, new labels, or new typographic roles.

**Font interaction with token mismatch (D-02):** Reading Mode tokenizes the same fence via Prism; Edit Mode (this phase) uses Lezer. Token boundaries may differ slightly (e.g., `async def` colored as two tokens in Prism, one in Lezer). This is ACCEPTED per D-02 — no theme parity work, no font adjustment. Both modes inherit the same `var(--font-monospace)` face.

## Color

Phase 5.3 declares **no new color rules**. All color comes from Obsidian's theme via two existing channels:

| Role | Value | Usage in Phase 5.3 |
|------|-------|---------------------|
| Dominant (60%) | `var(--background-primary)` (Obsidian theme var) | Editor background surrounding the fence — not painted by this phase |
| Secondary (30%) | `var(--background-secondary)` (Obsidian theme var) | Fence background (Obsidian renders) — not painted by this phase |
| Accent (10%) | Obsidian `HighlightStyle` token colors (theme-provided) | Syntax highlighting inside the fence: keywords, strings, numbers, comments, function names, types. Exact hue depends on user's active theme — not prescribed by this phase |
| Destructive | not used | No destructive action in this surface |
| Bracket match | Obsidian's default bracket-match decoration (theme-provided) | Ephemeral: brief visual emphasis when the caret is next to a `{`, `[`, `(` that has a matching partner |
| Autocomplete popover | N/A for Phase 5.3 | `@codemirror/autocomplete` is present in Obsidian's bundle (external), but Phase 5.3 does NOT enable the autocompletion extension — only `closeBrackets()` (sub-extension of LanguageSupport). Users see NO completion popover inside the fence |
| Whitespace-copy fallback | N/A — no color impact | Fallback only affects indent math on Enter; no tokens colored, no decorations emitted |

**Accent reserved for:**
- Obsidian's active `HighlightStyle` token colors (keywords, strings, numbers, comments, types, function names) rendered inside the `## Code` fence when a supported language pack is active.
- Obsidian's active bracket-match decoration color when caret is adjacent to a matched bracket inside a supported-language fence.
- **NOT** used for any plugin-authored element. Phase 5.3 emits zero custom-styled DOM.

**Theme verification method:**
- **Light theme:** Obsidian Settings → Appearance → Theme: "Default" light. Open an `lc-slug` Python note in Live Preview. Inside the `## Code` fence, verify keywords (`def`, `class`, `return`, `if`, `else`) render in the theme's keyword color, strings in the string color, comments (`# ...`) in the comment color. Matches the highlighting a user sees on any other Python fence in their vault outside an LC note.
- **Dark theme:** Settings → Appearance → Theme: "Default" dark. Same verification; colors shift to dark-mode palette automatically via Obsidian's `HighlightStyle` — no plugin intervention.
- **Sampling cadence:** Part of D-14 live-smoke gate after Wave 1 (primary implementation). Repeat per language: Python, Java, C++, JavaScript, TypeScript, Go, Rust. Spot-check C# / Ruby unsupported fallback: no syntax colors applied, fence content is plain monospace.

**Why no raw hex, no new CSS variables:** `@codemirror/lang-*` packs emit Lezer node tags (`tags.keyword`, `tags.string`, etc.) that CM6 maps to CSS classes via the active `HighlightStyle`. Obsidian supplies the `HighlightStyle` as part of its theme. The plugin never touches the color math. This is identical to how every other Obsidian plugin that uses CM6 language packs (Code-Styler, Lezer-tree-viewer) operates.

## Copywriting Contract

Phase 5.3 introduces **no user-facing copy**. Exhaustive list:

| Element | Copy |
|---------|------|
| Primary CTA | not applicable — no buttons, no commands, no menu items |
| Empty state heading | not applicable — no empty state (pack either renders or falls back; both states are invisible) |
| Empty state body | not applicable |
| Error state | **silent** — pack-load failures fall back to whitespace-copy indent without surfacing any Notice (CF-19 parity with 5.2's `python3Highlighter` silent Prism-load fallback) |
| Destructive confirmation | not applicable — no destructive actions |
| Loading / pending state | **no user-visible loading indicator** — first fence-enter per language pays ~50–150 ms pack-init latency; during that window the fence renders as plain markdown (Pitfall 4). Accepted per RESEARCH Open Question 1 |
| New commands | **zero** — no new command-palette entries |
| New Notices | **zero** — zero Notice copy added. Pack-load errors are swallowed silently |
| New settings | **zero** — no new setting rows, toggles, or text inputs |
| Tooltip / hover hint | **zero** — no hover copy on fence content |
| Screen reader ARIA | inherited from Obsidian's editor — no plugin-authored `aria-*` attributes |

### Error surfacing (by deliberate non-decision)

- **Pack-load failure (network-unrelated — disk read or module-resolution error):** Registry's `loadPack` rejects → `swapLanguage` catches → installs `whitespaceCopyIndentExtension` silently. No Notice. No console error (swallowed in the catch handler). Rationale: pack-load failures during `npm install` would block the build; at runtime they're effectively impossible (packs are bundled into `main.js` per D-08). If one does fire, the user experiences whitespace-copy indent — a strict subset of the intended behavior, not a crash.
- **Grammar parse failure on malformed code:** Lezer grammars are tolerant by design — malformed input produces `⚠` error nodes in the tree, but indent math and bracket matching continue to work against the last-valid partial tree. User sees no error; syntax highlighting may momentarily drop on the malformed region, which matches the user's mental model ("my code is broken; the editor is showing me it's broken by un-coloring it").
- **View-destroyed race (user closes note while pack loads):** Guarded by `view.destroyed` check in `swap()` before `view.dispatch()` (RESEARCH Pitfall 3). No Notice, no crash.

### Loading / pending state (deliberate non-decision, inherited from RESEARCH Pitfall 4)

- **No spinner.** **No disabled editor.** **No "Loading python grammar..." placeholder.** **No visual indicator of pack-load progress.**
- Rationale:
  1. Pack-init latency is ~50–150 ms per language on first use. A spinner shown for sub-200 ms would itself be a UX regression (spinner-flash).
  2. Default language can be warm-loaded at `onload()` via `warmDefaultPack` (Pattern 2 in RESEARCH) — the most common case pays zero runtime latency.
  3. During the load window, the user sees plain-markdown rendering of the fence content — readable, non-broken, just un-highlighted for ~100 ms.
- **Contract:** If a future phase wants visible load-progress feedback, it MUST add it in a way that does NOT trigger for sub-500ms loads (prevent spinner-flash). Phase 5.3 does NOT add any such surface.

### Carried-forward copy invariants

- **No `LeetCode:` command prefix** — no new commands in this phase, so N/A.
- **Notice copy locked (CF-19 from Phase 5)** — no new Notices in this phase.
- **Sentence case (`eslint-plugin-obsidianmd/ui/sentence-case`)** — no new UI strings, so N/A.

## Interaction Contract (the phase's primary surface)

### Mount Point

| Context | Mount Path | Source-of-Truth File |
|---------|-----------|---------------------|
| Edit Mode (Live Preview) | CM6 editor extension via `plugin.registerEditorExtension(buildCodeFenceLanguageExtension(plugin))` | `src/main/codeFenceLanguageExtension.ts` (NEW in Phase 5.3) |
| Edit Mode (Source Mode) | Same CM6 editor extension — no branching | Same as above |
| Reading Mode | **NOT TOUCHED by Phase 5.3.** Phase 5.2's `python3Highlighter.ts` (Prism alias) continues to own Reading-Mode highlighting | `src/main/python3Highlighter.ts` (Phase 5.2 — **UNTOUCHED**) |

The 5.3 extension and Phase 5.1's `codeActionsEditorExtension` coexist on every `EditorView`. They do not interact: 5.1 emits a widget decoration at the closer-line boundary; 5.3 reconfigures a `Compartment` holding nested-language state. No overlap, no ordering sensitivity.

### Gate: when language-aware behavior activates

Language-aware behavior (indent rules, bracket matching, auto-close, syntax highlighting) activates **if and only if** all of the following are true:

1. The active note has non-empty `lc-slug` frontmatter (verified via `state.field(editorInfoField)?.file` + `app.metadataCache.getFileCache(file)?.frontmatter['lc-slug']`) — parity with Phase 5.1's gate (CF-13, D-12).
2. The note body contains an `## Code` H2 heading followed (before the next H2) by a fenced code block with an opener and closer — parity with Phase 5.1's `findCodeFence` logic (D-13).
3. The caret position is strictly between the opener line and the closer line (`caretLine > fence.openerLine && caretLine < fence.closerLine`) — NOT on boundary lines. Boundary lines (opener, closer) revert to empty compartment (markdown behavior).
4. The fence's language tag (extracted via `/^\s*\`\`\`\s*(\S+)?/` on the opener line and normalized via `resolveLangSlug`) maps to a supported pack id or resolves to the fallback path. Empty/missing tag falls back to `settings.defaultLanguage` before mapping (parity with `starterCodeInjector` default-lang behavior).

All four conditions FALSE → the compartment stays empty → Obsidian's default markdown behavior applies → indistinguishable from pre-5.3 behavior.

**Scoping parity with Phase 5.1:** The same four-condition gate rejects:
- Non-LC notes (no `lc-slug`).
- LC notes with no `## Code` fence.
- LC notes with fences ONLY outside `## Code` (e.g., example fences under `## Problem`).
- Caret positions outside the fence (in prose, in another section, in a different fence).

**Known fallback (RESEARCH D-13 compromise):** If CM6 caret-line walking cannot determine the enclosing H2 section efficiently (edge cases: caret exactly on an H2 heading line, extremely large notes where section-walking becomes a perf concern), the phase accepts "any supported fence in an lc-slug note activates the pack" as the D-13 fallback. User-visible impact: example Python code in `## Problem` would also get auto-indent — minor UX oddity, not a blocker. See CONTEXT §D-13.

### Per-language behavior contract

The packs own the behavior; this table documents what the user experiences so downstream agents can verify against it during D-14 live smoke.

| Language (pack) | Enter-key indent | Dedent trigger | Bracket auto-close | Syntax highlighting |
|-----------------|------------------|----------------|--------------------|--------------------|
| Python (`python`, `python3` → `@codemirror/lang-python`) | After `:` at end of line → new line indents one level | `else:` / `elif` / `except:` / `finally:` / `return ` at start of line → dedent relative to matching block | `(` / `[` / `{` / `"` / `'` / `"""` (triple-quote) → matching close inserted | Keywords (`def`, `class`, `return`, `if`, `else`, `for`, `while`, `import`, `from`, `as`), strings (single/double/triple), numbers, comments (`# ...`), function & class names |
| Java (`java` → `@codemirror/lang-java`) | After `{` → indent one level | `}` at start of line → dedent | `(` / `[` / `{` / `"` → matching close | Keywords (`class`, `public`, `private`, `static`, `void`, `return`, `if`, `else`, `for`), types, strings, numbers, comments (`//`, `/* */`) |
| C / C++ (`c`, `cpp` → `@codemirror/lang-cpp`) | After `{` → indent one level | `}` at start of line → dedent | `(` / `[` / `{` / `"` / `'` → matching close | Keywords, types, strings, numbers, preprocessor directives (`#include`, `#define`), comments |
| JavaScript (`javascript` → `@codemirror/lang-javascript()`) | After `{` / `(` / `[` → indent one level | `}` / `)` / `]` at start of line → dedent | `(` / `[` / `{` / `"` / `'` / `` ` `` (template literal) → matching close | Keywords, strings, numbers, template literals, regex literals, comments, object property names |
| TypeScript (`typescript` → `@codemirror/lang-javascript({ typescript: true })`) | Same as JavaScript | Same as JavaScript | Same as JavaScript | Same as JavaScript + type annotations highlighted (interface, type, as, satisfies, generics) |
| Go (`golang` → `@codemirror/lang-go`) | After `{` → indent one level | `}` at start of line → dedent | `(` / `[` / `{` / `"` / `'` / `` ` `` (raw string) → matching close | Keywords (`func`, `package`, `import`, `return`, `if`, `else`, `for`, `var`, `type`, `struct`, `interface`), strings, numbers, comments |
| Rust (`rust` → `@codemirror/lang-rust`) | After `{` → indent one level | `}` at start of line → dedent | `(` / `[` / `{` / `"` / `'` / `r#"..."#` → matching close | Keywords (`fn`, `let`, `mut`, `pub`, `impl`, `trait`, `match`, `use`, `mod`, `struct`, `enum`), lifetimes (`'a`), strings, numbers, attributes, comments |
| **Unsupported** (C#, Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL, SQL, Elixir, Erlang, Racket, any other LC slug) | **New line inherits previous non-empty line's leading whitespace** (whitespace-copy only; D-06) | **None** (no grammar awareness) | **None** (Obsidian's markdown default applies — i.e., no `{}` auto-close inside the fence) | **None** (plain-monospace rendering — no token colors) |

**Unsupported-language fallback visible signs:**
- Fence content renders as plain monospace (no keyword colors, no string colors). To the user, this looks identical to how a `csharp` fence rendered in Obsidian BEFORE Phase 5.3 — i.e., no regression from prior state.
- Enter inherits previous line's leading spaces/tabs — a usability improvement over zero-indent default, but without the "smart" dedent behavior that supported languages offer.
- Typing `{` does NOT produce a matching `}`. This is deliberate (D-06) — the fallback is "whitespace-copy only, no bracket handling."
- **No crash.** **No disabled editor.** **No error Notice.** (Success Criterion 3 from ROADMAP §Phase 5.3.)

### Coexistence with Phase 5.1 Run/Submit buttons

Phase 5.1's `CodeActionsWidget` (button row anchored below the closer fence line) must remain visible, correctly positioned, and functional when the 5.3 compartment swaps in/out. Specifically:

- **No visual flicker of the button row** on compartment reconfigure (RESEARCH Pitfall 12).
- **No shift in button position** when language highlighting turns on or off.
- **Clicks on Run / Submit** continue to dispatch to `plugin.runFromActive()` / `plugin.submitFromActive()` — unaffected by the language compartment.
- **Focus / tab order** — 5.1's buttons remain reachable via Tab after the fence. Language compartment does not introduce new tab stops inside the fence (autocomplete popover not enabled).

**D-14 live-smoke verification required:** Open an lc-slug Python note in Live Preview → caret enters fence → python highlighting appears → verify Run/Submit buttons do not reposition. Caret exits fence → compartment clears → verify buttons stay put.

### Coexistence with Phase 5.2 Reading-Mode python3 alias

- **Zero interaction.** Phase 5.2's `python3Highlighter.ts` is a `MarkdownPostProcessor` that runs in Reading Mode only. Phase 5.3's `codeFenceLanguageExtension.ts` is a CM6 editor extension that runs in Live Preview + Source Mode only. The two surfaces are disjoint (RESEARCH Open Question 5).
- Reading Mode python3 fences continue to highlight via Prism (5.2).
- Edit Mode python3 fences highlight via Lezer (5.3).
- Token mismatch between the two modes is ACCEPTED (D-02). No user action or plugin code bridges the two.

### Idempotency

- **One compartment per editor view.** `buildCodeFenceLanguageExtension(plugin)` is called by CM6 once per view; the compartment is fresh per view. Multiple views of the same file have independent compartments (CM6 expected behavior).
- **One pack instance per language across all views.** `languagePackRegistry.ts` module-level Map<PackId, Promise<LanguageSupport>> dedupes: first call triggers `import()`, subsequent calls reuse the cached promise.
- **Transaction guard in ViewPlugin.update:** `if (prev.desiredPack !== next.desiredPack)` prevents redundant compartment swaps on scroll / focus-change transactions that carry selectionSet but no actual caret-position-meaningful change (RESEARCH Pitfall 5). Load-bearing guard — without it, every keystroke would re-dispatch a compartment reconfigure.

### First-render behavior (mounted view, caret already in fence)

When an already-open lc-slug note is mounted in a fresh EditorView with the caret already inside the `## Code` fence (e.g., user restart, workspace restore), `StateField.create` computes the initial `FenceState` and `ViewPlugin` constructor detects `desiredPack !== 'none'` and triggers the async swap. Pack loads during the first ~50–150 ms of view mount; fence content renders as plain markdown for that window, then swaps to language-aware rendering. **Accepted per RESEARCH Open Question 1 / Pitfall 4.** `warmDefaultPack(settings.defaultLanguage)` called at `plugin.onload()` eliminates this window for the user's default language.

## Accessibility

| Dimension | Treatment |
|-----------|-----------|
| Keyboard behavior | Obsidian's native editor keybindings (Enter, Tab, arrows, Home/End) continue to work. Language packs may register additional indent-related keybindings via their default keymap — all are keyboard-only and non-exclusive with Obsidian's defaults |
| Focus indicator | Inherited from Obsidian — no plugin-authored focus styling. The fence itself does not gain new focus stops; caret moves inside the fence via normal arrow-key navigation |
| Tab key inside the fence | Obsidian's editor Tab-indent behavior continues to apply; language pack's `indentOnInput` may augment dedent behavior on specific tokens. No conflict |
| Screen reader | No `aria-*` changes. Fence content announced as-is by the screen reader via the editor's native accessibility tree |
| Color contrast | Token colors are inherited from Obsidian's active theme `HighlightStyle`. Community themes are responsible for theme contrast; Phase 5.3 does not override or modify token colors |
| Motion | `closeBrackets()` auto-insert is instantaneous (no animation). Bracket-match decoration is a momentary color emphasis on the matching character (inherited from CM6 default; no animation). Compartment reconfigure is an instantaneous state swap — no transition, no fade. No `prefers-reduced-motion` concern |
| Touch target | Not applicable (`isDesktopOnly: true`; pointer-primary input) |
| Keyboard shortcut collisions | No new hotkeys registered by this phase. Language packs' default keymaps are scoped to `Prec.default` — do not override Obsidian's higher-precedence bindings (CM6 precedence system) |

## Out of Scope (documented to prevent drift)

The following are **explicitly NOT** part of Phase 5.3's design contract, even though downstream agents may be tempted to add them:

- **New commands, new Notices, new settings, new DOM, new CSS.** Zero additions across all of these (CONTEXT §Claude's Discretion confirms).
- **C# language support.** Rejected per D-04 — no first-party pack, and `@replit/codemirror-lang-csharp` fails the third-party registry vetting bar. Users selecting C# get whitespace-copy fallback (D-06).
- **Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket, any other LC language outside the pragmatic 8.** All fall back to whitespace-copy (D-06).
- **Edit-Mode ↔ Reading-Mode highlighting theme parity.** Token mismatch between Prism (5.2 Reading) and Lezer (5.3 Edit) is ACCEPTED per D-02. No cross-mode color reconciliation.
- **Autocompletion popover inside the fence.** `@codemirror/autocomplete`'s `autocompletion()` extension is NOT enabled in this phase — only the `closeBrackets()` sub-extension (bundled inside `LanguageSupport`). Users see no completion UI.
- **Linting / LSP / debugger inside the fence.** Hard out per PROJECT.md "IDE-style features" exclusion.
- **Language-specific snippets.** Out of 5.3 scope; candidate for a future "snippet library" phase not on the v1 roadmap.
- **Dynamic language-pack install at runtime** (download unknown language from a registry on demand). Forbidden by CF-05 (no remote code eval). All 6 packs bundled into `main.js`.
- **Configurable indent width / tab-vs-space per language inside the plugin.** Obsidian's per-vault Editor settings already cover this.
- **Applying LanguageSupport globally to every code fence in every note.** D-12 gates on `lc-slug` frontmatter; non-LC notes receive zero plugin-side modification. Over-reach trigger for plugin reviewers.
- **Loading indicator / spinner / "compiling grammar" placeholder.** By deliberate non-decision (see "Loading / pending state" above). First-use ~100 ms latency is accepted.
- **Status indicator in the UI showing which pack is active** (debug affordance). Not required for v1 — users can tell by whether their code is highlighted.
- **Per-note default language override setting.** Use `settings.defaultLanguage` + fence tag. No new UI control.
- **Any plugin-authored decorations inside the fence** (custom highlight, custom widget, custom line wrap). Phase 5.3 only supplies the `LanguageSupport` to CM6's compartment — all decorations come from the pack + Obsidian's theme `HighlightStyle`.
- **Modifying Phase 5.1's `codeActionsEditorExtension.ts` or Phase 5.2's `python3Highlighter.ts`.** Zero changes to either. Phase 5.3 is purely additive (CONTEXT §"What to avoid").
- **Bumping `manifest.json` `minAppVersion`.** Stays at `1.10.0` (locked in Phase 2 Plan 01); Phase 5.3 does not move it.
