<objective>
Close the MEDIUM-severity gap **G-COPY-TO-CODE-LANG-DRIFT** surfaced during the
Phase 5.3 live-smoke UAT: when the user opens a previous submission via the
SubmissionDetailModal and clicks "Copy to ## Code", the fence body and fence
opener tag are rewritten correctly (`copyToCode` already routes through
`forceInjectCodeSection` → `codeBlockFor` → `lcSlugToFenceTag`), but the note's
`lc-language` frontmatter and the chevron label are NOT updated. Result: the
note's stored language and the actual code in the fence drift apart, and a
subsequent Run/Submit dispatches with the wrong slug to the LC API.

Fix: extend `copyToCode` to also write `lc-language` via
`app.fileManager.processFrontMatter` immediately after the `vault.process`
fence rewrite resolves. This mirrors Step C of `switchFenceLanguage`
(`src/main.ts:801-803`) — the same atomic-shape pattern the chevron uses,
but invoked from the copy-to-code path.

Purpose: keep `lc-language` as the single source of truth for LC API dispatch
(per CONTEXT D-03 and D-12). The chevron label refresh comes for free —
Plan 05's `languageRefreshEffect` wiring observes `metadataCache.on('changed')`
and rebuilds the StateField when the frontmatter write lands, so no widget
code change is needed here.

Output: Updated `copyToCode` signature internals, unchanged caller surface
(SubmissionDetailModal still passes `(app, file, code, langSlug)`), new unit
test file `tests/graph/copyToCode.langSync.test.ts`. Existing copyToCode
tests (overwrite assertion, silent-copy assertion) continue to pass unchanged.

The companion live-smoke verification of this fix end-to-end happens in
Plan 05.3-07 (the chevron-affected re-test checkpoint).
</objective>

**Modify `src/graph/copyToCode.ts`.**

Update the `copyToCode` function to write `lc-language` to frontmatter
immediately AFTER `app.vault.process` resolves. The change happens entirely
inside this function — the public signature stays
`copyToCode(app: App, file: TFile, code: string, langSlug: string): Promise<void>`

Implementation contract (planner-locked — implementer follows it verbatim):

1. After `await app.vault.process(file, ...)` resolves, read the current
   `lc-language` from `app.metadataCache.getFileCache(file)?.frontmatter`.
   Cast through `Record<string, unknown>` exactly as `switchFenceLanguage`
   does (`src/main.ts:746-748`).

2. Same-slug short-circuit: if `currentLangSlug === langSlug`, return without
   calling `processFrontMatter`. Avoids touching the vault when the note is
   already at the submission's language (Test 2 path).

3. Unknown-slug short-circuit: if `!LC_LANG_SLUGS.has(langSlug)`, return
   without calling `processFrontMatter`. The fence body has already been
   rewritten by `forceInjectCodeSection` (existing contract), but we refuse
   to write a non-canonical LC slug into `lc-language` because that field
   is consumed by `runFromActive` / `submitFromActive` for LC API dispatch.
   Writing `cobol` would corrupt that contract. (Test 3 path.)

4. Otherwise: call
   `await app.fileManager.processFrontMatter(file, fm => { fm['lc-language'] = langSlug; })`.
   Mirror the exact shape from `switchFenceLanguage` Step C
   (`src/main.ts:801-803`). Errors propagate naturally — do NOT try/catch.

5. Add a doc-block comment that traces the fix to G-COPY-TO-CODE-LANG-DRIFT
   and references `switchFenceLanguage` Step C as the pattern source. Begin
   the comment with `// G-COPY-TO-CODE-LANG-DRIFT (gap-closure 05.3-06):`.

**Why we do NOT extract a shared `atomicallyUpdateFenceLanguage` helper:**

The init prompt suggested either reusing `switchFenceLanguage` or extracting
a shared helper. After tracing the call shapes:

- `switchFenceLanguage` (main.ts:738) does THREE things — (A) fetch starter
  code via `client.getProblemDetail`, (B) atomic CM6 dispatch rewriting
  opener tag + fence body, (C) write `lc-language`. It depends on
  `LeetCodeClient`, an active `MarkdownView`, and the CM6 EditorView.
- `copyToCode` does ZERO of (A) and (B) the same way — its body is supplied
  by the caller, and the rewrite goes through `vault.process` not a CM6
  dispatch (different architectural tier — vault file mutation vs. live
  editor dispatch).
- The ONLY shared step is (C) — three lines of `processFrontMatter`. A
  shared helper for three lines adds an indirection without saving real
  code, and would force `copyToCode` to depend on a helper file that
  conceptually belongs to the chevron-switch path.

Decision: inline the three lines into `copyToCode` with a doc comment that
references `switchFenceLanguage` Step C for the lineage. If a third call
site for the frontmatter-write step appears in a future phase, extract then.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.3.06-01 | Tampering | langSlug parameter | mitigate | Unknown-slug guard via `LC_LANG_SLUGS.has(langSlug)` prevents writing non-canonical values into `lc-language`; downstream Run/Submit dispatch sees only the validated set documented in src/solve/languages.ts |
| T-05.3.06-02 | Tampering | Frontmatter race | accept | Vault.process and processFrontMatter run sequentially; if a foreign edit lands between them the frontmatter write reflects the latest user-visible state. Same race surface as switchFenceLanguage Step C — accepted there, accepted here |
| T-05.3.06-03 | Information disclosure | metadataCache read | accept | Reads only the same `lc-language` value already displayed elsewhere in the UI; no PII, no external transmission |
| T-05.3.06-04 | Denial of service | processFrontMatter rejection | accept | Propagates to caller; SubmissionDetailModal.performCopy currently does not catch (it `await`s and then closes); failure surfaces as an unhandled rejection only if the modal is mid-close. Existing behavior of `copyToCode` already had no rejection handler — this plan does not regress |
