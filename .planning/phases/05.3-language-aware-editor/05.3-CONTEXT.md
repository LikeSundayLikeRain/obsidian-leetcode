# Phase 5.3: Language-Aware Editor - Context

**Gathered:** 2026-05-11
**Status:** Ready for research + planning

<domain>
## Phase Boundary

Phase 5.3 turns the `## Code` fenced block in Edit Mode (Live Preview) on `lc-slug` notes into a language-aware editing surface — IDE-grade auto-indentation, dedent rules, bracket matching, auto-close brackets, and CM6-native syntax highlighting — for a pragmatic set of 8 first-party-supported LeetCode languages. Unsupported languages fall back cleanly to whitespace-copy indentation with no crash.

Phase 5.2 D-13 shipped `python3 → python` grammar aliasing in **Reading Mode** via Prism. Phase 5.3 extends the language awareness to **Edit Mode** via CM6's `@codemirror/lang-*` packs. Phase 5.1 established the CM6 extension infrastructure (`codeActionsEditorExtension.ts`, the `lc-slug` gate + `findCodeFence` helper, the `registerEditorExtension` call); Phase 5.3 adds a sibling extension that registers additional languages into Obsidian's markdown nested-parser for the `## Code` fence.

**In scope (locked by ROADMAP.md §Phase 5.3 success criteria + discussion D-01..D-13):**

1. Pressing Enter inside the `## Code` fence on an `lc-slug` note triggers language-aware indentation via the language pack's indent rules (Python `:` → indent; `else`/`elif`/`except`/`finally` → dedent; C-family `{` → indent, `}` → dedent).
2. Bracket matching, auto-close brackets, and `@codemirror/lang-*` syntax highlighting are ACTIVE inside the `## Code` fence — full `LanguageSupport` per language.
3. Supported languages: **Python (and python3), Java, C++, C, JavaScript, TypeScript, Go, Rust** — the pragmatic 8 covered by FIRST-PARTY `@codemirror/lang-*` packs. **C#** is EXCLUDED (no official pack; third-party maintenance is weak).
4. Unsupported languages (C#, Ruby, Swift, Kotlin, Scala, PHP, MySQL/SQL, Bash, Dart, etc.): fall back to whitespace-copy indent (new line inherits previous line's leading whitespace, no grammar awareness, no bracket auto-close, no crash).
5. Scoped to the `## Code` fence only (D-13); example fences in `## Problem` stay on Obsidian's default markdown behavior.
6. Gated on `lc-slug` frontmatter (D-12); non-LC notes are untouched.
7. Live Preview + Source Mode parity (same CM6 editor infrastructure as Phase 5.1). Reading Mode is untouched (Phase 5.2's `python3Highlighter.ts` Prism alias continues to own Reading Mode).
8. Prism (Reading) ↔ Lezer (Edit) highlighting token mismatch is ACCEPTED (D-02) — no theme parity attempt in this phase.
9. Language packs are LAZY-INITIALIZED on first supported fence encounter in an `lc-slug` note (D-07, D-08) — startup latency deferral, not bundle-size reduction.
10. Bundle cap: `main.js` ≤ 700kB post-build. Enforced by a new build gate script (D-09).
11. No regression in Phase 5.1 edit-mode Run/Submit buttons or Phase 5.2's Reading-Mode python3 Prism alias. The 5.1 button-widget Decoration (`CodeActionsWidget`, `side: 1`/`side: -1` inline anchoring) and the 5.3 language extension are independent extensions on the same CM6 EditorView.

**Explicitly out of scope for Phase 5.3:**

- **C# support** — skipped per D-04; planner does NOT pull in `@replit/codemirror-lang-csharp` or any third-party pack. C# users get whitespace-copy fallback.
- **Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket** — all LC languages outside the pragmatic 8 get the fallback path; no per-language coverage work.
- **True code-splitting / ESM chunks** — Obsidian's single-CJS-`main.js` model and store review policy rule this out. D-08 mandates inline-bundling via async wrappers.
- **Edit-Mode / Reading-Mode theme parity** — D-02 accepts the token mismatch. If a user notices `def foo():` highlights differently in each mode, that is deliberate for v1.
- **Buttons under non-`## Code` fences** — parity with 5.1 D-02. Example fences in `## Problem` stay on Obsidian's default markdown rendering.
- **Apply LanguageSupport globally to every code fence in every note** — D-12 gates on `lc-slug`; plugin does not modify non-LC notes (over-reach plugin reviewers flag).
- **Language-specific keyboard shortcuts, snippets, linter integration, LSP** — hard out per PROJECT.md `IDE-style features` exclusion.
- **Dynamic language-pack install / download at runtime** — store policy forbids remote code evaluation (CF-05). All 6 packs ship inside `main.js`.
- **Whitespace-copy + default bracket auto-close for unsupported fences** — D-06 locks unsupported to whitespace-copy ONLY. No bracket closing for C#, Ruby, etc.

<decisions>
## Implementation Decisions

### Feature Depth

- **D-01:** **Full `@codemirror/lang-*` `LanguageSupport` per language.** Ship indent rules + bracket matching + auto-close brackets + syntax highlighting — the standard CM6 LanguageSupport bundle. Closest to the "IDE-grade" ROADMAP promise. Indent-only or indent+brackets alternatives rejected as "half-baked" relative to the phase goal.

- **D-02:** **Accept Prism (Reading Mode) ↔ Lezer (Edit Mode) highlighting token mismatch.** Different grammars tokenize slightly differently — e.g., Prism might color `async def` tokens differently from Lezer. Users rarely side-by-side compare; both modes look "highlighted" at a glance. No theme parity work in Phase 5.3. Flag as a potential v1.1 polish item if users complain.

- **D-03:** **Bracket auto-close stays SCOPED to the supported-language fence, not global.** `closeBrackets()` is a sub-extension of the LanguageSupport, not a top-level editor extension — it fires only when the caret is inside the nested language region. Outside the fence (prose), Obsidian's default markdown bracket behavior wins. Parity with 5.1 scoping discipline: plugin does not change prose-typing behavior.

### Supported Languages & Mapping

- **D-04:** **Pragmatic 8 via FIRST-PARTY packs.** Languages covered: Python, Java, C++, C, JavaScript, TypeScript, Go, Rust. All from `@codemirror/lang-python`, `-java`, `-cpp`, `-javascript`, `-go`, `-rust`. C# is EXCLUDED — no official pack; `@replit/codemirror-lang-csharp` is a less-maintained third-party option rejected to avoid unvetted dependency risk in a plugin-store-ready ship.

- **D-05:** **Standard `langSlug → pack` map:**
  ```
  python       → @codemirror/lang-python
  python3      → @codemirror/lang-python      (same grammar, matches 5.2 D-13 Prism alias)
  java         → @codemirror/lang-java
  cpp          → @codemirror/lang-cpp
  c            → @codemirror/lang-cpp         (shared C/C++ parser)
  javascript   → @codemirror/lang-javascript()
  typescript   → @codemirror/lang-javascript({ typescript: true })
  golang       → @codemirror/lang-go
  rust         → @codemirror/lang-rust
  ```
  Researcher verifies LC's actual `langSlug` strings (cross-reference Phase 5.2 D-12's pinned language list and `starterCodeInjector` / `languages.ts` canonical mapping) before planning; if a slug differs from the proposed map, researcher updates the mapping with rationale.

- **D-06:** **Unsupported language fallback: whitespace-copy indent only.** On Enter, new line inherits previous line's leading whitespace — no grammar awareness, no bracket auto-close, no crash. Matches ROADMAP §Phase 5.3 success criterion 3. A single shared `whitespaceCopyIndentExtension` covers C#, Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket, and anything else LC may add in the future. No per-language work for these.

### Bundle Strategy

- **D-07:** **Lazy-load language packs via dynamic `import()` on first supported fence encounter in an `lc-slug` note.** First time the editor enters a Python fence, the python pack initializes; subsequent Python fences reuse the cached module. Startup cost: ZERO. Runtime cost: ~80–150ms pack init per language on first use. Pack-init lifecycle managed by a registry module (planner names; `src/main/languagePackRegistry.ts` or similar).

- **D-08:** **esbuild inline-bundles dynamic imports via `splitting: false` (retain current CJS single-`main.js`).** Obsidian plugin-store policy requires a single `main.js`; ESM chunk-splitting is REJECTED as a ship blocker (would break store review). The async-wrapper around the `import(...)` defers pack INITIALIZATION but does NOT reduce bundle size. Net effect of D-07+D-08: ~380kB of LanguageSupport code stays in `main.js`, but is only parsed/executed when an LC note with a supported fence renders — Obsidian startup stays fast.

- **D-09:** **Bundle cap: `main.js` ≤ 700kB post-build.** Enforced by a new `scripts/check-bundle-size.sh` gate (extends the Phase 5 Plan 06 `prerelease-check.sh` pattern). The script: `stat` the built `main.js`, fail if > 700_000 bytes. Added to the `npm run build` tail OR to `prerelease-check.sh` (planner picks). If the cap trips during implementation, the escalation path is: (a) drop the largest unused pack (rust is likely lowest-value), (b) re-discuss D-04 to shed a language, (c) spike a code-splitting workaround — in that order.

### Scoping (CM6)

- **D-10:** **Augment Obsidian's markdown `codeLanguages` nested-parser map.** Obsidian's markdown editor already uses `@codemirror/lang-markdown` with a `codeLanguages` callback that routes fenced content to nested languages (same mechanism markdown-it uses for nested code blocks). Phase 5.3 registers additional language packs into this map — CM6 automatically applies LanguageSupport INSIDE the fence only. This is the idiomatic CM6 pattern and is why D-03's bracket-scoping "just works" — nested language is caret-range-scoped by construction.
  - Access path: researcher verifies during research phase. Options: (a) Obsidian's `app.workspace.getActiveViewOfType(MarkdownView)?.editor.cm as EditorView` → find the markdown language `Compartment` on the existing extension set; (b) register a CM6 `languageDataProp` override that adds additional `codeLanguages` entries; (c) a `Facet` extension that augments the existing nested-parser registry.
  - CLAUDE.md §7 confirms `view.editor.cm as EditorView` is internal but sanctioned for CM6 editor access.

- **D-11:** **Plan B if D-10's markdown-codeLanguages augmentation is inaccessible: whole-editor `Compartment` swap on fence-focus.** When caret enters the `## Code` fence, swap the editor's language via `Compartment.reconfigure()`; swap back to markdown on exit. Caret-position tracking via CM6 `Transaction.filter` + `editorInfoField` caret query. Researcher documents the pivot in RESEARCH.md with explicit rationale if Plan A is blocked. Plan C (custom range-scoped extension) is NOT approved — would drop D-01's "full LanguageSupport" intent; re-enter discuss-phase if both A and B fail.

- **D-12:** **`lc-slug` frontmatter gate — parity with 5.1.** LanguageSupport registration / activation only fires for editors whose file has `lc-slug` set. Non-LC notes receive zero plugin-side behavior modification. Matches 5.1 D-06 + CF-13. Implementation hook: same `editorInfoField` + `metadataCache.getFileCache({ path })` pattern as `codeActionsEditorExtension.ts::buildDecorations`.

- **D-13:** **Target ONLY the `## Code` fence, not every supported fence in the lc-slug note.** Parity with 5.1 D-02 `findCodeFence` logic — first fence under `## Code` H2 wins; example fences in `## Problem` stay plain. The nested-parser augmentation approach (D-10) needs a way to enforce this section boundary; researcher picks mechanism:
  - **Preferred:** `codeLanguages` callback inspects the fence's enclosing section (via Lezer tree ancestry or caret-position heuristic); returns LanguageSupport for `## Code` fences, `undefined` (→ plain markdown) for other fences.
  - **Acceptable compromise:** if the nested-parser can't do section-aware routing, researcher may accept "any supported fence in lc-slug note" as D-13 fallback, documenting the widening in CONTEXT supplement. User-facing impact: example Python code in `## Problem` would also get auto-indent — minor UX oddity, not a blocker.

### Testing & Verification

- **D-14:** **Live-verify gate between waves — LOCKED lesson inherited from 5-UAT G1 and 5.1 D-10.** Unit tests on the language pack registry, fallback logic, and `lc-slug` gate are required (logic confidence), but they are NOT sufficient to verify "no layout corruption" or "indent behavior feels right." Planner MUST include a human-verified live-smoke checkpoint after each wave that touches CM6 layout or typing. The smoke surface: dev vault open → Cmd-R reload → open 2–3 `lc-slug` notes in different languages (Python, Java, C++) → type multi-line code → verify indent, dedent, bracket matching, bracket auto-close, syntax color → switch a note to an unsupported language (e.g., Ruby) → verify whitespace-copy fallback and no crash → toggle to Reading Mode → verify python3 highlighter (5.2) still works → scroll through, verify 5.1's Run/Submit buttons still appear below `## Code`.

- **D-15:** **Bundle-size gate (`scripts/check-bundle-size.sh`) is a hard CI-style gate, run on every `npm run build` AND on every live-smoke.** If the gate trips, the plan it tripped on is BLOCKED pending D-09 escalation. The gate is not just a prerelease check — it's a per-plan check so bundle regression is caught at the wave boundary, not at ship time.

- **D-16:** **Regression tests for Phase 5.1 + Phase 5.2:**
  - `tests/main/codeActionsEditorExtension.test.ts` — 5.1 edit-mode button extension tests MUST stay green.
  - `tests/main/python3Highlighter.test.ts` (if it exists; 5.2 Plan 05 tests) — Reading-Mode python3 Prism alias tests MUST stay green.
  - A new integration-style test verifies: in an `lc-slug` note with the 5.3 language extension registered, the 5.1 `CodeActionsWidget` still paints below the `## Code` fence. Planner decides whether this is a unit test (stubbing CM6) or a live-smoke-only check; 5.1 precedent says live-smoke is the acceptance bar for widget placement (D-14).

### Claude's Discretion

User delegated all low-level CM6 implementation choices within the D-01..D-13 guardrails. Specifically open:

- Registry module naming (`src/main/languagePackRegistry.ts` vs `src/main/languageSupport.ts` vs other).
- Whether the per-language lazy-init returns a `Promise<LanguageSupport>` or a synchronous "pending" placeholder that swaps in on resolve.
- First-render behavior while the pack is still loading: (a) plain markdown rendering (indent-as-markdown) for 50–200ms until pack init resolves, then swap to language-aware rendering, OR (b) whitespace-copy fallback until pack resolves. Researcher picks; (a) is likely fine given the small pack-init cost.
- Pack-init caching across notes: does the registry cache per-language across all notes, or per-note? Per-language cache is obvious; planner confirms.
- Exact `codeLanguages` augmentation mechanism for D-10 (see D-10 bullet list of options).
- Source Mode vs Live Preview parity: if one mode rejects the nested-parser augmentation, researcher's call whether to split code paths or ship one mode first.
- Python-specific behaviors: Python's LanguageSupport handles `:` indent and `else`/`elif`/`except`/`finally` dedent natively. No custom keymap needed — researcher confirms.
- Whether to preload the user's `settings.defaultLanguage` pack at plugin `onload()` (one-shot warm) or wait for fence encounter. D-07 says "on first fence encounter"; researcher may upgrade this to "onload warm for default language + fence for others" if UX benefits materialize.

### Carried Forward (not re-asked)

- **CF-05:** Zero Required `eslint-plugin-obsidianmd` violations. No `innerHTML`, no `fetch`, no default hotkeys, no remote code evaluation. Dynamic `import()` of bundled modules is NOT remote eval (module is in `main.js`).
- **CF-06:** `vault.process()` + `processFrontMatter()` only for note writes. Phase 5.3 adds ZERO vault writes — pure editor-extension surface (reads the doc, renders decorations, no mutations).
- **CF-07:** `createEl()` / `createDiv()` discipline for all plugin-authored DOM. Language packs are third-party; their internal DOM output is accepted as plugin-store-compliant (standard `@codemirror/*` packs; audited by Obsidian plugin reviewers for similar plugins).

Phase 5.3 extends `src/main/` (peer of `codeActionsEditorExtension.ts`, `codeActionsPostProcessor.ts`, `codeBlockButtonRow.ts`, `python3Highlighter.ts`, `fileOpenHook.ts`). Planner may name modules `src/main/codeFenceLanguageExtension.ts` + `src/main/languagePackRegistry.ts` or equivalent; no new top-level `src/` folder.
- **CF-13:** `lc-slug` frontmatter drives LC-note detection. Same key used by 5.1 `codeActionsEditorExtension`, 5.2 `fileOpenHook`. Phase 5.3 reuses.
- **CF-19:** Notice copy locked. Phase 5.3 surfaces NO new Notices. Pack-init failures are silent (same pattern as 5.2's `python3Highlighter`'s silent Prism fallback).
