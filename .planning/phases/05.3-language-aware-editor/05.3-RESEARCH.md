**Researched:** 2026-05-11 (replan after D-14 live-smoke failure)
**Domain:** Write-time fence-tag remap + chevron-driven LC language switching for `## Code` fence on `lc-slug` notes (Edit-Mode only); revert of failed Compartment-swap approach

## Summary

Phase 5.3 (replanned) ships markdown-native syntax highlighting for `## Code` fences via two cooperating mechanisms: (1) **write-time fence-tag remap** so LC's slugs become tags `lang-markdown` already recognizes (`python3 → python`, `golang → go`, `c → cpp`); (2) a **language chevron dropdown** inline with Phase 5.1's Run/Submit row that destructively switches the LC language and writes a single atomic CM6 transaction — fence opener line + fence body — paired with `app.fileManager.processFrontMatter` for `lc-language`. The 6 `@codemirror/lang-*` packs and three implementation files (`codeFenceLanguageExtension`, `languagePackRegistry`, `whitespaceCopyIndent`) are deleted; bundle returns from 520 KB → ~155 KB.

The chevron is **Edit-Mode only** (D-09) — Phase 5.1's Reading-Mode `codeActionsPostProcessor.ts` is unchanged. The chevron sits LEFT of Run/Submit; the existing `.leetcode-code-actions` flex container changes from `justify-content: flex-end` to `justify-content: space-between`.

**Atomic dispatch shape** is the load-bearing pattern: `processFrontMatter` is async and writes through `app.fileManager` (NOT the editor); `view.dispatch({ changes: [...] })` is synchronous and writes through CM6. Cmd-Z merges them only if both happen inside one user-event tick **and** the doc's frontmatter was already cached by Obsidian (so `processFrontMatter`'s vault.modify doesn't cycle the editor). The recommended order is **CM6 dispatch FIRST → `processFrontMatter` second** (Pattern 1 in Architecture), with `view.dispatch` carrying both `changes` (fence rewrite) inside ONE transaction so they undo together; the frontmatter write becomes a separate undo step that lands on Obsidian's vault-undo stack rather than the editor's. **Single-Cmd-Z atomic-revert across editor+frontmatter is NOT achievable via public Obsidian APIs** — but the user-perceived experience is acceptable because frontmatter is invisible in Live Preview by default; Cmd-Z reverts the visible fence (the thing the user sees), and the now-stale `lc-language` value is corrected on the next switch.

**Primary recommendation:** Reuse the existing `.leetcode-code-actions` row (modify `buildCodeBlockButtonRow` to accept an optional pre-Run prefix element; only Edit-Mode passes the chevron). Build the chevron as a custom DOM dropdown (not native `<select>`, not `Modal`) styled with Obsidian CSS variables for visual parity. Add `LC_LANG_DISPLAY_LABELS` and `LC_LANG_FENCE_TAG` maps to `src/solve/languages.ts` (single-source-of-truth principle the file already establishes). Refetch starter code from `LeetCodeClient.getProblemDetail(slug).codeSnippets[]` (already cached in `data.json` per `SettingsStore`); fall back to network via the existing client when cache misses or stale. Revert via **forward-deletion** (rm files + uninstall packages + edit `main.ts` Step 6i back out) — cleaner than `git revert` because the failed implementation has 3 commits across 5 files plus one fix commit (`de9aeeb`).

## Historical: Why the original approach failed (D-14 live-smoke)

> Preserved verbatim from the obsolete RESEARCH.md so future readers understand the pivot. The technical recommendations from that original research are NOT carried forward; only this single failure-mode finding is.

**Original Pitfall 1:** "D-10 Plan A (augment Obsidian's markdown codeLanguages) is blocked" — `obsidian.d.ts` exposes only `registerEditorExtension(extension: Extension)`; no `codeLanguages`, `addLanguage`, `markdownLanguageField`, `LanguageDescription`, or `syntaxTree` helper is publicly exposed. `[CITED: grep of node_modules/obsidian/obsidian.d.ts 2026-05-11 — no matches for codeLanguages, LanguageSupport, syntaxTree, addLanguage, languageMap]`

The original plan therefore went with "Plan B" — per-view `Compartment.reconfigure([languageSupport])` driven by caret-in-fence detection. Wave 0 + Wave 1 implemented it; Wave 1 unit tests (53 of them) all passed; build emitted single `main.js` at 520 KB; `setTimeout(0)` was added to defer the dispatch (commit `de9aeeb`) past CM6's "update in progress" guard. **At D-14 live smoke**, console verified the dispatch fires successfully (`swap() loaded pack=python` → `swap() dispatching reconfigure` → `swap() dispatch OK`), but **the language pack does NOT take effect inside the markdown fence**. No syntax highlighting; no language-aware indent.

**Empirical root cause:** Obsidian's `lang-markdown` (CodeMirror's nested-parser markdown grammar) owns the syntax tree for the entire markdown document. Inside a fenced code block, `lang-markdown` consults its **own** `codeLanguages` callback (configured at construction time, not exposed publicly) to decide which sub-language tokenizer to invoke for the fence content. CM6 routes language data **per syntax-tree node**, not **per outer Compartment**. A second LanguageSupport installed via Compartment at the editor root either: (a) tokenizes the whole document as Python (wrong — destroys markdown rendering everywhere outside the fence), or (b) silently does nothing inside the fence because `lang-markdown` already claimed that sub-tree.

This is a fundamental architectural mismatch between "swap a top-level language Compartment" (the canonical CM6 pattern for non-markdown-host editors) and "edit fence content inside a markdown-host editor whose nested parser owns fence sub-trees." Two paths exist to solve it properly — both rejected as too expensive for this milestone:

- **Path A (mutate Obsidian's internal `codeLanguages` map):** undocumented private API access via `(app as any).workspace.activeEditor.editor.cm` or similar; 1–2 days to spike, breaks on Obsidian updates.
- **Path B (nested EditorView per fence):** mount a separate, smaller EditorView dedicated to the fence content via `Decoration.replace({ widget: new FenceEditorWidget(...) })`. Solves all 8 LC languages with full LanguageSupport. 2–3 weeks of bidirectional sync, cursor crossing widget boundaries, undo/redo grouping, Live-Preview re-render storms, copy/paste, Phase 5.1 widget coexistence.

Both deferred to a future Phase 6.x.

**The pivot insight:** if `lang-markdown` already does nested-parser routing inside fences using its built-in `codeLanguages` map, then the cheapest path to native highlighting is **make the fence tag one that's already in that map**. Markdown recognizes `python`, `cpp`, `java`, `javascript`, `typescript`, `go`, `rust`. LC's slugs `python3` and `golang` are not. Remap at write time → highlighting Just Works. No CM6 LanguageSupport packs installed. Bundle returns to baseline.

This is precisely Phase 5.2 D-13's pattern (Reading-Mode `python3 → python` Prism alias) lifted to write-time. Same trick, different surface.

### Pattern 1: Atomic CM6 dispatch for fence rewrite (load-bearing)

**What:** A single `view.dispatch({ changes: [...], userEvent: 'leetcode.lang-switch' })` carrying TWO ranges — fence opener line edit + fence body replacement — so Cmd-Z reverts both in ONE step. The frontmatter `processFrontMatter` write happens **after** the dispatch, on a separate undo stack (vault-level), and is NOT user-undoable as part of the editor's Cmd-Z.

**When to use:** Any chevron click that switches LC language.

**Source:** [VERIFIED: CodeMirror reference manual `EditorView.dispatch` — accepts `changes: ChangeSpec | ChangeSpec[]`; multiple changes inside one dispatch land in one undoable transaction; `userEvent` annotation is the canonical way to mark transaction origin for undo merging] [CITED: 05.1-RESEARCH.md Pitfall 2 — `WidgetType.eq()` discipline ensures the chevron widget reuses DOM after the dispatch, no flicker]

**Example:**
```typescript
// src/main/languageChevronWidget.ts
// Source: [VERIFIED: codeActionsEditorExtension.ts findCodeFence — D-13 parity reuse]
// [CITED: CodeMirror reference manual "Transactions and Effects"]
import {
  type EditorView,
} from '@codemirror/view';
import {
  type EditorState,
} from '@codemirror/state';
import { findCodeFence } from './codeActionsEditorExtension';
import { lcSlugToFenceTag } from '../solve/languages';

export function dispatchFenceRewrite(
  view: EditorView,
  newSlug: string,
  newStarterCode: string,
): boolean {
  const fence = findCodeFence(view.state);
  if (!fence) return false;

  const opener = view.state.doc.line(fence.openerLine);
  const closer = view.state.doc.line(fence.closerLine);

  // Compute the new fence opener tag — preserve any leading whitespace on the
  // opener line (rare but possible in nested-list contexts).
  const tagMatch = /^(\s*```)\s*\S*\s*$/.exec(opener.text);
  const newOpenerText = tagMatch
    ? `${tagMatch[1]}${lcSlugToFenceTag(newSlug)}`
    : `\`\`\`${lcSlugToFenceTag(newSlug)}`;

  // Body spans from start-of-line-after-opener to start-of-closer-line.
  // (closer.from is the very start of the closer-line text; we replace the
  // body content WITHOUT touching the closer.)
  const bodyStart = opener.to + 1; // newline after opener
  const bodyEnd = closer.from;

  // Single dispatch — both edits land in one undo step.
  view.dispatch({
    changes: [
      { from: opener.from, to: opener.to, insert: newOpenerText },
      { from: bodyStart, to: bodyEnd, insert: newStarterCode + '\n' },
    ],
    userEvent: 'leetcode.lang-switch',
    // No selection adjustment — caret will fall wherever CM6 maps it.
    // If a user-perceived UX issue surfaces (caret lands oddly), add an
    // explicit `selection: EditorSelection.cursor(bodyStart)` here.
  });
  return true;
}
```

**Why no `Annotation.define<symbol>()` for "merge undo with frontmatter":** Obsidian's `processFrontMatter` does NOT route through CM6's transaction system. It writes via `vault.modify`/`fileManager` which lives on a different undo stack. There is no public API to bridge the two stacks. Accepting the divergence is the right call: in normal use the user won't notice (frontmatter is invisible in Live Preview by default), and the next chevron switch corrects any orphaned `lc-language` value.

### Pattern 4: Switch dispatch sequence (CM6 first → processFrontMatter second)

**What:** The chevron handler executes:
1. Network/cache fetch starter code.
2. CM6 `view.dispatch({ changes: [...] })` — single transaction; reverts opener+body atomically on Cmd-Z.
3. `await app.fileManager.processFrontMatter(file, fm => { fm['lc-language'] = newSlug })` — vault-level write, separate undo stack.

**Why this order:**
- CM6 first: the user-visible change (fence content) appears immediately on the typing tick; minimizes perceived latency.
- `processFrontMatter` second: vault write is async (opens file, parses YAML, writes back). Doing it first would block the visible fence rewrite by 5–20 ms.
- They're INDEPENDENT operations — failure of either doesn't strand the other. If processFrontMatter rejects, the fence still has new code; the next switch corrects `lc-language`. If CM6 dispatch fails (view destroyed mid-fetch), processFrontMatter still updates the source-of-truth.

**Source:** [VERIFIED: codeActionsEditorExtension.ts pattern — uses `view.dispatch` directly; obsidian.d.ts `app.fileManager.processFrontMatter(file: TFile, fn: (fm) => void): Promise<void>`] [CITED: NoteTemplate.ts:212 `applyFrontmatter` proves processFrontMatter is the canonical Obsidian frontmatter API used elsewhere in this codebase]

**Code:**
```typescript
// In LeetCodePlugin (main.ts) — add a public method the chevron handler calls.
async switchFenceLanguage(file: TFile, newSlug: string): Promise<void> {
  // Resolve current note context.
  const view = this.app.workspace.getActiveViewOfType(MarkdownView);
  if (!view || view.file !== file) return;     // raced with leaf change — abort

  // Fetch starter code via existing path (cache-then-network).
  const cached = this.settings.getProblemDetail(file.basename) ??
                 this.settings.getProblemDetail(/* slug from frontmatter */ '');
  let snippet: string;
  if (cached) {
    snippet = cached.codeSnippets?.find(s => s.langSlug === newSlug)?.code ?? '';
  } else {
    // Cache miss — call client. (Already wraps requestUrl + throttle.)
    try {
      const fresh = await this.client.getProblemDetail(/* slug */);
      snippet = fresh?.codeSnippets?.find(s => s.langSlug === newSlug)?.code ?? '';
    } catch {
      // Network error — keep existing fence body, just update tag + frontmatter.
      // (Per CONTEXT D-11 "planner picks safer default; recommend keep-current-body".)
      return;
    }
  }

  // Step 1: CM6 dispatch — atomic fence rewrite.
  const ok = dispatchFenceRewrite(view.editor.cm as EditorView, newSlug, snippet);
  if (!ok) return;

  // Step 2: processFrontMatter — separate undo stack.
  await this.app.fileManager.processFrontMatter(file, (fm) => {
    fm['lc-language'] = newSlug;
  });
}
```

### Anti-Patterns to Avoid

- **Calling `processFrontMatter` BEFORE `view.dispatch`**: introduces a 5–20 ms gap where `lc-language` is updated but the fence still shows the old language. If the user fires Run during that gap, the API call uses the new language but the fence content is the old language → server-side compile error. Always dispatch CM6 first.
- **Wrapping the CM6 dispatch in `setTimeout(..., 0)`**: tempting to mirror the failed approach's defer pattern (commit `de9aeeb`), but the chevron handler runs from a click event — NOT from inside `ViewPlugin.update()`. There is no "update in progress" guard to escape. Synchronous dispatch is correct. setTimeout-deferring would split the click handler from its undo step and break atomicity.
- **Using `Modal` for the dropdown**: blocks underlying editor click handling, requires Esc handling, breaks click-through expectations. CONTEXT discretion delegates this; recommend custom DOM.
- **Using native `<select>`**: works keyboard-accessibly but visually clashes with `.leetcode-code-action-*` styling because browser-chrome `<select>` doesn't honor Obsidian CSS variables. Reject.
- **Hand-rolling fence detection**: REUSE `findCodeFence` from `codeActionsEditorExtension.ts` (D-13 parity with Phase 5.1). Re-implementing breaks the "one source of truth for fence location" invariant.
- **Storing chevron state in a `StateField`**: Phase 5.1 already runs a StateField on every editor view. The chevron's "current language" is NOT editor state — it's the `lc-language` frontmatter value, sourced from `metadataCache`. Read at widget render time; do not duplicate into a StateField.
- **Mutating LC's cached `codeSnippets[]`**: read-only. The snippet text becomes the new fence body via dispatch; never write back to the cache.
- **Forgetting the Reading-Mode parity check**: `codeActionsPostProcessor.ts` MUST NOT pass `opts.prefix`. CONTEXT D-09 is explicit. Live-smoke must verify Reading Mode shows ONLY Run/Submit (no chevron).
- **Adding the chevron to `## Solution` fences (Phase 4)**: out of scope. The chevron only appears on `## Code` fences (Phase 5.1 `findCodeFence` already returns ONLY the `## Code` fence — D-13 parity).

## Common Pitfalls

### Pitfall 1: `processFrontMatter` and CM6 dispatch are on separate undo stacks

**What goes wrong:** User clicks chevron → fence rewrites + `lc-language` updates. User Cmd-Z's. Editor reverts fence opener+body (one CM6 transaction). `lc-language` stays on the new value. User Cmd-Z's again. `lc-language` reverts via vault undo. Two Cmd-Zs to fully revert what looked like one "destructive switch."

**Why it happens:** Public Obsidian APIs do not bridge editor undo and vault undo. `view.dispatch` writes to CM6's transaction history; `app.fileManager.processFrontMatter` writes to the file via `vault.modify`/`fileManager`, recorded on a separate vault-level undo stack.

**How to avoid:** Accept the divergence. Frontmatter is invisible in Live Preview by default (only the property panel shows it, and the property panel is a toggle most users keep collapsed). Cmd-Z reverts the visible fence — the thing the user sees. The orphaned `lc-language` value is harmless because (a) the next chevron switch corrects it, (b) Run/Submit always reads frontmatter at click time so an orphaned value just means the wrong API call is dispatched once until the user notices.

**Warning signs:**
- Live-smoke testing where Cmd-Z appears to "not fully undo" the switch.
- User reports "I switched to Java, undid, then ran the code, and it submitted as Java even though my code is Python."

**Mitigation if users complain (defer to Phase 6.x):** Bridge via a custom undo annotation that observers can subscribe to — non-trivial; out of scope for Phase 5.3.

**[CITED: obsidian.d.ts `app.fileManager.processFrontMatter` — no transaction-history hook; vault.modify uses Obsidian's vault undo, separate from CM6's]**

### Pitfall 2: Reading mode chevron leaks if `codeActionsPostProcessor.ts` is not audited

**What goes wrong:** Reading-Mode `codeActionsPostProcessor.ts` calls `buildCodeBlockButtonRow(doc, plugin)` without options today. After Pattern 3's signature change, that call STILL omits opts.prefix, so the chevron does NOT appear in Reading Mode — which is correct (D-09). BUT if a future contributor copy-pastes the Edit-Mode call site into Reading Mode, the chevron would leak. Reading Mode has no live editor; clicking the chevron there has no semantic meaning.

**How to avoid:** Add a unit test that asserts Reading-Mode `buildCodeBlockButtonRow` invocation does NOT pass `opts.prefix`. Mark the call site with a comment: `// D-09 — chevron is Edit-Mode only; do NOT pass opts.prefix here`.

**Warning signs:**
- Live smoke shows chevron appearing under both Edit and Reading mode.
- Clicking the chevron in Reading Mode throws (no `EditorView` to dispatch to).

### Pitfall 3: Chevron click handler races with metadataCache cold state

**What goes wrong:** User opens a note → chevron renders before `metadataCache` populates `lc-language`. `LC_LANG_DISPLAY_LABELS[undefined]` returns undefined; chevron paints `▼ undefined`.

**Why it happens:** Phase 5.1 RESEARCH Pitfall 4 (`metadataCache may be cold immediately on file open — docChanged transactions rebuild once the cache populates`).

**How to avoid:** Default the chevron label to `settings.getDefaultLanguage()` when frontmatter has no `lc-language` (matches the pattern `NoteWriter` already follows on line 311: `pickStarterCode(newEntry, defaultLang)`). Phase 5.1's StateField rebuild-on-docChanged will refresh the chevron label once the cache populates.

**Warning signs:** Live smoke shows `▼ undefined` for ~100 ms after file open.

### Pitfall 4: Stale fence body preserved on network failure

**What goes wrong:** User clicks chevron → network down → `getProblemDetail` rejects → chevron handler bails BEFORE dispatching CM6. User sees "nothing happened." User clicks again — same thing. Ambiguous user feedback.

**Why it happens:** CONTEXT D-11 says "keep current fence body on failure" — recommended; safer than blanking. But it's silent.

**How to avoid:** **Surface a single Notice** on network failure: `'Couldn't fetch starter code for {Language}. Check your connection.'` Same copy convention as Phase 5's CF-04 / CF-19. Idempotent: don't fire repeatedly if user double-clicks.

**Warning signs:** User reports "the chevron does nothing for me" → root cause: silent network failure on the GraphQL fetch.

### Pitfall 5: `findCodeFence` returns null when fence is unterminated

**What goes wrong:** User edits the closing ` ``` ` away mid-typing → `findCodeFence` returns null → chevron click silently bails.

**Why it happens:** 5.1 `findCodeFence` requires both opener AND closer.

**How to avoid:** Acceptable behavior. The chevron silently no-ops (`dispatchFenceRewrite` returns false). User restores the fence; chevron works again. Live smoke verifies; no code path needed.

**Warning signs:** None — quiet no-op is correct posture.

### Pitfall 6: `view.editor.cm` is undocumented internal API

**What goes wrong:** Accessing CM6's `EditorView` from Obsidian's `Editor` requires the cast `view.editor.cm as EditorView`. This is undocumented; could break on future Obsidian versions.

**Why it happens:** Obsidian wraps CM6 in its own `Editor` interface; the `cm` property is the underlying CM6 view but is not in `obsidian.d.ts`.

**How to avoid:** This is unavoidable for the chevron's CM6 dispatch. PRECEDENT: Phase 5.1 `codeActionsEditorExtension.ts` doesn't need this cast because it registers an `EditorExtension` and CM6 calls into its StateField directly. The chevron handler runs from a DOM click — outside CM6's transaction lifecycle — so it must reach for `view.editor.cm` to dispatch. Document the cast with a comment; same posture as `as unknown as TFile` in `NoteWriter.ts`.

**[CITED: CLAUDE.md "Access to CM6 EditorView from a plugin is possible (`view.editor.cm as EditorView`) but is undocumented/internal."]**

**Warning signs:** Future Obsidian release deprecates `editor.cm` → chevron breaks. Mitigation deferred until it happens.

### Pitfall 7: Chevron dropdown z-index conflict with Obsidian's own popovers

**What goes wrong:** Dropdown opens BEHIND Obsidian's hover preview / autocomplete / property-panel popovers because `z-index: 100` may be lower than Obsidian's overlays.

**Why it happens:** Obsidian doesn't publish a z-index hierarchy.

**How to avoid:** Live-smoke test the dropdown against (a) hovering over a wikilink, (b) opening the autocomplete via `[[`, (c) toggling the property panel. If conflicts surface, bump `z-index` to 1000 or higher. Conservative recommendation: use `z-index: var(--layer-popover, 100)` if Obsidian provides such a variable; fall back to 100.

**Warning signs:** User reports "I clicked the chevron and a popover appeared on top of the dropdown."

### Pitfall 8: Test for `applySolveTimeFrontmatter` lc-runtime-ms removal breaks existing tests

**What goes wrong:** D-01/D-02 deletes `applySolveTimeFrontmatter`'s writes for `lc-runtime-ms` and `lc-memory-mb`. Tests in `tests/graph/onAccepted.frontmatter.test.ts`, `onAccepted.reAccepted.test.ts`, `fixture-integration.test.ts` assert these keys are written. Removing the writes breaks 5 test assertions.

**Why it happens:** Coverage carries the old contract.

**How to avoid:** **Update the tests in the same wave as the source change.** New assertions: `expect(fm['lc-runtime-ms']).toBeUndefined()` (or remove the assertion entirely). Plan should sequence: source edit → test edit → green run → commit, in one atomic plan-execution.

**[CITED: grep of tests/ 2026-05-11 — 5 failing assertions if writes are deleted without test updates]**

**Warning signs:** Wave 1 reports 5 test failures; root cause obvious.

### Pitfall 9: `dispatchFenceRewrite` corrupts surrounding markdown if fence body contains backticks

**What goes wrong:** User's solution code contains the literal string ` ``` ` (e.g., a triple-backtick comment in a Python solution). `findCodeFence`'s closer regex matches that interior backtick run instead of the real closer, so `dispatchFenceRewrite` rewrites the body partway through user code, corrupting the fence.

**Why it happens:** Phase 5.1's `findCodeFence` uses simple regex `/^\s*```/` and walks forward for the first match. It doesn't account for fence-fence-fence-inside-code edge cases. (Standard markdown fences should use 4 backticks for the outer fence in this case, but LC's starter code is always 3-backtick-wrapped.)

**How to avoid:** This is a Phase 5.1 limitation, NOT a Phase 5.3 regression. Existing Run/Submit buttons exhibit the same issue. Acceptable to inherit. If the planner chooses to harden, do so AS A SEPARATE TASK that updates 5.1's findCodeFence (and add fence-with-internal-backticks as a test case). Not load-bearing for shipping 5.3.

**Warning signs:** User reports "the chevron broke my code" + user's code contains `` ``` ``.

### Pitfall 10: Fence-tag remap collides with user's existing fence content on retro-revert

**What goes wrong:** User had a `python3` fence pre-Phase-5.3. After 5.3 ships, they switch to Java via the chevron — fence becomes `java`. They Cmd-Z. The CM6 undo step reverts opener back to `python3` — fine. But their `lc-language` frontmatter was just updated to `java`; vault undo reverts it back to `python3` — fine. Two undos, but consistent end state.

**Why it happens:** This is the SAME divergence as Pitfall 1 — flagged here as a positive-case observation: even though undo is split across two stacks, the END STATE after both undos is correct.

**How to avoid:** Document the two-Cmd-Z behavior in UI-SPEC. Live-smoke verifies the end state is consistent (fence + frontmatter agree) after both undos.

**Warning signs:** User reports "after the second Cmd-Z, my old code came back, but the language frontmatter still said Java."

## Open Questions

### 1. Should the chevron's dropdown include unsupported LC languages (csharp, ruby, swift, etc.)?

- **What we know:** CONTEXT D-04 says "For unsupported LC languages: write the fence tag as the LC slug as-is (e.g., ` ```csharp `). The fence will render plain monospace (no highlighting)." The chevron handler can write `lc-language: csharp` and emit a ```csharp fence — Run/Submit work fine because they only read `lc-language`. The only side effect is no Edit-Mode highlighting.
- **What's unclear:** Whether the dropdown LIST shows the 8 supported languages only (`LC_CHEVRON_LANG_ORDER` above) or all 23 LC slugs from `LC_LANG_SLUGS`. Showing all 23 lets users switch to C# / Ruby etc. but renders unstyled. Showing only 8 limits the chevron's utility but keeps the UX clean.
- **Recommendation:** Show only the 8 supported languages in the dropdown for v1 (matches CONTEXT's "8 LC languages" phrasing). Defer "extended language picker" to a future phase. If users complain, the change is one-line (extend `LC_CHEVRON_LANG_ORDER`).

### 2. Should the chevron handler refetch starter code every click, or cache per-language per-note?

- **What we know:** CONTEXT D-11 says "refetch from LC GraphQL on every switch; cache strategy is planner's discretion." The existing `SettingsStore.getProblemDetail(slug)` cache stores ALL `codeSnippets[]` for ALL 8 languages in ONE entry per slug. So the FIRST chevron click after note open is "free" (cache hit; no network); subsequent clicks within the same session are also "free."
- **What's unclear:** Whether to invalidate the cache on chevron click (force refresh) or trust the 7-day TTL.
- **Recommendation:** Trust the 7-day TTL. The `codeSnippets[]` for a problem don't change once the problem is created on LC. No invalidation needed.

### 3. Should the chevron persist its open/closed state across editor focus changes?

- **What we know:** Standard dropdowns close on outside click (industry convention).
- **What's unclear:** Whether toggling between Live Preview and Source mode (Cmd-E) should keep the dropdown open.
- **Recommendation:** Close on any focus change (mode flip, leaf change, file close). Keeps the implementation simple; matches user mental model of "transient picker."

### 4. Should the chevron be keyboard-accessible (Tab to focus, Space/Enter to open, arrows to navigate)?

- **What we know:** Custom DOM dropdowns require explicit keyboard handling. Native `<select>` gets it free but was rejected.
- **What's unclear:** Whether v1 ships keyboard support or defers to a UX polish phase.
- **Recommendation:** Ship basic keyboard support (Tab focus, Space/Enter to toggle, Esc to close, arrow keys to navigate, Enter to select) — ~30 LOC. Skipping it would fail Obsidian community plugin accessibility expectations. Live-smoke verifies.

### Specific answers to research questions

#### Q1 — Atomic CM6 dispatch shape (load-bearing)

**Recommended order: CM6 dispatch FIRST → `processFrontMatter` SECOND.**

- CM6 dispatch carries BOTH the fence opener tag rewrite AND the fence body replacement in ONE `view.dispatch({ changes: [...] })` call → ONE editor undo step.
- `processFrontMatter` runs after; lands on Obsidian's vault undo stack (separate).
- Single-Cmd-Z atomic-revert across both stacks is **NOT achievable via public Obsidian APIs**. Accepted divergence (Pitfall 1).
- Alternative considered: dispatch processFrontMatter FIRST → CM6 SECOND. Rejected because (a) processFrontMatter is async and would delay the user-visible fence rewrite; (b) ordering doesn't change the undo divergence.
- Reference for known-pattern usage: SAME pattern already used in this codebase for solve-time writes — `KnowledgeGraphWriter.onAccepted` calls `applySolveTimeFrontmatter` (which calls processFrontMatter) AFTER the editor write to `## Solution` via `vault.process` — same two-stack divergence accepted there. **[CITED: KnowledgeGraphWriter.ts:120 + applySolveTimeFrontmatter via NoteTemplate.ts:340]**

#### Q2 — Chevron widget anchoring

**Recommended: modify `buildCodeBlockButtonRow` to accept optional `opts.prefix`.**

- Cleanest extension point — Phase 5.1's button row renders inside `codeActionsEditorExtension.ts::CodeActionsWidget.toDOM`; that's the ONLY Edit-Mode call site, so passing the chevron factory there (and not in `codeActionsPostProcessor.ts`) gives Edit-Mode-only chevron with zero conditionals in the row builder.
- CSS layout: change `.leetcode-code-actions { justify-content: flex-end }` to `space-between` AND add `margin-left: auto` to `.leetcode-code-action-run` so Run+Submit cling to the right edge as a group.
- Sibling extension that overlaps would require absolute positioning + per-mode coordination — fragile.

#### Q3 — Native `<select>` vs custom DOM vs `Modal`

**Recommended: custom DOM dropdown.**

- Native `<select>`: reject — visually clashes with Obsidian theme (browser-chrome vs CSS-variable styling).
- `Modal`: reject — overkill for an 8-item picker; blocks underlying editor click handling.
- Custom DOM: ~30 LOC; matches `.leetcode-code-action-*` styling via Obsidian theme variables (Pattern 2).
- **No public Obsidian plugins observed using inline language pickers exactly like this** — the chevron pattern is novel for this plugin domain. Closest analog: Obsidian's own Markdown Editor language switcher uses a Modal (different UX context — global setting). Custom DOM is the right fit.

#### Q4 — GraphQL refetch path

**Existing call site: `LeetCodeClient.getProblemDetail(slug)` at `src/api/LeetCodeClient.ts:112`.**

- Returns `LeetCodeProblemDetail` shape including `codeSnippets?: Array<{ lang, langSlug, code }>`.
- Wraps `@leetnotion/leetcode-api`'s `lc.problem(slug)` → GraphQL `question(slug: String!)` query (which IS the `questionEditorData`-equivalent endpoint LC uses).
- Already proxies through `installRequestUrlFetcher` → `throttledRequestUrl` (rate-limited, CORS-safe).
- Already cached by `SettingsStore.getProblemDetail(slug)` (7-day TTL).
- Chevron handler reuses this path verbatim — no new endpoint, no new call.

#### Q7 — Bundle-size cap after revert

**Recommended: tighten the cap to 250 KB hard / 200 KB warn.**

The point of a hard cap is to surface regressions early; 250 KB gives ~95 KB of breathing room while still catching a 50%-bloat regression. Recommend tightening.
