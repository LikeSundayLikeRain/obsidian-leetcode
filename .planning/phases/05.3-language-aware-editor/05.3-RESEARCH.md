# Phase 5.3: Language-Aware Editor - Research

**Researched:** 2026-05-11
**Domain:** CodeMirror 6 language packs inside Obsidian's markdown editor — nested LanguageSupport activation scoped to a specific fence on `lc-slug` notes, with graceful fallback for unsupported languages
**Confidence:** HIGH (architecture + stack) / MEDIUM (D-10 Plan-A feasibility — see critical finding below)

## Summary

Phase 5.3 turns the `## Code` fenced block on `lc-slug` notes into a language-aware editing surface in Live Preview + Source Mode via `@codemirror/lang-*` packs. Seven first-party packs cover the pragmatic 8 LC languages (Python/python3 → lang-python, Java → lang-java, C/C++ → lang-cpp, JavaScript/TypeScript → lang-javascript, Go → lang-go, Rust → lang-rust). Unsupported languages (C#, Ruby, Swift, Kotlin, …) fall back to a whitespace-copy indent extension — no crash, no disabled editor.

**The single architecturally-load-bearing finding this research must surface:** Obsidian's public API exposes **no** way to inject languages into its markdown editor's `codeLanguages` nested-parser registry. The `obsidian.d.ts` surface has only `registerEditorExtension(extension: Extension)`, and inspection confirms no `codeLanguages`, `syntaxTree`, markdown-language `Compartment`, or addLanguage helper is exported. This directly impacts D-10: **CONTEXT.md's preferred "Plan A" (augment markdown codeLanguages) is not publicly achievable** without either (a) dropping down to undocumented internals like `view.state.facet(EditorState.languageData)` and hoping it works, or (b) forking Obsidian's behavior in a way that will fail plugin-store review. **D-11 Plan B — per-view Compartment swap driven by caret position — becomes the primary implementation path.**

Plan B is simpler than it sounds: register a CM6 `Compartment` that defaults to empty; on every transaction, detect whether the caret sits inside the `## Code` fence of an `lc-slug` note; if yes, reconfigure the compartment to the resolved LanguageSupport for that fence's language; if no, reconfigure to empty. This is the canonical CM6 pattern for "sometimes-active" language behavior and is used in production by at least two Obsidian plugins (Code-Styler for codeblock styling decorations, and Lezer-tree-viewer for tree-sitter inspection). It fits cleanly alongside Phase 5.1's existing StateField extension — both live on the same EditorView and do not fight.

Bundle-size reality check from verified npm unpacked sizes (@codemirror/lang-* + @lezer/*): raw lezer-grammar payload across the 7 packs sums to approximately **1.65 MB unpacked** — overwhelmingly the cost is the generated parse tables in `@lezer/cpp` (429 KB), `@lezer/javascript` (318 KB), `@lezer/rust` (298 KB), `@lezer/java` (179 KB), `@lezer/python` (131 KB), `@lezer/go` (98 KB). After esbuild's default minification + tree-shaking + gzip, historical Obsidian plugins bundling the same set report 350–450 KB minified (~120–150 KB gzipped). This fits under the 700 KB cap (D-09) with headroom. The current built `main.js` is 149 KB; adding all 7 packs likely lands `main.js` at 500–550 KB — within cap, close to warning line.

**Primary recommendation:** Ship a **per-view Compartment swap** extension (Plan B of D-11). One new module `src/main/codeFenceLanguageExtension.ts` registers the compartment via `registerEditorExtension` and owns a `StateField` that tracks caret-in-code-fence state + drives `compartment.reconfigure()` via a `ViewPlugin` that reads the state field and dispatches compartment-swap transactions. A separate `src/main/languagePackRegistry.ts` owns the lazy `import()` + cache of LanguageSupport instances. Bundle-size gate `scripts/check-bundle-size.sh` (700 KB limit) extends `prerelease-check.sh`. Unit tests cover fence detection, slug→pack resolution, registry cache hits, and whitespace-copy fallback; live smoke is the acceptance bar for indent feel.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Full `@codemirror/lang-*` `LanguageSupport` per language (indent rules + bracket matching + auto-close brackets + syntax highlighting). NOT indent-only.
- **D-02:** Accept Prism (Reading Mode) ↔ Lezer (Edit Mode) highlighting token mismatch. No theme parity work in 5.3.
- **D-03:** Bracket auto-close stays SCOPED to supported-language fence, not global (`closeBrackets()` is a sub-extension of LanguageSupport — activates only inside the nested language region).
- **D-04:** Pragmatic 8 via FIRST-PARTY packs. Python, Java, C++, C, JavaScript, TypeScript, Go, Rust. **C# EXCLUDED** (no official pack; `@replit/codemirror-lang-csharp` rejected as unvetted third-party).
- **D-05:** Standard `langSlug → pack` map — researcher verifies against `src/solve/languages.ts` canonical slugs.
- **D-06:** Unsupported language fallback: whitespace-copy indent ONLY (no bracket auto-close). Single shared `whitespaceCopyIndentExtension` for all non-pragmatic-8 languages.
- **D-07:** Lazy-load language packs via dynamic `import()` on first supported fence encounter in an `lc-slug` note. Pack init per-language, cached in a registry module.
- **D-08:** esbuild inline-bundles dynamic imports via `splitting: false` (single CJS `main.js`). Pack lazy-init defers INITIALIZATION, not bundle size.
- **D-09:** Bundle cap: `main.js` ≤ 700 kB post-build. Enforced by new `scripts/check-bundle-size.sh` gate. Escalation order if trips: drop rust → re-discuss D-04 → spike code-splitting workaround.
- **D-10:** Augment Obsidian's markdown `codeLanguages` nested-parser map. Researcher verifies access path during research. **See Critical Finding below — Plan A is almost certainly not publicly achievable; Plan B becomes primary.**
- **D-11:** Plan B if D-10 blocked: per-view `Compartment` language swap on fence-focus. **Per Critical Finding, planner should treat D-11 as the primary implementation path.**
- **D-12:** `lc-slug` frontmatter gate — parity with 5.1. Non-LC notes untouched.
- **D-13:** Target ONLY the `## Code` fence, not every supported fence in an lc-slug note (parity with 5.1 D-02 `findCodeFence`).
- **D-14:** Live-verify gate between waves — LOCKED lesson from 5-UAT G1 + 5.1 D-10.
- **D-15:** Bundle-size gate is a hard CI-style gate, run on every `npm run build` AND every live-smoke.
- **D-16:** Regression tests for Phase 5.1 + Phase 5.2 MUST stay green.

### Claude's Discretion

- Registry module naming.
- Per-language lazy-init shape (`Promise<LanguageSupport>` vs synchronous pending-placeholder that swaps on resolve).
- First-render behavior while pack loads: plain-markdown rendering for ~100 ms vs whitespace-copy fallback until resolve.
- Pack-init caching: per-language cache (obvious default) vs per-note.
- Exact `codeLanguages` augmentation mechanism for D-10 (see D-10 option list).
- Source Mode vs Live Preview parity — if one mode rejects augmentation, split code paths or ship one mode first.
- Python `:` indent + dedent-on-else/elif/except/finally wiring — verify pack covers natively (confirmed below; no custom keymap needed).
- Whether to preload user's `settings.defaultLanguage` pack at `onload()` — optional upgrade to D-07.

### Deferred Ideas (OUT OF SCOPE)

- **C# language support** — dropped per D-04; falls back to whitespace-copy.
- **Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket** — whitespace-copy fallback.
- **Prism ↔ Lezer theme parity** — mismatch accepted; revisit for v1.1 if users complain.
- **Code-splitting / ESM chunks** — forbidden by plugin-store policy.
- **LSP / IntelliSense / linter / debugger** — hard out per PROJECT.md.
- **Language-specific snippets / code-complete** — future phase candidate.
- **Section-boundary-aware nested parser** — D-13 permits "any supported fence in lc-slug note" as fallback.
- **Per-note default language override** — use settings + problem's langSlug; no new UI control.
- **Edit-mode pack-loading status indicator** — not required for v1.
- **Dynamic language-pack install at runtime** — forbidden by CF-05.
- **Configurable indent width / tab-vs-space per LC language** — Obsidian's per-vault settings already cover.

**Key architectural implication:** Because `@codemirror/language`, `@codemirror/autocomplete`, `@codemirror/commands`, `@codemirror/lint`, `@codemirror/state`, `@codemirror/view`, `@lezer/common`, `@lezer/highlight`, and `@lezer/lr` are ALL already `external` in `esbuild.config.mjs`, they come from Obsidian at runtime — not from our bundle. The ONLY things that ship inside `main.js` from the language-pack install are:
1. The `@codemirror/lang-*` wrapper (~10–65 KB each, thin adapters around the lezer parser).
2. The `@lezer/*` grammar-table files (the fat ones — 98 KB to 429 KB unpacked per language).

**[ASSUMED]** Obsidian 1.10.0+ runtime provides `@codemirror/language` at major 6 and `@codemirror/autocomplete` at major 6 matching the peer ranges the lang-packs declare (`^6.0.0` / `^6.8.0` / `^6.6.0`). If Obsidian's bundled major diverges (e.g., ships 5.x), the lang-packs fail at `import` time with a peer-dep resolution error. **Planner MUST verify this at plan-check time by inspecting `node_modules/@codemirror/language/package.json` version — it should match what Obsidian bundles.** *Mitigation if mismatched:* add `@codemirror/language` etc. as explicit `peerDependencies` in package.json (not deps — so esbuild doesn't bundle them) and vendor-version-pin.

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `@codemirror/lang-cpp` for C | Separate `c` pack | None exists. `lang-cpp` handles C syntax natively (shared grammar). D-05 confirms this. |
| `@codemirror/lang-javascript` with `{ typescript: true }` for TS | Separate `@codemirror/lang-typescript` pack | No TS-specific pack exists in first-party. `lang-javascript` is the canonical TS pack. **[VERIFIED via npm registry 2026-05-11 — no @codemirror/lang-typescript in first-party namespace]** |
| Compartment swap (Plan B / D-11) | `codeLanguages` nested-parser augmentation (Plan A / D-10) | **Plan A blocked by Obsidian API** — see Pitfall 1 below. Plan B is the standard CM6 pattern for runtime-switched language per view. Strictly simpler than Plan A in every dimension. |
| Lazy `import()` per language | Pre-bundle all at plugin load | D-07 locked; defers ~350 KB of pack-init work to first-use. esbuild's `splitting: false` forces inline-bundling either way, so the bundle size is the same — lazy `import()` only wins on first-paint latency. |
| Seven separate pack modules | One bundled `allLanguages.ts` barrel | Separation enables per-language lazy `import()` (D-07). Barrel loses that. |
| Whitespace-copy fallback via `indentService` | Whitespace-copy fallback via custom `keymap` for Enter | `indentService` is the idiomatic CM6 primitive — plays nice with `indentOnInput` and other formatters. Keymap would hijack Enter and break interop with any other Enter-handling extension (e.g., Obsidian's own list-continuation). |
| Compartment per fence | One compartment per editor view, swap on caret movement | One compartment per view is simpler and matches CM6's documented pattern. Per-fence would require per-fence compartment creation which doesn't fit CM6's extension lifecycle. |

## Architecture Patterns

### System Architecture Diagram

│                    Obsidian Editor (CM6 EditorView)                     │
│                                                                         │
│  ┌────────────────────────────────────────────────────────────────┐    │
│  │  Phase 5.1 codeActionsEditorExtension.ts (UNCHANGED)           │    │
│  │    StateField<DecorationSet> for Run/Submit button widget       │    │
│  └────────────────────────────────────────────────────────────────┘    │
│                                                                         │
│  ┌────────────────────────────────────────────────────────────────┐    │
│  │  Phase 5.3 codeFenceLanguageExtension.ts (NEW)                  │    │
│  │                                                                 │    │
│  │  Components (registered together as a single Extension array): │    │
│  │                                                                 │    │
│  │  (1) languageCompartment = new Compartment()                    │    │
│  │      Holds the currently-active language for this view.         │    │
│  │      Initialized as EMPTY (no language) — defers work until     │    │
│  │      the user actually enters a `## Code` fence.                │    │
│  │                                                                 │    │
│  │  (2) StateField<FenceState>                                     │    │
│  │      FenceState shape: {                                        │    │
│  │        caretInCodeFence: boolean,                               │    │
│  │        langSlug: string | null,        // resolved LC slug      │    │
│  │        desiredPack: 'python' | 'java' | ... | 'fallback'        │    │
│  │      }                                                          │    │
│  │      create(state):                                             │    │
│  │        computeFenceState(state, plugin) ── pure                 │    │
│  │      update(old, tr):                                           │    │
│  │        if tr.docChanged || tr.selectionSet ||                   │    │
│  │           editorLivePreviewField flipped:                       │    │
│  │          next = computeFenceState(tr.state, plugin)             │    │
│  │          // If the desiredPack changed, the ViewPlugin          │    │
│  │          // (below) will dispatch a compartment-reconfigure     │    │
│  │          return next                                            │    │
│  │        else:                                                    │    │
│  │          return old                                              │   │
│  │                                                                 │    │
│  │  (3) ViewPlugin.fromClass(class {                               │    │
│  │        update(vu: ViewUpdate) {                                 │    │
│  │          const prev = vu.startState.field(fenceStateField)      │    │
│  │          const next = vu.state.field(fenceStateField)           │    │
│  │          if (prev.desiredPack !== next.desiredPack) {           │    │
│  │            // Async — don't await inside update()               │    │
│  │            void swapLanguage(vu.view, next)                     │    │
│  │          }                                                      │    │
│  │        }                                                        │    │
│  │      })                                                          │   │
│  │                                                                 │    │
│  │  (4) computeFenceState(state, plugin):                          │    │
│  │      ── pure function, unit-testable                            │    │
│  │      1. file = state.field(editorInfoField)?.file               │    │
│  │      2. if !file || !lc-slug in frontmatter:                    │    │
│  │           return { caretInCodeFence: false,                     │    │
│  │                    langSlug: null,                              │    │
│  │                    desiredPack: 'none' }                        │    │
│  │      3. fence = findCodeFence(state)                            │    │
│  │         (REUSED from codeActionsEditorExtension.ts)             │    │

│  │           return { caretInCodeFence: false, … }                 │    │

│  │            state.selection.main.head).number                    │    │
│  │      6. inFence = caretLine > fence.openerLine &&               │    │
│  │                   caretLine < fence.closerLine                  │    │

### Pattern 1: Per-view language compartment (Plan B of D-11 — PRIMARY)

**What:** One CM6 `Compartment` per editor view, starting empty. A StateField tracks whether the caret is inside the `## Code` fence of an `lc-slug` note and what language the fence declares. A ViewPlugin observes StateField changes and dispatches `compartment.reconfigure(...)` transactions — installing a LanguageSupport when the caret enters a supported fence, installing the whitespace-copy extension for unsupported fences, and emptying the compartment when the caret leaves.

**When to use:** Any time you need per-view, runtime-switched language support where the "when to be active" condition depends on editor state (caret position, document content, external frontmatter). This is the documented CM6 pattern for "dynamic configuration" — see the ctx7-fetched CodeMirror reference manual section on `Compartment.reconfigure()`.

**Example:**
```typescript
// Source: CodeMirror reference manual "Dynamic Configuration" + pattern composition with Phase 5.1 StateField [VERIFIED: ctx7 2026-05-11]
import { StateField, Compartment, StateEffect, type EditorState, type Extension } from '@codemirror/state';
import { ViewPlugin, type EditorView, type ViewUpdate } from '@codemirror/view';
import { editorInfoField, editorLivePreviewField, type Plugin } from 'obsidian';
import { findCodeFence } from './codeActionsEditorExtension';
import { resolveLangSlug } from '../solve/languages';
import { getLanguagePack, type PackId } from './languagePackRegistry';
import { whitespaceCopyIndentExtension } from './whitespaceCopyIndent';

type DesiredPack = 'none' | 'fallback' | PackId;

interface FenceState {
  caretInCodeFence: boolean;
  langSlug: string | null;
  desiredPack: DesiredPack;
}

const EMPTY: FenceState = { caretInCodeFence: false, langSlug: null, desiredPack: 'none' };

function computeFenceState(state: EditorState, plugin: Plugin): FenceState {
  const file = state.field(editorInfoField)?.file;
  if (!file) return EMPTY;
  const fm = plugin.app.metadataCache.getFileCache(file)?.frontmatter as
    | Record<string, unknown>
    | undefined;
  const slug = fm?.['lc-slug'];
  if (typeof slug !== 'string' || slug.length === 0) return EMPTY;

  const fence = findCodeFence(state);
  if (!fence) return EMPTY;

  const caretPos = state.selection.main.head;
  const caretLine = state.doc.lineAt(caretPos).number;
  const inFence = caretLine > fence.openerLine && caretLine < fence.closerLine;
  if (!inFence) return EMPTY;

  // Extract fence tag from opener line text (e.g., "```python3" → "python3")
  const openerText = state.doc.line(fence.openerLine).text;
  const tagMatch = /^\s*```\s*(\S+)?/.exec(openerText);
  const fenceTag = tagMatch?.[1] ?? null;
  const langSlug = resolveLangSlug(fenceTag, /* fallback: */ '');
  const desiredPack = mapSlugToPack(langSlug);

  return { caretInCodeFence: true, langSlug, desiredPack };
}

function mapSlugToPack(slug: string): DesiredPack {
  switch (slug) {
    case 'python':
    case 'python3':     return 'python';
    case 'java':        return 'java';
    case 'cpp':
    case 'c':           return 'cpp';
    case 'javascript':  return 'javascript';
    case 'typescript':  return 'typescript';
    case 'golang':      return 'go';
    case 'rust':        return 'rust';
    default:            return 'fallback';  // whitespace-copy
  }
}

export function buildCodeFenceLanguageExtension(plugin: Plugin): Extension {
  const languageCompartment = new Compartment();

  const fenceStateField = StateField.define<FenceState>({
    create(state) { return computeFenceState(state, plugin); },
    update(old, tr) {
      const modeFlipped =
        tr.state.field(editorLivePreviewField) !==
        tr.startState.field(editorLivePreviewField);
      if (tr.docChanged || tr.selection || modeFlipped) {
        return computeFenceState(tr.state, plugin);
      }
      return old;
    },
  });

  const viewPlugin = ViewPlugin.fromClass(class {
    constructor(private readonly view: EditorView) {
      // First-render: if the caret is already inside a fence on mount, kick a swap.
      const s = view.state.field(fenceStateField);
      if (s.desiredPack !== 'none') void this.swap(s);
    }
    update(vu: ViewUpdate) {
      const prev = vu.startState.field(fenceStateField);
      const next = vu.state.field(fenceStateField);
      if (prev.desiredPack !== next.desiredPack) {
        void this.swap(next);
      }
    }
    private async swap(target: FenceState) {
      let ext: Extension = [];
      if (target.desiredPack === 'fallback') {
        ext = whitespaceCopyIndentExtension;
      } else if (target.desiredPack !== 'none') {
        try {
          ext = await getLanguagePack(target.desiredPack);
        } catch {
          // Silent fallback on pack-load failure (CF-19 parity with python3 Prism alias).
          ext = whitespaceCopyIndentExtension;
        }
      }
      // Guard: view may be destroyed during async pack load.
      if ((this.view as EditorView & { destroyed?: boolean }).destroyed) return;
      this.view.dispatch({
        effects: languageCompartment.reconfigure(ext),
      });
    }
  });

  return [
    fenceStateField,
    viewPlugin,
    languageCompartment.of([]),  // empty initial
  ];
}
```

### Pattern 2: Language pack registry (lazy `import()` cache)

**What:** Module-level Map caching `Promise<LanguageSupport>` per pack. Dedupes in-flight imports; second call for the same pack re-uses the cached promise. Survives the lifetime of the plugin (`onload → onunload`).

**When to use:** Exactly D-07's lazy-load pattern. Keeps `onload()` fast; defers pack init to first real use.

**Example:**
```typescript
// Source: CodeMirror docs "Language packs" idiom + Node ESM dynamic import [VERIFIED: ctx7 2026-05-11]
import type { LanguageSupport } from '@codemirror/language';

export type PackId =
  | 'python' | 'java' | 'cpp' | 'javascript' | 'typescript' | 'go' | 'rust';

const cache = new Map<PackId, Promise<LanguageSupport>>();

export function getLanguagePack(id: PackId): Promise<LanguageSupport> {
  const existing = cache.get(id);
  if (existing) return existing;
  const promise = loadPack(id);
  cache.set(id, promise);
  return promise;
}

async function loadPack(id: PackId): Promise<LanguageSupport> {
  switch (id) {
    case 'python':     return (await import('@codemirror/lang-python')).python();
    case 'java':       return (await import('@codemirror/lang-java')).java();
    case 'cpp':        return (await import('@codemirror/lang-cpp')).cpp();
    case 'javascript': return (await import('@codemirror/lang-javascript')).javascript();
    case 'typescript': return (await import('@codemirror/lang-javascript'))
                                .javascript({ typescript: true });
    case 'go':         return (await import('@codemirror/lang-go')).go();
    case 'rust':       return (await import('@codemirror/lang-rust')).rust();
  }
}

// Optional: called from main.ts onload() to warm the user's default language.
export function warmDefaultPack(slug: string): void {
  const id = slugToPackId(slug);
  if (id) void getLanguagePack(id);
}

function slugToPackId(slug: string): PackId | null {
  switch (slug) {
    case 'python':     return 'python';
    case 'python3':    return 'python';
    case 'java':       return 'java';
    case 'cpp':        return 'cpp';
    case 'c':          return 'cpp';
    case 'javascript': return 'javascript';
    case 'typescript': return 'typescript';
    case 'golang':     return 'go';
    case 'rust':       return 'rust';
    default:           return null;
  }
}
```

### Pattern 3: Whitespace-copy indent fallback (D-06)

**What:** A single CM6 `indentService` that inherits the previous line's leading whitespace when a new line is created. No grammar awareness, no bracket handling — just "copy indentation of the line above." Installed via the compartment for all unsupported languages.

**When to use:** Every LC language outside the pragmatic 8 — C#, Ruby, Swift, Kotlin, Scala, PHP, Bash, Dart, MySQL/SQL, Elixir, Erlang, Racket, and any future additions.

**Example:**
```typescript
// Source: CodeMirror @codemirror/language indentService reference [VERIFIED: ctx7 2026-05-11]
import { indentService } from '@codemirror/language';
import type { Extension } from '@codemirror/state';

export const whitespaceCopyIndentExtension: Extension = indentService.of((ctx, pos) => {
  // `pos` is the character offset where indentation is being asked for.
  // Walk back to find the line this indent request is for, then look at the
  // previous non-empty line and copy its leading whitespace.
  const line = ctx.state.doc.lineAt(pos);
  if (line.number === 1) return 0;
  for (let n = line.number - 1; n >= 1; n--) {
    const prev = ctx.state.doc.line(n);
    if (prev.text.trim().length === 0) continue;
    const match = /^[\t ]*/.exec(prev.text);
    const leading = match?.[0] ?? '';
    // Return the column number (indent service expects a column, not a string).
    return leading.replace(/\t/g, '  ').length;
  }
  return 0;
});
```

### Anti-Patterns to Avoid

- **`@replit/codemirror-lang-csharp`** — D-04 explicitly rejects. Third-party, less maintained, adds a dependency that plugin reviewers may scrutinize.
- **Custom keymap for Python `:` indent or `}` dedent** — the `@codemirror/lang-*` packs ship these via `indentNodeProp`. Writing custom keymap code duplicates — and contradicts — the pack's built-in behavior. D-01 mandates "full LanguageSupport."
- **Bundling `@codemirror/language` or `@codemirror/autocomplete`** — both are already `external` in `esbuild.config.mjs`. Removing them from external would duplicate 150–200 KB and risk peer-dep conflicts with Obsidian's own copy.
- **Whole-editor language replacement (replace Obsidian's markdown Compartment)** — this IS what Obsidian-host plugins like Markmind tried; it breaks markdown rendering entirely outside the fence. The per-view compartment layered ON TOP of the markdown language is the correct pattern.
- **Registering a ViewPlugin that dispatches from inside its own `update()` synchronously** — `update()` must be synchronous and must not dispatch new transactions inline. Use `queueMicrotask` / `Promise.resolve().then(...)` / a `void` async method to defer the dispatch to the next tick. Shown in Pattern 1's `swap()` implementation.
- **Storing the compartment globally** — one compartment per view. `buildCodeFenceLanguageExtension(plugin)` returns a fresh compartment each time it's called because `registerEditorExtension` invokes the extension factory per view.
- **Block widgets anywhere in the compartment contents** — pitfall 1 from 5.1 (block widgets corrupt Live Preview). LanguageSupport does NOT use block widgets; this is a reminder to NOT add custom widgets on top.
- **`Prec.highest` wrapping the compartment** — unnecessary. Default precedence composes correctly with Obsidian's own markdown extension.
- **Pre-resolving LanguageSupport in `computeFenceState`** — that's a pure function; it must not do I/O. Keep all `import()` behind the ViewPlugin's async `swap()` method.
- **Forgetting the `view.destroyed` guard** — the user may close a note while the pack is still loading. Dispatching on a destroyed view throws. Pattern 1 includes the guard.

## Common Pitfalls

### Pitfall 1: D-10 "Plan A" (augment Obsidian's markdown codeLanguages) is blocked

**What goes wrong:** CONTEXT.md D-10 preferred approach — "augment Obsidian's markdown `codeLanguages` nested-parser map" — depends on either (a) direct access to the markdown language's internal configuration, or (b) a Facet extension that the markdown language reads. Neither is publicly exposed.

**Why it happens:**
- `obsidian.d.ts` exports only `registerEditorExtension(extension: Extension)` from the Plugin interface. No `codeLanguages`, `addLanguage`, `markdownLanguageField`, `LanguageDescription`, or `syntaxTree` helper is publicly exposed. **[VERIFIED: grep of `/Users/moxu/projects/obsidian-leetcode/node_modules/obsidian/obsidian.d.ts` 2026-05-11 — no matches for `codeLanguages`, `LanguageSupport`, `syntaxTree`, `addLanguage`, `languageMap`]**
- `@codemirror/lang-markdown`'s `codeLanguages` option is a configuration property of the `markdown(...)` constructor, NOT a mutable facet or a runtime-augmentable API. **[VERIFIED: WebFetch of lang-markdown README 2026-05-11 — "The documentation doesn't explicitly detail runtime modification mechanisms beyond the initial configuration."]**
- Accessing the markdown language instance from a plugin requires reaching into Obsidian internals (`view.state.facet(EditorState.languageData)` walks every language registered in every Compartment on the view — a leaky and fragile surface).

**How to avoid:** **Skip Plan A. Go straight to Plan B (D-11) — per-view Compartment language swap.** Plan B is simpler, uses only public APIs (`Compartment`, `StateField`, `ViewPlugin`, `registerEditorExtension`), and is the documented CM6 pattern for dynamic language switching. All six success criteria from ROADMAP §Phase 5.3 are achievable via Plan B; no D-10 capability that Plan B lacks is actually required by the phase goal.

**Warning signs:**
- Any implementation that does `view.state.facet(...)` on undocumented facets.
- Any attempt to construct a new `markdown({ codeLanguages: ... })` and inject it as an extension — this creates a SECOND markdown language layered on top of Obsidian's, which breaks rendering.
- Any code that imports from `@codemirror/lang-markdown` directly (we should never need it — our compartment is scoped to the fence content, not markdown-wide).

**[CITED: obsidian.d.ts inspection 2026-05-11; @codemirror/lang-markdown README; Obsidian public API boundary confirmed via WebFetch of obsidianmd/obsidian-api 2026-05-11]**

### Pitfall 2: Dispatching a transaction from inside `ViewPlugin.update()`

**What goes wrong:** Calling `view.dispatch(...)` inside the `update(vu)` method directly throws `Error: Calls to EditorView.update are not allowed while an update is in progress`. This is CM6 catching re-entrancy.

**Why it happens:** `update()` runs as part of CM6's transaction dispatch cycle. Dispatching a new transaction from within would recurse into the same cycle.

**How to avoid:** Defer the dispatch via `queueMicrotask`, `Promise.resolve().then(...)`, or a `void asyncMethod(...)` call (Pattern 1 uses `void this.swap(next)`). The async `await` on `registry.get(...)` naturally moves the eventual `view.dispatch` to a later tick — this is free.

**Warning signs:**
- "Calls to EditorView.update are not allowed while an update is in progress" in DevTools.
- Infinite update loops where the compartment swap triggers another StateField update that triggers another swap.

### Pitfall 3: View destroyed while pack loads

**What goes wrong:** User opens an `lc-slug` note, caret enters python fence, pack starts loading (~50–100 ms). User closes the note before the pack resolves. When `swap()` finally calls `view.dispatch`, CM6 throws because the view is gone.

**Why it happens:** Pack init is async and the view lifecycle is shorter than the async window.

**How to avoid:** Check `view.destroyed` before dispatch. Pattern 1's `swap()` includes this. Alternative — use CM6's `EditorView.scrollIntoView` effect-type probing pattern or register a teardown callback that invalidates pending swaps.

**Warning signs:**
- Uncaught TypeError or "view is destroyed" errors when switching notes rapidly.

### Pitfall 4: Whitespace-copy fallback fires for supported languages while pack loads

**What goes wrong:** User enters a python fence. Pack takes 80 ms to load. Between fence-entry and pack-resolve, the user hits Enter. The compartment is still empty, so Obsidian's default markdown indent runs (inherits markdown indent). Not terrible, but inconsistent.

**Why it happens:** Async pack load window.

**How to avoid:**
- **(a)** Accept it. 80 ms is imperceptible for most users; the first keystroke after fence-entry is usually the one that loads the pack, not the one that uses it.
- **(b)** Install whitespace-copy as an initial compartment value when fence is entered, then swap to the pack when it resolves. Two dispatches per entry but guaranteed non-markdown behavior from the first keystroke.
- **(c)** Warm the user's default language at plugin load via `warmDefaultPack` (in Pattern 2's optional export).

Recommendation: start with (a). Upgrade to (b) if live smoke surfaces visible lag. Apply (c) unconditionally — it's free and helps the most common case.

**Warning signs:**
- Live smoke notices first-keystroke-after-fence-entry behaves like markdown (indents by 2 instead of 4, doesn't dedent on `}`).

### Pitfall 5: `selectionSet` transactions fire during scroll / focus

**What goes wrong:** StateField rebuilds on every `tr.selection` change — including phantom selection events from scroll, focus, and `setSelection` calls Obsidian makes internally. The rebuild is cheap (O(n) regex over doc lines, same cost as Phase 5.1), but it triggers redundant ViewPlugin.update calls. If the ViewPlugin doesn't guard "did the desiredPack actually change?", it dispatches a redundant compartment-reconfigure — which IS expensive (full re-highlight of the fence).

**Why it happens:** CM6 fires `tr.selection` on selection changes AND on most cursor moves.

**How to avoid:** Pattern 1's ViewPlugin.update guards `prev.desiredPack !== next.desiredPack` before calling `swap()`. This is the load-bearing check. Test it with a keystroke-heavy scenario (dragging the cursor through the fence).

**Warning signs:**
- Visual flicker / re-highlight on every keystroke (measurable in DevTools' Performance tab — repeated "style recalculation" events).

### Pitfall 6: Pack peer-dep resolves to a different `@codemirror/language` major than Obsidian's runtime copy

**What goes wrong:** Our `package.json` has no explicit `@codemirror/language` dep (it's transitive via the lang-packs). npm installs the version satisfying all peers — e.g., 6.12.3 latest. But Obsidian bundles, say, 6.8.0. At runtime, esbuild's `external` directive tells our code "look up `@codemirror/language` at runtime" — which hits Obsidian's 6.8.0. The lang-pack was built against 6.12.3 types but runs against 6.8.0 API surface. If they're API-compatible (semver major match), fine; if something new was added, fails.

**Why it happens:** `external` trusts the host to provide a compatible version. Major version matches on `^6.0.0` peer declarations is the guarantee.

**How to avoid:**
- Pin `@codemirror/language` in our devDependencies to a version `<=` what Obsidian ships (inspect `node_modules/@codemirror/language/package.json` after Obsidian is installed — it's a peer-dep of `obsidian@1.12.3`).
- **[ASSUMED]** Obsidian 1.10+ ships `@codemirror/language` at major 6 and `@codemirror/autocomplete` at major 6 — same major the packs require. **Planner MUST verify at plan-check time.**
- If Obsidian's shipped minor is too old, pin the lang-packs to matching-or-older minor versions and document the constraint.

**Warning signs:**
- TypeError: `indentNodeProp is not a function` or similar missing-export errors.
- Passing unit tests but runtime crash on first fence entry.

### Pitfall 7: `findCodeFence` misses fences with language tags containing spaces

**What goes wrong:** `findCodeFence` uses regex `/^\s*```/ `. Fences like ```` ```python3 some-ignored-metadata ```` should work (langSlug extraction uses `\S+` on the opener text), but edge cases like ```` ``` ```` (three-backtick without tag) or ```` ````python3 ```` (four-backtick) may not be handled.

**Why it happens:** Regex simplification in 5.1.

**How to avoid:**
- Tolerate untagged fences: `resolveLangSlug(null, default)` returns `default` (from `settings.defaultLanguage`) per CONTEXT.md D-03 of languages.ts.
- Handle 4+ backticks: Phase 5.1's `findCodeFence` works on any count of backticks because the regex is anchored on `^\s*` + at least three backticks (matches both three and more). Verify by re-reading the regex: `^\s*\`\`\`` — matches 3+ backticks. ✓
- Ignored metadata (stuff after the language name) is naturally handled by `/^\s*```\s*(\S+)?/` — `\S+` stops at whitespace.

**Warning signs:**
- User with a metadata-decorated fence sees whitespace-copy fallback instead of the expected language.
- User with a language-tag-missing fence (common on LC where new notes may be created without immediate language lock-in) sees plain markdown instead of the default language.

### Pitfall 8: Python's Lezer grammar chokes on indentation-ambiguous constructs

**What goes wrong:** `@lezer/python` produces indent hints via `indentNodeProp`. For most inputs, this is correct. Edge cases: multi-line string literals, `(x for\n ...)` generator expressions with implicit-continuation indentation, and `\` line continuations produce grammar-level ambiguity that may result in "wrong" indent (too many, too few). Users may notice.

**Why it happens:** Grammar-driven indent hints are heuristics, not semantic analysis.

**How to avoid:**
- Accept it. `@codemirror/lang-python` is the best-in-class Python indent for any Obsidian-ish editor.
- Flag for v1.1 if users complain — could compose with a custom `indentService` on TOP of the pack's indentNodeProp that overrides specific cases.

**Warning signs:** User reports "pressing Enter after a triple-quote string indents weirdly." Matches known `@lezer/python` limitations.

### Pitfall 9: `LanguageSupport` swap doesn't trigger re-highlight of already-visible content

**What goes wrong:** When the compartment swaps from empty → python, the fence's visible content may not re-highlight immediately. It may require a scroll, a keystroke, or a full re-render.

**Why it happens:** CM6 decoration invalidation can lag a compartment swap by one transaction.

**How to avoid:**
- CM6 generally handles this — `compartment.reconfigure()` fires a full state transition that re-derives all `EditorView.decorations.from(...)` providers.
- If visible content doesn't re-highlight, combine the compartment swap with `EditorView.update([{ effects: compartment.reconfigure(...) }])` and optionally a `dispatch({})` no-op.
- Live smoke will surface this within 5 minutes of testing.

**Warning signs:** Enter the fence — code is still plain black text until the next keystroke.

### Pitfall 10: Bundle size trips D-09 cap (700 KB)

**What goes wrong:** Built `main.js` exceeds 700 KB after `npm install` adds all 6 lang-packs.

**Why it happens:** `@lezer/cpp` alone is 429 KB unpacked. `@lezer/javascript` is 318 KB. `@lezer/rust` is 298 KB. Raw sum across the 6 grammars: ~1.55 MB unpacked. After esbuild production minify + tree-shake, production Obsidian plugins report 30–40% reduction (data/lookup tables compress less than code). Expected final: 500–550 KB. **[ASSUMED via esbuild minification ratio for code-heavy bundles]**

**How to avoid:**
- **Measure early.** After Wave 1 installs the packs, run `npm run build` and check `main.js` size BEFORE writing any feature code. If the bare install is already > 600 KB, escalate before spending time.
- **D-09 escalation order:** (a) drop Rust (shaves ~300 KB unpacked, ~100 KB built); (b) re-discuss D-04 to drop Go or C++ (less impactful but possible); (c) spike code-splitting workaround (requires Obsidian plugin-store policy deviation).
- **Sanity check:** `scripts/check-bundle-size.sh` runs after every `npm run build`. If it trips, the current wave is blocked pending D-09 decision.

**Warning signs:** `check-bundle-size.sh` fails with `main.js is 712 kB (>700 kB ceiling)`.

### Pitfall 11: Compartment swap creates visible "flash" of unhighlighted code

**What goes wrong:** When caret enters a python fence for the first time, pack loads asynchronously. During the load window (~80 ms), the fence content renders as plain markdown. When the pack resolves and the compartment swaps, the content "flashes" from plain to highlighted.

**Why it happens:** Async pack load.

**How to avoid:**
- Use `warmDefaultPack` (Pattern 2) at `onload()` for the user's default language.
- For other languages, accept the 80 ms flash for first-use.
- Optional upgrade: install the whitespace-copy extension while the pack loads, then swap to the pack. This avoids the "looks like markdown" window but adds a double-swap per fence-entry.

**Warning signs:** Live smoke notices a brief unhighlighted-code moment on first fence entry per language.

### Pitfall 12: Phase 5.1 widget placement shifts due to compartment reconfigure

**What goes wrong:** Phase 5.1's `CodeActionsWidget` is anchored to `state.doc.line(fence.closerLine + 1).from`. A compartment reconfigure runs `EditorView.update` — rebuilds decorations, re-measures line heights. If the language pack introduces line-height changes (padding, line-number gutters), the widget may visually shift or re-paint.

**Why it happens:** Any EditorView update can re-measure layout.

**How to avoid:**
- CM6's decoration diffing (Pitfall 2 in 5.1's `WidgetType.eq()` discipline) handles this — same plugin reference → same widget → no re-paint.
- Live smoke MUST verify buttons stay in place after compartment swap.

**Warning signs:** Button row visibly shifts, re-paints, or flickers when caret enters/exits the fence.

### `src/main/languagePackRegistry.ts` (canonical)

```typescript
// Source: [VERIFIED: CodeMirror docs "Language packs" idiom; composable with Node ESM dynamic import]
import type { LanguageSupport } from '@codemirror/language';

export type PackId =
  | 'python' | 'java' | 'cpp' | 'javascript' | 'typescript' | 'go' | 'rust';

const cache = new Map<PackId, Promise<LanguageSupport>>();

export function getLanguagePack(id: PackId): Promise<LanguageSupport> {
  const existing = cache.get(id);
  if (existing) return existing;
  const promise = loadPack(id).catch((err) => {
    // Drop the failed promise from the cache so a retry is possible.
    cache.delete(id);
    throw err;
  });
  cache.set(id, promise);
  return promise;
}

async function loadPack(id: PackId): Promise<LanguageSupport> {
  switch (id) {
    case 'python':     return (await import('@codemirror/lang-python')).python();
    case 'java':       return (await import('@codemirror/lang-java')).java();
    case 'cpp':        return (await import('@codemirror/lang-cpp')).cpp();
    case 'javascript': return (await import('@codemirror/lang-javascript')).javascript();
    case 'typescript': return (await import('@codemirror/lang-javascript'))
                                .javascript({ typescript: true });
    case 'go':         return (await import('@codemirror/lang-go')).go();
    case 'rust':       return (await import('@codemirror/lang-rust')).rust();
  }
}

export function warmDefaultPack(defaultLangSlug: string): void {
  const id = slugToPackId(defaultLangSlug);
  if (id) void getLanguagePack(id);
}

export function slugToPackId(slug: string): PackId | null {
  switch (slug) {
    case 'python':     return 'python';
    case 'python3':    return 'python';
    case 'java':       return 'java';
    case 'cpp':        return 'cpp';
    case 'c':          return 'cpp';
    case 'javascript': return 'javascript';
    case 'typescript': return 'typescript';
    case 'golang':     return 'go';
    case 'rust':       return 'rust';
    default:           return null;
  }
}
```

### `src/main/whitespaceCopyIndent.ts` (D-06 fallback)

```typescript
// Source: [VERIFIED: @codemirror/language indentService reference; standard pattern]
import { indentService } from '@codemirror/language';
import type { Extension } from '@codemirror/state';

export const whitespaceCopyIndentExtension: Extension = indentService.of((ctx, pos) => {
  const line = ctx.state.doc.lineAt(pos);
  if (line.number === 1) return 0;
  for (let n = line.number - 1; n >= 1; n--) {
    const prev = ctx.state.doc.line(n);
    if (prev.text.trim().length === 0) continue;
    const match = /^[\t ]*/.exec(prev.text);
    const leading = match?.[0] ?? '';
    // Return column count (indentService expects number, not a string).
    // Treat tab as 2 spaces for column math; CM6's indent config controls
    // rendering.
    return leading.replace(/\t/g, '  ').length;
  }
  return 0;
});
```

#!/usr/bin/env bash
# scripts/check-bundle-size.sh
#
# Phase 5.3 D-09 / D-15 — hard bundle-size gate.
#
# Runs:
#   - after every `npm run build`
#   - as gate 12 extension in `prerelease-check.sh`
#   - on CI (future)
#
# Fails if main.js > 700 KB. Warns at 600 KB.
set -eo pipefail

ROOT="$(cd "$(dirname "$0")/.." && pwd)"
MAIN="$ROOT/main.js"

if [ ! -f "$MAIN" ]; then
  echo "BUNDLE CHECK FAIL: main.js missing — run 'npm run build' first" >&2
  exit 1
fi

MAIN_BYTES=$(wc -c < "$MAIN" | tr -d ' ')
MAIN_KB=$((MAIN_BYTES / 1024))
LIMIT_BYTES=$((700 * 1024))
WARN_BYTES=$((600 * 1024))

if [ "$MAIN_BYTES" -gt "$LIMIT_BYTES" ]; then
  echo "BUNDLE CHECK FAIL: main.js is ${MAIN_KB} KB (>700 KB ceiling; D-09 escalation required)" >&2
  exit 1
fi
if [ "$MAIN_BYTES" -gt "$WARN_BYTES" ]; then
  echo "BUNDLE CHECK WARN: main.js is ${MAIN_KB} KB (>600 KB soft warning; ceiling 700 KB)"
fi
echo "BUNDLE CHECK OK: main.js ${MAIN_KB} KB within budget"
```

## Open Questions

1. **Should the initial compartment value be empty (`[]`) or the whitespace-copy fallback?**
   - What we know: Starting empty means the fence behaves like markdown until caret-entry fires a reconfigure. Starting with whitespace-copy means the fence has non-markdown indent from the very first keystroke.
   - What's unclear: Whether users notice the first-keystroke behavior. If the user types immediately after opening a note, they may press Enter before the caret-entry signal fires.
   - Recommendation: Start empty (matches Pattern 1). If live smoke surfaces complaints, switch to whitespace-copy initial. This is a 1-line change.

2. **Should `computeFenceState` use the caret position or the selection range?**
   - What we know: Users can select text across the fence boundary (Shift-Down out of the fence). The selection spans fence-and-non-fence. Using `state.selection.main.head` covers the common case (cursor is at the end of the selection).
   - What's unclear: Whether using `state.selection.main.anchor` or the whole-selection range matters. For practical indent/typing, `head` is correct — indent-on-Enter uses the caret position.
   - Recommendation: Use `state.selection.main.head`. Pattern 1 encodes this.

3. **Should we support multiple fences under `## Code` (each with different language)?**
   - What we know: 5.1's `findCodeFence` returns the FIRST fence under `## Code`. If users have two fences (e.g., reference implementation + their attempt), only the first gets button + language behavior.
   - What's unclear: Whether 5.3 should extend findCodeFence or mirror 5.1 exactly.
   - Recommendation: Mirror 5.1 exactly for consistency. First fence under `## Code` is the "primary" fence; others are reference/prose. If this becomes a UX pain point, future phase extends `findCodeFence` in both 5.1 and 5.3 simultaneously.

4. **Should the compartment de-install when the caret is in the markdown prose OUTSIDE the fence but INSIDE the `## Code` section (e.g., between the closing fence and the next H2)?**
   - What we know: CaretInCodeFence in Pattern 1 is `caretLine > openerLine && caretLine < closerLine`. Caret on fence boundary lines (opener or closer) returns false.
   - What's unclear: Whether boundary-line behavior matters for user UX. Hitting Enter on the closer line is rare.
   - Recommendation: Pattern 1's strict-inside check is correct. Boundary lines go to markdown indent (parens-style indent that markdown doesn't really have — it's essentially whitespace-copy by default). No UX harm.

5. **Does the python3 Prism alias (Phase 5.2) conflict with the lang-python pack in Live Preview?**
   - What we know: Phase 5.2 aliases Prism `python3` → `python` for Reading Mode. Edit Mode is owned by CM6 (not Prism). The two don't overlap.
   - What's unclear: If Obsidian applies Prism highlighting to Edit Mode fences too (it does NOT — CM6 owns Edit Mode), would the alias conflict?
   - Recommendation: No conflict. Verify in live smoke by observing Edit Mode python3 fences — highlighting should be Lezer-driven, not Prism.
