<objective>
Fix GitHub issue #16: embedded login captures cookies but the plugin remains logged out because LC sometimes sets `csrftoken` as a host-only cookie (Domain unset → stored under `leetcode.com`, not `.leetcode.com`). The current `cookies.get({ domain: '.leetcode.com' })` filter omits host-only cookies, `extractAuthCookies()` returns null, the user closes the window, the silent-cancel path resolves null, and `data.json` is never written.

Switch the cookie filter to `{ url: 'https://leetcode.com/' }` (Electron returns "all cookies that would be sent on a request to that URL" — exactly what the API client also needs by construction), add a 30-second capture-timeout watchdog so future silent failures surface a distinct Notice instead of looking like a cancellation, and lock both behaviors with an automated unit test.

Purpose: Restore reliable embedded login for users hitting the host-only-csrftoken path AND prevent the next "looks like cancel but isn't" failure mode from being invisible.

Output:
- Patched `src/auth/BrowserWindowLogin.ts` with URL-based filter, timeout watchdog, tagged-union result, and an extracted `tryCaptureCookies` testable helper.
- Patched `src/auth/AuthService.ts` switching on the new tagged-union and emitting the timeout Notice.
- New `tests/browser-window-login-filter.test.ts` proving the filter shape.
- Single atomic commit referencing issue #16.
</objective>
